Sockets — TCP ו-UDP

Socket הוא "השקע" שדרכו שתי תוכניות מדברות זו עם זו ברשת — בדיוק כמו שקע חשמל שמחבר מכשיר לקיר. כאן נבנה את אותה אפליקציית client-server פעמיים: פעם מעל TCP (חיבור מתמשך ואמין) ופעם מעל UDP (בלי חיבור, מהיר). נראה את כל קריאות-המערכת שורה-שורה, ונסכם בטבלאות לזיכרון.

🧭
קונטקסט — Client / Server ושני הפרוטוקולים ב-client-server, ה-שרת מחכה לפנייה בכתובת ופורט ידועים, וה-לקוח יוזם את הקשר. שניהם משתמשים ב-socket כנקודת-הקצה. מתחת לזה יש שני פרוטוקולי תעבורה: TCP — אמין, מסודר, מבוסס-חיבור (כמו שיחת טלפון); ו-UDP — מהיר, בלי חיבור, בלי הבטחות (כמו לזרוק גלויה לתיבת דואר).

1️⃣ ה-Socket API — קריאות המערכת

גם TCP וגם UDP בנויים מאותו ארגז-כלים של קריאות מערכת (system calls). נכיר אותן פעם אחת, ואז נראה איך כל פרוטוקול מרכיב אותן אחרת:

קריאהמה עושה
socket(AF_INET, type, 0)יוצר socket ומחזיר file descriptor. AF_INET = IPv4; type = SOCK_STREAM (TCP) או SOCK_DGRAM (UDP).
struct sockaddr_inמבנה שמחזיק כתובת: sin_family (AF_INET), sin_addr (IP), sin_port (פורט).
htons(port)"host-to-network short" — ממיר את הפורט ל-byte order של הרשת (big-endian). חובה, כי מחשבים שונים שומרים מספרים אחרת.
INADDR_ANY(בשרת) "האזן על כל כתובות ה-IP של המכונה" — לא משנה דרך איזה כרטיס רשת יגיע הלקוח.
inet_pton(AF_INET, ip, &addr)(בלקוח) "presentation-to-network" — ממיר כתובת IP מטקסט ("127.0.0.1") לייצוג בינארי.
bind(fd, &addr, len)קושר את ה-socket לכתובת+פורט מקומיים. בשרת — כך לקוחות יודעים לאן לפנות.
close(fd)סוגר את ה-socket ומשחרר את המשאבים.
🔤
השאר תלוי בפרוטוקול listen/accept/connect ו-send/recv שייכים ל-TCP; sendto/recvfrom שייכים ל-UDP. נכיר כל אחד במקומו.

2️⃣ TCP — חיבור מתמשך

📞
הרעיון TCP הוא חיבור מתמשך (כמו שיחת טלפון): קודם "מרימים טלפון" ויוצרים חיבור, ואז מדברים חופשי. השרת עושה bind → listen → accept, הלקוח עושה connect. אחרי שהחיבור קיים — send/recv בלי כתובת (החיבור כבר "זוכר" מי הצד השני).
socketSOCK_STREAM
→
bindכתובת+פורט
→
listenמאזין לחיבורים
→
accept→ clientfd
→
recv / sendעל החיבור
🌊
למה קוראים בית-בית? — TCP הוא זרם (stream) ב-TCP אין גבולות הודעה: ה-recv עלול להחזיר חצי הודעה, או שתי הודעות מודבקות יחד. הנתונים זורמים כמו ברז. לכן צריך להגדיר מפריד משלנו — כאן \n — ולקרוא בית-בית עד שמגיעים אליו, וזו הודעה שלמה.

2.1 server_main.cpp — שרת TCP

cpp · TCP server
#include <iostream>
#include <string>
#include <cstring>
#include <memory>
#include <unistd.h>
#include <arpa/inet.h>
#include <sys/socket.h>
#include "parser/CommandParser.h"
#include "storage/FileDataStorage.h"
#include "commands/RecordCommand.h"
#include "commands/ReadCommand.h"

int main(int argc, char* argv[]) {
    int port = std::stoi(argv[1]);

    FileDataStorage storage("data/sensor.txt");
    CommandParser parser;
    parser.registerCommand("RECORD", std::make_shared<RecordCommand>(storage));
    parser.registerCommand("READ",   std::make_shared<ReadCommand>(storage));

    int serverfd = socket(AF_INET, SOCK_STREAM, 0);      // STREAM

    struct sockaddr_in addr;
    memset(&addr, 0, sizeof(addr));
    addr.sin_family = AF_INET;
    addr.sin_addr.s_addr = INADDR_ANY;
    addr.sin_port = htons(port);
    bind(serverfd, (struct sockaddr*)&addr, sizeof(addr));
    listen(serverfd, 1);                                 // listen

    while (true) {
        int clientfd = accept(serverfd, nullptr, nullptr); // accept
        std::string line;
        char c;
        while (recv(clientfd, &c, 1, 0) > 0) {           // recv בית-בית
            if (c == '\n') {
                std::string response = parser.parse(line);
                send(clientfd, response.c_str(), response.size(), 0);
                line.clear();
            } else {
                line += c;
            }
        }
        close(clientfd);
    }
    return 0;
}
🔤
פירוק השרת
  • listen(serverfd, 1) — מסמן את ה-socket כ"מאזין", עם תור באורך 1 לחיבורים ממתינים.
  • accept(...) — חוסם עד שלקוח מתחבר, ומחזיר clientfd חדש — socket נפרד לאותו לקוח (ה-serverfd ממשיך להאזין לבאים).
  • recv(clientfd, &c, 1, 0) — קורא בית אחד. מצטברים ל-line עד \n, ואז parser.parse(line) ושולחים תשובה ב-send.
  • הלולאה החיצונית חוזרת ל-accept — לקוח אחד בכל פעם.

2.2 client.cpp — לקוח TCP ב-C++

cpp · TCP client
#include <iostream>
#include <string>
#include <cstring>
#include <unistd.h>
#include <arpa/inet.h>
#include <sys/socket.h>

int main(int argc, char* argv[]) {
    std::string ip = argv[1];
    int port = std::stoi(argv[2]);

    int sock = socket(AF_INET, SOCK_STREAM, 0);

    struct sockaddr_in addr;
    memset(&addr, 0, sizeof(addr));
    addr.sin_family = AF_INET;
    addr.sin_port = htons(port);
    inet_pton(AF_INET, ip.c_str(), &addr.sin_addr);

    connect(sock, (struct sockaddr*)&addr, sizeof(addr));   // connect

    std::string line;
    while (std::getline(std::cin, line)) {
        line += "\n";
        send(sock, line.c_str(), line.size(), 0);
        char buffer[4096];
        int bytes = recv(sock, buffer, sizeof(buffer) - 1, 0);
        if (bytes <= 0) break;
        buffer[bytes] = '\0';
        std::cout << buffer;
    }
    close(sock);
    return 0;
}
🔗
הלקוח — connect פעם אחת הלקוח בונה את כתובת השרת (עם inet_pton), עושה connect פעם אחת, ואז בלולאה: שולח שורה ב-send ומקבל תשובה ב-recv — בלי כתובת, כי החיבור קיים.

2.3 client.py — לקוח TCP בפייתון

python · TCP client
import sys, socket

ip = sys.argv[1]
port = int(sys.argv[2])

s = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
s.connect((ip, port))                                # connect

while True:
    line = input()
    s.sendall((line + '\n').encode())
    response = s.recv(4096).decode()
    print(response, end='')
🐍
אותו רעיון, פחות שורות בפייתון ה-API זהה במהות: socket(AF_INET, SOCK_STREAM) → connect((ip, port)) → sendall/recv. צריך .encode() (טקסט→בייטים) בשליחה ו-.decode() בקבלה. sendall דואג לשלוח את כל הבייטים.

3️⃣ UDP — בלי חיבור

✉️
הרעיון UDP הוא בלי חיבור (כמו לשלוח גלויות): אין "הרמת טלפון" — כל הודעה נשלחת לבד עם כתובת היעד עליה. השרת עושה רק bind (בלי listen/accept), והלקוח בלי connect. כל שליחה/קבלה היא sendto/recvfrom שנושאת את הכתובת.
socketSOCK_DGRAM
→
bindכתובת+פורט
→
listen
accept
→
recvfrom / sendto+כתובת לקוח
📦
כל חבילה = הודעה שלמה בניגוד ל-TCP, ב-UDP נשמרים גבולות ההודעה: sendto אחד = recvfrom אחד = חבילה (datagram) שלמה. לכן אין לולאת בית-בית — קוראים את כל החבילה בבת אחת ומתייחסים אליה כשורה שלמה.

3.1 server_main.cpp — שרת UDP

cpp · UDP server
#include <iostream>
#include <string>
#include <cstring>
#include <memory>
#include <unistd.h>
#include <arpa/inet.h>
#include <sys/socket.h>
#include "parser/CommandParser.h"
#include "storage/FileDataStorage.h"
#include "commands/RecordCommand.h"
#include "commands/ReadCommand.h"

int main(int argc, char* argv[]) {
    int port = std::stoi(argv[1]);

    FileDataStorage storage("data/sensor.txt");
    CommandParser parser;
    parser.registerCommand("RECORD", std::make_shared<RecordCommand>(storage));
    parser.registerCommand("READ",   std::make_shared<ReadCommand>(storage));

    int sock = socket(AF_INET, SOCK_DGRAM, 0);           // DGRAM

    struct sockaddr_in addr;
    memset(&addr, 0, sizeof(addr));
    addr.sin_family = AF_INET;
    addr.sin_addr.s_addr = INADDR_ANY;
    addr.sin_port = htons(port);
    bind(sock, (struct sockaddr*)&addr, sizeof(addr));
    // אין listen, אין accept

    while (true) {
        char buffer[4096];
        struct sockaddr_in client;
        socklen_t clientLen = sizeof(client);

        int bytes = recvfrom(sock, buffer, sizeof(buffer) - 1, 0,
                             (struct sockaddr*)&client, &clientLen);  // recvfrom
        buffer[bytes] = '\0';

        std::string line(buffer);                        // חבילה שלמה = שורה
        std::string response = parser.parse(line);       // ← זהה ל-TCP!

        sendto(sock, response.c_str(), response.size(), 0,
               (struct sockaddr*)&client, clientLen);    // sendto
    }
    close(sock);
    return 0;
}
🔤
פירוק השרת
  • SOCK_DGRAM במקום SOCK_STREAM — וזהו, אין listen ו-אין accept ואין clientfd נפרד.
  • recvfrom(...) — מקבל חבילה וגם ממלא את כתובת השולח (client), כדי לדעת למי לענות.
  • sendto(...) — שולח את התשובה בחזרה לאותה כתובת שקיבלנו מ-recvfrom.
🎯
שים לב — לוגיקת ה-business זהה השורה parser.parse(line) זהה לחלוטין ל-TCP. רק שכבת התעבורה (איך הבייטים נעים) השתנתה — כל הקוד של הפקודות (RECORD/READ, האחסון) נשאר אותו דבר. הפרדת אחריות נקייה בין רשת ללוגיקה.

3.2 client.cpp — לקוח UDP ב-C++

cpp · UDP client
#include <iostream>
#include <string>
#include <cstring>
#include <unistd.h>
#include <arpa/inet.h>
#include <sys/socket.h>

int main(int argc, char* argv[]) {
    std::string ip = argv[1];
    int port = std::stoi(argv[2]);

    int sock = socket(AF_INET, SOCK_DGRAM, 0);

    struct sockaddr_in addr;
    memset(&addr, 0, sizeof(addr));
    addr.sin_family = AF_INET;
    addr.sin_port = htons(port);
    inet_pton(AF_INET, ip.c_str(), &addr.sin_addr);

        line += "\n";
        sendto(sock, line.c_str(), line.size(), 0,
               (struct sockaddr*)&addr, addrLen);            // sendto

        int bytes = recvfrom(sock, buffer, sizeof(buffer) - 1, 0,
                            (struct sockaddr*)&addr, &addrLen); // recvfrom
        buffer[bytes] = '\0';
        std::cout << buffer;
    }
    close(sock);
    return 0;
}
🔌
הלקוח — בלי connect מכינים את כתובת השרת (inet_pton) אבל לא עושים connect. בכל סבב: sendto עם כתובת השרת, ו-recvfrom לקבלת התשובה. הכתובת נלווית לכל הודעה.

3.3 client.py — לקוח UDP בפייתון

python · UDP client
import sys, socket

ip = sys.argv[1]
port = int(sys.argv[2])

s = socket.socket(socket.AF_INET, socket.SOCK_DGRAM)     # DGRAM
# אין connect

while True:
    line = input()
    s.sendto((line + '\n').encode(), (ip, port))          # sendto
    response, server = s.recvfrom(4096)                   # recvfrom
    print(response.decode(), end='')
🐍
בפייתון SOCK_DGRAM, ללא connect. sendto((data, (ip, port))) שולח עם כתובת, ו-recvfrom(4096) מחזיר גם את הנתון וגם את כתובת השולח (response, server).

5️⃣ סיכום — מתי כל אחד

🎯
המשפט אחד לזכור TCP = חיבור (SOCK_STREAM): השרת bind→listen→accept, הלקוח connect, ואז send/recv בלי כתובת — זרם בלי גבולות, אז קוראים בית-בית עד \n. UDP = בלי חיבור (SOCK_DGRAM): השרת רק bind, הלקוח בלי connect, וכל sendto/recvfrom נושא כתובת — כל חבילה הודעה שלמה.
בחר TCP כש...
חשובה אמינות וסדר — שכל בית יגיע, פעם אחת, בסדר הנכון. (אתרים, העברת קבצים, רוב הבקשות.)
בחר UDP כש...
חשובה מהירות ואפשר להרשות אובדן חבילה בודדת. (סטרימינג וידאו, משחקים, שיחות קול, DNS.)
🧩
הנקודה הגדולה שתי הגרסאות מריצות את אותה אפליקציה (RECORD/READ, אותו parser.parse, אותו אחסון) — רק שכבת התעבורה מתחלפת. זה מדגים הפרדה נקייה: לוגיקת העסק לא יודעת ולא אכפת לה אם הבייטים הגיעו ב-TCP או UDP.
