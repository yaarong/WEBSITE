MVC בפועל — בונים את אתר "Foo News" צעד-צעד

בהרצאה 5 ראינו את תורת ה-MVC (Model / View / Controller) ברמת התיאוריה. עכשיו נבנה אפליקציית Express אמיתית מאפס — אתר חדשות בשם Foo News — ונראה איך כל חלק נכנס למקומו: מ-"Hello world" פשוט, דרך routes מודולריים, controllers, תבניות EJS, העברת data דינמי, model שמחזיק את הנתונים, פרמטרים ב-URL, ועד טופס שיוצר כתבה חדשה דרך POST.

🗺️
קונטקסט כללי — למה הרצאה שלמה על דוגמה אחת? ב-MVC קל ללכת לאיבוד: "איזה קובץ קורא למי?". הדרך הכי טובה להבין היא לבנות. נתחיל מהקוד המינימלי ביותר ובכל שלב נוסיף בדיוק רכיב אחד — וכך נראה בדיוק איזו בעיה כל רכיב פותר. בסוף יהיה לנו אתר עובד עם הפרדה נקייה בין שכבות, בדיוק כמו שדיברנו עליו בתיאוריה.
📌
המשך בהרצאה 8 בדוגמה הזו ה-Model מחזיק את הכתבות במערך פשוט בתוך הקוד (בזיכרון). בהרצאה 8 נחליף אותו במסד נתונים אמיתי (MongoDB / Mongoose), נוסיף שכבת services וקונפיגורציית סביבות — אבל כל שאר המבנה (routes, controller, views) יישאר זהה. זה בדיוק היופי של MVC: מחליפים שכבה בלי לגעת בשאר.

חלק א' · מבנה הפרויקט ו-MVC בפועל

1. נקודת התחלה — אפליקציית Express מינימלית

🧭
קונטקסט לפני שמדברים על MVC, נתחיל מהדבר הכי קטן שעובד: שרת Express שמחזיר טקסט. הפרויקט נקרא mynews ובהתחלה יש בו רק node_modules, app.js וקבצי ה-package.
js · app.js — הצעד הראשון
const express = require('express')
var app = express()

app.get('/article', (req, res) => {
    res.send('Hello world')
})

app.listen(8080)
🔤
פירוק שורה-שורה
  • require('express') — טוען את ספריית Express (ראינו require בהרצאה 5/6 — סגנון CommonJS).
  • express() — יוצר את אובייקט האפליקציה app.
  • app.get('/article', handler) — "כשמגיעה בקשת GET לנתיב /article, הרץ את ה-handler". ה-handler מקבל req (הבקשה) ו-res (התשובה).
  • res.send('Hello world') — שולח טקסט חזרה לדפדפן.
  • app.listen(8080) — מאזין לבקשות בפורט 8080.

גלישה ל-localhost:8080/article מציגה "Hello world". יש לנו שרת — אבל הכל דחוס בקובץ אחד. ככל שהאתר יגדל, זה יהפוך לבלתי-נסבל. בואו נתחיל לסדר.

2. הוצאת ה-routes לקובץ נפרד — express.Router

🧭
קונטקסט — הבעיה אתר אמיתי הוא עשרות נתיבים (כתבות, משתמשים, תגובות...). לדחוס את כולם ב-app.js זה כאוס. הפתרון: Router — "תת-אפליקציה" שמרכזת קבוצת נתיבים בקובץ נפרד. זו בדיוק האנקפסולציה שדיברנו עליה בהרצאה 6 (modules).

app.js — מאציל את כל מה שמתחיל ב-/articles לקובץ ה-router:

js · app.js
const express = require('express')
var app = express()

const articles = require('./routes/articles');
app.use('/articles', articles);

app.listen(8080)

routes/articles.js — מגדיר את הנתיבים ומייצא את ה-router:

js · routes/articles.js
const express = require('express')
var router = express.Router()

router.get('/', (req, res) => {
    res.send('Hello world')
});

module.exports = router
🔤
איך מתחברים הנתיבים? app.use('/articles', articles) אומר: "כל בקשה שמתחילה ב-/articles — תטופל ע"י ה-router הזה". בתוך ה-router, router.get('/') הוא יחסי לקידומת — כלומר הוא מטפל ב-/articles/ בפועל. ככה ה-router לא צריך לדעת באיזו קידומת הוא "תלוי" — הפרדה נקייה.

3. וריאציית ה-MVC — דיאגרמת הזרימה

🧭
קונטקסט — מי מדבר עם מי? ראינו ב-MVC: Model (הנתונים), View (התצוגה), Controller (התיווך). בווריאציה הנפוצה ל-web ה-Controller הוא הצומת המרכזי: הוא מקבל את הבקשה, מבקש נתונים מה-Model, מעביר אותם ל-View שמייצר את ה-HTML, ומחזיר את התוצאה לדפדפן. ה-View וה-Model לא מדברים ישירות — הכל עובר דרך ה-Controller. 
 ┌─────────────────────┐
 │      Browser        │   1. request   ┌────────────┐
 │  (המשתמש רואה את    │ ─────────────▶ │            │
 │   ה-View ופועל בו)  │                │ Controller │
 │                     │ ◀───────────── │            │
 └─────────────────────┘   6. response  └────────────┘
                          (מכיל את התצוגה)  ▲ │   ▲ │
                                            │ │   │ │
                            5. data viz ──┐ │ │   │ │ ┌── 2. invoke
                                          │ │ │   │ │ │
                                   4. data│ │ │   │ │ │3. data
                                          ▼ │ │   │ ▼ │
                                       ┌──────────┐ ┌────────┐
                                       │   View   │ │ Model  │
                                       └──────────┘ └────────┘
🔢
ששת השלבים
  1. request — הדפדפן שולח בקשה ל-Controller.
  2. invoke — ה-Controller פונה ל-Model ומבקש נתונים.
  3. data — ה-Model מחזיר את הנתונים ל-Controller.
  4. data — ה-Controller מעביר את הנתונים ל-View.
  5. data visualization — ה-View מייצר תצוגה (HTML) ומחזיר ל-Controller.
  6. response — ה-Controller שולח את ה-HTML המוגמר חזרה לדפדפן.
בדיוק את הזרימה הזו נבנה עכשיו בקוד — שלב אחרי שלב.

4. Controller — הוצאת הלוגיקה מה-route

🧭
קונטקסט ה-route צריך רק לומר "איזה URL → איזו פונקציה". את הלוגיקה עצמה (מה לעשות בבקשה) מוציאים ל-Controller נפרד. שוב — הפרדת אחריות (SRP מהרצאה 2).

routes/articles.js — מצביע על פונקציה ב-controller במקום על handler אנונימי:

js · routes/articles.js
const express = require('express')
var router = express.Router()

const articles = require('../controllers/articles')
router.get('/', articles.getArticle);

module.exports = router

controllers/articles.js — מחזיק את פונקציות הטיפול:

js · controllers/articles.js
const getArticle = (req, res) => {
    res.send('Hello world')
}

module.exports = { getArticle }
🔤
נקודה עדינה — מעבירים את הפונקציה, לא קוראים לה שים לב: router.get('/', articles.getArticle) — בלי סוגריים. מעבירים את הפונקציה עצמה כ-callback, ו-Express יקרא לה עם req, res כשתגיע בקשה. זו בדיוק אותה מלכודת onclick = show מהרצאה 6 — getArticle() עם סוגריים היה מריץ מיד ומעביר את התוצאה במקום את הפונקציה.

5. View — תבנית EJS (view engine)

🧭
קונטקסט — הבעיה עד עכשיו החזרנו טקסט עם res.send. אבל אתר אמיתי מחזיר HTML, ועדיף HTML דינמי (עם נתונים משתנים). הפתרון: template engine — קובץ HTML עם "חורים" שממלאים בנתונים. נשתמש ב-EJS.

קודם — קובץ views/article.ejs סטטי (HTML רגיל לגמרי):

html · views/article.ejs
<!DOCTYPE html>
<html lang="en">
<head>
    <meta charset="UTF-8">
    <meta name="viewport" content="width=device-width, initial-scale=1.0">
    <title>Document</title>
</head>
<body>
    Hello world
</body>
</html>

ה-controller מרנדר אותו עם res.render במקום res.send:

js · controllers/articles.js
const getArticle = (req, res) => {
    res.render('article.ejs', {})
}

module.exports = { getArticle }

וצריך להגיד ל-Express שמנוע התבניות הוא EJS — ב-app.js:

js · app.js
const express = require('express')
var app = express()

const articles = require('./routes/articles');
app.use('/articles', articles);

app.set('view engine', 'ejs');   // ← מגדיר את EJS כמנוע התבניות

app.listen(8080)
🔤
res.send מול res.render
  • res.send('...') — מחזיר טקסט/HTML כמו שהוא, ללא עיבוד.
  • res.render('article.ejs', { ... }) — מריץ את ה-View: לוקח את התבנית, ממלא בה את הנתונים מהאובייקט השני, ומחזיר את ה-HTML המוגמר. כברירת מחדל EJS מחפש את הקובץ בתיקיית views/.
זה בדיוק שלב 4→5→6 בדיאגרמה: ה-controller מעביר data ל-view, ה-view מייצר תצוגה, וזו חוזרת כ-response.

חלק ב' · נתונים דינמיים

6. העברת data ל-View

🧭
קונטקסט עד כה ה-view היה סטטי. עכשיו הכוח האמיתי: מעבירים נתונים מה-controller ל-view, וה-view "מזריק" אותם ל-HTML. כאן נכנס תחביר ה-EJS.

בתבנית — מוסיפים "חור" עם <%= foo %>:

html · views/article.ejs (קטע ה-body)
<body>
    Hello world <%= foo %>
</body>

ב-controller — מעבירים את הערך של foo:

js · controllers/articles.js
const getArticle = (req, res) => {
    res.render('article.ejs', { foo : 42 })
}

module.exports = { getArticle }

התוצאה בדפדפן: Hello world 42. הערך 42 עבר מה-controller, דרך אובייקט הנתונים, אל תוך התבנית.

🔤
תחביר EJS — <%= ... %> <%= expression %> = "הערך את הביטוי בתוך ה-JS, והדפס את התוצאה ל-HTML". המשתנים הזמינים בתוך התבנית הם בדיוק המפתחות של האובייקט שהעברנו ל-render (כאן: foo). יש עוד תגיות EJS — למשל <% ... %> להרצת קוד בלי הדפסה (לולאות, if) — אבל <%= %> הוא הנפוץ ביותר.

7. אובייקט שלם ל-View + תבנית article.ejs מלאה

🧭
קונטקסט כתבה היא לא מספר אחד — יש לה כותרת, מחבר, תאריך ותוכן. אז נעביר אובייקט שלם ל-view, וה-view יציג כל שדה במקומו.

ב-controller — מעבירים אובייקט כתבה כ-foo:

js · controllers/articles.js
const getArticle = (req, res) => {
    res.render('article.ejs', { foo : {
        id : 1,
        title : 'My cake',
        author: 'John Doe',
        published: 'February 11, 2024',
        content: 'Lorem ipsum'
    } })
}

module.exports = { getArticle }

ב-view — ניגשים לשדות עם foo.title, foo.author וכו':

html · views/article.ejs (body מלא)
<body>
    <header>
        <h1>Foo News</h1>
    </header>
    <main>
        <article>
            <h1><%= foo.title %></h1>
            <p>Published on: <%= foo.published %></p>
            <p>Author: <%= foo.author %></p>
            <p><%= foo.content %></p>
        </article>
    </main>
</body>

התוצאה: כותרת Foo News, מתחתיה My cake, התאריך, המחבר והתוכן — כל אחד מהשדה המתאים באובייקט.

8. Model — שכבת הנתונים

🧭
קונטקסט — הבעיה כרגע הכתבה כתובה בתוך ה-controller. זה לא נכון: ה-controller צריך לתאם, לא להחזיק נתונים. את הנתונים מוציאים ל-Model — השכבה האחראית על הגישה לנתונים. ה-controller רק מבקש מהמודל.

ה-controller מבקש מהמודל במקום להחזיק את הכתבה בעצמו:

js · controllers/articles.js
const articles = require('../models/articles')

const getArticle = (req, res) => {
    const id = 1
    res.render('article.ejs', { foo : articles.getArticle(id) })
}

module.exports = { getArticle }

models/articles.js — מחזיק את הכתבות (כרגע: מערך בזיכרון) ומספק פונקציית גישה:

js · models/articles.js
const articles = [
    {
        id : 1,
        title : 'My cake',
        author: 'John Doe',
        published: 'February 11, 2024',
        content: 'Lorem ipsum'
    },
    {
        id : 2,
        title : 'Not my cake',
        author: 'Not John Doe',
        published: 'Not February 11, 2024',
        content: 'Not Lorem ipsum'
    }
]

const getArticle = (id) => {
    return articles.filter((article) => article.id == id)[0]
}

module.exports = { getArticle }
🔤
איך עובד getArticle articles.filter((article) => article.id == id) מחזיר מערך של כל הכתבות שה-id שלהן תואם. מכיוון שהוא ייחודי, ניקח את האיבר הראשון עם [0]. (בהמשך, עם מסד נתונים אמיתי בהרצאה 8, זה יוחלף בשאילתה — אבל ה-חתימה של הפונקציה תישאר זהה, וה-controller לא ירגיש בכלל.)
🔗
זו בדיוק ה-DAL מהרצאה 5 ה-Model כאן הוא ה-Data Access Layer: השכבה היחידה שיודעת איפה ואיך שמורים הנתונים. ה-controller לא יודע אם זה מערך, קובץ או מסד נתונים — הוא רק קורא ל-getArticle(id). זו ההפרדה שמאפשרת לנו להחליף את הזיכרון ב-MongoDB בלי לשבור כלום.

9. Route params — /:id ו-req.params

🧭
קונטקסט — הבעיה ה-id מקובע ל-1 בקוד — תמיד מוצגת אותה כתבה. אבל אנחנו רוצים ש-/articles/1 יציג כתבה 1 ו-/articles/2 יציג כתבה 2. הפתרון: פרמטר בנתיב.

ב-controller — קוראים את ה-id מתוך הבקשה:

js · controllers/articles.js
const articles = require('../models/articles')

const getArticle = (req, res) => {
    const id = req.params.id          // ← מגיע מה-URL
    res.render('article.ejs', { foo : articles.getArticle(id) })
}

module.exports = { getArticle }

ב-route — מגדירים את הפרמטר עם נקודתיים:

js · routes/articles.js
const express = require('express')
var router = express.Router()

const articles = require('../controllers/articles')
router.get('/:id', articles.getArticle);

module.exports = router
🔤
:id ו-req.params הנקודתיים ב-'/:id' מגדירות פרמטר נתיב (route parameter) — חלק מה-URL שמשתנה. את הערך קוראים מ-req.params.id. כך /articles/2 נותן req.params.id === '2' (כמחרוזת!), וה-controller ישלוף את כתבה 2. שים לב: זה שונה מ-query string (?id=2) שנקרא מ-req.query, ומ-body של POST שנקרא מ-req.body (נראה מיד).

חלק ג' · יצירת תוכן (טפסים ו-POST)

10. טופס יצירה — new_article.ejs

🧭
קונטקסט עד כה רק קראנו כתבות (GET). עכשיו נאפשר ליצור כתבה חדשה. לזה צריך טופס HTML שאוסף קלט מהמשתמש ושולח אותו לשרת ב-POST (זוכרים מהרצאה 5: GET לקריאה, POST לשליחת נתונים שמשנים מצב).
html · views/new_article.ejs
<body>
    <header>
        <h1>Foo News</h1>
    </header>
    <main>
        <form method="post" action="/articles/create">
            <input name="title"     placeholder="Article title">
            <input name="published" type="date" placeholder="Publication date">
            <input name="author"    placeholder="Who wrote it?">
            <input name="content"   placeholder="Actual article goes here">
            <button>Create</button>
        </form>
    </main>
</body>
🔤
שלושת המאפיינים החשובים של <form>
  • method="post" — הטופס נשלח כבקשת POST (לא GET).
  • action="/articles/create" — הכתובת שאליה הטופס נשלח.
  • name="..." בכל <input> — קריטי! זה השם שתחתיו הערך יגיע לשרת (ב-req.body.title וכו'). בלי name, השדה פשוט לא נשלח.
ה-<button> בתוך טופס שולח אותו אוטומטית. type="date" נותן בורר תאריך מובנה של הדפדפן.

11. ה-routes ל-create — GET (טופס) + POST (יצירה)

🧭
קונטקסט יצירת כתבה היא שני נתיבים: אחד מציג את הטופס (GET /articles/create), והשני מקבל את הטופס המלא ויוצר את הכתבה (POST /articles/create).
js · routes/articles.js
const express = require('express')
var router = express.Router()

const articles = require('../controllers/articles')
router.get('/create',  articles.createArticleForm);   // הצגת הטופס
router.post('/create', articles.createArticle);        // קבלת הטופס
router.get('/:id',     articles.getArticle);           // הצגת כתבה

module.exports = router
🪤
סדר הנתיבים חשוב! (תוספת חשובה) שים לב ש-/create מוגדר לפני /:id. זה לא מקרי: Express בודק נתיבים לפי הסדר ולוקח את ההתאמה הראשונה. אילו /:id היה ראשון, גלישה ל-/articles/create הייתה נתפסת על ידו — ו-req.params.id היה מקבל את הערך "create"! לכן נתיבים ספציפיים (/create) תמיד באים לפני נתיבים עם פרמטר כללי (/:id).
💡
אותו URL, מתודות שונות GET /create ו-POST /create הם אותה כתובת אבל שתי פעולות שונות לגמרי — בדיוק כמו שראינו ב-REST בהרצאה 5: המתודה (GET/POST) היא חלק מהזהות של הבקשה, לא רק ה-URL.

12. ה-controller ל-create + express.urlencoded

🧭
קונטקסט שתי פונקציות חדשות ב-controller: אחת מציגה את טופס היצירה, השנייה קוראת את שדות הטופס, מוסיפה כתבה דרך ה-Model, ומפנה את המשתמש לכתבה החדשה.
js · controllers/articles.js
const articles = require('../models/articles')

const getArticle = (req, res) => {
    const id = req.params.id
    res.render('article.ejs', { foo : articles.getArticle(id) })
}

const createArticleForm = (req, res) => {
    const id = req.params.id
    res.render('new_article.ejs', {})        // רק מציג את הטופס
}

const createArticle = (req, res) => {
    const title     = req.body.title
    const published = req.body.published
    const author    = req.body.author
    const content   = req.body.content
    const id = articles.addArticle(title, published, author, content)
    res.redirect('/articles/' + id);          // הפניה לכתבה החדשה
}

module.exports = { getArticle, createArticleForm, createArticle }

כדי ש-req.body יהיה זמין, צריך middleware ב-app.js שיפענח את גוף הטופס:

js · app.js
const express = require('express')
var app = express()
app.use(express.urlencoded({ extended: true }));   // ← מפענח גוף טפסים

const articles = require('./routes/articles');
app.use('/articles', articles);

app.set('view engine', 'ejs');

app.listen(8080)
🔤
למה צריך express.urlencoded? כשטופס נשלח ב-POST, הנתונים מגיעים בגוף הבקשה (body) בקידוד application/x-www-form-urlencoded (למשל title=New+article&author=Jane+Smith). Express לא מפענח את זה לבד. ה-middleware express.urlencoded() קורא את הגוף, מפענח אותו, וממלא את req.body באובייקט נוח. בלעדיו, req.body יהיה undefined. (extended: true מאפשר אובייקטים מקוננים.)
🔤
res.redirect — למה לא פשוט לרנדר? אחרי יצירה מוצלחת אנחנו עושים res.redirect('/articles/' + id) — מבקשים מהדפדפן לגשת לכתבה החדשה (GET). זה ה-PRG pattern שנפרק מיד בסעיף הבא. (בדוגמה זו addArticle ב-Model פשוט מוסיף למערך ומחזיר id — בהרצאה 8 הוא ייצור רשומה במסד הנתונים.)

13. מה קורה ברשת — POST, redirect, ותבנית PRG

🧭
קונטקסט נפתח את לשונית Network ב-DevTools ונראה מה באמת עף על החוט כשלוחצים "Create". זה ממחיש בדיוק את מחזור הבקשה-תשובה שלמדנו, ומגלה תבנית עיצוב חשובה.

כשממלאים את הטופס (New article / 12/02/2024 / Jane Smith / Wow wow wow) ולוחצים Create, רואים שתי בקשות ב-Network: create ואז 3.

בקשה 1 — ה-POST. שדות הטופס נשלחים בגוף הבקשה (Form Data / Payload):

http · בקשת ה-POST
POST /articles/create HTTP/1.1
Content-Type: application/x-www-form-urlencoded

# Form Data (כפי שמוצג ב-DevTools):
title:     New article
published:  2024-02-12
author:    Jane Smith
content:   Wow wow wow

# אותו דבר, מקודד (view URL-encoded):
title=New+article&published=2024-02-12&author=Jane+Smith&content=Wow+wow+wow

התשובה ל-POST — 302 Found (הפניה):

http · תשובת ה-redirect
HTTP/1.1 302 Found
X-Powered-By: Express
Location: /articles/3
Content-Type: text/html; charset=utf-8

בקשה 2 — ה-GET האוטומטי. הדפדפן רואה את ה-302 + Location וניגש אוטומטית לכתובת החדשה:

http · ה-GET שאחרי ההפניה
GET /articles/3 HTTP/1.1
→ 200 OK
→ <!DOCTYPE html> ... <h1>New article</h1> ...   (הכתבה החדשה!)
🔁
תבנית PRG — Post / Redirect / Get הזרימה היא: POST (יוצרים כתבה) → השרת מחזיר Redirect (302) → הדפדפן עושה GET לכתבה החדשה. למה לא פשוט להחזיר את ה-HTML ישירות מה-POST? כי אז רענון הדף היה שולח את ה-POST שוב — ויוצר כתבה כפולה! עם PRG, הדף הסופי הגיע דרך GET רגיל, אז רענון בטוח לחלוטין. (זוכרים מהרצאה 5: GET הוא idempotent ובטוח, POST לא.)
🔤
קודי הסטטוס שראינו 302 Found = "המשאב נמצא במקום אחר, לך לשם" — הפניה זמנית, עם הכתובת ב-header Location. 200 OK = הבקשה הצליחה והנה התוכן. שניהם הופיעו בעמודת ה-Status ב-Network.

★ סיכום ההרצאה

🎯
התמונה המלאה — מה בנינו
  • app.js — נקודת הכניסה: מרים את Express, רושם middleware (urlencoded), מחבר את ה-router, ומגדיר את מנוע ה-view (ejs).
  • routes/ — ממפה URL+method → פונקציית controller (express.Router). סדר הנתיבים חשוב: ספציפי לפני /:id.
  • controllers/ — הלוגיקה: מקבל בקשה, מבקש מה-Model, מעביר ל-View. קורא קלט מ-req.params (URL) ו-req.body (טופס).
  • models/ — שכבת הנתונים (ה-DAL): כאן הכתבות במערך, עם getArticle/addArticle. השכבה היחידה שיודעת איפה הנתונים.
  • views/ — תבניות EJS: <%= foo.title %> מזריק נתונים ל-HTML. res.render ממלא ומחזיר.
  • זרימת MVC: request → controller → (invoke→data) model → (data) view → response. ה-controller הוא הצומת המרכזי.
  • טפסים: <form method="post" action="...">" + name לכל שדה → req.body. תבנית PRG (POST→302→GET) מונעת יצירה כפולה ברענון.
➡️
בהרצאה 8 נחליף את ה-Model מהמערך בזיכרון במסד נתונים אמיתי: MongoDB (ו-Mongoose), נוסיף שכבת services, פונקציות אסינכרוניות (async/await), CRUD מלא, וקונפיגורציית סביבות (dotenv, NODE_ENV). כל המבנה שבנינו היום יישאר — נחליף רק את שכבת הנתונים.
