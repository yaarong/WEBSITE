1. מה זו מערכת בקרת גרסאות (VCS) ולמה צריך אותה

🧭
קונטקסט — מה אנחנו לומדים כאן לפני שנגיע ל-Git עצמו, צריך להבין איזו בעיה הוא בא לפתור. הנושא הזה מסביר מהי מערכת בקרת גרסאות (Version Control System) — סוג של תוכנה שמלווה אותנו כמתכנתים לכל אורך הקריירה, ובלעדיה עבודת צוות על קוד היתה כמעט בלתי אפשרית.

כשאנחנו מפתחים תוכנה, הקוד שלנו מתפתח כל הזמן: מוסיפים פיצ'רים, מתקנים באגים, משנים קבצים, מוחקים קוד ישן. אחרי כמה שבועות הפרויקט נראה אחרת לגמרי ממה שהיה בהתחלה. השאלה היא: איך עוקבים אחרי כל השינויים האלה בצורה מסודרת?

מערכת בקרת גרסאות (VCS) היא כלי שעוקב אחרי הגרסאות של הקוד שלנו ומנהל אותן. היא נותנת לנו כמה יכולות קריטיות:

  • היסטוריה ו-Rollback — לראות בדיוק מה השתנה, מתי, ועל ידי מי; ובמקרה הצורך לחזור לגרסה ישנה שעבדה. שברתם משהו? אפשר פשוט לחזור אחורה.
  • השוואות (Comparisons) — להשוות בין שתי גרסאות ולראות בדיוק אילו שורות נוספו, נמחקו או שונו.
  • גיבויים (Backups) — הקוד שמור גם במקום מרוחק. אם המחשב נשרף — הפרויקט לא הולך לאיבוד.
  • עבודת צוות — ובמיוחד סנכרון — כמה מפתחים יכולים לעבוד על אותו פרויקט במקביל, וה-VCS דואג למזג את העבודה של כולם בלי שאחד ידרוס את השני.
💡
דוגמה מהחיים (משלי) תחשבו על Google Docs: אתם וחבר עורכים את אותו מסמך, ויש "היסטוריית גרסאות" שמאפשרת לחזור לכל נקודה בזמן. VCS עושה בדיוק את זה — אבל לקוד, ובצורה הרבה יותר חזקה ומדויקת (ברמת השורה הבודדת).

2. גרסאות לאורך זמן

🧭
קונטקסט הרעיון המרכזי של VCS הוא שמירת נקודות ציון (snapshots) של הפרויקט לאורך ציר הזמן. נבין את המודל המנטלי הזה לפני שנדבר על איך Git מממש אותו.

נדמיין ציר זמן. בכל פעם שאנחנו מסיימים שלב משמעותי בעבודה, אנחנו "מצלמים" את מצב הפרויקט ושומרים אותו כגרסה:

   V1 ───────▶ V2 ───────▶ V3 ───────▶ ...
   │           │           │
 גרסה        גרסה        גרסה
 ראשונה      שנייה       שלישית
                                          ⟶ Time (זמן)

כל נקודה (V1, V2, V3...) היא גרסה שמורה של כל הפרויקט באותו רגע. בעולם של Git נקרא לנקודות האלה commits. היתרון: בכל רגע אפשר "לקפוץ" חזרה לכל נקודה ולראות איך הפרויקט נראה אז.

💡
נקודה חשובה הגרסאות לא "מוחקות" אחת את השנייה — הן נשמרות בנוסף. ההיסטוריה כולה נשמרת, וזה בדיוק מה שמאפשר rollback והשוואות.

3. מודל Client–Server (לקוח–שרת)

🧭
קונטקסט — למה זה כאן? כדי להבין איך Git מסנכרן קוד בין כמה מפתחים, צריך קודם להבין את מודל לקוח–שרת — אחד המודלים הבסיסיים ביותר במדעי המחשב. הוא יחזור שוב ושוב בקורס הזה (ובכל מערכת מבוזרת).

נניח ש-Alice רוצה לשלוח הודעה "Hi" ל-Bob. שני המחשבים שלהם הם לקוחות (clients). אבל איך ההודעה עוברת ביניהם? היא לא עוברת ישירות — היא עוברת דרך שרת (server) מרכזי באמצע:

   Alice's client              Server               Bob's client
   ┌──────────────┐         ┌──────────┐         ┌──────────────┐
   │ Send to Bob: │  ─────▶ │  SERVER  │  ─────▶ │ From Alice:  │
   │   "Hi"       │         │          │         │   "Hi"       │
   └──────────────┘         └──────────┘         └──────────────┘
     "From Alice to Bob: Hi"        "For Bob from Alice: Hi"

הרעיון המרכזי, שמסומן במצגת במסגרת אדומה:

⭐
השרת משרת את הלקוחות (Server serves clients) השרת הוא נקודה מרכזית שכל הלקוחות מתחברים אליה. הלקוח מבקש משהו, והשרת מספק אותו (serve = להגיש/לשרת). מכאן השם "server".

קוד צד-לקוח מול קוד צד-שרת

חלוקת התפקידים יוצרת שני סוגי קוד:

  • Client-side code — הקוד שרץ אצל המשתמש (אצל Alice ואצל Bob): קליטת ההודעה שהוקלדה, הצגת ההודעה שהתקבלה.
  • Server-side code — הקוד שרץ על השרת המרכזי: קבלת ההודעה מ-Alice, וניתובה ל-Bob.

דוגמה: דפדפן ושרת אינטרנט

המודל הזה הוא בדיוק מה שקורה כשאתם גולשים באינטרנט:

🌐
הדפדפן הוא תוכנת לקוח דפדפן (browser) הוא תוכנת לקוח (client program) שמתקשרת עם תוכנת שרת (server program). כשאתם מקלידים https://www.google.com ולוחצים Enter — הדפדפן (הלקוח) שולח בקשה לשרת של Google, והשרת מחזיר בתגובה את דף האינטרנט.
💡
הרחבה (משלי) אותו דבר קורה בכל אפליקציה כמעט: WhatsApp, אינסטגרם, Gmail — בכולן יש לקוח (האפליקציה במכשיר שלכם) ושרת מרכזי באמצע. ב-Git, כפי שנראה מיד, ה"שרת" הוא GitHub/GitLab, וה"לקוח" הוא המחשב שלכם.

4. Git — מערכת בקרת גרסאות מבוזרת

🧭
קונטקסט עכשיו, אחרי שהבנו מהי VCS ומהו מודל לקוח–שרת, נכיר את Git — מערכת בקרת הגרסאות הנפוצה ביותר בעולם, ואת זו שנשתמש בה לאורך כל הקורס.
  • Git היא VCS מבוזרת (distributed) — לכל מפתח יש עותק מלא של כל המאגר, כולל כל ההיסטוריה, אצלו במחשב. זה שונה ממערכות "מרוכזות" שבהן ההיסטוריה קיימת רק בשרת.
  • Git עוזרת לנהל ולעקוב אחרי שינויים בקוד המקור במהלך פיתוח התוכנה.
  • צריך להתקין את Git על המחשב כדי להשתמש בה (היא תוכנה שרצה אצלכם, לא שירות אינטרנט).
🔑
"מבוזר" — למה זה כל כך חשוב (משלי) כי לכל מפתח יש עותק מלא, אפשר לעבוד גם בלי חיבור לאינטרנט — לבצע commits, לראות היסטוריה, ליצור ענפים. רק כשרוצים לסנכרן עם אחרים (push/pull) צריך רשת. זה גם אומר שאם השרת המרכזי נופל — אף אחד לא איבד את הקוד, כי לכולם יש עותק.
📦
הבחנה חשובה: Git לעומת GitHub Git = התוכנה שרצה אצלכם במחשב ומנהלת גרסאות.
GitHub / GitLab / BitBucket = שירותי אירוח (שרתים) שמאחסנים מאגרי Git ברשת ומאפשרים שיתוף פעולה. הם ה"שרת" במודל לקוח–שרת. Git יכולה לעבוד גם בלעדיהם, לגמרי מקומית.

5. יצירת מאגר מקומי — git init

🧭
קונטקסט מכאן מתחיל ה"זרימה" המעשית של עבודה עם Git. הצעד הראשון בכל פרויקט חדש הוא להפוך תיקייה רגילה למאגר (repository) ש-Git עוקב אחריו. זה מה שעושים עם הפקודה git init.

Repository (מאגר, בקיצור "repo") הוא פשוט הפרויקט שלנו ביחד עם כל היסטוריית הגרסאות שלו. כדי ליצור מאגר מקומי חדש על המחשב, נריץ את רצף הפקודות הבא בטרמינל:

bash · terminal
% mkdir hello_world
% cd hello_world
% ls
% git init
Initialized empty Git repository in /hello_world/.git/
% ls

הסבר על הפקודות (סינטקס)

  • mkdir hello_world — make directory: יוצר תיקייה חדשה בשם hello_world.
  • cd hello_world — change directory: נכנס לתוך התיקייה.
  • ls — list: מציג את תוכן התיקייה (כרגע ריקה).
  • git init — ⭐ הפקודה המרכזית: מאתחל מאגר Git חדש בתיקייה הנוכחית. הפלט Initialized empty Git repository מאשר שזה הצליח.

הקסם של git init מסתתר בתיקייה נסתרת אחת. אם נריץ ls -a (שמציג גם קבצים נסתרים) נראה אותה:

bash · terminal
% ls -a
.    ..    .git
📁
התיקייה .git — הלב של המאגר git init יוצר תיקייה נסתרת בשם .git (הנקודה בהתחלה הופכת אותה לנסתרת). כל ההיסטוריה, ההגדרות והאובייקטים של Git שמורים בתוכה. אם תמחקו את .git — התיקייה תחזור להיות תיקייה רגילה, ותאבדו את כל היסטוריית הגרסאות. בהמשך ההרצאה (בפרק Internals) נציץ פנימה ונראה מה יש שם.

6. מאגר מרוחק (Remote) — GitHub / GitLab / BitBucket

🧭
קונטקסט מאגר מקומי טוב לעבודה אישית, אבל בשביל גיבוי ושיתוף עם אחרים צריך מאגר מרוחק — כזה שיושב על שרת ברשת. כאן נכנסים שירותים כמו GitHub. זהו ה"שרת" מהמודל שראינו קודם.

גם מאגר מרוחק "מאותחל" (init), אבל הפעם דרך ממשק הווב של השירות (GitHub וכו') ולא דרך הטרמינל. בעת יצירת מאגר חדש ב-GitHub ממלאים טופס:

  • Owner / Repository name — מי הבעלים ומה שם המאגר.
  • Public / Private — האם כל אחד באינטרנט יכול לראות את המאגר (Public), או רק אנשים שאתם מאשרים (Private).
  • Add a README file — מוסיף קובץ README.md ראשוני שמתאר את הפרויקט.
  • Add .gitignore — תבנית קבצים שלא נרצה ש-Git יעקוב אחריהם (נרחיב בהמשך).
  • Choose a license — רישיון שימוש בקוד.
  • זה גם יקבע את main כשם ענף ברירת המחדל.

אחרי היצירה, יש לנו מאגר מרוחק עם קובץ README.md ו-commit ראשון ("Initial commit"). קובץ README נכתב בפורמט Markdown (סיומת .md). לדוגמה:

markdown · README.md
# MyDemo
This is my demo repository
💡
קצת על Markdown (משלי) Markdown היא שפת סימון פשוטה לעיצוב טקסט. # כותרת = כותרת גדולה, ## כותרת = כותרת קטנה יותר, **מודגש** = טקסט מודגש, - פריט = רשימה. GitHub מציג קבצי .md מעוצבים יפה אוטומטית — לכן ה-README הוא "כרטיס הביקור" של כל פרויקט.

7. שכפול מאגר — git clone

🧭
קונטקסט — תרחיש Alice יש מאגר מרוחק על השרת. עכשיו Alice רוצה לעבוד עליו במחשב שלה. היא צריכה להוריד עותק מקומי מלא — זה מה ש-git clone עושה.

git clone מוריד את המאגר המרוחק במלואו (כל הקבצים + כל ההיסטוריה) ויוצר ממנו עותק מקומי על המחשב. אחרי ה-clone, ל-Alice יש "עותק מקומי משלה של המאגר" (Alice's local copy of the repository):

   Server (remote)                     Alice's computer (client)
   ┌────────────────┐                  ┌────────────────┐
   │ Project / Repo │   git clone      │ Project / Repo │
   │  README.md     │  ───────────▶    │  README.md     │
   └────────────────┘   (עותק מקומי)   └────────────────┘
bash · terminal
git clone https://github.com/username/MyDemo.git

ב-VS Code אפשר לעשות את אותו דבר גרפית: בפאנל Source Control לוחצים על Clone Repository ומדביקים את כתובת ה-URL של המאגר. אחרי ה-clone, התיקייה MYDEMO נפתחת ובתוכה הקובץ README.md.

💡
init מול clone (משלי) git init = מתחילים פרויקט חדש מאפס (מאגר ריק).
git clone = מורידים פרויקט קיים שכבר נמצא בשרת.
ברוב המקרים בעבודת צוות תשתמשו ב-clone כדי להצטרף לפרויקט שכבר קיים.

8. מצבי קבצים: Untracked, Modified, Ignored

🧭
קונטקסט — אחד הנושאים הכי חשובים בהרצאה Git לא מתייחס לכל הקבצים אותו דבר. לכל קובץ יש מצב (state), ו-Git מציג אותו (ב-VS Code, באות צבעונית ליד שם הקובץ). הבנת המצבים האלה היא הבסיס להבנת כל מה שבא אחר כך — staging, commit ו-push.

שינוי קובץ קיים → "Modified" (M)

Alice משנה את README.md (למשל מוסיפה "2" בסוף השורה). ברגע ששינתה קובץ ש-Git כבר עוקב אחריו, הקובץ מסומן כ-Modified (האות M ב-VS Code):

markdown · README.md (שונה)
# MyDemo
This is my demo repository2

קובץ חדש → "Untracked" (U)

Alice יוצרת קובץ חדש לגמרי, hello.txt. Git עדיין לא עוקב אחרי הקובץ הזה — הוא Untracked (האות U):

❗
Untracked = "אני לא יודע מי אתה" כשיוצרים קובץ חדש, Git רואה שהוא קיים אבל לא מנהל את הגרסאות שלו עדיין. הוא ממתין שנגיד לו במפורש "תתחיל לעקוב אחרי הקובץ הזה" (זה מה ש-git add עושה, נראה מיד).

קובץ שלא רוצים לעקוב אחריו → ".gitignore" → "Ignored"

לפעמים יש קבצים שאנחנו בכוונה לא רוצים ש-Git יעקוב אחריהם. Alice יוצרת קובץ hello2.txt, ומוסיפה את שמו לקובץ מיוחד בשם .gitignore. מעכשיו הקובץ נחשב Ignored (מתעלמים ממנו):

.gitignore
hello2.txt

הקובץ .gitignore מכיל רשימה של קבצים ותבניות ש-Git יתעלם מהם לחלוטין — הם לא יופיעו כ-Untracked ולא ייכנסו לעולם ל-commits.

💡
למה צריך .gitignore? (משלי + דוגמאות נפוצות) יש קבצים שאסור או שאין טעם לשמור ב-Git: סיסמאות וקבצי הגדרות רגישים, קבצים שנוצרים אוטומטית (תוצרי קומפילציה), תיקיות ענק של ספריות חיצוניות. דוגמאות לתבניות נפוצות ב-.gitignore:
.gitignore · דוגמה מורחבת
# קבצי תלויות (לא שומרים ב-Git — מתקינים מחדש)
node_modules/

# תוצרי בנייה / קומפילציה
build/
dist/
*.class
*.exe

# קבצי סביבה רגישים (סיסמאות, מפתחות API)
.env

# קבצים זמניים של מערכת ההפעלה / העורך
.DS_Store
*.log

# התעלמות מכל קובץ עם סיומת מסוימת
*.tmp
🔤
סינטקס של .gitignore name.txt — קובץ ספציפי.  |  folder/ — תיקייה שלמה.  |  *.log — כל הקבצים עם הסיומת .log (הכוכבית = "כל דבר").  |  # הערה — שורה שמתחילה ב-# היא הערה ומתעלמים ממנה.

9. Staging ו-Commit — git add + git commit

🧭
קונטקסט — איך שומרים גרסה? ראינו שקבצים יכולים להיות Modified או Untracked. עכשיו נלמד איך "מקבעים" אותם לגרסה שמורה. התהליך הוא בן שני שלבים: קודם staging (בחירת מה ייכנס לגרסה), ואז commit (שמירת הגרסה בפועל). שימו לב — זה אחד הדברים שמבלבלים מתחילים, אז נסביר לאט.

שלב 1: git add — העברת קבצים ל-Staging

git add מסמן קבצים כ"מועמדים להיכנס ל-commit הבא". בעגה של Git זה נקרא staging (ב-VS Code זה הכפתור + = "Stage Changes", והקובץ עובר ממצב U/M למצב Staged, האות A = Added):

bash · terminal
# להוסיף קובץ ספציפי ל-staging
git add hello4.txt

# להוסיף את כל השינויים בבת אחת
git add .
🎯
למה בכלל צריך staging? (הרעיון המרכזי) ה-staging area נותן לנו שליטה מדויקת על מה ייכנס ל-commit. דוגמה מהמצגת: Alice יצרה גם hello3.txt וגם hello4.txt, אבל החליטה לבצע git add רק ל-hello4.txt. כך ה-commit הבא יכיל את hello4.txt בלבד — ואילו hello3.txt יישאר Untracked. זה מאפשר לקבץ שינויים קשורים ל-commit אחד הגיוני, במקום "לזרוק הכל ביחד".

שלב 2: git commit — קיבוע הגרסה

אחרי ש-staging מוכן, git commit שומר את כל מה שב-staging כגרסה חדשה ומקובעת בהיסטוריה המקומית. לכל commit יש הודעה (message) שמתארת מה השתנה:

bash · terminal
# commit עם הודעה
git commit -m "My first commit"
🔤
סינטקס -m = message. הטקסט בגרשיים אחריו הוא הודעת ה-commit. הודעה טובה מתארת בקצרה מה השינוי עושה (למשל "Add login button" ולא "fix"). אחרי commit, הקבצים חוזרים למצב נקי — Unmodified (אין יותר אות צבעונית לידם).

ב-VS Code אפשר פשוט לכתוב הודעה בתיבת הטקסט וללחוץ על ✓ Commit. אם לוחצים Commit כשהשינויים עדיין לא ב-staging, VS Code יבצע עבורכם גם add וגם commit ביחד.

⚠️
קריטי: commit הוא מקומי בלבד! commit שומר את הגרסה רק על המחשב שלכם (במאגר המקומי). השרת המרוחק (GitHub) לא יודע עליו כלום עדיין! אם תיכנסו ל-GitHub אחרי commit, לא תראו שום שינוי. כדי שהשינוי יגיע לשרת — צריך push (הנושא הבא).

10. דחיפה למאגר המרוחק — git push

🧭
קונטקסט עשינו commits מקומיים. עכשיו נרצה להעלות אותם לשרת — כדי לגבות אותם וכדי שחברי צוות יוכלו לראות. זה התפקיד של git push.

git push דוחף (שולח) את ה-commits המקומיים אל המאגר המרוחק. רק אחרי push, השינויים מופיעים בשרת:

   Alice's computer (client)              Server (remote)
   ┌────────────────┐     git push       ┌────────────────┐
   │ local commits  │  ──────────────▶   │  remote repo   │
   └────────────────┘                    └────────────────┘
bash · terminal
git push

ב-VS Code: בתפריט ··· של Source Control בוחרים Push.

🔍
שימו לב לדוגמה מהמצגת Alice עשתה commit שכלל את hello4.txt בלבד (לא את hello3.txt). אחרי ה-push, בשרת מופיעים README.md, hello.txt, .gitignore ו-hello4.txt — אבל hello3.txt עדיין נשאר Untracked ולא הגיע לשרת, כי הוא מעולם לא נכנס ל-commit. זה בדיוק הכוח של staging: שליטה מדויקת על מה מתפרסם.
📜
היסטוריית ה-Commits ב-GitHub (וגם מקומית עם git log) אפשר לראות את כל ה-commits לפי סדר כרונולוגי — מי עשה אותם, מתי, ועם איזו הודעה. למשל: "Initial commit" → "My first commit" → "My second commit". זו ה"היסטוריה" שדיברנו עליה בהתחלה.

11. המודל המלא: ארבעת האזורים של Git

🧭
קונטקסט — התמונה הגדולה זו אולי השקופית החשובה ביותר בהרצאה. עד עכשיו ראינו את הפקודות אחת-אחת. עכשיו נחבר הכל למודל אחד שמראה איפה כל קובץ נמצא בכל רגע. שווה לחזור למודל הזה בכל פעם שאתם מתבלבלים.

Git מחלק את העולם לארבעה אזורים, ולקובץ יש מצב בכל אחד מהם:

אזור מצבי הקובץ באזור מה זה אומר
Working directory
(תיקיית העבודה)		 Untracked / Unmodified / Modified הקבצים שאתם רואים ועורכים בפועל על הדיסק.
Staging area
(אזור הבמה)		 Staged קבצים שסומנו עם git add ומחכים להיכנס ל-commit הבא.
Local repository
(מאגר מקומי)		 Committed גרסאות מקובעות שנשמרו עם git commit — על המחשב שלכם.
Remote repository
(מאגר מרוחק)		 Committed אותן גרסאות, אבל על השרת — אחרי git push.

מסע של קובץ דרך כל האזורים

נעקוב אחרי קובץ foo.txt לאורך כל החיים שלו, בדיוק כפי שמופיע בדיאגרמת הזרימה במצגת:

 Working directory          │ Staging │ Local   │ Remote
 Untracked Unmodified Modified│ Staged  │Committed│Committed
 ────────────────────────────────────────────────────────────
 (1) foo.txt נוצר ........... Untracked
 (2) git add ............................ Staged
 (3) git commit ................................. Committed
 (4) חוזר ל-Unmodified ◀── "working copy is unmodified again"
 (5) עורכים שוב ............ Modified
 (6) git add ............................ Staged
 (7) git commit ................................. Committed
 (8) git push ............................................... Committed (remote)

שלב אחר שלב:

  1. נוצר קובץ חדש foo.txt → המצב הוא Untracked.
  2. git add → הקובץ עובר להיות Staged (ב-VS Code: סימן +, האות A).
  3. git commit → השינוי נשמר במאגר המקומי כ-Committed.
  4. אחרי ה-commit, עותק העבודה חוזר להיות Unmodified ("נקי").
  5. עורכים את הקובץ שוב → הוא הופך ל-Modified.
  6. git add שוב → Staged (מוכן ל-commit).
  7. git commit שוב → Committed, וחוזר חלילה.
  8. לבסוף git push → השינויים מגיעים ל-מאגר המרוחק.
⚡
קיצור דרך (מהמצגת) אפשר לבצע commit ישירות לקבצים Modified, ו-Git יעשה את ה-"add to staging" עבורכם. כלומר לקבצים שכבר במעקב (tracked) אפשר לדלג על git add ולהשתמש ב-git commit -a -m "...". שימו לב: זה עובד רק לקבצים tracked — קבצים Untracked חייבים git add מפורש.
bash · קיצור add+commit
# -a עושה add אוטומטי לכל הקבצים ה-tracked שהשתנו, ואז commit
git commit -a -m "Update foo.txt"

12. עבודת צוות: clone, push, pull

🧭
קונטקסט — נכנס שחקן שני עד עכשיו רק Alice עבדה. עכשיו Bob רוצה להצטרף לפרויקט. נראה את המחזור המלא של שיתוף פעולה — וכאן נכיר את הפקודה האחרונה בחבילת הבסיס: git pull.

Bob מצטרף — git clone

Bob רוצה לעבוד על הפרויקט מקומית, אז הוא מבצע git clone מהשרת. עכשיו יש שני עותקים מקומיים נפרדים: של Alice ושל Bob. שימו לב: Bob מקבל רק את מה שיש בשרת (למשל בלי hello3.txt ו-hello2.txt שמעולם לא עלו).

Bob עובד — git add + commit + push

Bob משנה קובץ (למשל hello.txt), מבצע add + commit מקומית, ואז git push כדי להעלות לשרת. עכשיו השרת מעודכן עם השינוי של Bob.

Alice מסתנכרנת — git pull

הבעיה: אצל Alice עדיין יש את הגרסה הישנה — היא לא יודעת על השינוי של Bob! כדי למשוך את השינויים מהשרת אל המאגר המקומי שלה, Alice מבצעת git pull:

   Server (remote)                        Alice's computer
   ┌────────────────┐      git pull        ┌────────────────┐
   │  remote repo   │  ──────────────▶     │  local repo    │
   │ (Bob's change) │   (מושך עדכונים)     │ (now updated)  │
   └────────────────┘                      └────────────────┘
bash · terminal
git pull
🔄
push מול pull — לזכור את הכיוון push = שולח את השינויים שלי מהמחשב לשרת (↑ למעלה).
pull = מושך שינויים של אחרים מהשרת למחשב שלי (↓ למטה).
מחזור עבודה טיפוסי: pull (לקבל את העדכונים האחרונים) → לעבוד → add → commit → push (לשתף).

13. ענפים (Branches), HEAD ופוינטרים

🧭
קונטקסט — אחד הכלים החזקים ביותר ב-Git עד עכשיו דיברנו על קו זמן אחד של commits. אבל מה אם רוצים לפתח פיצ'ר חדש בלי לסכן את הקוד היציב? מה אם כמה אנשים עובדים על דברים שונים במקביל? התשובה היא ענפים (branches). זה הנושא שהופך את Git ממערכת גיבוי לכלי עבודה אמיתי.

איך commits באמת מקושרים

בתוך Git, ה-commits מקושרים בשרשרת: כל commit "מצביע" אחורה ל-commit שלפניו (ה-parent שלו). זה יוצר קו רציף:

  Commit ◀── Commit ◀── Commit ◀── Commit
                                      ▲
                                      │
                              main branch pointer

מהו ענף (branch)?

📌
ענף הוא בסך הכל מצביע (pointer) ל-commit זו אחת התובנות החשובות בהרצאה: ענף הוא רק מצביע (pointer) שמצביע על commit מסוים. הוא לא "עותק" של הקוד. ענף ברירת המחדל נקרא master או main, וה-pointer שלו מצביע תמיד על ה-commit האחרון בענף.

HEAD — איפה אני נמצא עכשיו

כשיוצרים ענף חדש (למשל Foo), נוצר מצביע נוסף. אבל איך Git יודע על איזה ענף אנחנו עובדים כרגע? בעזרת מצביע מיוחד בשם HEAD:

👉
HEAD = "על איזה ענף אני נמצא ברגע זה" HEAD הוא מצביע שמצביע על הענף הפעיל הנוכחי. אם HEAD מצביע על Foo — אתם "על" ענף Foo, וכל commit חדש יקדם את המצביע של Foo. אם תעברו ל-main, ה-HEAD יצביע על main.

עבודה על כמה ענפים

כשנמצאים על ענף Foo ועושים commits, רק המצביע של Foo מתקדם — בעוד main נשאר במקום. כך נוצרת "התפצלות": שני קווי פיתוח מקבילים שיוצאים מאותה נקודה. כל ענף מתקדם בנפרד מבלי להפריע לשני.

                             Foo ──▶ Commit ◀── Commit   ◀── HEAD
                                                  │
  Commit ◀── Commit ◀── Commit ◀───────────────────┘
                          ▲
                         main

יצירת ענף בפועל

ב-VS Code: תפריט ··· → Branch → Create Branch.... מאחורי הקלעים Git מריץ:

bash · terminal
# יצירת ענף חדש בשם MyBranch ומעבר אליו (-b = create + switch)
git checkout -b MyBranch

# (הפקודה המודרנית והמומלצת יותר, עושה אותו דבר):
git switch -c MyBranch
🔤
סינטקס git checkout = מעבר בין ענפים/גרסאות. הדגל -b = branch, כלומר "צור ענף חדש וגם עבור אליו". במצגת מופיעה הצורה המלאה שה-GUI מריץ: git checkout -q -b MyBranch --no-track HEAD (-q = quiet, שקט).

commit ו-push לענף החדש

עושים commit בענף MyBranch, ואז דוחפים אותו לשרת. בפעם הראשונה צריך לציין לאן לדחוף את הענף החדש:

bash · terminal
# push ראשון של ענף חדש — קושר אותו לענף מרוחק תואם
git push -u origin MyBranch
🔤
סינטקס origin = השם המוסכם של המאגר המרוחק (השרת). -u (או --set-upstream) = "קשור את הענף המקומי לענף המרוחק התואם", כך שבפעמים הבאות אפשר פשוט git push בלי לפרט. אחרי זה יהיו לנו שני ענפים בשרת: main ו-MyBranch, ו-GitHub יציין ש-MyBranch הוא "1 commit ahead of main".

14. מיזוג Fast-forward

🧭
קונטקסט סיימנו לפתח בענף הצדדי — עכשיו נרצה להחזיר את השינויים ל-main. הפעולה הזו נקראת מיזוג (merge). נתחיל מהמקרה הפשוט ביותר: fast-forward.

Fast-forward merge קורה במקרה הפשוט: ה-main לא השתנה מאז שיצרנו את הענף. במצב כזה אין שום סתירה — Git פשוט "מזיז את המצביע של main קדימה" עד שהוא משיג את הענף. זהו, מוזג.

  לפני המיזוג:
                          MyBranch ──▶ Commit (חדש)   ◀── HEAD
                                          │
  Commit ◀── Commit ◀── Commit ◀───────────┘
                          ▲
                         main (נשאר מאחור)

  אחרי fast-forward — main פשוט "קופץ קדימה":
  Commit ◀── Commit ◀── Commit ◀── Commit
                                      ▲
                                  main = MyBranch

השלבים בפועל (כפי שבמצגת):

  1. Checking out the main branch — קודם עוברים חזרה לענף main (תפריט ··· → Checkout to... → main).
  2. Merging 'MyBranch' into main — ··· → Branch → Merge Branch... ובוחרים את MyBranch.
  3. התוצאה: success — main עכשיו מכיל את כל ה-commits של MyBranch.
  4. אפשר עכשיו למחוק את MyBranch (Delete MyBranch) — וההיסטוריה שלו נשארת ב-main (כי ה-commits כבר חלק מ-main).
bash · terminal
git checkout main          # לעבור ל-main
git merge MyBranch         # למזג את MyBranch לתוך main
git branch -d MyBranch     # למחוק את הענף שמוזג (-d = delete)
💡
למה אפשר למחוק את הענף בלי לאבד כלום (משלי) זכרו — ענף הוא רק מצביע. אחרי מיזוג ל-main, ה-commits עצמם כבר חלק מההיסטוריה של main. מחיקת הענף מוחקת רק את המצביע, לא את ה-commits. ההיסטוריה של main אכן מכילה את כל ה-commits מהענף שנמחק.

15. עבודה לפי פיצ'ר (Topic / Feature branches)

🧭
קונטקסט זהו הדפוס עבודה (workflow) הסטנדרטי בתעשייה: כל פיצ'ר חדש מקבל ענף משלו. נושא קצר אבל חשוב מאוד — זה בדיוק איך עובדים בצוותים אמיתיים.

השיטה נקראת "Branch by feature" — לכל פיצ'ר ענף נפרד. הזרימה המלאה:

  1. יצירת ענף חדש ומעבר אליו (ענף לכל פיצ'ר).
  2. מימוש השינויים שהפיצ'ר דורש:
    • commit + push לענף הפיצ'ר לפי הצורך (כמה פעמים שרוצים).
  3. כשמסיימים:
    • עוברים לענף main.
    • ממזגים את שינויי ענף הפיצ'ר לתוך main.
    • מוחקים את ענף הפיצ'ר.
    • דוחפים את השינויים (push).
🛡️
למה זה כל כך טוב (משלי) ענף main נשאר תמיד יציב ועובד, כי כל הניסויים והפיתוחים קורים בענפים נפרדים. אם פיצ'ר נכשל — פשוט מוחקים את הענף שלו, ו-main לא נפגע. בנוסף, כמה מפתחים יכולים לעבוד במקביל, כל אחד על ענף משלו, בלי לדרוך אחד לשני.

16. מיזוג Three-way

🧭
קונטקסט ב-fast-forward הנחנו ש-main לא השתנה. אבל בעבודת צוות אמיתית, בזמן שעבדנו על הענף שלנו — מישהו אחר כבר שינה את main. במצב כזה Git לא יכול פשוט "לקפוץ קדימה". כאן נכנס three-way merge.

המצב: גם main התקדם (commit חדש, מסומן בכתום בדיאגרמה) וגם הענף Foo התקדם — בשני כיוונים נפרדים מאותה נקודת מוצא. Git לא יכול פשוט להזיז מצביע; הוא צריך לאחד את שני קווי הפיתוח.

                        Foo ──▶ Commit ◀── Commit
                                             │
  Commit ◀── Commit ◀── Commit ◀── Commit(כתום) ◀── HEAD/main
                          ▲
                          └── נקודת מוצא משותפת
🔀
למה "three-way"? Git מסתכל על שלוש נקודות כדי למזג: (1) נקודת המוצא המשותפת (ה-commit האחרון ששני הענפים חלקו), (2) הקצה של הענף שלי, (3) הקצה של main. מתוך השלוש הוא מחשב את התוצאה הממוזגת. במצגת מופיעה הערה חשובה: "main לא בהכרח ייראה כך אחרי המיזוג" — Git יוצר commit מיזוג מיוחד שמחבר את שני הקווים.

17. Pull Requests

🧭
קונטקסט עד עכשיו מיזגנו ענפים בעצמנו. אבל בפרויקטים אמיתיים, לרוב לא ממזגים ישירות ל-main. במקום זה משתמשים ב-Pull Request (בקיצור PR) — מנגנון שמארגן ומבקר מיזוגים. זהו לב ליבה של שיתוף פעולה ב-GitHub.

למה לא פשוט למזג ישירות? המצגת נותנת שתי סיבות:

  • לפעמים אי אפשר למזג "סתם ככה" — יש ענפים מוגנים (protected branches) שרק אנשים מורשים יכולים למזג אליהם.
  • לפעמים אפשר, אבל לא רוצים — כי אנחנו רוצים שאנשים אחרים יבדקו את הקוד שלנו (code review) לפני שהוא נכנס.
📝
מהו Pull Request על ידי יצירת pull request, אנחנו מארגנים את מיזוגי הקוד. PR הוא בקשה רשמית: "אני רוצה למזג את הענף שלי לתוך main — בואו נבדוק ונאשר ביחד". הוא מאפשר דיון, תגובות, בדיקות אוטומטיות (CI), ואישור — לפני שהקוד נכנס ל-main.

הזרימה של Pull Request (מהמצגת)

  1. Alice עושה commit ו-push לענף פיצ'ר (למשל Foo-feature-branch).
  2. GitHub מציע כפתור "Compare & pull request". Alice יוצרת PR.
  3. ב-PR בוחרים: base: main (לאן ממזגים) ו-compare: Foo-feature-branch (מה ממזגים).
  4. אם אפשר למזג אוטומטית — GitHub מציג "Able to merge" בירוק. אם לא — "Can't automatically merge".
  5. אחרים מגיבים, מבקשים שינויים, ובסוף לוחצים "Merge pull request".

18. קונפליקטים (Merge Conflicts) ופתרונם

🧭
קונטקסט — מה כל מתחיל מפחד ממנו קונפליקט קורה כשמנסים למזג, ושני אנשים שינו את אותה שורה באותו קובץ בצורה שונה. Git לא יכול להחליט לבד איזו גרסה נכונה — אז הוא עוצר ומבקש מאיתנו להכריע. זה נשמע מפחיד, אבל זה לגמרי שגרתי ופתיר. נבין בדיוק איך זה נראה ואיך פותרים.

איך נוצר קונפליקט

דוגמה מהמצגת: שני מפתחים שינו את hello3.txt מקומית. המפתח השמאלי מיזג את שינוייו ל-main (שם כתוב עכשיו FooBar), בעוד המפתח הימני, על הענף שלו, כתב Foo / Bar / Foobar. כשהמפתח הימני עושה git pull מ-main — Git מגלה שאותה שורה שונתה בשני מקומות בצורה שונה ⟵ קונפליקט.

⚠️
ההודעה שתראו "There are merge conflicts. Resolve them before committing." — יש קונפליקטים שצריך לפתור לפני שאפשר לבצע commit.

איך נראה קובץ עם קונפליקט

Git מסמן את אזור הקונפליקט בקובץ עצמו, עם "סמני קונפליקט" מיוחדים:

hello3.txt · קובץ עם קונפליקט
<<<<<<< HEAD
Foo
Bar
Foobar
=======
FooBar
>>>>>>> 76ffa775f1bc2399a6f688864844fc1a
🔤
קריאת סמני הקונפליקט <<<<<<< HEAD — מכאן מתחילה הגרסה שלי (הענף הנוכחי).
======= — קו ההפרדה בין שתי הגרסאות.
>>>>>>> <hash> — עד כאן הגרסה השנייה (זו שמושכים/ממזגים, מזוהה לפי ה-hash).
לפתור = למחוק את הסמנים ולהשאיר את הטקסט הסופי הנכון (אולי לשלב את שתי הגרסאות).

פתרון הקונפליקט

ב-VS Code יש עורך מיזוג נוח: בצד שמאל "Theirs" (origin/main), בצד ימין "Yours" (הענף שלי), ולמטה "Result" — התוצאה הסופית. בוחרים שורות מכל צד או כותבים ידנית (למשל FooBar + Foobar2), עד שכתוב "0 Conflicts Remaining". אז לוחצים Mark as resolved / Commit merge.

✅
הזרימה השלמה של PR עם קונפליקט (מהמצגת) 1. נוצר PR אבל "main branch changed ⟹ conflicts exist".
2. לוחצים Resolve conflicts ופותרים.
3. אחרי הפתרון: "This branch has no conflicts with the base branch" — אפשר למזג.
4. ב-PR אפשר גם להמשיך להגיב ולבקש שינויים (למשל "No reason for two Foobars, please choose one") — וניתן להוסיף עוד commits לענף; ה-PR מתעדכן אוטומטית.
5. בסוף — Merge pull request, וההיסטוריה של main כוללת את כל ה-commits ואת commit המיזוג.

19. Git Internals — איך Git שומר הכל

🧭
קונטקסט — מציצים מתחת למכסה המנוע עד עכשיו השתמשנו ב-Git כ"קופסה שחורה". בחלק הזה נראה איך הוא באמת עובד בפנים — מה נמצא בתיקיית .git וכיצד Git מאחסן קבצים, גרסאות ו-commits. זה הופך את כל הקסם להגיוני, ועוזר להבין למה דברים מתנהגים כמו שהם מתנהגים.

הצצה לתיקיית .git

ניצור תיקייה, קובץ foo.txt, ונריץ git init. נפתח את התיקייה הנסתרת .git ונראה את המבנה הפנימי:

.git/ · מבנה התיקייה
.git/
├── hooks/         # סקריפטים שרצים אוטומטית באירועים (לפני commit וכו')
├── info/
├── objects/       # ⭐ כאן נשמרים כל האובייקטים: blobs, trees, commits
│   ├── info/
│   └── pack/
├── refs/          # מצביעים
│   ├── heads/     # ⭐ הענפים המקומיים (main, mybranch...)
│   └── tags/
├── config         # הגדרות המאגר
├── description
├── FETCH_HEAD
└── HEAD           # ⭐ מצביע לענף הפעיל הנוכחי
💡
VS Code מסתיר את .git כברירת מחדל VS Code מסתיר את תיקיית .git (דרך ההגדרה files.exclude עם התבנית **/.git). כדי לראות אותה במצגת, שינו את ההגדרה ל-"**/.git": false.

שלושת סוגי האובייקטים של Git

Git שומר הכל כ"אובייקטים" בתוך objects/, ולכל אובייקט יש מזהה ייחודי — hash מסוג SHA-1 (40 תווים הקסדצימליים). יש שלושה סוגים עיקריים:

אובייקטמה הוא שומר
blob (Binary Large Object)את התוכן של קובץ (רק התוכן — לא את שם הקובץ!).
tree (עץ)"תיקייה" — רשימת שמות קבצים ותתי-תיקיות, וה-hash של ה-blob/tree של כל אחד.
commitהצבעה ל-tree (תמונת מצב של כל הפרויקט) + מי המחבר + מתי + הודעה + ה-parent.

הדגמה: blob

נכתוב bar לתוך foo.txt ונריץ git add foo.txt. מיד נוצר אובייקט חדש ב-objects/, בתיקייה ששמה שני התווים הראשונים של ה-hash (ba) וקובץ ששמו שאר ה-hash:

bash · בדיקת האובייקט
# -p מדפיס את התוכן (print), -t מדפיס את הסוג (type)
$ git cat-file -p ba0e16
bar

$ git cat-file -t ba0e16
blob
🔤
סינטקס git cat-file = פקודת "הצצה" לאובייקט לפי ה-hash שלו (מספיק להעתיק כמה תווים ראשונים). -p = print (תוכן), -t = type (סוג). שימו לב: התוכן הגולמי של האובייקט בדיסק דחוס (אם תפתחו אותו ישירות תראו ג'יבריש) — cat-file הוא מה שמפענח אותו.

הדגמה: commit יוצר tree + commit

נריץ git commit -m "First commit". עכשיו נוצרו שני אובייקטים חדשים — אובייקט tree ואובייקט commit:

bash · חקירת ה-commit
# האובייקט tree — מקשר שם קובץ ל-blob שלו
$ git cat-file -p 899c3e
100644 blob ba0e162e1c47469e3fe4b393a8bf8c569f302116    foo.txt

$ git cat-file -t 899c3e
tree

# האובייקט commit — מצביע ל-tree, ומכיל מטא-דאטה
$ git cat-file -p 03d1b9
tree 899c3ea59c69ba62f93d1c01b9c8d0a11f12beea
author a <a> 1727467926 +0300
committer a <a> 1727467926 +0300

First commit

$ git cat-file -t 03d1b9
commit

אז ה-commit הראשון יוצר את שרשרת ההפניות הבאה:

  ┌─────────────┐
  │ First commit│  03d1b9942500d28fcd5d3f4ad0be18f926a66fe0
  └──────┬──────┘
         ▼
  Tree: 899c3ea59c69ba62f93d1c01b9c8d0a11f12beea
         ▼
  Blob: ba0e162e1c47469e3fe4b393a8bf8c569f302116   ("bar")

commit שני — תיקיות מקוננות ו-parent

נוסיף תיקייה mydir עם קבצים, ונעשה commit שני. עכשיו נראה משהו יפה: ל-tree של ה-commit השני יש tree מקונן (עבור mydir), ול-commit השני יש שדה parent שמצביע על ה-commit הראשון:

bash · commit שני
$ git cat-file -p 20babc      # ה-commit השני
tree 1fc0ad423e328231d2bb8b1f25a8920bbfd38d53
parent 03d1b9942500d28fcd5d3f4ad0be18f926a66fe0   <-- מצביע ל-commit הראשון!
author a <a> 1727469188 +0300
committer a <a> 1727469188 +0300

Second commit

$ git cat-file -p 1fc0ad      # ה-tree הראשי
100644 blob ba0e162e...    foo.txt
040000 tree e53225397...   mydir          <-- tree מקונן (תיקייה בתוך תיקייה)
🔗
התובנה הגדולה: Git הוא גרף של אובייקטים כל commit מצביע ל-parent שלו (ה-commit הקודם) — וכך נבנית שרשרת ההיסטוריה. כל commit מצביע גם ל-tree (תמונת מצב מלאה של הפרויקט), שמצביע ל-blobs (תוכן הקבצים) ול-trees מקוננים (תיקיות). הכל מקושר דרך hashes: 
  First commit  ◀────parent──── Second commit
   03d1b9...                      20babc...
      │                              │
      ▼                              ▼
  Tree 899c3e               Tree 1fc0ad ──┐
      │                       │           │
      ▼                       ▼           ▼
  Blob ba0e16 ◀───────────────┘     Tree e53225
   ("bar")                              │   │
                                        ▼   ▼
                              Blob ba0e16   Blob 43837c
🧠
למה ה-hash מבוסס תוכן זה גאוני (משלי) ה-hash של אובייקט מחושב מהתוכן שלו. לכן: (1) שני קבצים זהים → אותו blob (Git לא משכפל תוכן זהה — חוסך מקום!). (2) אם תוכן משתנה ולו בתו אחד → ה-hash משתנה לגמרי → לא ניתן "לזייף" היסטוריה בלי שיבחינו. זה גם מה שמבטיח את שלמות (integrity) ההיסטוריה כולה.

refs ו-HEAD — איך Git יודע איפה כל ענף

אחרי commit, נוצר הקובץ .git/refs/heads/main. מה יש בו? פשוט את ה-hash של ה-commit האחרון בענף:

.git/refs/heads/main
e6c93d1d926e927baf63fb662d6454743e59e6d5

כשיוצרים ענף חדש (git checkout -b mybranch), נוצר קובץ נוסף .git/refs/heads/mybranch — עם אותו hash (כי הענף מתחיל מאותה נקודה). וקובץ HEAD מצביע על הענף הפעיל:

.git/HEAD
ref: refs/heads/mybranch
📌
עכשיו זה מתחבר! זוכרים ש"ענף הוא רק מצביע" ו"HEAD מצביע על הענף הפעיל"? עכשיו אנחנו רואים את זה ממש בקבצים: ענף = קובץ קטן ב-refs/heads/ שמכיל hash של commit. HEAD = קובץ שמכיל ref: לענף הפעיל. כשעושים commit חדש, Git פשוט מעדכן את ה-hash בקובץ הענף. כשעוברים ענף, Git משנה את התוכן של HEAD. הכל קבצי טקסט קטנים — אין כאן שום קסם.

נראה את זה: עושים commit על mybranch. הקובץ refs/heads/mybranch מתעדכן ל-hash החדש (040785b...), בעוד refs/heads/main נשאר עם ה-hash הישן (e6c93d1...). שני הענפים מצביעים עכשיו על commits שונים — בדיוק התמונה התיאורטית שראינו בפרק הענפים, אבל הפעם בקבצים אמיתיים.

20. נספח: דף פקודות מהיר (Cheat-sheet)

📋
תוספת שלי לסיכום ריכוז של כל הפקודות מההרצאה במקום אחד, לשליפה מהירה.
bash · Git cheat-sheet
# -- התחלה -------------------------------
git init                       # מאגר חדש מקומי
git clone <url>                 # שכפול מאגר קיים מהשרת

# -- מצב נוכחי ---------------------------
git status                     # אילו קבצים השתנו / staged
git log                        # היסטוריית commits
git log --oneline --graph      # היסטוריה קומפקטית עם גרף ענפים

# -- מחזור עבודה בסיסי -------------------
git add <file>                  # קובץ ספציפי ל-staging
git add .                      # כל השינויים ל-staging
git commit -m "message"        # קיבוע גרסה מקומית
git commit -a -m "message"     # add (ל-tracked) + commit ביחד
git push                       # העלאה לשרת
git pull                       # משיכת עדכונים מהשרת

# -- ענפים -------------------------------
git branch                     # רשימת ענפים
git checkout -b <name>          # יצירת ענף + מעבר אליו
git switch <name>               # מעבר לענף קיים (מודרני)
git merge <name>                # מיזוג ענף לתוך הנוכחי
git branch -d <name>            # מחיקת ענף שמוזג
git push -u origin <name>       # push ראשון של ענף חדש

# -- הצצה לפנים (internals) --------------
git cat-file -p <hash>          # הדפסת תוכן אובייקט
git cat-file -t <hash>          # הדפסת סוג אובייקט (blob/tree/commit)
🎯
סיכום ההרצאה במשפט Git שומר גרסאות של הקוד כשרשרת commits (כל אחד הוא תמונת מצב + מצביע ל-parent). אנחנו מקדמים קבצים דרך האזורים working → staging → local → remote בעזרת add → commit → push, מסנכרנים עם אחרים דרך pull/push, ומפתחים במקביל בעזרת ענפים שממוזגים חזרה (fast-forward או three-way), לרוב דרך Pull Requests — וכל זה בנוי על גרף פשוט של אובייקטים (blob/tree/commit) המזוהים ב-hash.
→ עמוד הבית כל ההרצאות והתרגולים תרגול 1 ← המשך לתרגול
