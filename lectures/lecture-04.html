הרצאה 4

ריבוי תהליכים וריבוי תהליכונים

איך גורמים לתוכנה לעשות כמה דברים במקביל? נכיר את שתי הדרכים: תהליכים (processes) שכל אחד עם זיכרון נפרד (fork, wait, exec), ותהליכונים (threads) שחולקים זיכרון ולכן קלים ומהירים יותר. ומכאן נגיע לבעיה הגדולה של מקביליות — race conditions — ולכלים שפותרים אותה: mutex, double-checked locking, volatile, ולבסוף הפשטות גבוהות יותר ב-Java (Thread, Runnable, Executor).

🗺️
קו המחשבה של ההרצאה עד עכשיו תוכנית הרצה שורה אחרי שורה (סדרתית). אבל מחשב מודרני יכול לעשות הרבה דברים בו-זמנית. ההרצאה בונה את זה בשלבים: קודם תהליך (היחידה הבסיסית של ריצה), אחר כך איך יוצרים עוד תהליכים (fork), אז למה תהליכונים עדיפים לעיתים (שיתוף זיכרון) — ומיד מגלים שהשיתוף הזה הוא גם הסכנה הגדולה (race conditions), שמולה נלמד מנעולים וסנכרון.

חלק א' · תהליכים

1. מהו תהליך (Process)

🧭
קונטקסט — אנלוגיה שכבר מכירים ההרצאה פותחת בהקבלה חכמה לעולם ה-OOP: בדיוק כמו שאובייקט הוא מופע (instance) של מחלקה, כך תהליך הוא מופע של תוכנית. זה נותן אינטואיציה מיידית: אפשר ליצור הרבה מופעים מאותה מחלקה — ובדיוק כך אפשר להריץ הרבה תהליכים מאותה תוכנית.
  • אובייקט הוא instance של מחלקה — ואפשר ליצור הרבה מופעים ממחלקה אחת.
  • תהליך הוא instance של תוכנית, שמורץ ע"י מערכת ההפעלה.
  • אפשר ליצור הרבה תהליכים של אותה תוכנית — או של תוכניות שונות.
  • כל תהליך מורץ בנפרד (separately) — יש לו זיכרון משלו.
  • כל תהליך מורץ סדרתית (sequentially) — פקודה אחרי פקודה.
🌐
דוגמה משלי כשאתה פותח שלושה חלונות של אותו דפדפן — ייתכן שירוצו שלושה תהליכים של אותה תוכנית (כרום אפילו מריץ תהליך נפרד לכל טאב, מסיבות אבטחה ויציבות). כל אחד עם הזיכרון שלו, כך שקריסה של טאב אחד לא מפילה את האחרים. זה היתרון הגדול של הפרדת זיכרון — וגם, כפי שנראה, מקור הקושי בתקשורת ביניהם.

2. מצבי תהליך (Process States)

🧭
קונטקסט תהליך לא תמיד "רץ". יש למחשב מעבד אחד (או כמה ליבות) שצריך להתחלק בין הרבה תהליכים, ולפעמים תהליך ממתין למשהו (קלט מהדיסק, מהרשת...). לכן לכל תהליך יש מצב (state), ומערכת ההפעלה מעבירה אותו בין המצבים. 
                    ┌─────────┐
                    │   New   │   נוצר זה עתה
                    └────┬────┘
                         ▼
        ┌──────────▶ ┌─────────┐ ◀──────────┐
        │            │  Ready  │            │
        │            └────┬────┘            │
   (האירוע קרה,            ▼          (נגמר הזמן /
    חזרה לתור)     ┌─────────────┐     הופסק זמנית)
        │          │   Running   │ ────────┘
        │          └──┬───────┬──┘
        │             ▼       ▼
   ┌─────────┐   ┌──────────────┐
   │ Waiting │◀──│  (ממתין ל-I/O)│
   └─────────┘   └──────────────┘
                         ▼
                  ┌────────────┐
                  │ Terminated │   הסתיים
                  └────────────┘
מצבמשמעות
Newהתהליך נוצר אבל עוד לא מוכן לרוץ.
Readyמוכן לרוץ וממתין ש-CPU יתפנה (בתור).
Runningכרגע מורץ בפועל על המעבד.
Waitingממתין לאירוע (למשל סיום קריאת קובץ). לא מבזבז CPU בינתיים.
Terminatedסיים את הריצה.
💡
המעברים החשובים שים לב ל"משולש" שבמרכז: מ-Running אפשר לחזור ל-Ready (כשנגמר חלון הזמן של התהליך — preemption), או לעבור ל-Waiting (כשצריך לחכות ל-I/O). מ-Waiting חוזרים ל-Ready (לא ישר ל-Running!) כי עדיין צריך לחכות לתור. רק Ready→Running מתרחש כשה-scheduler בוחר את התהליך.

3. fork — יצירת תהליך חדש

🧭
קונטקסט איך תהליך יוצר תהליך נוסף? בעולם Unix/Linux עושים זאת עם קריאת המערכת fork. הרעיון מפתיע בהתחלה: fork משכפל את התהליך הקיים — נוצר עותק כמעט מדויק, ושני התהליכים ממשיכים מאותה נקודה.
cpp · fork
pid_t pid = fork();
  • נוצר יחס הורה/ילד (parent/child).
  • התהליך שקרא ל-fork הוא ההורה.
  • הילד משכפל את מרחב הזיכרון של ההורה.
  • ואז שניהם ממשיכים לרוץ מאותה שורת קוד — אבל בנפרד.
🔤
הטריק החשוב — מה fork מחזיר fork נקרא פעם אחת אבל "חוזר פעמיים" — פעם בכל תהליך, וכל אחד מקבל ערך אחר. כך התהליכים מבדילים מי הם:
ערך pidבאיזה תהליךמשמעות
pid > 0ההורההערך הוא ה-process id של הילד שזה עתה נוצר.
pid == 0הילדהילד תמיד מקבל 0 (כך הוא יודע שהוא הילד).
pid < 0—כישלון (לא נוצר תהליך).
🍴
למה "fork" (מזלג)? — אינטואיציה משלי השם מתאר התפצלות של נתיב הריצה לשניים, כמו מזלג בכביש. עד ה-fork יש נתיב אחד; אחריו — שני נתיבים שרצים במקביל, כל אחד עם העתק משלו של הזיכרון. בזכות ערך-ההחזרה השונה, אותו קוד בדיוק "יודע" אם הוא ההורה או הילד, ומריץ ענף שונה ב-if.

4. wait ו-exec — מה עושים אחרי fork

🧭
קונטקסט אחרי שיצרנו ילד, יש שתי שאלות: (1) האם ההורה צריך לחכות שהילד יסיים? (2) האם הילד צריך להמשיך להריץ את אותו קוד, או להחליף את עצמו בתוכנית אחרת לגמרי? לשתי השאלות יש קריאות מערכת ייעודיות.

wait — ההורה ממתין שהילד יסיים:

cpp · wait
pid = wait(&status);
  • מרגע הקריאה — ריצת ההורה נעצרת עד שהילד מסתיים.
  • לחלופין, ההורה יכול להמשיך לרוץ בלי לחכות (פשוט לא קוראים ל-wait).

exec — תהליך מחליף את הזיכרון שלו בתוכנית אחרת:

  • תהליך יכול להחליף את הזיכרון הקיים שלו ע"י טעינת תוכנית אחרת לתוכו, באמצעות קריאת המערכת exec (ובמשפחה שלה, כמו execlp).
  • השילוב הקלאסי הוא fork + exec: יוצרים ילד, ואז הילד מחליף את עצמו בתוכנית חדשה — כך מריצים תוכנית אחרת מתוך התוכנית שלנו.
🐚
איך ה-shell עובד — דוגמה/הקשר משלי זה בדיוק מה שעושה ה-terminal! כשמקלידים פקודה, ה-shell עושה fork (יוצר ילד), הילד עושה exec לתוכנית שביקשת (למשל ls), וההורה (ה-shell) עושה wait עד שהיא מסתיימת — ואז מציג שוב את ה-prompt. בדיוק לכן בדוגמה שנראה מיד, הילד מריץ execlp("/bin/sh", ...) — הוא הופך ל-shell.

5. דוגמה מלאה — fork, getpid, exec, wait

הקוד הבא ממחיש את הכל יחד. שים לב ל-getpid() (ה-id של התהליך הנוכחי) ו-getppid() (ה-id של ההורה):

cpp · a.cpp
#include <iostream>
#include <unistd.h>

using namespace std;

int main() {
    cout << getpid() << " " << getppid() << endl;
    int pid = fork();
    if (pid == 0) {
        cout << "Child: " << getpid() << " " << getppid() << endl;
        execlp("/bin/sh", "/bin/sh", NULL);
    }
    else if (pid > 0) {
        cout << "Parent: " << pid << " " << getpid() << " " << getppid() << endl;
        wait(NULL);
        cout << "Child is done, I'm done" << endl;
    }
}

מה קורה כאן, שורה-שורה:

קודהסבר
getpid() / getppid()מחזירים את ה-id של התהליך הנוכחי / של ההורה שלו.
int pid = fork();מכאן והלאה רצים שני תהליכים במקביל מאותה שורה.
if (pid == 0)הענף שרק הילד נכנס אליו. הילד מדפיס את זהותו, ואז execlp מחליף אותו ב-shell.
else if (pid > 0)הענף שרק ההורה נכנס אליו. שים לב: ב-pid יש כאן את ה-id של הילד.
wait(NULL);ההורה ממתין שהילד (ה-shell) יסתיים. רק אז ידפיס "Child is done".

פלט לדוגמה (מהמצגת — שים לב לקשרי ההורה/ילד דרך ה-id-ים):

console · ריצה
$ g++ a.cpp
$ echo $$
4853                         ← ה-PID של ה-shell הנוכחי
$ ./a.out
5271 4853                    ← התהליך שלנו (5271), ההורה שלו = ה-shell (4853)
Parent: 5272 5271 4853       ← ההורה: id-של-הילד=5272, אני=5271, ההורה-שלי=4853
Child: 5272 5271             ← הילד: אני=5272, ההורה-שלי=5271
$ echo $$
5272                         ← עכשיו אנחנו בתוך ה-shell החדש (הילד)! ה-PID שלו 5272
$ exit
exit
Child is done, I'm done      ← רק אחרי exit מה-shell — wait() משחרר את ההורה
🔍
מה הפלט מלמד אותנו ה-id של הילד (5272) שמודפס ע"י ההורה (Parent: 5272...) זהה ל-id שהילד מדפיס על עצמו (Child: 5272...) — זה ה-pid שהוחזר מ-fork. וה-shell החדש (אחרי ה-exec) הוא אותו תהליך 5272. כל עוד לא הקלדנו exit, ה-wait של ההורה חוסם אותו — ולכן "Child is done" מופיע אחרון.

6. Multiprocessing — יתרונות וחסרונות

🧭
קונטקסט עכשיו שיודעים ליצור תהליכים, אפשר לפרק משימה גדולה להרבה תהליכים ייעודיים. אבל לגישה הזו יש מחיר — וזה בדיוק מה שיוביל אותנו אל ה-threads בחלק הבא.
  • אפשר לפרק משימות גדולות לתהליכים קטנים וייעודיים.
  • אבל לכל תהליך יש מרחב זיכרון משלו — וזו בעיה:
  • איך שני תהליכים יתקשרו? (אם הזיכרון נפרד, אי-אפשר פשוט לקרוא משתנה משותף — צריך מנגנון IPC כמו pipes, sockets, shared memory).
  • Context switch יקר — מעבר של ה-CPU מתהליך לתהליך דורש שמירה וטעינה של הרבה מצב (כולל מרחב הזיכרון), וזה לוקח זמן.
⚖️
המתח שמוביל ל-threads הפרדת הזיכרון נותנת בידוד ובטיחות (תהליך אחד לא יכול להשחית את הזיכרון של השני), אבל גובה מחיר בתקשורת וביצועים. מה אם נרצה את היתרונות של מקביליות בלי המחיר הכבד הזה? כאן נכנסים התהליכונים.

7. מהו thread ולמה

🧭
קונטקסט תהליכון (thread) הוא "חוט ריצה" בתוך תהליך. בניגוד לתהליכים נפרדים, threads של אותו תהליך חולקים זיכרון — וזה הופך אותם לקלים, מהירים, ונוחים לתקשורת. זהו הפתרון למחיר ש-multiprocessing גבה.
  • לכל תהליך יש thread אחד לפחות — ה-thread הראשי (main), שנוצר אוטומטית.
  • תהליך יכול להחזיק כמה threads במקביל (concurrently).
  • לכל thread יש בנפרד: registers, stack, ו-program counter (ה"מצב הפרטי" של הריצה).
  • כל ה-threads של התהליך חולקים: ה-heap, מקטעי ה-code/data, וקבצים פתוחים.
           ┌───────────────── Process ─────────────────┐
           │   משותף לכל ה-threads:                      │
           │   ┌──────────┐  ┌──────────┐  ┌──────────┐  │
           │   │   Heap   │  │Code/Data │  │  Files   │  │
           │   └──────────┘  └──────────┘  └──────────┘  │
           │                                             │
           │   Thread 1        Thread 2        Thread 3  │
           │   ┌────────┐      ┌────────┐      ┌────────┐│
           │   │ stack  │      │ stack  │      │ stack  ││  ← פרטי
           │   │ regs   │      │ regs   │      │ regs   ││  ← לכל
           │   │  PC    │      │  PC    │      │  PC    ││  ← thread
           │   └────────┘      └────────┘      └────────┘│
           └─────────────────────────────────────────────┘

למה threads? (Why?)

  • כיוון שהם חולקים חלק מזיכרון התהליך — הם קלים יותר ומהירים ליצירה מתהליכים.
  • Context switch קל יותר ביניהם, ויש למעבד שיפורים מובנים לכך.
  • קוד שרץ ב-threads שונים של אותו תהליך יכול להחליף מידע בקלות (פשוט דרך זיכרון משותף — אין צורך ב-IPC מורכב).
⚠️
החיסרון שמסתתר ביתרון הזיכרון המשותף הוא בדיוק מה שעושה threads נוחים — אבל גם מסוכנים. כששני threads ניגשים לאותו נתון בו-זמנית, נוצרת אפשרות ל-race condition (פרק 10). כל מה שנלמד בהמשך — mutex, locks — נועד לרסן בדיוק את הסכנה הזו.

8. pthread — threads ב-C

🧭
קונטקסט ב-C/C++ על Linux, ה-API הקלאסי ל-threads הוא POSIX threads (pthread). שתי הקריאות המרכזיות: pthread_create ליצירת thread ש"מריץ פונקציה", ו-pthread_join להמתנה שה-thread יסיים.
  • pthread_create — מריץ פונקציה בשם func עם ארגומנטים args ב-thread נפרד.
  • pthread_join — thread יכול לחכות ל-thread אחר שיסיים. ריצת ה-thread הנוכחי נעצרת עד שה-thread שמחכים לו מסתיים (בדיוק כמו wait לתהליכים).

דוגמה מעשית — שרת UDP מרובה-threads: הלולאה הראשית מקבלת הודעות, ולכל לקוח פותחת thread נפרד שמטפל בו (כדי שלקוח אטי לא יחסום את כל השאר):

cpp · הלולאה הראשית — thread לכל לקוח
while (true) {
    struct sockaddr_in from;
    unsigned int from_len = sizeof(struct sockaddr_in);
    char buffer[4096];
    int bytes = recvfrom(sock, buffer, sizeof(buffer), 0,
                         (struct sockaddr *)&from, &from_len);
    if (bytes < 0) {
        perror("error reading from socket");
    }

    char* p = new char[bytes];
    strncpy(p, buffer, bytes);

    Data* data = new Data();
    strncpy(data->buffer, buffer, bytes);
    data->buffer_size = bytes;
    data->from = from;

    pthread_t tid;
    pthread_attr_t attr;
    pthread_attr_init(&attr);
    pthread_create(&tid, &attr, handle_client, data);
}
cpp · מבנה Data + פונקציית ה-thread
int sock;
time_t timev;

class Data {
public:   // to make code shorter
    char buffer[4096];
    int buffer_size;
    struct sockaddr_in from;
};

void *handle_client(void* param) {
    Data* data = (Data*)param;

    time(&timev);
    cout << "Lech lishon: " << timev << endl;
    sleep(5);
    time(&timev);
    cout << "Gooooood morning: " << timev << endl;

    cout << "The client sent: " << data->buffer << endl;
    int sent_bytes = sendto(sock, data->buffer, data->buffer_size, 0,
                            (struct sockaddr *)&data->from, sizeof(data->from));
    if (sent_bytes < 0) {
        perror("error writing to socket");
    }
}
🔤
הסבר סינטקס — נקודות מפתח
  • pthread_create(&tid, &attr, handle_client, data) — יוצר thread שמריץ את handle_client, ומעביר לו את data כפרמטר. tid יתמלא במזהה ה-thread.
  • פונקציית ה-thread חייבת להיות בחתימה void* func(void* param) — מקבלת ומחזירה מצביע גנרי. בתוכה עושים cast: Data* data = (Data*)param;.
  • למה מקצים Data חדש ב-new לכל לקוח? כדי שלכל thread יהיה עותק משלו של הנתונים. אם כל ה-threads היו קוראים מאותו buffer מקומי שמשתנה בלולאה — הם היו דורסים זה את נתוני זה (race!).
  • sleep(5) כאן ממחיש שהטיפול בלקוח עשוי לקחת זמן — ובזכות ה-thread, השרת ממשיך לקבל לקוחות נוספים בינתיים.

9. std::thread ואי-דטרמיניזם

🧭
קונטקסט C++ המודרני נותן ממשק נקי יותר מ-pthread: המחלקה std::thread. הדוגמה הבאה גם מלמדת לקח קריטי — סדר הריצה של threads אינו צפוי (non-deterministic).
cpp · 20 threads עם std::thread
#include <iostream>
#include <thread>
#include <unistd.h>
using namespace std;

void foo(int a) {
    sleep(5);
    cout << a << endl;
}

int main() {
    thread threads[20];
    for (int i = 0; i < 20; i++) {
        threads[i] = thread(foo, i);
    }
    for (int i = 0; i < 20; i++) {
        threads[i].join();
    }
}
קודהסבר
thread(foo, i)יוצר thread שמריץ foo(i). כאן יוצרים 20 כאלה.
threads[i].join()הלולאה השנייה ממתינה לכל ה-threads שיסיימו — אחרת main היה מסתיים לפניהם.

הפלט — שלוש הרצאות, שלוש תוצאות שונות:

console · שלושה ניסיונות
ניסיון 1:   1 0 2 4 6 7 8 9 10 11 12 13 15 17 18 3 5 14 16 19
ניסיון 2:   0 3 6 5 9 11 13 14 12 15 16 17 18 19 1 10 4 2 8 7
ניסיון 3:   5 17 18 1 7 8 3 9 10 11 12 13 14 15 16 19 6 4 2 0
🎲
הלקח הקריטי — אין שום הבטחה על הסדר כל 20 ה-threads "ישנים" 5 שניות ואז מדפיסים. הם מתעוררים בערך באותו זמן ומתחרים על ה-CPU וה-cout — ומערכת ההפעלה מחליטה מי רץ מתי. לכן הסדר משתנה מהרצה להרצה. אסור להניח סדר ריצה בין threads. אם הקוד שלך תלוי בסדר מסוים — יש לך באג שלפעמים "עובד במזל". זה בדיוק מוביל אותנו ל-race conditions.

10. Race Condition וקטע קריטי (Critical Section)

🧭
קונטקסט — הבעיה המרכזית של מקביליות זה הלב של ההרצאה. כשהזיכרון משותף וסדר הריצה לא צפוי — מה קורה כששני threads נוגעים באותו נתון בו-זמנית? התשובה: באגים מסוכנים וקשים לשחזור.
📖
הגדרות Race condition = מצב שבו שני threads מתחרים על גישה למשאב משותף (מבנה נתונים או התקן) שאסור שייגש אליו יותר מ-thread אחד בו-זמנית. Critical section (קטע קוד קריטי) = קטע קוד שמכיל race condition — קטע שצריך להגן עליו כך שרק thread אחד יריץ אותו בכל רגע.

הדוגמה הקלאסית — שני threads (אחד עם thread_id=1, השני עם thread_id=2) מריצים את אותו קוד, כש-x מאותחל ל-0:

cpp · קטע קריטי
if (x < 1) {
    x++;
    cout << thread_id << " was first\n";
}
else
    cout << thread_id << " was second\n";

הכוונה: רק thread אחד יהיה "first" והשני "second". הבעיה: בסביבה מרובת-threads ההתנהגות לא-דטרמיניסטית.

הריצה ה"טובה" (כמו שציפינו):

  1. Thread A בודק if (x<1) → אמת → מבצע x++ (עכשיו x=1).
  2. Thread B בודק if (x<1) → שקר (x=1 כבר) → מריץ את ה-else. ✓ אחד first, אחד second.

הריצה ה"רעה" (race condition):

  1. Thread A בודק if (x<1) → אמת.
  2. לפני ש-A הספיק לבצע x++ — מתחלף ל-Thread B.
  3. Thread B בודק if (x<1) → גם אמת (x עדיין 0!).
  4. שניהם קיבלו אמת → שניהם יהיו "first". ✗ שגיאה!
⏱️
למה זה כל-כך ערמומי — הסבר משלי השורה if (x<1) x++; נראית "אטומית" אבל היא לא — היא מתורגמת לכמה פקודות מכונה (קרא את x, השווה, הגדל, כתוב חזרה). thread switch יכול לקרות באמצע. וכיוון שזה תלוי בתזמון מדויק, הבאג עלול להופיע פעם ב-10,000 הרצות — ולעולם לא במחשב המפתח. זו אחת מסיבות הכאב הגדולות בתכנות מקבילי.

11. בעיית ה-Singleton וה-Lazy Initialization

🧭
קונטקסט עכשiו נראה את ה-race condition בדוגמה אמיתית ומפורסמת: תבנית ה-Singleton (מחלקה עם מופע יחיד). הדרך שבה ההרצאה מפתחת את הפתרון — צעד אחרי צעד, כשכל "תיקון" חושף בעיה חדשה — היא שיעור מצוין בחשיבה מקבילית.
cpp · Car.h
class Car {
private:
    bool inUse;
    Car();               // prevent the class instantiation
    static Car* instance;
public:
    static Car* getInstance();
    bool Occupy();
};
cpp · Car.cpp
#include "Car.h"

Car* Car::instance = NULL;

Car::Car() {
    inUse = false;
}

Car* Car::getInstance() {
    if (!instance)
        Car::instance = new Car;
    return instance;
}

bool Car::Occupy() {
    if (!inUse) {
        inUse = true;
        return true;
    }
    return false;
}

הקונסטרקטור פרטי כדי שאי-אפשר יהיה ליצור Car מבחוץ — רק דרך getInstance, שמבטיח מופע יחיד. אבל getInstance מכיל race condition: שני threads יכולים לראות !instance כאמת בו-זמנית ולייצר שני אובייקטים.

פתרון 1 — Eager initialization:

cpp · eager
static Car* instance = new Car;   // נוצר מראש, לפני כל גישה

פותר את ה-race (נוצר פעם אחת בטעינה), אבל מבזבז: נוצר גם אם לעולם לא נשתמש בו.

פתרון 2 — נעילה (lock):

cpp · lock סביב הכל
// lock
if (!instance)
    instance = new Car;
// release lock

החיסרון? אנחנו רוצים lazy initialization (לא ליצור אם לא נשתמש), והרי יוצרים פעם אחת בלבד — אז למה לנעול בכל קריאה? נעילה היא יקרה, וכאן היא מתבצעת בכל גישה ל-getInstance.

פתרון 3 — בדיקה לפני נעילה:

cpp · נעילה רק אם צריך
if (!instance) {
    // lock
    instance = new Car;
    // release lock
}

נעילה רק כשעדיין אין מופע. אבל חזר ה-race: שני threads יכולים לעבור את ה-if החיצוני, ואז בזה אחר זה ליצור שני אובייקטים.

פתרון 4 — Double-Checked Locking:

cpp · בדיקה כפולה
if (!instance) {
    // lock
    if (!instance)
        instance = new Car;
    // release lock
}

בודקים פעמיים — פעם בלי נעילה (מהיר, למקרה הרגיל), ופעם בתוך הנעילה (לבטיחות). הגענו? עדיין לא!

🧨
הבעיה הסמויה — הקומפיילר עלול לשנות סדר התרחיש: (1) Thread 1 נועל ומתחיל ליצור את המופע. (2) הקומפיילר מחליט קודם להקצות זיכרון ולהצביע עליו ב-instance, ורק אחר כך לאתחל את האובייקט. (3) בין שני הצעדים — מתרחש thread switch. (4) Thread 2 רואה ש-instance כבר לא NULL, מדלג על ה-if, ומשתמש באובייקט שעדיין לא אותחל! 💥

פתרון 5 — volatile + דגל בוליאני:

cpp · עם booleanVar
if (!booleanVar) {
    // lock
    if (!booleanVar) {
        instance = new Car;
        booleanVar = true;     // נקבע רק אחרי שהאובייקט מוכן
    }
    // release lock
}
  • volatile מצהיר שהמשתנה עשוי לשמש ב-threads שונים, כך שאי-אפשר להסתמך על ערך שמור — וזה מונע מהקומפיילר לשנות את סדר הפעולות כמו קודם.
  • ה-booleanVar נקבע ל-true רק אחרי שהאובייקט נוצר במלואו — כך thread אחר לא יראה "מוכן" לפני הזמן.

12. Mutex — איך נועלים בפועל

🧭
קונטקסט דיברנו על "// lock" ו-"// release lock" כהערות. עכשיו נראה איך מממשים אותן באמת. דרך נפוצה היא mutex (mutual exclusion — הדרה הדדית): מנעול שרק thread אחד יכול להחזיק בכל רגע.
במקום ההערההקוד האמיתי
// lockpthread_mutex_lock(&lock); — נועל באמצעות משתנה ה-lock.
// release lockpthread_mutex_unlock(&lock); — משחרר את הנעילה.

אבל מהו משתנה ה-lock? מצהירים, מאתחלים ומשמידים אותו כך:

cpp · pthread mutex
// הצהרה
pthread_mutex_t lock;

// אתחול (פעם אחת)
if (pthread_mutex_init(&lock, NULL) != 0) {
    // error
}

// שימוש (יכול לקרות הרבה פעמים)
pthread_mutex_lock(&lock);
// ... critical section ...
pthread_mutex_unlock(&lock);

// השמדה (פעם אחת)
pthread_mutex_destroy(&lock);
  • init ו-destroy נקראים פעם אחת בלבד.
  • lock ו-unlock יכולים להיקרא הרבה פעמים.
💡
P.S. — trylock קיימת גם pthread_mutex_trylock: במקום לחכות עד שהמנעול ישתחרר, היא מחזירה שגיאה מיד אם המנעול כבר תפוס. שימושי כשלא רוצים לחסום את ה-thread — למשל "אם המשאב תפוס, לך תעשה משהו אחר ותחזור".

13. Threads ב-Java

🧭
קונטקסט Java נותנת הפשטה גבוהה יותר מ-pthread. יש שתי דרכים עיקריות ליצור thread: לרשת מ-Thread, או לממש את Runnable. שתיהן דורשות לממש את run(), ולהפעיל עם start().
java · המחלקות הבסיסיות
public interface Runnable {
    void run();
}

public class Thread extends Object implements Runnable {
    // ...
    Thread();
    Thread(Runnable target);
    public void run();
    public void start();
    public final void join() throws InterruptedException;
}

דרך 1 — ירושה מ-Thread:

java · extends Thread
public class Foo extends Thread {
    public void run() {
        // ...
    }
    public static void main(String args[]) {
        Foo foo = new Foo();
        foo.start();
    }
}

דרך 2 — מימוש Runnable:

java · implements Runnable
public class Foo implements Runnable {
    public void run() {
        // ...
    }
    public static void main(String args[]) {
        Thread t = new Thread(foo);
        t.start();
    }
}
🔤
start() מול run() — טעות נפוצה קוראים ל-start(), לא ל-run() ישירות! start() יוצר thread חדש ומריץ בו את run(). אם תקרא ל-run() ישירות — היא פשוט תרוץ ב-thread הנוכחי, בלי שום מקביליות.

מתי איזו דרך? מעדיפים בדרך-כלל Runnable — כי ב-Java אפשר לממש כמה interfaces אבל לרשת רק ממחלקה אחת. מימוש Runnable משאיר את אפשרות הירושה פנויה ומפריד את "המשימה" מ"מנגנון ה-thread".

14. Executor ו-Thread Pools

🧭
קונטקסט יצירת thread לכל משימה היא יקרה ולא נשלטת (מה אם יש מיליון משימות?). ה-Executor מפריד בין הגשת משימה לבין הדרך שבה היא תרוץ — ומאפשר להשתמש במאגר threads (thread pool) קבוע במקום ליצור threads בלי הגבלה.
java · interface Executor
public interface Executor {
    void execute(Runnable command);
}

מהתיעוד: אובייקט שמריץ משימות Runnable שהוגשו לו, ומנתק את הגשת המשימה מהמכניקה של איך כל משימה תרוץ (כמה threads, תזמון וכו'). במקום:

java · הדרך הישנה
new Thread(new RunnableTask()).start();

משתמשים ב:

java · עם Executor
Executor executor = anExecutor;
executor.execute(new RunnableTask1());
executor.execute(new RunnableTask2());

דוגמה — שני thread pools בגודל 2, ו-8 משימות:

java · FixedThreadPool
final long start = System.currentTimeMillis();
Executor executor  = Executors.newFixedThreadPool(2);
Executor executor2 = Executors.newFixedThreadPool(2);

Runnable r = () -> {
    try {
        Thread.sleep(1000);
        Thread t = Thread.currentThread();
        System.out.println(t.getId() + t.getName() + " " +
                (System.currentTimeMillis() - start));
    } catch (InterruptedException e) {
        e.printStackTrace();
    }
};

executor.execute(r);  executor2.execute(r);
executor.execute(r);  executor2.execute(r);
executor.execute(r);  executor2.execute(r);
executor.execute(r);  executor2.execute(r);

🏊
מה הפלט מלמד + סוגי pools עם pool בגודל 2, רק 2 משימות רצות בו-זמנית בכל pool; השאר ממתינות בתור. לכן הגל הראשון מסתיים ב-~1000ms והשני ב-~2000ms. שלושת הסוגים העיקריים:
  • newFixedThreadPool(n) — מספר קבוע של n threads.
  • newSingleThreadExecutor() — thread יחיד; המשימות רצות אחת-אחרי-השנייה לפי הסדר.
  • newCachedThreadPool() — יוצר threads לפי הצורך וממחזר פנויים; גדל ומתכווץ דינמית.
זהו, בעצם, יישום של תבנית עיצוב: ה-Executor מסתיר מאיתנו את ניהול ה-threads, בדיוק כפי ש-Runnable מפריד את המשימה מהמנגנון.

★ סיכום ההרצאה

🎯
התמונה המלאה
  • תהליך = מופע של תוכנית, עם זיכרון נפרד, רץ סדרתית; עובר בין מצבים New→Ready→Running→Waiting→Terminated.
  • fork משכפל תהליך; מחזיר >0 להורה (id הילד), 0 לילד, <0 בכישלון. wait ממתין לילד, exec מחליף את הזיכרון בתוכנית אחרת.
  • Multiprocessing: בידוד טוב, אבל תקשורת קשה ו-context switch יקר.
  • Thread = חוט ריצה בתוך תהליך. חולק heap/code/data/files, אבל לכל אחד stack/registers/PC משלו. קל ומהיר, נוח לתקשורת.
  • אבל — סדר ריצה לא-דטרמיניסטי + זיכרון משותף = race conditions ב-קטעים קריטיים.
  • הפתרון: mutex (pthread_mutex_lock/unlock), ובמקרים עדינים גם double-checked locking + volatile (כמו ב-Singleton).
  • Java: Thread/Runnable (תמיד start() ולא run()), ומעליהם Executor + thread pools (fixed/single/cached) לניהול נשלט של המקביליות.
