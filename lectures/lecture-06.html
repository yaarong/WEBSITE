בהרצאה הקודמת השתמשנו ב-JavaScript בלי לעצור על השפה עצמה. עכשיו נכיר אותה לעומק: המאפיינים והקוויארקים שלה, מנוע ההרצה, ואז שפת הליבה — ההבדלים העדינים של var/let/const ו-hoisting, אובייקטים, ה-async (callbacks, Promises, async/await) ו-prototypes.

1. מהי JavaScript — מאפיינים

🧭
קונטקסט לפני שכותבים קוד, חשוב להבין את האופי של השפה — מאפיינים שמסבירים למה היא מתנהגת כמו שהיא מתנהגת.
  • שפה אינטואיטיבית — אבל לא טריוויאלית (יש מלכודות עדינות).
  • רצה על הרבה פלטפורמות (דפדפן, שרת, מובייל...).
  • לא באמת קשורה ל-Java (השם הוא בעיקר שיווק מהשנים ההן).
  • רוב היסודות סטנדרטיים (if/else, לולאות וכו').
  • Weakly-typed (טיפוסים גמישים), interpreted (מפורשת, עם JIT-compilation).
  • Case sensitive — foo ≠ Foo.
  • Single-threaded, event-driven (thread אחד + event loop — בדיוק כמו שראינו בהרצאה 5).
  • Multi-paradigm — תומכת בפרוצדורלי, מונחה-עצמים, ופונקציונלי.

2. הקוויארקים — אל תנצל לרעה!

⚠️
"JavaScript can be weird, but DON'T abuse that" השפה מתירנית מאוד — אבל רק כי אפשר לעשות משהו לא אומר שצריך. הנה שלושה "היתרים" שעדיף להימנע מהם:
js · קוויארקים
// 1. נקודה-פסיק אופציונלית (אבל תמיד תכתוב אותה!)
foo = 1;
foo = 1

// 2. הצהרת משתנה אופציונלית (אבל תמיד הצהר!)
foo = 1;
var foo = 1

// 3. הצהרה כפולה — מותרת ולא זורקת שגיאה
var foo = 1;
var foo;
console.log(foo); // 1   (foo נשאר 1, לא נמחק!)
💡
הלקח תמיד כתוב נקודה-פסיק, תמיד הצהר משתנים (ועדיף עם let/const ולא var), ואל תצהיר אותו משתנה פעמיים. הסלחנות של השפה היא מקור עיקרי לבאגים — וכל החלק הזה של ההרצאה בא להראות איפה היא מסתתרת.

3. מנוע JS — Interpreted, JIT, V8

🧭
קונטקסט — איך הקוד באמת רץ השם הרשמי של JavaScript הוא ECMAScript, עם עדכונים שנתיים מאז 2015 (גרסת ES6 / ES2015 ואילך הביאו את רוב מה שנלמד היום). אבל איך הקוד מורץ?

JavaScript היא שפה מפורשת (interpreted), והמסלול הוא:

 קוד JS
   │ 1. ה-engine מנתח (parse) את הקוד לזיהוי שגיאות
   ▼      (באמצעות Abstract Syntax Tree — AST)
 AST
   │ 2. הומר ל-bytecode
   ▼
 Bytecode
   │ 3. מורץ ע"י JavaScript VM,
   ▼      ואפשר לייעל אותו עם JIT-compiler
 ריצה מהירה
  • JavaScript מורצת ע"י מנועים (engines) ייעודיים.
  • בעבר המנועים היו רק בדפדפנים (להרצת JS בצד הלקוח).
  • היום JS פופולרית גם בצד השרת — למשל מנוע V8 של Chrome משמש את Node.js.
🔤
Engine APIs — מה ייחודי לכל סביבה שפת הליבה זהה בכל המנועים, אבל לכל מנוע יכולה להיות פונקציונליות ייחודית שהיא חלק מ-API: console.log היא דוגמה אחת; alert(...) היא דוגמה אחרת — היא חלק מה-WebAPI של הדפדפן ולא קיימת ב-Node.js. כלומר, אותו קוד "core" ירוץ בכל מקום, אבל קריאות API מסוימות תלויות-סביבה.

4. Hello World + הפרדת HTML/JS

JavaScript משובצת בדף HTML בתוך תג <script>:

html + js · hello world
<html>
<body>
    <script type="text/javascript">
        var x = 'Hello world';
        alert(x);
    </script>
</body>
</html>

alert(x) מציג תיבת הודעה בדפדפן עם הטקסט. (שים לב: רק קוד בתוך <script> מורץ — טקסט מחוץ לו הוא סתם תוכן הדף.)

עדיף להפריד את ה-JS לקובץ נפרד (כמו שמפרידים CSS) — קריא יותר וניתן למחזור:

html · script חיצוני
<html>
<head></head>
<body>
    <script type="text/javascript" src="foo.js"></script>
</body>
</html>
🔗
חיבור להרצאה 2 זו בדיוק הפרדת אחריות (separation of concerns) שראינו ב-SOLID: HTML = מבנה, CSS = עיצוב, JS = התנהגות. ערבוב של שלושתם באותו קובץ הוא code smell.

5. var / let / const ו-scoping

🧭
קונטקסט — אחד הנושאים החשובים ביותר לפני ES6 הייתה רק var, וההתנהגות שלה מבלבלת. ES6 הוסיפה את let ו-const שמתנהגות צפוי יותר. ההבדל המרכזי: scope (תחום הקיום).

var אינו block-scoped — בלוק if לא יוצר scope חדש:

js · var בתוך if
var x = 1;
console.log(x); // 1

if (x == "1"){       // true! (== ממיר טיפוסים; === היה מחזיר false)
    var x = 2;       // אותו x! לא חדש
    console.log(x);  // 2
}

console.log(x); // 2   ← השתנה! ה-x בשורה 1 ובתוך ה-if הם אותו משתנה

var כן function-scoped — פונקציה כן יוצרת scope:

js · var בתוך function
var x = 1;
function foo() {
    var x = 2;       // x נפרד! (scope של הפונקציה)
    console.log(x);  // 2
}
foo();
console.log(x); // 1   ← לא השתנה

let הוא block-scoped — כל בלוק {} יוצר scope:

js · let
let x = 1;
console.log(x); // 1

if (x == "1"){
    let x = 2;       // x חדש! (scope של הבלוק)
    console.log(x);  // 2
}

function foo() {
    let x = 3;       // x חדש נוסף
    console.log(x);  // 3
}
foo();

console.log(x); // 1   ← לא הושפע משום בלוק/פונקציה

const — כמו let אבל לא ניתן להשמה מחדש:

js · const
const foo = 1;
foo = 1;  // Uncaught TypeError: Assignment to constant variable.
🌍
מלכודת — משתנים לא-מוצהרים הם גלובליים! אם משתמשים במשתנה בלי להצהיר אותו בכלל, JS מצהירה אותו אוטומטית כגלובלי:
js
function foo() {
    x = 2;            // x לא מוצהר → הופך לגלובלי!
    console.log(x);   // 2
}
foo();
console.log(x); // 2   ← נגיש מבחוץ! (זיהום ה-global scope)
זו אחת הסיבות המרכזיות לכתוב תמיד let/const ולהפעיל "use strict".
💡
כלל אצבע (דוגמה משלי) ב-JS מודרני: השתמש ב-const כברירת מחדל, ב-let רק כשצריך לשנות את הערך, וב-var אף פעם. זה מונע את רוב המלכודות לעיל. זוכרים את לולאת ה-let i מקטע האירועים? עם var i כל ה-handlers היו "רואים" את אותו i הסופי — באג קלאסי שבדיוק let פותר.

6. אובייקטים (JSON) ומתודות

🧭
קונטקסט אובייקטים ב-JS הם פשוט אוספי מפתח-ערך, בתחביר שנקרא JSON (JavaScript Object Notation). הם דינמיים — אפשר להוסיף ולשנות שדות תוך כדי ריצה.
js · אובייקט דינמי
const o = {foo: "Bar", caz: 3, bar: 'abc'}
console.log(o);   // {foo: 'Bar', caz: 3, bar: 'abc'}

o.caz = 4;        // שינוי שדה קיים
console.log(o);   // {foo: 'Bar', caz: 4, bar: 'abc'}

o.xyz = '5';      // הוספת שדה חדש תוך כדי ריצה!
console.log(o);   // {foo: 'Bar', caz: 4, bar: 'abc', xyz: '5'}

o = {}            // TypeError! (const — אי-אפשר להשמה מחדש)
🔤
const על אובייקט — נקודה עדינה const o מונע השמה מחדש של o (o = {} אסור), אבל כן מאפשר לשנות את תוכן האובייקט (o.caz = 4 מותר). ה-const "נועל" את ההפניה, לא את התוכן.

מתודות באובייקט + הגישה אליהן:

js · מתודות
const o = {
    foo: 1,
    caz() { return this.foo },           // קיצור מתודה
    //caz2: () => { return this.foo },   // ❌ arrow — אין לה this נכון!
    bar: function(x) { return x+1 }
}
console.log(o.foo)      // 1
console.log(o['foo'])   // 1   (גישה גם בסוגריים מרובעים)
console.log(o.caz())    // 1
console.log(o.bar(1))   // 2
⚠️
אל תשתמש ב-arrow functions כמתודות! ל-arrow functions יש מגבלות מובנות — אין להן גישה ל-this של האובייקט (ה-this שלהן נקבע מההקשר החיצוני). לכן caz2 המקוונן לא היה עובד. למתודה שצריכה this — השתמש ב-caz() או ב-function רגיל.

7. Hoisting (הרמה)

🧭
קונטקסט לפני שהקוד רץ, ה-engine סורק אותו ו"מרים" (hoists) את ההצהרות לראש ה-scope שלהן. חשוב: מורמות רק ההצהרות (declarations), לא האתחול (initialization).

עם var — הצהרה בלי אתחול = undefined (לא שגיאה):

js · hoisting של var
// זה כל הקוד:
console.log(foo); // undefined   ← לא שגיאה! ההצהרה "הורמה" למעלה
var foo;

// מול: משתנה שלא קיים בכלל
console.log(bar); // ReferenceError: bar is not defined

Hoisting מרים את ההצהרה, לא את הערך:

js
console.log(foo); // undefined   ← ההצהרה הורמה, אבל לא ה-='bar'
var foo = 'bar';
console.log(foo); // bar
🪤
let ו-const — Temporal Dead Zone ל-let/const ההתנהגות שונה — גישה לפני ההצהרה זורקת שגיאה (הם ב-"אזור מת זמני", TDZ):
js
console.log(foo); // ReferenceError: foo is not defined
let foo = 'foo';
זה התנהגות בריאה יותר — עוד סיבה להעדיף let/const על var.

הצהרת פונקציה מול ביטוי פונקציה — הבדל קריטי:

js · function declaration — עובד
foo(); // foo   ← עובד! הצהרת פונקציה מורמת במלואה
function foo(){
    console.log("foo");
}
js · function expression — נכשל
foo(); // TypeError: foo is not a function
var foo = function (){          // רק ההצהרה var foo מורמת (=undefined),
    console.log("foo");         // ההשמה של הפונקציה עוד לא קרתה!
}
🔤
ההבדל הצהרת פונקציה (function foo(){}) מורמת במלואה — אפשר לקרוא לה לפני ההגדרה. ביטוי פונקציה (var foo = function(){}) — רק ה-var foo מורם (כ-undefined), וקריאה לפני ההשמה נותנת "foo is not a function". וגם בתוך פונקציה: var פנימי מורם לראש הפונקציה, ולכן console.log(foo) לפני var foo='foo' יחזיר undefined ולא את הערך החיצוני.

8. Async: Callbacks, Promises, async/await

🧭
קונטקסט ראינו בהרצאה 5 את ה-event loop, Promises ו-async/await. כאן נחזק את ההבנה דרך הזווית של ההיסטוריה: callbacks → Promises → async/await, וננתח דוגמת סדר-ריצה מדויקת.

הדרך הישנה — Callbacks: מעבירים פונקציה שתורץ "כשיהיה מוכן":

js · callback עם XMLHttpRequest
function foo(callback){
    const xhr = new XMLHttpRequest();
    xhr.onload = function() {
        callback(xhr.responseText)   // קוראים ל-callback כשהתשובה הגיעה
    }
    xhr.open("GET", "http://www.foo.com");
    xhr.send();
}

function bar(data) {
    console.log(data)
}

foo(bar);   // מעבירים את bar כ-callback

Event loop — תזכורת: קוד סינכרוני רץ על ה-Stack, פעולות אסינכרוניות עוברות ל-APIs, וה-callbacks שלהן נכנסים ל-Task queue. ה-event loop מעביר אותם ל-Stack רק כשהוא ריק. לכן:

js · מה קודם?
setTimeout(function (){ console.log('foo'); }, 2000);
console.log('bar');
// פלט: bar ואז foo

דוגמת Promise מלאה — נתח את סדר הפלט:

js · סדר ריצה
console.log(1)
let foo = new Promise((resolve, reject) => {
    console.log(2)
    setTimeout(function() {
        console.log(3)
        resolve('done');
    }, 3000);
    console.log(4)
})

console.log(7)

function bar(data) {
    console.log(6)
    console.log(data)
}

console.log(8)
foo.then(bar);
console.log(9)
🔢
הפלט: 1, 2, 4, 7, 8, 9, 3, 6, done — למה?
  • קוד סינכרוני קודם: 1. ה-Promise executor רץ מיד וסינכרונית: 2, מתזמן את ה-setTimeout, 4. ממשיכים: 7, 8, רושמים את foo.then(bar), 9.
  • אחרי 3 שניות (כשה-Stack ריק): ה-callback של ה-timeout רץ → 3, ואז resolve('done') מפעיל את bar → 6, ואז console.log(data) → done.
השורה המספרית (3000ms) רצה אחרונה — בדיוק כמו ש-fetch(...).then(...) רץ אחרי הקוד הסינכרוני שאחריו.

async/await — פונקציות async תמיד מחזירות Promise:

js · async/await
let foo = async () => "bar";
// זהה ל: function foo() { return Promise.resolve("bar"); }

var bar = await foo()   // await "פותח" את ה-Promise ומחזיר את הערך
console.log(bar)        // bar

await ממתין שה-Promise ייפתר ומחזיר את הערך — קוד אסינכרוני שנראה סינכרוני וקריא (ראינו בהרצאה 5).

9. Prototypes ושרשרת הפרוטוטייפ

🧭
קונטקסט — מה באמת קורה מאחורי class ל-JavaScript יש class ו-extends כמו ב-Java — אבל זה הטעיה. מתחת למכסה המנוע, JS היא prototype-based, וה-class הוא רק "סוכר תחבירי" (syntactic sugar) מעל מנגנון ה-prototypes.
js · class ו-extends
class Foo {
    caz = 1
    constructor (foo) {
        this.bar = foo
    }
    baz() {
        return this.bar
    }
}

class Foo2 extends Foo {
    constructor(foo2) {
        super(foo2)
    }
}

const f = new Foo2(3)
console.log(f)   // Foo2 {caz: 1, bar: 3}, [[Prototype]]: Foo

איך זה עובד באמת:

  • לכל אובייקט יש תכונה מיוחדת — prototype.
  • ערך ה-prototype הוא גם אובייקט, ולכן גם לו יש prototype — וכך נוצרת שרשרת פרוטוטייפ (prototype chain).
  • כשמחפשים תכונה של אובייקט: מחפשים קודם באובייקט עצמו, ואם לא נמצא — מחפשים ב-prototype שלו, וחוזרים על זה עד שמגיעים ל-null.
 f (Foo2)  ──prototype──▶  Foo.prototype  ──prototype──▶  Object.prototype  ──▶  null
 {caz, bar}                {baz, constructor}            {toString, valueOf, ...}

 f.baz()  →  לא ב-f? → חפש ב-Foo.prototype → נמצא! ✓
 f.toString()  →  לא ב-f? → לא ב-Foo? → ב-Object.prototype → נמצא! ✓
🔗
בדיקה בקונסול Object.getPrototypeOf(c) מחזיר את ה-prototype של אובייקט. אם תמשיך לטפס (getPrototypeOf(getPrototypeOf(c))) תגיע בסוף ל-null — קצה השרשרת. זה בדיוק מה ש-JS עושה בכל גישה לתכונה. הסיכום: JavaScript היא prototype-based; ה-class רק נראה כמו ירושה קלאסית אבל מתורגם לשרשרת prototypes.

★ סיכום ההרצאה

🎯
התמונה המלאה
  • JS: weakly-typed, interpreted (JIT, V8/Node), case-sensitive, single-threaded event-driven. סלחנית — אבל אל תנצל זאת לרעה.
  • scoping: var = function-scoped + hoisted ל-undefined; let/const = block-scoped + TDZ. השתמש ב-const/let, לעולם לא var. משתנה לא-מוצהר = גלובלי (סכנה!).
  • Hoisting: הצהרות עולות לראש ה-scope (לא האתחול). function declaration עולה במלואה; function expression לא.
  • אובייקטים דינמיים; אל תשתמש ב-arrow כמתודה (אין this).
  • Async: callbacks → Promises → async/await; הכל מעל event loop יחיד.
  • Prototypes: JS היא prototype-based; class הוא syntactic sugar מעל prototype chain.
