ממשק משתמש (GUI) — מבניית רכיבים ב-code, דרך XML/HTML ו-CSS, ועד React

עד עכשיו עסקנו בעיקר בצד השרת (Node, Express, MVC). ההרצאה הזו עוסקת בצד שהמשתמש רואה — ה-GUI (Graphical User Interface). נתחיל מהשאלה "מה בכלל מרכיב ממשק גרפי?", נראה למה לא בונים UI ישירות ב-code, נכיר את שפות ה-Markup (XML, HTML), CSS לעיצוב, את ה-Listener pattern (תכנות מונחה-אירועים), ולבסוף נעלה רמה: למה צריך frameworks מודרניים ונבנה את אתר ה-Foo News מחדש ב-React.

🗺️
קונטקסט כללי — איך הכל מתחבר ההרצאה משווה כל הזמן בין שלוש פלטפורמות: Android (מובייל), Web (דפדפן), ו-.NET/WPF (דסקטופ). זה לא מקרי — הרעיונות זהים בכולן: רכיבים גרפיים הם אובייקטים, בונים אותם בשפת תיאור (markup), מעצבים בנפרד (styling), ומגיבים לפעולות המשתמש (events). ברגע שמבינים את התבנית במקום אחד — מזהים אותה בכל מקום. בסוף נגיע ל-React, שלוקח את כל הרעיונות האלה ועוטף אותם בצורה מודרנית.

חלק א' · מה זה GUI ואיך (לא) בונים אותו

1. מהו GUI — Web, Mobile, Desktop

🧭
קונטקסט GUI = Graphical User Interface — הממשק הגרפי שדרכו המשתמש מתקשר עם התוכנה (כפתורים, שדות, תפריטים, תמונות), בניגוד לממשק טקסטואלי (CLI / טרמינל). לכל סוג של פלטפורמה יש "מערכת GUI" משלה, אבל העקרונות משותפים.

שלושת העולמות המרכזיים שנשווה ביניהם לאורך ההרצאה:

פלטפורמהאיפה רץטכנולוגיה אופיינית
Web GUIבתוך דפדפןHTML + CSS + JavaScript
Mobile GUIאפליקציית מוביילAndroid (Java/Kotlin + XML)
Desktop GUIתוכנת מחשב.NET / WPF (C# + XAML)

נתמקד בעיקר ב-Web (כי לשם נגיע עם React), אבל נשווה כל הזמן ל-Android ו-.NET כדי לראות שהרעיונות אוניברסליים.

2. Native מול Cross-Platform

🧭
קונטקסט — דילמה הנדסית כשבונים אפליקציה שצריכה לרוץ על כמה מערכות (iOS, Android, Windows...), יש שתי גישות עקרוניות. ההבדל ביניהן הוא קלאסי בהנדסת תוכנה: ביצועים והתאמה מקסימלית מול שיתוף קוד ומהירות פיתוח.
גישהרעיוןיתרונותחסרונות
Native פותחים לכל פלטפורמה בנפרד, בשפה/כלים שלה ביצועים מעולים, חוויית משתמש מקומית, גישה מלאה ל-APIs של המכשיר קוד נפרד לכל פלטפורמה (כפילות, תחזוקה כפולה)
Cross-Platform "Write once, run anywhere" — קוד אחד לכל הפלטפורמות קוד משותף לכל הפלטפורמות הנתמכות קשה להגיע לאיכות של native (ביצועים/חוויה)

דוגמאות: Native — Kotlin/Java לאנדרואיד, Swift ל-iOS. Cross-Platform — .NET MAUI, וגם React Native ו-Flutter. (ה-Web עצמו הוא במובן מסוים ה-cross-platform האולטימטיבי: אתר אחד רץ בכל דפדפן בכל מכשיר.)

3. רכיבים גרפיים הם אובייקטים — ובנייה ב-code

🧭
קונטקסט — תובנת המפתח כל פריט שאתם רואים על המסך — חלון, כפתור, תווית (label), שדה טקסט — הוא אובייקט בקוד. לאובייקט יש "members" (שדות ומתודות) שאחראים על הציור שלו ועל המאפיינים שלו, ואנחנו יכולים לשנות אותם כדי להתאים את הרכיב. זה הבסיס להכל: GUI הוא בסופו של דבר עץ של אובייקטים.

הדרך ה"נאיבית" לבנות ממשק היא ליצור את האובייקטים ידנית בקוד. ב-Android (Java/Kotlin) זה נראה כך:

java · Android — בניית UI ב-code
public class MainActivity extends Activity {

    @Override
    protected void onCreate(Bundle savedInstanceState) {
        super.onCreate(savedInstanceState);

        TextView textView = new TextView(this);
        textView.setText("Hello World!");

        LinearLayout linearLayout = new LinearLayout(this);

        linearLayout.addView(textView);
        setContentView(linearLayout);
    }
}
🔤
מה קורה כאן?
  • new TextView(this) — יוצרים אובייקט תווית (להצגת טקסט) ו-setText קובע את תוכנו.
  • new LinearLayout(this) — container שמסדר רכיבים בשורה/טור. addView מוסיף את התווית לתוכו.
  • setContentView(linearLayout) — קובע את העץ הזה כתוכן של המסך.
שימו לב לרעיון: בונים עץ — layout שמכיל רכיבים. זה ה"DOM" של אנדרואיד.

ואותו דבר ב-Web (JavaScript) — מוסיפים צומת טקסט ל-DOM (ראינו DOM בהרצאה 6):

js · Web — בניית UI ב-code
var txt = document.createTextNode("Hello world");
document.body.appendChild(txt);

4. הבעיה בבנייה ב-code → הפרדת UI מלוגיקה

⚠️
למה בנייה ב-code היא לא best practice אפשר לבנות UI כך — אבל בדרך כלל לא כדאי:
  • קשה לקרוא ולתחזק — קוד הבנייה מתערבב עם הלוגיקה, וקשה "לראות" את מבנה המסך מתוך השורות.
  • דורש ידע בתכנות — ובמיוחד בשפה הספציפית. מעצב/ת UI לא אמור/ה לכתוב Java כדי לסדר כפתור.
💡
הפתרון — שפת Markup מפרידים את קוד ה-UI מקוד הלוגיקה, ומתארים את הרכיבים בשפת Markup — שפה שמתארת אילו אובייקטים ליצור, בפורמט קריא וגמיש. חשוב: עדיין נוצרים אובייקטים מאחורי הקלעים, אז עדיין אפשר לעבוד איתם ב-code כשצריך — פשוט לא חייבים. זו בדיוק הפרדת האחריות (SRP) מהרצאה 2, ואותו רעיון של "תיאור מול ביצוע".

חלק ב' · שפות Markup — XML ו-HTML

5. XML — eXtensible Markup Language

🧭
קונטקסט XML היא שפת markup לשמירה ולהעברה של נתונים בצורה מובְנית. הנתונים מאורגנים בעזרת תגיות (tags), ו-"eXtensible" (ניתנת להרחבה) פירושו שאת שמות התגיות אתם ממציאים בעצמכם — אין רשימה קבועה. זה ההבדל הגדול מ-HTML, שבה התגיות קבועות מראש (<p>, <div>...).

הדוגמה המינימלית:

xml · המבנה הבסיסי
<?xml version="1.0" encoding="UTF-8"?>
<foo>
    <bar>Hello, World!</bar>
</foo>
🔤
אנטומיה של XML
  • שורת ה-<?xml ... ?> בראש — הכרזה (prolog) שמציינת גרסה וקידוד.
  • כל אלמנט נפתח בתגית <foo> ונסגר ב-</foo>. מה שביניהן הוא התוכן.
  • אלמנטים מקוננים זה בתוך זה ויוצרים עץ (כאן bar הוא ילד של foo).
  • חייב להיות שורש אחד (root) שעוטף את הכל.

דוגמה עשירה יותר — מאגר כתבות חדשות (שימו לב: זה אותו "Foo News" שנפגוש שוב ושוב):

xml · news_articles.xml
<?xml version="1.0" encoding="UTF-8"?>
<news_articles>
    <article>
        <headline>World's Largest Foo Created in Foobar Event</headline>
        <author>Baz Caz</author>
        <category>Mystery</category>
        <publication_date>2024-01-11</publication_date>
    </article>
    <article>
        <headline>Unicorn Called Bar in West of Foobar</headline>
        <author>Foo foo</author>
        <category>Nature</category>
        <publication_date>2024-01-10</publication_date>
    </article>
</news_articles>

שימו לב כמה זה קריא: גם בלי לדעת כלום על הפורמט, ברור שיש שתי כתבות ולכל אחת כותרת, מחבר, קטגוריה ותאריך. זו המטרה של markup.

6. XML באנדרואיד — Layouts ו-Resource Files

🧭
קונטקסט אנדרואיד משתמשת ב-XML המון — בעיקר ל-הגדרת ה-UI (layouts) ול-resources (מחרוזות, צבעים, סגנונות). במקום לבנות את המסך ב-Java, מתארים אותו בקובץ XML, וה-Activity רק "טוענת" אותו.

במקום כל קוד הבנייה מסעיף 3, ה-Activity מצטמצמת לשורה אחת:

java · MainActivity.java
public class MainActivity extends Activity {

    @Override
    protected void onCreate(Bundle savedInstanceState) {
        super.onCreate(savedInstanceState);
        setContentView(R.layout.activity_main);   // ← טוען את ה-XML
    }
}

וה-UI מתואר בקובץ XML נפרד:

xml · res/layout/activity_main.xml
<?xml version="1.0" encoding="utf-8"?>
<LinearLayout
    xmlns:android="http://schemas.android.com/apk/res/android"
    android:layout_width="match_parent"
    android:layout_height="match_parent"
    android:orientation="vertical">

    <TextView
        android:layout_width="match_parent"
        android:layout_height="wrap_content"
        android:text="Hello world" />

</LinearLayout>
🔤
תחביר ומושגים
  • R.layout.activity_main — R הוא מחלקה אוטומטית שאנדרואיד מייצרת, שמפנה לכל ה-resources. R.layout.activity_main = "קובץ ה-layout בשם activity_main".
  • android: — namespace (מרחב שמות) שמבדיל את מאפייני אנדרואיד; מוגדר ב-xmlns:android="...".
  • match_parent = "תפוס את כל רוחב/גובה ההורה"; wrap_content = "תפוס בדיוק כמה שצריך לתוכן".
  • orientation="vertical" — סידור הילדים בטור (אפשר גם horizontal).
  • תגית שנסגרת ב-/> (כמו <TextView ... />) היא תגית ריקה — בלי ילדים.

7. HTML — HyperText Markup Language

🧭
קונטקסט HTML היא שפת ה-markup של ה-Web: היא מתארת את המבנה והתוכן של עמוד. הדפדפן לוקח את קובץ ה-HTML ומצייר אותו ויזואלית. בדומה ל-XML היא משתמשת בתגיות — אבל כאן התגיות קבועות ומוגדרות מראש (html, head, body, p, button...).
🎯
כלל אצבע HTML יכולה להכיל גם אלמנטים של עיצוב — אבל עדיף שלא. ההפרדה הנכונה: HTML למבנה/תוכן, CSS לעיצוב, JavaScript להתנהגות. נראה את זה בפועל בחלק העיצוב.

אותו "Hello world" כמו ב-Android, אבל ב-HTML:

html · index.html
<!DOCTYPE html>
<html>
    <head></head>
    <body>
        Hello world
    </body>
</html>
🔤
השלד הקבוע של כל עמוד
  • <!DOCTYPE html> — מצהיר שזה מסמך HTML5.
  • <html> — שורש המסמך.
  • <head> — מטא-מידע (כותרת, קישור ל-CSS, קידוד) — לא מוצג בגוף העמוד.
  • <body> — התוכן שהמשתמש רואה בפועל.
זה בדיוק ה-article.ejs מהרצאה 7 — קובץ EJS הוא בסך הכל HTML עם "חורים" דינמיים.

8. רכיבי GUI — אותם רעיונות, שלוש שפות

🧭
קונטקסט לכל פלטפורמה יש סט רכיבים בסיסיים: תווית להצגת טקסט, שדה לקלט, וכפתור. השמות שונים — הרעיון זהה לחלוטין. הנה אותו ממשק ("Foo / Bar / Foobar") בשלוש השפות.
רכיבAndroid (XML)Web (HTML).NET (XAML)
תווית (טקסט להצגה)<TextView><span><Label>
שדה קלט<EditText><input><TextBox>
כפתור<Button><button><Button>
Container/Layout<LinearLayout><div><StackPanel>

Android:

xml · Android
<LinearLayout
    xmlns:android="http://schemas.android.com/apk/res/android"
    android:layout_width="match_parent"
    android:layout_height="match_parent"
    android:orientation="vertical">

    <TextView  android:text="Foo" ... />
    <EditText  android:text="Bar" ... />
    <Button    android:text="Foobar" ... />
</LinearLayout>

Web:

html · Web
<html>
    <head></head>
    <body>
        <span>Foo</span>
        <input value="Bar"></input>
        <button>Foobar</button>
    </body>
</html>

.NET (XAML):

xml · .NET / WPF
<StackPanel>
    <Label>Foo</Label>
    <TextBox>Bar</TextBox>
    <Button>Foobar</Button>
</StackPanel>
🔗
למדו לזהות את התבנית ברגע שמזהים ש-"תווית/קלט/כפתור בתוך container" חוזר בכל הפלטפורמות — לימוד פלטפורמה חדשה הופך לתרגום מילון, לא ללימוד מאפס. זה בדיוק כוחה של חשיבה בתבניות (patterns).

חלק ג' · עיצוב (Styling) ו-CSS

9. עיצוב ב-Web — מ-inline ועד קובץ CSS

🧭
קונטקסט — מסע בשלושה שלבים נראה את אותו עיצוב (כפתור ירוק) בשלוש דרכים, מהגרועה לטובה. זה ממחיש בדיוק למה מפרידים עיצוב מתוכן.

שלב 1 — inline style (עיצוב ישירות על האלמנט). עובד, אבל מתערבב בתוכן ולא ניתן לשימוש חוזר:

html · inline (לא מומלץ)
<button style="background-color:#00FF00;
               color:#FF0000;
               font-size:18pt;
               margin:10px">Foo</button>
<button>Bar</button>
<button>Foobar</button>

שלב 2 — בלוק <style> ב-<head>. עכשיו כל הכפתורים מקבלים את העיצוב במכה אחת:

html · internal CSS
<html>
<head>
    <style>
        button {
            background-color:#00FF00;
            color:#FF0000;
            font-size:18pt;
            margin:10px
        }
    </style>
</head>
<body>
    <button>Foo</button>
    <button>Bar</button>
    <button>Foobar</button>
</body>
</html>

שלב 3 — קובץ CSS חיצוני. ההפרדה המלאה: ה-HTML נקי לגמרי, והעיצוב חי בקובץ נפרד שאפשר לשתף בין עמודים:

css · index.css
button {
    margin:10px;
    background-color:#00FF00;
    color:#FF0000;
    font-size:18pt
}
html · index.html — מקשר ל-CSS
<html>
<head>
    <link href="index.css" rel="stylesheet">
</head>
<body>
    <button>Foo</button>
    <button>Bar</button>
    <button>Foobar</button>
</body>
</html>
🔤
סלקטורים — type, class, id ב-CSS, כל כלל מתחיל ב-selector שקובע על מי הוא חל:
  • button { ... } — type selector: כל אלמנטי ה-<button>.
  • .green-bg { ... } — class selector (נקודה): כל אלמנט עם class="green-bg". ניתן לשימוש חוזר על הרבה אלמנטים.
  • #danger-btn { ... } — id selector (סולמית): האלמנט היחיד עם id="danger-btn". id אמור להיות ייחודי בעמוד.

שילוב של השלושה — שימו לב איך אלמנט יכול לקבל גם class וגם id:

css · index.css
button {
    background-color:#00FF00;
    color:#FF0000;
    font-size:18pt;
    margin:10px;
}
#danger-btn {
    background-color:#0000FF;
}
.green-bg {
    background-color:#000000;
}
html · index.html
<button>Foo</button>
<button class="green-bg">Bar</button>
<button class="green-bg" id="danger-btn">Foobar</button>

אבל רגע — לכפתור השלישי יש שלושה כללים שנוגעים בצבע הרקע (type, class, ו-id). מי מנצח? לכאן נכנס Specificity.

10. CSS Specificity — מי מנצח כשיש התנגשות?

🧭
קונטקסט כשכמה כללים מגדירים את אותו מאפיין על אותו אלמנט, הדפדפן צריך להכריע. הוא עושה זאת לפי specificity — "ניקוד ספציפיות": כלל ספציפי יותר גובר על כללי יותר. זה אחד המקורות הנפוצים ביותר ל"למה לעזאזל הצבע לא משתנה?!".

סדר העדיפות, מהחזק לחלש (אפשר לחשוב על זה כ-4 ספרות (a, b, c, d)):

#סוגמשקל (a,b,c,d)
1Inline styles (style="...")(1, 0, 0, 0)
2ID selectors (#id)(0, 1, 0, 0)
3Class / attribute / pseudo-class (.cls, :hover)(0, 0, 1, 0)
4Type / pseudo-element (button, ::before)(0, 0, 0, 1)
5Universal selector (*)(0, 0, 0, 0)
🃏
שני חוקי-זהב
  • !important הוא ג'וקר — גובר כמעט על הכל (ולכן יש להשתמש בו במשורה, כמוצא אחרון).
  • כששתי הגדרות באותה ספציפיות בדיוק — האחרונה מנצחת (סדר הופעה בקובץ).

דוגמאות חישוב (בקירוב נוח: id=100, class=10, type=1):

11. עיצוב ב-Android + String Resources (i18n)

🧭
קונטקסט אותו עיקרון בדיוק כמו ב-Web: אפשר לעצב inline (מאפיינים על האלמנט), אבל עדיף לרכז את העיצוב ב-style נפרד ולהפנות אליו — בדיוק כמו class ב-CSS.

מגדירים <style> בקובץ resources, ומפנים אליו מהכפתור עם style="@style/myBtn":

xml · res/values/styles.xml
<?xml version="1.0" encoding="utf-8"?>
<resources>
    <style name="myBtn">
        <item name="android:background">#00FF00</item>
        <item name="android:textColor">#FF0000</item>
        <item name="android:textSize">18sp</item>
        <item name="android:layout_margin">10dp</item>
    </style>
</resources>
xml · activity_main.xml
<Button
    android:layout_width="wrap_content"
    android:layout_height="wrap_content"
    android:text="Foo"
    style="@style/myBtn" />
🌍
String Resources — והקשר ל-i18n (בינאום) אנדרואיד מעודדת לא לכתוב טקסט "קשיח" ב-layout, אלא להפנות ל-string resource: android:text="@string/foo", כשהמחרוזות מוגדרות בקובץ נפרד. למה? כי כך אפשר לספק קובץ מחרוזות שונה לכל שפה — וכל האפליקציה מתורגמת בלי לגעת ב-layout.
xml · res/values/strings.xml (אנגלית)
<resources>
    <string name="app_name">My Application</string>
    <string name="foo">Foo</string>
    <string name="bar">Bar</string>
    <string name="foobar">Foobar</string>
</resources>

קובץ מחרוזות אחר (למשל ספרדית) ימפה את אותם שמות לערכים Hola, Cómo, estás — ואותו layout בדיוק יוצג בשפה אחרת לגמרי. השם נשאר @string/foo, רק הערך מתחלף.

חלק ד' · תכנות מונחה-אירועים (Event-Driven)

13. Listener Pattern — Event & Event Handler

🧭
קונטקסט — איך UI "מגיב" עד עכשיו בנינו ועיצבנו רכיבים סטטיים. אבל ממשק צריך להגיב למשתמש (לחיצה, הקלדה, גלילה). זה נעשה בתכנות מונחה-אירועים: אנחנו לא "שואלים כל הזמן" אם נלחץ כפתור — אנחנו נרשמים מראש לאירוע, והמערכת קוראת לקוד שלנו כשהוא קורה. זוהי תבנית ה-Listener (מאזין).
🔤
שני מושגי המפתח
  • Event (אירוע) — משהו שקרה (נלחץ כפתור, הסתיימה טעינה, הוקלד מקש).
  • Event Handler (מטפל) — הקוד שאנחנו רושמים מראש, שירוץ כשהאירוע מתרחש.
אנחנו "מאזינים" לאירוע ומספקים handler. ה-handler הוא callback — פונקציה שמישהו אחר יקרא לה בעתיד (בדיוק כמו ה-handlers של Express מהרצאה 7!).

Web — שלוש דרכים (מהפשוטה לטובה):

html · 1) onclick כמאפיין (inline)
<button id="myButton"
        onclick="alert('Button Clicked!')">Click me</button>
js · 2) addEventListener (מומלץ)
<button id="myButton">Click me</button>
<script type="text/javascript">
    const btn = document.getElementById('myButton');
    btn.addEventListener('click', function() {
        alert('Button Clicked!');
    });
</script>
js · 3) השמה ל-onclick
btn.onclick = function() {
    alert('Button Clicked!');
};
🪤
המלכודת מהרצאה 6 — שוב! בדרך 3 כותבים btn.onclick = myFunc — בלי סוגריים. רושמים את הפונקציה עצמה כ-handler. אם נכתוב btn.onclick = myFunc() (עם סוגריים), נריץ אותה מיד ונשים את הערך המוחזר במקום ה-handler. זו בדיוק אותה מלכודת onclick = show מהרצאה 6, ואותו עיקרון של "מעבירים פונקציה, לא קוראים לה" מ-Express בהרצאה 7.

Android — אותו רעיון בדיוק עם setOnClickListener:

java · Android — Listener
Button myButton = findViewById(R.id.myButton);
myButton.setOnClickListener(new View.OnClickListener() {
    @Override
    public void onClick(View v) {
        Toast.makeText(MainActivity.this,
            "Button Clicked!",
            Toast.LENGTH_SHORT).show();
    }
});
🔤
אותה תבנית, מבטא שונה setOnClickListener מקבל אובייקט View.OnClickListener שמממש את המתודה onClick — וזה ה-handler. (Toast הוא הודעה קופצת קטנה.) בדיוק כמו addEventListener('click', handler) ב-Web: רושמים מאזין, מספקים handler, המערכת קוראת לו בלחיצה. Listener Pattern במלוא הדרו.

חלק ו' · Frameworks מודרניים — React

15. הבעיה והפתרון

🧭
קונטקסט הבעיה: ככל שממשקים נעשים מורכבים יותר (נתונים שמשתנים, אלפי רכיבים, אינטראקציות), פיתוח ב-HTML/JS גולמי הופך קשה מאוד — צריך לעדכן את ה-DOM ידנית בכל שינוי, ולשמור על סנכרון בין הנתונים לתצוגה.
הפתרון: Web UI Frameworks — למשל React, Vue, Angular. הם נותנים לנו לתאר את ה-UI כפונקציה של הנתונים, והם דואגים לעדכן את ה-DOM אוטומטית כשהנתונים משתנים.

16. React מהיסוד — createElement, JSX, ו-Babel

🧭
קונטקסט לפני שנשתמש ב-React "האמיתי", נבין מה הוא עושה מתחת למכסה — בעזרת אותו "Hello" שלוש פעמים: ב-JS גולמי, עם React.createElement, ולבסוף עם JSX.

1) JS גולמי — יוצרים אלמנט ידנית ומוסיפים ל-DOM (כמו בסעיף 3):

js · DOM גולמי
<div id="root"></div>
<script>
    const h = document.createElement('h1')
    h.textContent = 'Hello'
    h.className = 'foo'
    document.getElementById('root').appendChild(h)
</script>

2) React.createElement — React יוצר ומנהל את האלמנט, ו-ReactDOM.render "מצייר" אותו לתוך ה-#root:

js · React.createElement
<div id="root"></div>
<script src="react.development.js"></script>
<script src="react-dom.development.js"></script>
<script>
    const h = React.createElement('h1', {
        className : 'foo'
    }, 'Hello')
    ReactDOM.render(h, document.getElementById('root'))
</script>
🔤
React.createElement(type, props, children) שלושת הארגומנטים: סוג האלמנט ('h1'), אובייקט ה-props (מאפיינים, כאן className), וה-children (התוכן, כאן 'Hello'). ReactDOM.render(element, container) מצייר את האלמנט לתוך ה-#root שב-HTML.

3) JSX — לכתוב createElement ביד מסורבל. JSX מאפשר לכתוב מה שנראה כמו HTML בתוך ה-JS:

jsx · עם Babel
<script src="babel.js"></script>
<script type="text/babel">
    const h = <h1>Hello {1+2}
        <div>
            <span className="foo">Hi</span>
        </div>
    </h1>
    ReactDOM.render(h, document.getElementById('root'))
</script>
🔤
JSX — תחביר
  • נראה כמו HTML, אבל זה לא — זה "סוכר תחבירי" שמתורגם ל-React.createElement.
  • { ... } — סוגריים מסולסלים "בורחים" חזרה ל-JavaScript: {1+2} ידפיס 3. ככה מזריקים ערכים דינמיים (בדיוק כמו <%= %> ב-EJS!).
  • שימו לב: className ולא class — כי class היא מילה שמורה ב-JavaScript.
⚙️
Transpilation — מה זה Babel? JavaScript מתפתחת ומוסיפה תחביר חדש (וגם JSX אינו תחביר JS חוקי שהדפדפן מבין). Transpilation = המרה של תחביר חדש ל-תחביר ישן יותר שכל הדפדפנים מבינים. Babel הוא ה-transpiler שממיר JSX (ו-ES חדש) ל-JavaScript "רגיל". זה קורה לפני שהקוד מגיע לדפדפן.

🧭
קונטקסט בפועל לא טוענים React דרך <script> — משתמשים בכלי שמקים פרויקט מסודר עם Babel, שרת פיתוח, בדיקות והכל מוכן. הקלאסי הוא create-react-app. נבנה איתו את Foo News מחדש.

19. Props — העברת נתונים לרכיב

🧭
קונטקסט — הבעיה כרגע כל <Article /> מציג בדיוק את אותו תוכן קשיח. אבל אנחנו רוצים שכל כתבה תציג נתונים משלה. הפתרון: props — "ארגומנטים" שמעבירים לרכיב, בדיוק כמו מאפיינים (attributes) של תגית HTML.

מעבירים props ב-App (נראה כמו attributes):

js · App.js
function App() {
    return (
        <div className="App">
            <Article id="1"
                     title="My title"
                     author="Bar caz"
                     category="Nature"
                     publication_date="2024-1-11" />
            <Article id="2"
                     title="My title2"
                     author="Bar caz2"
                     category="Nature2"
                     publication_date="2024-1-12" />
        </div>
    );
}

והרכיב מקבל אותם ומזריק ל-JSX עם { }:

js · Article.js
import './Article.css'
import { ReactComponent as Umbrella } from './umbrella.svg';

function Article({ id, title, author, category, publication_date }) {
    return (
        <a href={"article.html?id=" + id}>
            <article>
                <h3>{title}</h3>
                <span>{author}, <time>{publication_date}</time></span>
                <div>{category}</div>
                <Umbrella />
            </article>
        </a>
    );
}

export default Article;
🔤
תחביר ה-props
  • הרכיב מקבל אובייקט אחד — props. כאן השתמשנו ב-destructuring כדי "לפרק" אותו ישירות לפרמטרים: function Article({ id, title, ... }).
  • בתוך ה-JSX, {title} מזריק את הערך — בדיוק כמו <%= foo.title %> ב-EJS מהרצאה 7!
  • גם בתוך מאפיין אפשר להשתמש ב-{ }: href={"article.html?id=" + id}.

20. רשימת רכיבים מתוך data — map

🧭
קונטקסט עדיין כתבנו כל <Article> ביד. אבל בדרך כלל הנתונים מגיעים ממקור נתונים (קובץ/שרת/מסד), ואנחנו רוצים לייצר רכיב לכל פריט אוטומטית. כאן נכנס map.

הנתונים בקובץ JSON (זוכרים את ה-XML מההתחלה? אותן כתבות, פורמט אחר):

json · src/data/db.json
[
    {
        "id" : 1,
        "title" : "World's Largest Foo Created in Foobar Event",
        "author" : "Baz Caz",
        "category" : "Mystery",
        "publication_date" : "2024-01-11",
        "icon" : "umbrella.png"
    },
    {
        "id" : 2,
        "title" : "Unicorn Called Bar in West of Foobar",
        "author" : "Grace Lewis",
        "category" : "Nature",
        "publication_date" : "2024-01-10",
        "icon" : "sunset.png"
    }
]

מייבאים את ה-data, ומריצים map שממיר כל אובייקט לרכיב <Article>:

js · App.js
import './App.css';
import Article from '../article/Article'
import articles from '../data/db.json'

function App() {
    return (
        <div className="App">
            {
                articles.map((article) =>
                    <Article {...article} />
                )
            }
        </div>
    );
}

export default App;
🔤
שני תחבירים חשובים
  • articles.map((article) => <Article ... />) — map עובר על המערך ומחזיר מערך חדש, כאן מערך של רכיבי <Article>. React יודע לרנדר מערך של רכיבים.
  • {...article} — spread של props: "פזר את כל מפתחות האובייקט כ-props". {...article} שקול ל-id={article.id} title={article.title} ... — קיצור אדיר.
🔗
השוו ל-vanilla מסעיף 14 זוכרים את הכפילות הנוראית? עכשיו שורה אחת של map מייצרת אינסוף כתבות מ-data. הוסיפו כתבה ל-JSON — היא תופיע אוטומטית. זה מה ש-React פותר.

21. State ו-useState — UI שמגיב לשינוי

🧭
קונטקסט — הקסם האמיתי עד כה ה-data היה קבוע. אבל ממשק אמיתי משתנה — מוסיפים כתבה, מוחקים, עורכים. ב-vanilla היינו צריכים לעדכן את ה-DOM ידנית. ב-React מספיק לעדכן את ה-state, ו-React מרנדר מחדש את מה שצריך אוטומטית. הכלי לכך: ה-Hook בשם useState.

מגדירים state עם רשימת הכתבות ההתחלתית:

js · App.js
import './App.css';
import Article from '../article/Article'
import articles from '../data/db.json'
import { useState } from 'react';

function App() {
    const [articlesList, setArticleList] = useState(articles)

    return (
        <div className="App">
            { articlesList.map((article) => <Article {...article} />) }
        </div>
    );
}

export default App;
🔤
useState — איך קוראים את השורה const [articlesList, setArticleList] = useState(articles):
  • useState(articles) — מאתחל state בערך ההתחלתי articles.
  • מחזיר זוג (array destructuring): articlesList = הערך הנוכחי, setArticleList = הפונקציה לעדכון הערך.
  • חוק ברזל: משנים state רק דרך פונקציית ה-set — לעולם לא בהשמה ישירה. זה מה שמסמן ל-React לרנדר מחדש.

עכשיו נוסיף כפתור Add שמוסיף כתבה. שמים handler עם onClick (האירועים מחלק ד'!):

js · App.js — גרסה סופית
import './App.css';
import Article from '../article/Article'
import articles from '../data/db.json'
import { useState } from 'react';

function App() {
    const [articlesList, setArticleList] = useState(articles)

    const addArticle = () => {
        const article = {
            "id" : 3,
            "title" : "Wowwwww! amazing!!!!",
            "author" : "aaa bbb",
            "category" : "Guess",
            "publication_date" : "2024-01-12"
        }
        setArticleList([...articlesList, article])
    }

    return (
        <div className="App">
            <button onClick={addArticle}>Add</button>
            { articlesList.map((article) => <Article {...article} />) }
        </div>
    );
}

export default App;
🔤
מה קורה בלחיצה?
  • onClick={addArticle} — רושמים את הפונקציה כ-handler (בלי סוגריים! המלכודת מסעיף 13).
  • setArticleList([...articlesList, article]) — יוצרים מערך חדש = כל הישנים (...articlesList) + החדש. לא משנים את הקיים — יוצרים חדש.
  • ברגע שה-state השתנה → React מרנדר מחדש את App → ה-map רץ שוב → הכתבה החדשה מופיעה. בלי שנגענו ב-DOM בכלל.
🎯
זה כל הרעיון של React במשפט אחד ה-UI הוא פונקציה של ה-state. אתם לא מעדכנים את המסך — אתם מעדכנים נתונים, ו-React דואג שהמסך תמיד ישקף אותם. זה בדיוק מה שהיה כל כך קשה ב-vanilla מסעיף 14, וזו הסיבה שה-frameworks המודרניים השתלטו על עולם ה-Web.

★ סיכום ההרצאה

🎯
התמונה המלאה
  • GUI = עץ של אובייקטים. אפשר לבנות ב-code, אבל עדיף ב-markup כדי להפריד תיאור מלוגיקה (SRP).
  • Markup: XML (תגיות שאתה ממציא, לנתונים/Android), HTML (תגיות קבועות, ל-Web). אותו רעיון: עץ של אלמנטים.
  • אותם רכיבים בכל מקום: תווית/קלט/כפתור/container — Android, Web ו-.NET רק משנים שמות.
  • Styling: תמיד אותו מסע — inline → סגנון מרוכז עם שם → הפניה. CSS class ≈ Android @style ≈ .NET StaticResource. וב-CSS — specificity מכריע התנגשויות (inline > id > class > type, ו-!important ג'וקר).
  • Events: Listener pattern — נרשמים מראש ל-event, מספקים handler. addEventListener ≈ Android setOnClickListener ≈ React onClick. תמיד: מעבירים פונקציה, לא קוראים לה.
  • למה Frameworks: UI גולמי = שכפול קוד ועדכון DOM ידני. React פותר: רכיבים לשימוש חוזר (components + props), data → רכיבים ב-map, ו-state (useState) שגורם לרינדור אוטומטי.
  • הרעיון המרכזי של React: ה-UI הוא פונקציה של ה-state. JSX (דרך Babel/transpilation) מאפשר לכתוב את התצוגה כמו HTML בתוך JS.
🔗
חיבורים להרצאות קודמות Foo News מלווה אותנו: XML (כאן) → Express/EJS (הרצאה 7) → React (כאן). ה-{title} ב-JSX הוא ה-<%= %> מ-EJS. ה-onClick={fn} בלי סוגריים הוא המלכודת מהרצאה 6 ומ-Express. הפרדת UI/לוגיקה היא ה-SRP מהרצאה 2. ?id=1 הוא ה-query string מהרצאה 5/7.
