1. מוטיבציה — אליס, בוב והבעיה

🧭
קונטקסט — למה מתחילים מסיפור? לפני שצוללים למונחים טכניים, ההרצאה בונה אינטואיציה דרך דוגמה: אליס רוצה לשלוח הודעה לבוב. נראה שזה פשוט — אבל ככל שנרצה "יותר" (תמונות, וידאו, הרבה משתתפים), נגלה שצריך מבנה. כל קושי שנפגוש כאן יוליד פתרון בהמשך ההרצאה.

התרחיש האידיאלי: אליס מקלידה "Hi" במחשב שלה, ההודעה עוברת בכבל למחשב של בוב, ובוב רואה "From Alice: Hi". במצב הזה אליס ובוב פשוט מחברים את המחשבים ישירות זה לזה בכבל. זה עובד טוב כי:

  • הם היחידים שמנסים לתקשר — אין למי "להתבלבל".
  • הם קרובים פיזית זה לזה — כבל אחד מספיק.

כשהאפליקציה תומכת רק בטקסט, התקשורת פשוטה מאוד:

  1. אליס "דוחפת" (pushes) את הטקסט אל הכבל.
  2. הטקסט נוסע לאורך הכבל אל הקצה השני.
  3. בוב מקבל את הטקסט שמגיע מהכבל ומציג אותו.
🤔
אבל מה אם נרצה יותר? מה קורה אם אליס רוצה לשלוח תמונה או וידאו ולא רק טקסט? עכשיו בוב צריך דרך להבין איזה סוג מידע הגיע ואיך לעבד אותו — וכדי שזה יקרה, אליס צריכה להרכיב הודעה מתאימה שמכריזה על סוג התוכן. זה בדיוק מה שמוביל אותנו למושג הבא: פרוטוקול.

2. פרוטוקול — שפת התקשורת

🧭
קונטקסט כדי ששני צדדים יוכלו לתקשר על סוגי מידע שונים, הם חייבים להסכים מראש על "שפה" משותפת — אילו הודעות קיימות, איך הן בנויות, ומה המשמעות שלהן. ה"שפה" הזו נקראת פרוטוקול.

פרוטוקול הוא אוסף של חוקים דיגיטליים להחלפת מידע. פרוטוקול מגדיר שלושה דברים:

  • Types — אילו סוגי הודעות קיימים.
  • Format (Syntax) — מהו המבנה / התחביר של ההודעות (איך הביטים מסודרים).
  • Semantics — מהי המשמעות של התחביר, כולל מתי לשלוח מה.

דוגמה למבנה הודעה (message format):

┌─────────────── The entire message format ───────────────┐
│                                                          │
│   ┌──── Header part ────┐   ┌──────── Body part ───────┐ │
│   ┌────────┬────────────┐   ┌──────────────────────────┐ │
│   │  Type  │  Sub-type  │   │         Content          │ │
│   └────────┴────────────┘   └──────────────────────────┘ │
└──────────────────────────────────────────────────────────┘
   └──────────── Syntax: חלוקת ההודעה לשדות ─────────────┘

והסמנטיקה (מה כל שדה אומר):

שדהמשמעותדוגמאות
Typeסוג התוכן הכלליtext, image, video
Sub-typeהקידוד / הפורמט המדויקgzip, png
Contentתוכן ההודעה עצמוהבייטים של התמונה
💡
Syntax מול Semantics — ההבדל בלב הפרק Syntax הוא הצורה: "להודעה יש שדה Type ואז שדה Sub-type ואז Content". Semantics הוא הפירוש: "אם כתוב image ב-Type ו-png ב-Sub-type — תפענח את ה-Content כתמונת PNG". אותו רצף ביטים בלי הסכמה על סמנטיקה הוא חסר משמעות. כשבוב מקבל הודעה כמו [Image | png | H4sIAAf…], הוא יודע להציג תמונה במקום אוסף תווים אקראי.

3. למה לא מחברים את כולם ישירות — ושכבות (Layers)

🧭
קונטקסט עד עכשיו דיברנו על אליס ובוב בלבד. אבל מה אם רוצים לתקשר עם הרבה אנשים? הפתרון של "כבל ישיר לכל אחד" קורס מיד — וזה מה שמכריח אותנו לחשוב בצורה מודולרית, כלומר בשכבות.

הבעיה בחיבור ישיר של כולם לכולם:

  • זה יוצר בלגן — כבלים בכל מקום.
  • זה מוגבל — מחשב בודד היה צריך המון כרטיסי רשת (אחד לכל מחבר).

הפתרון ההנדסי הקלאסי לבעיה מורכבת: לפרק אותה לחלקים קטנים ופשוטים יותר, כל אחד אחראי על היבט אחר. את הפירוק הזה אנחנו עושים לשכבות (layers):

  • כל שכבה אחראית על היבט מסוים של המשימה הגדולה.
  • שכבות מקבלות ומספקות שירותים לשכבות אחרות (כל שכבה נשענת על זו שמתחתיה ומשרתת את זו שמעליה).
🏢
דוגמה משלי — משלוח חבילה בדואר תחשוב על שליחת חבילה: אתה כותב מכתב (שכבת אפליקציה), פקיד הדואר שם אותו במעטפה ומדביק בול (שכבת תעבורה), מרכז המיון מחליט לאיזו עיר לשלוח (שכבת רשת), והנהג מעביר בפועל מתחנה לתחנה (שכבת קישור). אף אחד לא צריך לדעת מה השני עושה בפנים — כל שכבה רק "מוסרת" לשכבה הבאה. בדיוק כך עובדת רשת מחשבים.

4. מודלי השכבות: OSI מול TCP/IP

🧭
קונטקסט קיימים שני "מפות דרכים" מקובלות לחלוקה לשכבות. OSI הוא המודל התיאורטי המלא (7 שכבות), ו-TCP/IP הוא המודל המעשי שעליו האינטרנט באמת רץ (5 שכבות בגרסה שנלמדת כאן). ה-OSI מאחד מספר שכבות עליונות אל תוך שכבת ה-Application של TCP/IP. 
      OSI Model                 TCP/IP Model
   ┌──────────────┐          ┌──────────────┐
   │ Application  │ ┐        │              │
   ├──────────────┤ │        │              │
   │ Presentation │ ├──────▶ │ Application  │
   ├──────────────┤ │        │              │
   │   Session    │ ┘        │              │
   ├──────────────┤          ├──────────────┤
   │  Transport   │ ───────▶ │  Transport   │
   ├──────────────┤          ├──────────────┤
   │   Network    │ ───────▶ │   Network    │
   ├──────────────┤          ├──────────────┤
   │     Link     │ ───────▶ │     Link     │
   ├──────────────┤          ├──────────────┤
   │   Physical   │ ───────▶ │   Physical   │
   └──────────────┘          └──────────────┘

תפקיד כל שכבה (במודל TCP/IP):

שכבהאחראית על...
Applicationפרוטוקולים לשליחת מידע בין אפליקציות רשת (HTTP, DNS וכו').
Transportפרוטוקולים לשליחת מידע בין תהליכים (processes) — כאן חיים UDP ו-TCP.
Networkפרוטוקולים לשליחת מידע ממקור ליעד (ניתוב בין רשתות) — כאן חי ה-IP.
Linkפרוטוקולים לשליחת מידע בין שכנים ברשת (מכשירים מחוברים ישירות).
Physicalאיך מעבירים בפועל ביטים "על הכבל" (חשמל, אור, גלי רדיו).
📌
איך זוכרים את הסדר מלמעלה למטה ב-TCP/IP: Application → Transport → Network → Link → Physical. ככל שיורדים — מתרחקים מהמשתמש ומתקרבים לחומרה. נשתמש בידע הזה מיד כשנבין איך כל שכבה מוסיפה את ה"מעטפה" שלה.

5. אנקפסולציה — איך המידע נארז שכבה-שכבה

🧭
קונטקסט — הרעיון המרכזי של ההרצאה זהו אולי החלק החשוב ביותר. כשמידע יורד מהאפליקציה אל הכבל, כל שכבה עוטפת אותו ב"מעטפה" משלה — מוסיפה כותרת (Header) עם המידע שהיא צריכה. בצד המקבל קורה התהליך ההפוך: כל שכבה מקלפת את המעטפה שלה לפני שהיא מעבירה למעלה. זה נקרא אנקפסולציה (encapsulation) ו-דה-אנקפסולציה.

בצד השולח (Alice), המידע יורד מלמעלה למטה וכל שכבה מוסיפה כותרת (H = Header) ושכבת הקישור מוסיפה גם זנב (T = Trailer):

Application │                         │  Data  │
Transport   │                  │ H_T │  Data  │   →  Segment / Datagram
Network     │           │ H_N │ H_T │  Data  │   →  Packet
Link        │ │ H_L │ H_N │ H_T │  Data  │ T_L │  →  Frame
Physical    0101010100011010111110101010...        →  Bits on the wire

בצד המקבל (Bob) זה קורה הפוך — הביטים עולים מלמטה למעלה, וכל שכבה מסירה (decapsulates) את הכותרת שלה עד שנשאר ה-Data המקורי שמגיע לאפליקציה.

השמות של היחידה בכל שכבה (מונחים חשובים מאוד למבחן):

שכבהשם היחידה (PDU)
TransportSegment (ב-TCP) / Datagram (ב-UDP)
NetworkPacket
LinkFrame
PhysicalBits
🪆
אנלוגיה — בובת מטריושקה / מעטפות בתוך מעטפות דמיין מכתב (Data) שאתה שם בתוך מעטפה (Transport), את המעטפה הזו בתוך מעטפה גדולה יותר (Network), ואותה בתוך עוד מעטפה (Link). כל "פותח" בצד השני מסיר מעטפה אחת בלבד — את שלו — ומעביר הלאה. אף שכבה לא נוגעת בכותרות של שכבה אחרת. זו הסיבה שהשכבות עצמאיות וניתנות להחלפה.

6. פורטים (Ports)

🧭
קונטקסט — בעיית שכבת התעבורה על מחשב אחד רצות בו-זמנית הרבה אפליקציות רשת (דפדפן, משחק, תוכנת שיחות...). כשהודעה מגיעה למחשב — למי מהן היא שייכת? הפתרון הוא פורט: מספר שמזהה את האפליקציה הספציפית. זוהי העבודה של שכבת ה-Transport.
  • תוכנית שרצה במחשב נקראת תהליך (process).
  • פנימית, מערכת ההפעלה מזהה תהליכים לפי process id.
  • חיצונית, אפליקציות רשת מזוהות לפי פורטים (ports).
  • טווח מספרי הפורטים: 0 – 65,535 (כלומר 2¹⁶ ערכים — פורט הוא מספר של 16 ביט).
  • הטווח 0 – 1,023 שמור לשירותים מוכרים (well-known ports), למשל 80 ל-HTTP, 443 ל-HTTPS, 22 ל-SSH.

אפליקציה "מאזינה" (listens) על פורט מסוים. לדוגמה: שרת מאזין על פורט 5656, ובמחשב הלקוח אפליקציה אחת מאזינה על 2222 ואחרת על 12345. הכותרת של שכבת התעבורה (H_T) מכילה פורט מקור ופורט יעד:

            ┌─────────── H_T (Transport header) ───────────┐
┌────────┐  ┌──────────┬──────────────┐
│  Data  │  │   5656   │    12345     │
└────────┘  └──────────┴──────────────┘
              Source port   Destination port

כשהמידע מגיע לצד השני, שכבת התעבורה קוראת את פורט היעד (12345) ויודעת בדיוק לאיזו אפליקציה למסור את ה-Data. התהליך הזה נקרא demultiplexing.

🏠
אנלוגיה — כתובת IP = בניין, פורט = מספר דירה ה-IP (שנכיר מיד) מביא את ההודעה אל הבניין הנכון (המחשב). הפורט אומר לאיזו דירה (אפליקציה) למסור אותה בתוך הבניין. בלי פורט, ההודעה הייתה מגיעה למחשב אבל אף אחד לא היה יודע למי היא מיועדת.

7. כתובת IP

🧭
קונטקסט — בעיית שכבת הרשת הפורט מזהה אפליקציה בתוך מחשב, אבל קודם צריך להגיע למחשב הנכון. במציאות המחשבים לא מחוברים ישירות לאותה רשת — הם מפוזרים בעולם. צריך דרך לזהות כל מחשב באופן ייחודי כדי למסור לו הודעה. זה תפקיד כתובת ה-IP, בשכבת הרשת.
  • כתובת באורך 32 ביט (IPv4) או 128 ביט (IPv6).
  • זוהי כתובת לוגית — מאפשרת חלוקה לרשתות והיררכיה (בניגוד לכתובת פיזית/חומרה).
  • שייכת לשכבת הרשת (internet layer).
  • מכשירים משתמשים בה כדי למסור מידע בין רשתות, ומאפשרת subnetting (חלוקת רשת לתת-רשתות).
  • דוגמה לכתובת: 172.16.254.1

בדיוק כמו הפורטים, גם כאן יש מקור ויעד — אבל הפעם בכותרת של שכבת הרשת (H_N):

 Alice: 1.2.3.4                              Bob: 188.10.6.1

            ┌─────────── H_N (Network header) ───────────┐
┌────────┐  ┌──────────┬──────────────┐
│ Data_T │  │ 1.2.3.4  │  188.10.6.1  │
└────────┘  └──────────┴──────────────┘
              Source IP    Destination IP

  → כשהחבילה מגיעה לבוב, הוא בודק שה-Destination IP = שלו ✓
    ורק אז מעלה את המידע לשכבות שמעליו.
🔢
למה IPv6? — דוגמה/הקשר משלי ל-IPv4 יש רק 2³² ≈ 4.3 מיליארד כתובות — וזה נגמר! העולם מתמלא במכשירים מחוברים (טלפונים, מקררים, רכבים...). IPv6 עם 128 ביט נותן 2¹²⁸ כתובות — מספר עצום (בערך מספר גרגרי החול בכדור הארץ, בריבוע). לכן המעבר ל-IPv6 הוא הכרחי לטווח הארוך.
🧭
לסכם את ה"כתובת המלאה" כדי שהודעה תגיע ליעד מדויק צריך שני דברים יחד: IP (איזה מחשב — שכבת רשת) + Port (איזו אפליקציה במחשב — שכבת תעבורה). הצמד (IP, Port) הוא בדיוק מה שמגדיר נקודת קצה לתקשורת — ובהמשך נראה שזה בדיוק מה שסוקט מייצג.

8. פרוטוקולי תעבורה: UDP מול TCP

🧭
קונטקסט בשכבת התעבורה יש שתי אפשרויות עיקריות, ושתיהן משתמשות בפורטים. ההבדל ביניהן הוא הבטחות מול מהירות: TCP מבטיח שהכל יגיע נכון ובסדר, אבל איטי יותר; UDP מהיר אבל "זורק ושוכח". הבחירה ביניהן תלויה באפליקציה.

מתי בוחרים מה? — דוגמה/הקשר משלי TCP: כשכל בייט קריטי — טעינת דף אינטרנט, הורדת קובץ, הודעות צ'אט. עדיף להמתין ולקבל הכל נכון.
UDP: כשמהירות חשובה יותר משלמות — שיחות וידאו, סטרימינג, משחקים מקוונים, DNS. אם פריים אחד אבד בשיחת זום — עדיף "לדלג" עליו מאשר לעצור הכל ולחכות לשליחה חוזרת.
🔤
"Delivers Messages מול Bytes" — נקודה שחשוב להבין ב-UDP כל sendto שולח הודעה אחת שלמה (datagram) — מה ששלחת זה מה שמתקבל, כיחידה. ב-TCP המידע הוא זרם בייטים רציף: אם תשלח "Hello" ואז "World", בצד השני הם עלולים להתקבל כ-"HelloWorld" בקריאה אחת, או מפוצלים אחרת. לכן ב-TCP צריך פרוטוקול אפליקטיבי שיודע איפה הודעה אחת נגמרת והבאה מתחילה.

9. מהו סוקט (Socket)

🧭
קונטקסט — מהתיאוריה אל הקוד כל החלק הראשון תיאר איך רשת עובדת. עכשיו נשאל: איך אנחנו, כמתכנתים, ניגשים לכל זה? לא נכתוב כל שכבה בעצמנו — מערכת ההפעלה עושה את זה. היא נותנת לנו ממשק נקי: הסוקט.
  • סוקט הוא נקודת קצה לתקשורת (a communication's endpoint).
  • Socket API הוא אוסף הקריאות שמאפשר תקשורת בין-תהליכית (IPC — Inter-Process Communication) מבוססת-סוקטים.

מבחינת מיקום, הסוקט יושב בדיוק בין שכבת האפליקציה לשכבת התעבורה — הוא ה"דלת" שדרכה האפליקציה שלנו מוסרת מידע למערכת ההפעלה, וזו כבר דואגת לכל השאר (פורטים, IP, אנקפסולציה...):

 ┌───────────────┐
 │  Application  │   ← הקוד שלנו
 ├───────────────┤
 │  ▓▓ Socket ▓▓ │   ← הממשק (API)
 ├───────────────┤
 │   Transport   │   ← UDP / TCP (מערכת ההפעלה)
 ├───────────────┤
 │    Network    │
 └───────────────┘

זוכרים את הצמד (IP, Port) מהפרק הקודם? סוקט הוא בדיוק הייצוג התוכנתי שלו. בהמשך נראה שסוקט UDP מזוהה ע"י (Src IP, Src Port).

10. זרימת UDP — לקוח ושרת

🧭
קונטקסט UDP הוא הפרוטוקול הפשוט יותר (ללא חיבור), אז נתחיל ממנו. נלמד את רצף הקריאות שכל צד מבצע. שים לב שאין כאן "חיבור" — הלקוח פשוט שולח, והשרת פשוט מקבל. כל הודעה עומדת בפני עצמה. 
      CLIENT                         SERVER
   ┌──────────┐                   ┌──────────┐
   │  socket  │                   │  socket  │
   └────┬─────┘                   └────┬─────┘
        │                              │
        │                         ┌────┴─────┐
        │                         │   bind   │   ← "אני על פורט X"
        │                         └────┬─────┘
   ┌────┴─────┐                   ┌────┴─────┐
   │  sendto  │ ────────────────▶ │ recvfrom │ ◀─┐  (loop:
   └────┬─────┘                   └────┬─────┘   │   ממתין
   ┌────┴─────┐                   ┌────┴─────┐   │   להודעה
   │ recvfrom │ ◀──────────────── │  sendto  │ ──┘   הבאה)
   └────┬─────┘                   └──────────┘
   ┌────┴─────┐
   │  close   │
   └──────────┘
קריאהמה היא עושה
socket()יוצר את נקודת הקצה (הסוקט).
bind()קושר את הסוקט לכתובת ופורט מקומיים — נחוץ בשרת כדי שלקוחות ידעו לאן לשלוח.
sendto()שולח הודעה אל כתובת+פורט יעד (היעד מצוין בכל קריאה — אין חיבור קבוע).
recvfrom()ממתין ומקבל הודעה, ומחזיר גם מי שלח (כדי שנוכל לענות).
close()סוגר את הסוקט ומשחרר את המשאבים.

11. UDP Echo ב-Python

🧭
קונטקסט — מהו "Echo server"? שרת Echo הוא הדוגמה הקלאסית ללימוד סוקטים: השרת פשוט מחזיר ללקוח את מה שהלקוח שלח (כאן — באותיות גדולות, כדי שנראה שמשהו קרה). זה מאפשר להתמקד במנגנון התקשורת בלי לוגיקה מסובכת.

שרת UDP (Python)

python · udp_echo_server.py
from socket import socket, AF_INET, SOCK_DGRAM

s = socket(AF_INET, SOCK_DGRAM)

src_ip = ''
src_port = 12345
s.bind((src_ip, src_port))

while True:
    data, sender_info = s.recvfrom(2048)
    s.sendto(data.upper(), sender_info)

הסבר שורה-שורה:

קודמה קורה
socket(AF_INET, SOCK_DGRAM)יוצר סוקט. AF_INET = משפחת כתובות IPv4. SOCK_DGRAM = סוג datagram → UDP. (סוקט = נקודת קצה לתקשורת.)
src_ip = ''מחרוזת ריקה = "האזן על כל כתובות ה-IP של המחשב" (כל כרטיסי הרשת).
s.bind((src_ip, src_port))קושר את הסוקט לכתובת+פורט. סוקט UDP מזוהה ע"י (Src IP, Src port). שים לב: הארגומנט הוא tuple אחד (ip, port), ומכאן הסוגריים הכפולים.
while True:לולאה אינסופית — השרת לעולם לא "מסיים", הוא תמיד מוכן ללקוח הבא.
data, sender_info = s.recvfrom(2048)ממתין עד שמגיעה הודעה. מחזיר שני דברים: ה-data שהתקבל, ו-sender_info = הזוג (IP, port) של מי ששלח. 2048 = כמות הבייטים המקסימלית לקריאה.
s.sendto(data.upper(), sender_info)שולח את התשובה חזרה אל השולח (כי שמרנו את sender_info). .upper() הופך לאותיות גדולות — ה-"echo".

לקוח UDP (Python)

python · udp_echo_client.py

s = socket(AF_INET, SOCK_DGRAM)

dst_ip = '127.0.0.1'
dst_port = 12345

s.sendto('Hello', (dst_ip, dst_port))
data, sender_info = s.recvfrom(2048)

קודמה קורה
dst_ip = '127.0.0.1'כתובת היעד. 127.0.0.1 = localhost (loopback) — "המחשב הזה עצמו". שימושי לבדיקה שהשרת והלקוח רצים על אותו מחשב.
s.sendto('Hello', (dst_ip, dst_port))שולח "Hello" אל היעד. בלקוח UDP אין צורך ב-bind — כשלא מציינים פורט מקור, מערכת ההפעלה בוחרת אוטומטית פורט פנוי.
data, sender_info = s.recvfrom(2048)ממתין לתשובה מהשרת (ה-"HELLO" באותיות גדולות).
s.close()סוגר את הסוקט. הלקוח סיים — בניגוד לשרת שממשיך בלולאה.
📦
איך זה נראה בפועל — אנקפסולציה חוזרת! כשהלקוח קורא ל-sendto('Hello', ...), ה-'Hello' יורד דרך השכבות וכל אחת עוטפת אותו (בדיוק כמו בפרק 5): 
Application:                                  │ Hello │
Transport:                  │ Dst Port=12345 │ Hello │
Network:    │ Dst IP=127.0.0.1 │ UDP header   │ Hello │
זו בדיוק התיאוריה מחלק א' — עכשיו אתה רואה אותה מתרחשת בקוד אמיתי.

12. UDP Echo ב-C++

🧭
קונטקסט אותו רעיון בדיוק, אבל ב-C++ עם ה-Socket API של מערכת ההפעלה (POSIX/Linux). כאן הדברים מפורשים יותר — אנחנו מנהלים בעצמנו מבני-נתונים, גודל buffer, ובדיקת שגיאות. זה נראה ארוך יותר, אבל זה בדיוק אותו רצף קריאות.

שרת UDP מלא (C++)

cpp · udp_echo_server.cpp
#include <iostream>
#include <sys/socket.h>
#include <stdio.h>
#include <netinet/in.h>
#include <arpa/inet.h>
#include <unistd.h>
#include <string.h>
using namespace std;

int main() {
    const int server_port = 5555;

    int sock = socket(AF_INET, SOCK_DGRAM, 0);
    if (sock < 0) {
        perror("error creating socket");
    }

    struct sockaddr_in sin;
    memset(&sin, 0, sizeof(sin));
    sin.sin_family = AF_INET;
    sin.sin_addr.s_addr = INADDR_ANY;
    sin.sin_port = htons(server_port);

    if (bind(sock, (struct sockaddr *) &sin, sizeof(sin)) < 0) {
        perror("error binding to socket");
    }

    struct sockaddr_in from;
    unsigned int from_len = sizeof(struct sockaddr_in);
    char buffer[4096];

    int bytes = recvfrom(sock, buffer, sizeof(buffer), 0,
                         (struct sockaddr *) &from, &from_len);
    if (bytes < 0) {
        perror("error reading from socket");
    }
    cout << "The client sent: " << buffer << endl;

    int sent_bytes = sendto(sock, buffer, bytes, 0,
                            (struct sockaddr *) &from, sizeof(from));
    if (sent_bytes < 0) {
        perror("error writing to socket");
    }

    close(sock);
    return 0;
}

פירוק החלקים החשובים (ובדיוק אותן 4 קריאות מהדיאגרמה):

cpp · socket()
int sock = socket(AF_INET, SOCK_DGRAM, 0);

    perror("error creating socket");
}

יוצר סוקט ומחזיר file descriptor (מספר שלם — ב-Unix הכל הוא קובץ, גם סוקט). אם הערך שלילי — קרתה שגיאה. SOCK_DGRAM = UDP, והארגומנט השלישי 0 = פרוטוקול ברירת מחדל למשפחה הזו.

cpp · bind()

sin.sin_port = htons(server_port);

    perror("error binding to socket");
}
🔤
הסבר סינטקס — מבנה הכתובת ב-C++
  • struct sockaddr_in — מבנה שמחזיק כתובת IPv4 + פורט.
  • memset(&sin, 0, sizeof(sin)) — מאפס את כל הבייטים של המבנה לפני השימוש (הרגל חשוב — מונע "זבל" בשדות שלא מילאנו).
  • sin.sin_family = AF_INET — מצהיר שזו כתובת IPv4.
  • INADDR_ANY — המקבילה ל-'' בפייתון: "האזן על כל הכתובות".
  • htons(server_port) — Host TO Network Short. ממיר את מספר הפורט מסדר-הבייטים של המחשב לסדר-הבייטים של הרשת (big-endian). חובה! מחשבים שונים שומרים מספרים בסדר שונה, והרשת דורשת סדר אחיד.
  • (struct sockaddr *) &sin — casting: ה-API הישן מצפה למצביע לטיפוס הגנרי sockaddr, אז אנחנו "ממירים" אליו את המצביע ל-sockaddr_in הספציפי שלנו.
cpp · recvfrom()
struct sockaddr_in from;
unsigned int from_len = sizeof(struct sockaddr_in);
char buffer[4096];

int bytes = recvfrom(sock, buffer, sizeof(buffer), 0,
                     (struct sockaddr *) &from, &from_len);
if (bytes < 0) {
    perror("error reading from socket");
}

ממתין להודעה. ה-buffer יתמלא במידע, from יתמלא בכתובת השולח (כדי שנדע למי לענות), והערך המוחזר bytes = כמה בייטים התקבלו בפועל.

cpp · sendto()
int sent_bytes = sendto(sock, buffer, bytes, 0,
                        (struct sockaddr *) &from, sizeof(from));
if (sent_bytes < 0) {
    perror("error writing to socket");
}

שולח bytes בייטים מתוך ה-buffer בחזרה אל from (השולח המקורי). זה ה-echo.

לקוח UDP מלא (C++)

cpp · udp_echo_client.cpp
#include <iostream>
#include <sys/socket.h>
#include <stdio.h>
#include <netinet/in.h>
#include <arpa/inet.h>
#include <unistd.h>
#include <string.h>
using namespace std;

int main() {
    const char* ip_address = "127.0.0.1";
    const int port_no = 5555;

    int sock = socket(AF_INET, SOCK_DGRAM, 0);
    if (sock < 0) {
        perror("error creating socket");
    }

    struct sockaddr_in sin;
    memset(&sin, 0, sizeof(sin));
    sin.sin_family = AF_INET;
    sin.sin_addr.s_addr = inet_addr(ip_address);
    sin.sin_port = htons(port_no);

    char data[] = "hello";
    int data_len = sizeof(data);
    int sent_bytes = sendto(sock, data, data_len, 0,
                            (struct sockaddr *) &sin, sizeof(sin));
    if (sent_bytes < 0) {
        perror("error writing to socket");
    }

    struct sockaddr_in from;
    unsigned int from_len = sizeof(struct sockaddr_in);
    char buffer[4096];
    int bytes = recvfrom(sock, buffer, sizeof(buffer), 0,
                         (struct sockaddr *) &from, &from_len);
    if (bytes < 0) {
        perror("error reading from socket");
    }
    cout << "The server sent: " << buffer << endl;

    close(sock);
    return 0;
}
🔤
ההבדל מהשרת — שתי שורות מפתח בלקוח אין bind ואין INADDR_ANY. במקום זה:
  • inet_addr("127.0.0.1") — ממיר מחרוזת IP (קריאה לאדם) למספר 32-ביט (הצורה שהרשת מבינה). זו המקבילה ל-dst_ip בפייתון.
  • הלקוח קורא ישר ל-sendto אל ה-sin של השרת — לא קושר את עצמו לפורט מסוים (מערכת ההפעלה תבחר).
⚠️
מלכודת נפוצה — cout << buffer ב-C++ מחרוזת חייבת להסתיים בתו '\0'. ה-recvfrom ממלא בייטים גולמיים אבל לא מוסיף '\0' אוטומטית. אם תדפיס cout << buffer ישירות, אתה עלול לראות "זבל" אחרי ההודעה. תיקון בטוח: לאפס את ה-buffer מראש (memset), או להשתמש במספר הבייטים המוחזר (buffer[bytes] = '\0';). זו נקודה חשובה שלא מופיעה במפורש במצגת אבל מבדילה קוד שעובד "במזל" מקוד נכון.

13. זרימת TCP — לקוח ושרת

🧭
קונטקסט — מה משתנה מ-UDP? TCP הוא מבוסס-חיבור. לפני ששולחים מידע, הלקוח והשרת "לוחצים ידיים" ופותחים חיבור יציב. לכן מופיעות פה קריאות חדשות: בלקוח connect, ובשרת listen ו-accept. אחרי שהחיבור פתוח, שולחים עם send/recv (בלי לציין יעד כל פעם — היעד כבר ידוע מהחיבור).

   ┌──────────┐                   ┌──────────┐

קריאה (חדשה ב-TCP)מה היא עושה
listen(n)הופך את הסוקט ל"מאזין" ומגדיר תור המתנה בגודל n לחיבורים נכנסים.
accept()חוסם עד שלקוח מתחבר, ואז מחזיר סוקט חדש ייעודי לאותו לקוח (סוקט המאזין נשאר פנוי ללקוחות נוספים).
connect()(בלקוח) פותח חיבור אל כתובת+פורט השרת.
send() / recv()שולח/מקבל על חיבור קיים — בלי לציין יעד בכל פעם (היעד ידוע מהחיבור).

14. TCP Echo ב-Python

שרת TCP (Python)

python · tcp_echo_server.py
import socket

server = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
server_ip = ''
server_port = 12345
server.bind((server_ip, server_port))
server.listen(5)

while True:
    client_socket, client_address = server.accept()
    print 'Connection from: ', client_address
    data = client_socket.recv(1024)
    while not data == '':
        print 'Received: ', data
        client_socket.send(data.upper())
        data = client_socket.recv(1024)

    print 'Client disconnected'
    client_socket.close()
קודמה קורה
SOCK_STREAMסוג stream → TCP (בניגוד ל-SOCK_DGRAM של UDP). זה ההבדל היחיד ביצירת הסוקט!
server.listen(5)מתחיל להאזין; עד 5 חיבורים ממתינים בתור.
client_socket, client_address = server.accept()ממתין ללקוח. מחזיר סוקט חדש (client_socket) שמייצג את החיבור הספציפי לאותו לקוח.
data = client_socket.recv(1024)קורא עד 1024 בייטים מהלקוח.
while not data == '':כל עוד מתקבל מידע — ממשיכים. כש-recv מחזיר מחרוזת ריקה, זה הסימן שהלקוח סגר את החיבור.
client_socket.send(data.upper())ה-echo — שולח חזרה באותיות גדולות (על סוקט החיבור, לא על סוקט השרת).

קודמה קורה
s.connect((dest_ip, dest_port))פותח חיבור אל השרת. זה החלק שלא קיים ב-UDP — מכאן והלאה היעד ידוע ולא צריך לציין אותו.
raw_input(...)(Python 2) קורא שורת טקסט מהמשתמש. הלקוח שולח הודעות עד שמקלידים quit.
s.send(msg) / s.recv(4096)שולח ומקבל על החיבור — בלי כתובת יעד, כי החיבור כבר קיים.
🐍
שים לב — זה קוד Python 2 המצגת משתמשת ב-print 'x' (בלי סוגריים) וב-raw_input — שניהם Python 2. ב-Python 3 זה יהיה print('x') ו-input(...), וגם צריך להעביר bytes ולא str (למשל s.send(msg.encode())). המנגנון זהה — רק התחביר השתנה בין הגרסאות.

15. TCP Echo ב-C++

🧭
קונטקסט השרת והלקוח ב-C++ זהים כמעט לחלוטין לגרסת ה-UDP — אותם socket, sockaddr_in, htons, bind. ההבדלים: SOCK_STREAM במקום SOCK_DGRAM, ובשרת מתווספות listen ו-accept, ובלקוח connect. וגם — משתמשים ב-send/recv במקום sendto/recvfrom.

שרת TCP (C++)

cpp · tcp_echo_server.cpp
int main() {
    const int server_port = 5555;
    int sock = socket(AF_INET, SOCK_STREAM, 0);
    if (sock < 0) {
        perror("error creating socket");
    }

    struct sockaddr_in sin;
    memset(&sin, 0, sizeof(sin));
    sin.sin_family = AF_INET;
    sin.sin_addr.s_addr = INADDR_ANY;
    sin.sin_port = htons(server_port);

    if (bind(sock, (struct sockaddr *) &sin, sizeof(sin)) < 0) {
        perror("error binding socket");
    }
    if (listen(sock, 5) < 0) {
        perror("error listening to a socket");
    }

    struct sockaddr_in client_sin;
    unsigned int addr_len = sizeof(client_sin);
    int client_sock = accept(sock, (struct sockaddr *) &client_sin, &addr_len);
    if (client_sock < 0) {
        perror("error accepting client");
    }

    char buffer[4096];
    int expected_data_len = sizeof(buffer);
    int read_bytes = recv(client_sock, buffer, expected_data_len, 0);
    if (read_bytes == 0) {
        // connection is closed
    } else if (read_bytes < 0) {
        // error
    } else {
        cout << buffer;
    }

    int sent_bytes = send(client_sock, buffer, read_bytes, 0);
    if (sent_bytes < 0) {
        perror("error sending to client");
    }

    close(sock);
    return 0;
}
🔤
שלוש הקריאות שמייחדות TCP בשרת
  • listen(sock, 5) — הופך לסוקט מאזין עם תור באורך 5.
  • accept(sock, ...) — מחזיר client_sock חדש לחיבור הספציפי. שים לב: recv/send פועלים על client_sock, לא על sock המקורי.
  • recv מחזיר 0 = הלקוח סגר את החיבור, ערך שלילי = שגיאה, ערך חיובי = מספר הבייטים שהתקבלו. שלושת המקרים מטופלים ב-if/else if/else.

לקוח TCP (C++)

cpp · tcp_echo_client.cpp
#include <iostream>
#include <sys/socket.h>
#include <stdio.h>
#include <netinet/in.h>
#include <arpa/inet.h>
#include <unistd.h>
#include <string.h>
using namespace std;

int main() {
    const char* ip_address = "127.0.0.1";
    const int port_no = 5555;

    int sock = socket(AF_INET, SOCK_STREAM, 0);
    if (sock < 0) {
        perror("error creating socket");
    }

    struct sockaddr_in sin;
    memset(&sin, 0, sizeof(sin));
    sin.sin_family = AF_INET;
    sin.sin_addr.s_addr = inet_addr(ip_address);
    sin.sin_port = htons(port_no);

    if (connect(sock, (struct sockaddr *) &sin, sizeof(sin)) < 0) {
        perror("error connecting to server");
    }

    char data_addr[] = "Im a message";
    int data_len = strlen(data_addr);
    int sent_bytes = send(sock, data_addr, data_len, 0);
    if (sent_bytes < 0) {
        // error
    }

    char buffer[4096];
    int expected_data_len = sizeof(buffer);
    int read_bytes = recv(sock, buffer, expected_data_len, 0);
    if (read_bytes == 0) {
        // connection is closed
    } else if (read_bytes < 0) {
        // error
    } else {
        cout << buffer;
    }

    close(sock);
    return 0;
}

הלקוח: יוצר סוקט TCP, מגדיר את כתובת השרת (inet_addr + htons), פותח חיבור עם connect, שולח הודעה עם send, מקבל תשובה עם recv, וסוגר. שים לב ל-strlen שמחשב את אורך המחרוזת (בלי ה-'\0') — בניגוד ל-sizeof שהיינו משתמשים בו עבור מערך.

🧪
איך מריצים את זה בפועל? — טיפ משלי פותחים שני טרמינלים. בראשון מריצים את השרת (הוא "נתקע" וממתין ב-accept/recvfrom). בשני מריצים את הלקוח. בגלל שהשתמשנו ב-127.0.0.1, שניהם רצים על אותו מחשב ומדברים דרך ה-loopback. אם תרצה להריץ על שני מחשבים שונים — תחליף את ה-IP של הלקוח לכתובת ה-IP האמיתית של השרת ברשת.

★ סיכום ההרצאה

🎯
התמונה המלאה — מהבעיה אל הקוד
  • פרוטוקול = חוקים משותפים (Types, Syntax, Semantics) שמאפשרים לשני צדדים להבין זה את זה.
  • שכבות מפרקות את המשימה המורכבת; OSI (7) מול TCP/IP (5).
  • אנקפסולציה — כל שכבה עוטפת בכותרת משלה: Segment/Datagram → Packet → Frame → Bits.
  • פורט מזהה אפליקציה במחשב (שכבת תעבורה); IP מזהה מחשב ברשת (שכבת רשת). הצמד (IP, Port) = יעד מלא.
  • UDP = מהיר, ללא חיבור, "best-effort". TCP = אמין, מבוסס-חיבור, אבל איטי יותר.
  • סוקט = נקודת קצה לתקשורת, הממשק שלנו אל הרשת. UDP: socket→bind→recvfrom/sendto. TCP: מוסיף listen/accept (שרת) ו-connect (לקוח).
  • זכור את הזיהוי הזה ב-API: SOCK_DGRAM=UDP, SOCK_STREAM=TCP; htons לפורט; INADDR_ANY בשרת מול inet_addr בלקוח.
