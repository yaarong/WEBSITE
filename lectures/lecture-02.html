1. אפליקציית News — הפרויקט לדוגמה

🧭
קונטקסט — למה מתחילים מדוגמה? כל החלק הזה נבנה סביב פרויקט אחד שמלווה אותנו: אפליקציית חדשות. נתחיל מקוד "שעובד" אבל מכוער, ולאורך ההרצאה נשפר אותו צעד-צעד. הרעיון: ללמוד עקרונות לא בהפשטה, אלא תוך כדי תיקון קוד אמיתי.

מטרות הפרויקט (Goals):

  • לפתח אפליקציית חדשות (News app).
  • בהמשך יהיו לנו כמה ממשקי משתמש (UIs) שונים.
  • נתנסה בהרבה טכנולוגיות.
  • המטרה: לכתוב קוד טוב, ביעילות — ולא רק קוד שעובד.

דרישות (Requirements): לאפליקציה יש תפריט שמאפשר: (1) להציג את כל הכתבות, (2) להוסיף כתבה חדשה. בעת הפעלה — מוצגות כל הכתבות הקיימות. הריצה נראית כך בטרמינל:

2. Add article

הקוד הראשוני — הכל בתוך main אחד

זהו הקוד שאיתו מתחילים. הוא עובד, אבל כפי שנראה מיד — הוא מלא בעיות:

cpp · main.cpp (הגרסה הראשונית)
#include <iostream>
#include <fstream>
using namespace std;

int main() {
    // הצגת כל הכתבות בהפעלה
    ifstream file("articles.txt");
    string line;
    while (getline(file, line)) {
        cout << line << endl;
    }
    file.close();

    int task;
    cout << "What would you like to do next?" << endl;
    cout << "1. Show all articles" << endl;
    cout << "2. Add article" << endl;
    cin >> task;

    while (true) {
        switch (task) {
            case 1: {
                ifstream file("articles.txt");
                string line;
                while (getline(file, line)) {
                    cout << line << endl;
                }
                file.close();
                break;
            }
            case 2: {
                string article;
                getline(cin, article);   // "בולע" את ה-\n שנשאר מ-cin >> task
                getline(cin, article);   // קורא את הכתבה בפועל

                ofstream file("articles.txt", ios_base::app);
                file << article << endl;
                file.close();
                break;
            }
            default: {
                cout << "Sorry, no can do" << endl;
                break;
            }
        }

        cout << "What would you like to do next?" << endl;
        cout << "1. Show all articles" << endl;
        cout << "2. Add article" << endl;
        cin >> task;
    }
}
🔤
סינטקס C++ שכדאי להכיר כאן ifstream = input file stream — קריאה מקובץ. ofstream = output — כתיבה לקובץ. ios_base::app = פתיחה במצב append (הוספה לסוף הקובץ במקום דריסה). getline(file, line) קורא שורה שלמה. cin >> task קורא מספר אבל משאיר את ה-\n בבאפר — ולכן צריך getline "מבזבז" כדי לנקות אותו (זה כבר ריח של קוד...).
⚠️
הקוד הזה מלא בעיות קוד הצגת הכתבות מופיע פעמיים (כפילות), קוד התפריט מופיע פעמיים, ה-main עושה הכל (קריאת קובץ + לוגיקה + UI), ואי אפשר לבדוק אותו. נשפר אותו צעד אחר צעד — תהליך שנקרא refactoring.

2. Refactoring — מהו

🔧
הגדרה Refactoring = שיפור המבנה הפנימי של הקוד בלי לשנות את ההתנהגות החיצונית שלו. הקוד ימשיך לעשות בדיוק את אותו דבר — אבל יהיה קריא, נקי וקל יותר לתחזוקה. עושים זאת בצעדים קטנים, אחד בכל פעם.

לאורך החלק הזה נשתמש בשתי גישות משלימות לזיהוי מה לשפר: (1) עקרונות עיצוב (DRY, KISS, SOLID...) שאומרים מה לשאוף אליו, ו-(2) code smells ("ריחות רעים") שמסמנים מה כנראה דורש תיקון.

3. עקרונות עיצוב: DRY, YAGNI, KISS

עיקרוןפירושהמשמעות
DRYDon't Repeat Yourselfאל תחזור על עצמך — כל ידע/לוגיקה צריך להופיע במקום אחד. כפילות = מקור לבאגים.
YAGNIYou Ain't Gonna Need Itאל תכתוב קוד "ליתר ביטחון" לפיצ'רים שאולי תצטרך. תבנה מה שצריך עכשיו.
KISSKeep It Simple, Sillyשמור על הפשטות. הפתרון הפשוט ביותר שעובד הוא בדרך כלל הטוב ביותר.

ויש עוד רבים. נתחיל מ-DRY כי הוא הכי בולט בקוד שלנו.

4. DRY בפועל

🧭
קונטקסט בקוד הראשוני, גוש "הדפסת התפריט" מופיע פעם בהתחלה ופעם בסוף הלולאה, וגוש "הצגת כתבות" מופיע ב-case 1 וגם לפני הלולאה. נסיר את הכפילות.

הטריק היפה: אם נאתחל int task = 1; ונשים את הדפסת התפריט רק בסוף הלולאה, אז האיטרציה הראשונה תבצע אוטומטית את case 1 (הצגת הכתבות) — ובכך נחסך גם הגוש הכפול שלפני הלולאה וגם הגוש הכפול של התפריט:

cpp · DRY fixed
int main() {
    int task = 1;          // מאותחל ל-1 → האיטרציה הראשונה מציגה את הכתבות

    while (true) {
        switch (task) {
            case 1: { /* ... הצגת כתבות ... */ break; }
            case 2: { /* ... הוספת כתבה   ... */ break; }
            default: {
                cout << "Sorry, no can do" << endl;
                break;
            }
        }

        // התפריט מופיע פעם אחת בלבד
        cout << "What would you like to do next?" << endl;
        cout << "1. Show all articles" << endl;
        cout << "2. Add article" << endl;
        cin >> task;
    }
}
💡
למה זה משנה (משלי) תחשבו שמחר רוצים לשנות את נוסח התפריט. בקוד הכפול — צריך לזכור לשנות בשני מקומות, ואם שוכחים אחד מהם נוצר באג. עם DRY — משנים במקום אחד, וזהו. כל כפילות היא חוב עתידי.

5. הערות (Comments)

🧭
קונטקסט חוסר הערות הוא code smell — אבל גם הערות מיותרות הן! נראה את שני הקצוות ואז את הדרך הנכונה.

הערות מיותרות (Pointless) — ❌

הערה שרק חוזרת על מה שהקוד כבר אומר בבירור היא רעש מיותר:

cpp · הערות מיותרות
// init task
int task = 1;

// loop forever
while (true) {
    // branch task
    switch (task) {
        case 1: {
            // handle file
            ifstream file("articles.txt");

הערות כמו // loop forever ליד while(true) לא מוסיפות שום מידע — הקוד כבר אומר את זה.

הערות משמעותיות (Meaningful) — ✅

הערה טובה מסבירה את ה-"למה" ואת הכוונה, לא את ה-"מה":

cpp · הערות משמעותיות
// loop forever, because the app never closes
while (true) {
    // perform a task based on user input
    switch (task) {
        // option 1: print all articles
        case 1: {
            // open the file that contains all articles
            ifstream file("articles.txt");

            // read all articles from the file and print to screen
            string line;
            while (getline(file, line)) {
                cout << line << endl;
            }

            // close file, because we no longer need it
            file.close();
            break;
        }
💡
כלל אצבע (משלי) הקוד אומר מה קורה; ההערה צריכה להסביר למה. אם אתם נאלצים להוסיף הערה כדי שהקוד יהיה מובן — לפעמים עדיף לשנות את הקוד עצמו (שם משתנה טוב, פונקציה עם שם ברור) במקום להוסיף הערה.

6. Code Smells — "ריחות רעים" בקוד

👃
מהו Code Smell Code smell הוא סימן בקוד שמרמז (אבל לא מבטיח) שיש בעיית עיצוב עמוקה יותר. כמו ריח רע במקרר — לא בהכרח משהו רקוב, אבל שווה לבדוק. הנה הרשימה מההרצאה:
  • Duplicated code — קוד כפול (כמו שראינו ב-DRY).
  • Too similar sub-classes — תתי-מחלקות דומות מדי.
  • Lack of comments — חוסר הערות.
  • Pointless comments — הערות מיותרות.
  • Long methods — מתודות ארוכות מדי.
  • Many parameters / יותר מדי מתודות לא-ציבוריות.
  • A class with too many variables — מחלקה עם יותר מדי משתנים.
  • Long/short identifiers — שמות ארוכים מדי או קצרים מדי.
  • Excessive return of data — החזרת יותר מידע מהנדרש.
  • Same name, different meaning / שמות לא עקביים.
  • Trivial modules / middleman — שכבות מתווכות מיותרות.
  • Downcasting — שובר את ההפשטה (abstraction).
  • Multiple inheritance — "יהלום הארור" (dreaded diamond).
  • Unnecessary complexity — למשל כפיית design pattern במקום שלא צריך.
  • The GOD class/function — עושה יותר מדי.
  • Freeloader — עושה מעט מדי.
  • Conditional complexity — יותר מדי הסתעפויות (if/switch).
  • Tight coupling with implementation — צימוד הדוק למימוש קונקרטי.
🧠
הקוד שלנו "מסריח" (משלי) ה-main שלנו הוא GOD function קלאסי (עושה הכל), עם conditional complexity (switch גדול) ו-tight coupling לקבצים ול-UI. בהמשך נתקוף בדיוק את הריחות האלה.

7. פיצול מתודות ארוכות

🧭
קונטקסט ה-main שלנו ארוך ועושה הרבה דברים שונים. הפתרון: לחלץ כל "משימה" לפונקציה נפרדת עם שם ברור. זה הופך את ה-main לקריא כמו תוכן עניינים.
cpp · main קריא אחרי חילוץ פונקציות
int main() {
    int task = 1;          // איזו פעולה המשתמש רוצה לבצע

    while (true) {
        switch (task) {
            case 1:
                print_articles();   // option 1: print all articles
                break;
            case 2:
                add_article();      // option 2: add a new article
                break;
            default:
                cout << "Sorry, no can do" << endl;
                break;
        }
        print_menu();
        cin >> task;
    }
}
💡
היתרון עכשiו אפשר לקרוא את ה-main ולהבין את הזרימה ב-3 שניות, בלי לצלול לפרטים. כל פונקציה אחראית על דבר אחד — וזה בדיוק מוביל אותנו לעיקרון הבא.

8. עקרונות SOLID

🧭
קונטקסט SOLID הוא ראשי תיבות של חמישה עקרונות עיצוב מונחה-עצמים, שמטרתם קוד גמיש, ניתן לתחזוקה והרחבה. נעבור עליהם אחד-אחד עם דוגמאות מהפרויקט.
אותעיקרוןבקצרה
SSingle Responsibilityאחריות אחת לכל מחלקה/פונקציה.
OOpen / Closedפתוח להרחבה, סגור לשינוי.
LLiskov Substitutionתת-מחלקה חייבת להחליף את מחלקת-העל בלי לשבור.
IInterface Segregationממשקים קטנים וממוקדים במקום אחד גדול.
DDependency Inversionתלות בהפשטות (interfaces), לא במימושים.

9. Single Responsibility Principle (S)

🎯
העיקרון לכל מחלקה/פונקציה צריכה להיות אחריות אחת בלבד — סיבה אחת להשתנות. היתרון: פחות באגים, קל יותר לבדוק, להבין ולתחזק. אם לפונקציה יש כמה אחריויות, שינוי באחת עלול לשבור את השנייה.

הפונקציה add_article() שלנו עושה שני דברים: קוראת קלט מהמשתמש וגם כותבת לקובץ. נפצל אותה לשתי פונקציות, כל אחת עם אחריות אחת:

cpp · פיצול לפי אחריות
// option 2: add a new article
case 2: {
    string article = get_article_from_user();   // אחריות 1: קלט
    add_article(article);                         // אחריות 2: שמירה
    break;
}

// אחריות אחת: לקרוא כתבה מהמשתמש ולהחזיר אותה
string get_article_from_user() {
    string article;
    getline(cin, article);   // מנקה את ה-\n שנותר. TODO: refactor
    getline(cin, article);
    return article;
}

// אחריות אחת: לשמור כתבה לקובץ
void add_article(string article) {
    ofstream file("articles.txt", ios_base::app);
    file << article << endl;
    file.close();
}
💡
למה זה חזק (משלי) עכשיו אם נרצה לקרוא כתבה ממקור אחר (קובץ, רשת, ממשק גרפי) — נשנה רק את get_article_from_user. ואם נרצה לשמור ל-DB במקום לקובץ — נשנה רק את add_article. כל שינוי מבודד לאחריות שלו.

10. Open/Closed + תבנית Command

🚪
Open/Closed Principle (O) קוד צריך להיות פתוח להרחבה אך סגור לשינוי. כלומר: כשרוצים להוסיף יכולת חדשה, נשאף שאפשר יהיה להרחיב (למשל ע"י ירושה/מימוש interface) בלי לשנות את הקוד הקיים (שכבר נבדק ועובד — ואולי בכלל לא נגיש לנו).

הבעיה ב-switch (task) שלנו: כל פעם שנרצה להוסיף פעולה חדשה, נצטרך לשנות את ה-switch ולהוסיף case. זה מפר את Open/Closed. הפתרון הקלאסי הוא תבנית Command.

📐
תבנית Command (Behavioral Pattern) תבנית התנהגותית שמנתקת (decouples) את הצורך להפעיל פעולה מ-הפעולה הקונקרטית עצמה. כל פעולה הופכת לאובייקט נפרד שמממש ממשק אחיד. התוצאה: גמישות והרחבה.

שלב 1: ממשק משותף לכל הפקודות:

cpp · ICommand
class ICommand {
public:
    virtual void execute() = 0;   // מתודה טהורה וירטואלית = ממשק
};
// הערה: למען הפשטות, מתעלמים כאן מ-virtual destructor

שלב 2: כל פעולה היא מחלקה שמממשת את הממשק:

cpp · פקודות קונקרטיות
class DisplayArticlesCommand : public ICommand {
public:
    void execute() {
        ifstream file("articles.txt");
        string line;
        // ...
    }
};

class AddArticleCommand : public ICommand {
public:
    void execute() {
        string title, content;
        cout << "Enter article title: ";
        // ...
        // BTW: להוסיף כתבה לאן? לקובץ? ל-DB? (נחזור לזה ב-DIP)
    }
};

שלב 3: רושמים את הפקודות ב-map וקושרים אותן למפתחות התפריט:

cpp · רישום הפקודות
map<string, ICommand*> commands;

ICommand* displayArticles = new DisplayArticlesCommand();
commands["1"] = displayArticles;

ICommand* addArticle = new AddArticleCommand();
commands["2"] = addArticle;

App app(commands);
app.run();

// TODO: לאפשר יציאה מסודרת (graceful exit)
delete displayArticles;
delete addArticle;

שלב 4: הלולאה הראשית הופכת לנקייה — אין יותר switch! פשוט מפעילים את הפקודה המתאימה:

cpp · run() ללא switch
void run() {
    int task;
    while (true) {
        cout << "What would you like to do next?" << endl;
        cout << "1. Show all articles" << endl;
        cout << "2. Add article" << endl;
        cin >> task;

        try {
            commands[task]->execute();
        }
        catch (...) {
            cout << "Sorry, no can do" << endl;
        }
    }
}
⭐
השגנו Open/Closed עכשיו כדי להוסיף פעולה חדשה — פשוט יוצרים מחלקת ICommand חדשה ומוסיפים אותה ל-map. לא נוגעים בקוד של run() בכלל. הקוד פתוח להרחבה, סגור לשינוי. (שימו לב גם איך ה-try/catch החליף את ה-default.)

11. Liskov Substitution Principle (L)

🔁
העיקרון (Robert C. Martin) "פונקציות שמשתמשות במצביעים/הפניות למחלקת-בסיס חייבות להיות מסוגלות להשתמש באובייקטים של מחלקות-נגזרות בלי לדעת זאת." במילים פשוטות: אם B יורש מ-A, אז בכל מקום שמצפה ל-A צריך להיות אפשר לשים B בלי לשבור כלום.

הדוגמה הקלאסית: ריבוע הוא מלבן? מבחינה לוגית "כן" (ריבוע is-a מלבן) — אבל זה לא בהכרח עיצוב טוב:

cpp · הבעיה של square/rectangle
// r מטיפוס Rectangle (אבל בפועל אולי Square שיורש ממנו)
r.setWidth(2);
r.setHeight(4);

if (r.area() == 8) { ... }   // עבור Rectangle נכון; עבור Square?!

אם Square יורש מ-Rectangle, וכדי לשמור על ריבועיות הוא דואג שכשמשנים רוחב גם הגובה משתנה — אז קוד שבדק area() == 8 (כי הוא חשב שיש מלבן 2×4) ייכשל, כי בריבוע הרוחב והגובה השתנו יחד ל-4×4=16. ה-Square "הפתיע" את מי שציפה ל-Rectangle ⟵ הפרה של Liskov.

💡
המסר "מה שנשמע הגיוני" (ריבוע is-a מלבן) הוא לא תמיד עיצוב טוב. ירושה צריכה לשמר את ההתנהגות שמצפים לה ממחלקת-העל, לא רק את היחס הלוגי.

12. Interface Segregation Principle (I)

✂️
העיקרון במקום ממשק אחד גדול — לפצל אותו לכמה ממשקים קטנים וממוקדים. כך מונעים ממחלקות מממשות להיאלץ לממש מתודות שאין להן צורך בהן (ולכתוב מימוש ריק/שגוי רק כדי "לספק" את הממשק).
💡
דוגמה (משלי) ממשק ענק IMachine עם print(), scan(), fax() מכריח מדפסת פשוטה לממש גם scan ו-fax שאין לה. עדיף לפצל ל-IPrinter, IScanner, IFax — וכל מכשיר מממש רק את מה שרלוונטי לו.

13. Dependency Inversion + Dependency Injection (D)

🔌
Dependency Inversion Principle להעדיף צימוד רופף (loose coupling) על פני צימוד הדוק. במקום שמחלקה תתלה במימוש קונקרטי — היא תתלה בממשק (interface), וכל מימוש שתואם לממשק יוכל להיכנס במקומו.

דוגמת המכונית מההרצאה:

  • מכונית חייבת מנגנון הצתה (ignition).
  • מכונית נאיבית תחזיק מפתח-הצתה (key-ignition) קונקרטי ותקרא עליו start().
  • אבל אם נרצה הצתה ללא מפתח (keyless)? נצטרך לשנות את המכונית — רע!
  • הפתרון: ליצור ממשק Ignition, ושגם key-ignition וגם keyless-ignition יממשו אותו. המכונית מחזיקה איבר מטיפוס Ignition — ויכולה לקבל כל מימוש.
💉
Dependency Injection — איך מיישמים את זה Dependency Injection (DI) = לא המחלקה יוצרת את התלויות שלה, אלא מקבלת אותן מבחוץ (בדרך כלל דרך ה-constructor). כך אפשר "להזריק" כל מימוש — כולל mock-ים לצורך בדיקות.

נחיל את זה על ה-App שלנו: במקום שתהיה תלויה ב-UI קונקרטי (טרמינל), היא תקבל ממשק IMenu ואת מפת הפקודות מבחוץ:

cpp · App עם Dependency Injection
class App {
private:
    IMenu* menu;                          // ממשק, לא UI קונקרטי
    map<string, ICommand*> commands;
public:
    // התלויות מוזרקות מבחוץ דרך ה-constructor
    App(IMenu* menu, map<string, ICommand*> commands)
        : menu(menu), commands(commands) {}

    void run() {
        while (true) {
            int task = menu->nextCommand();      // הממשק מספק את הקלט
            try {
                commands[task]->execute();
            }
            catch (...) {
                menu->displayError("Sorry, no can do");
            }
        }
    }
};
⭐
מה הרווחנו ה-App כבר לא יודעת אם ה-UI הוא טרמינל, חלון גרפי או ווב — היא מדברת רק מול IMenu (עם ConsoleMenu כמימוש אחד אפשרי). זוכרים את המטרה "יהיו כמה UIs"? עכשיו זה אפשרי בלי לגעת ב-App. וזה גם מה שיהפוך את הקוד לניתן לבדיקה — הנושא הבא.

14. בדיקות (Tests) וסוגיהן

🧭
קונטקסט — הבעיה היסודית הקוד המקורי היה main אחד גדול — לא ניתן לבדיקה, ולא נכתבו לו בדיקות בכלל. איך בכלל בודקים קוד שצמוד למקלדת ולמסך? אחרי כל ה-refactoring (פיצול לפונקציות/מחלקות + DI), הקוד סוף-סוף ניתן לבדיקה. עכשיו נבין סוגי בדיקות.

בדיקות לפי היקף

סוגמה בודק
Unit testsיחידה בודדת (פונקציה/מחלקה) במנותק.
Integration testsהשילוב בין כמה יחידות.
End-to-end testsזרימה מלאה דרך כל המערכת, מקצה לקצה.
Acceptance testsהאם המערכת עומדת בדרישות הלקוח.

בדיקות לפי סוג התרחיש

סוגדוגמה
Sanity (שפיות)1+1 == 2 — המקרה הבסיסי עובד.
Negative (שלילי)1+1 != 3 — שגיאה אכן נכשלת/נזרקת כמצופה.
Boundary (גבול)מה קורה ב-1 + max? (קצוות הטווח החוקי).
Edge cases (קצה)1/0 — מקרים חריגים/בעייתיים.

15. TDD — Test Driven Development

🚦
מהו TDD מתודולוגיית פיתוח שבה כותבים את הבדיקה לפני הקוד. הבדיקות "מובילות" (drive) את פיתוח הפתרון. המחזור נקרא Red → Green → Refactor. 
   1. כותבים בדיקה
        └─ אין עדיין קוד → הבדיקה נכשלת  ──▶  🔴 RED

   2. כותבים קוד פשוט שיגרום לבדיקה לעבור
        └─ הבדיקה עוברת                  ──▶  🟢 GREEN

   3. Refactor (משפרים את הקוד) ──▶ חוזרים לשלב 1
        ↺

דוגמה מלאה עם gtest (Google Test)

נפתח במחלקת Article. שלב Red — קודם הבדיקה, לפני שהמחלקה קיימת:

cpp · Sanity test (RED)
#include <gtest/gtest.h>
#include "Article.h"

TEST(ArticleTest, Getters) {
    Article article(1, "Test Title", "Test Content");
    EXPECT_EQ(article.getId(), 1);
    EXPECT_EQ(article.getTitle(), "Test Title");
    EXPECT_EQ(article.getContent(), "Test Content");
}

זה לא יתקמפל בכלל (unknown type name 'Article') — זה ה-RED. שלב Green — כותבים את המחלקה המינימלית שתעבור:

cpp · Article.h (GREEN)
#include <string>
using namespace std;

class Article {
private:
    int id;
    string title;
    string content;
public:
    Article(int id, const string& title, const string& content);
    int getId() const;
    const string& getTitle() const;
    const string& getContent() const;
};

Negative test — בדיקה ששגיאה אכן נזרקת

cpp · Negative test
// כותרת ריקה - אמורה לזרוק חריגה
TEST(ArticleTest, EmptyTitleArticle) {
    EXPECT_THROW(Article(2, "", "Empty Title Content"),
                 std::invalid_argument);
}

הבדיקה נכשלת (הקוד "throws nothing"), אז מוסיפים את הבדיקה ל-constructor — וזה הופך ל-GREEN:

cpp · המימוש שמספק את הבדיקה
Article::Article(int id, const string& title, const string& content)
    : id(id), title(title), content(content)
{
    if (title == "")
        throw invalid_argument("Title cannot be empty");
}

Boundary test — בדיקת גבול

cpp · Boundary test
// אורך כותרת מקסימלי (נניח 50). 51 תווים = חריגה
TEST(ArticleTest, MaxTitleLengthArticle) {
    const std::string maxTitle(51, 'A');  // מחרוזת של 51 פעמים 'A'
    EXPECT_THROW(Article(1, maxTitle, "Content"),
                 std::invalid_argument);
}
📊
דיאגרמת UML של Article - id: int, - title: string, - content: string (פרטיים, מסומנים -) | + Article(...), + getId(), + getTitle(), + getContent() (ציבוריים, מסומנים +). ב-UML: - = private, + = public.
🔗
למה TDD ו-refactoring הולכים יד ביד (משלי) הבדיקות הן "רשת ביטחון": ברגע שיש לכם בדיקות שעוברות (GREEN), אתם יכולים לבצע refactoring בביטחון — אם שברתם משהו, בדיקה תיכשל מיד ותדע. בלי בדיקות, כל refactoring הוא הימור.

16. משבר התוכנה (The Software Crisis)

🧭
קונטקסט חלק א' עסק באיך לכתוב מודול קוד טוב. חלק ב' מתרחק ושואל: איך מנהלים פרויקט תוכנה שלם מההתחלה ועד הסוף? כדי להבין למה צריך בכלל "הנדסה" כאן, נתחיל מהבעיה ההיסטורית שהולידה את התחום.

משבר התוכנה = חוסר היכולת לכתוב את התוכנה בזמן הצפוי. התסמינים:

  • הלקוחות לא מרוצים.
  • תוכנה גרועה שלא עומדת בדרישות.
  • כתוצאה מכך, פרויקטים: חורגים מתקציב/זמן, קשים לתחזוקה, או פשוט נכשלים.
💬
ציטוט — Edsger W. Dijkstra (1972) "הסיבה המרכזית היא שהמכונות נעשו חזקות בכמה סדרי גודל! בלי מכונות, תכנות לא היה בעיה כלל; כשהיו לנו מחשבים חלשים, תכנות נהיה בעיה קטנה; ועכשיו שיש לנו מחשבים ענקיים — תכנות נהיה בעיה ענקית באותה מידה." (מתוך "The Humble Programmer", הרצאת פרס טיורינג).

17. מהי הנדסת תוכנה

🏗️
הגדרה (Wikipedia) "הנדסת תוכנה היא היישום השיטתי של גישות הנדסיות לפיתוח תוכנה." כלומר — לא "סתם לכתוב קוד", אלא תהליך מסודר ושיטתי, בדיוק כמו שמהנדס בונה גשר לפי תכנון ולא מאלתר.

18. שלבי פרויקט תוכנה

הניהול של פרויקט תוכנה מורכב מרצף שלבים:

 Requirements ─▶ Solution ─▶ Implementation ─▶ Testing ─▶ Delivery/  ─▶ Maintenance
   analysis        design                                  deployment
שלבמה קורה בו
Requirements analysis
(ניתוח דרישות)		לתקשר עם הלקוח, להבין את הבעיה, ולהגדיר מהו הפתרון הרצוי. אי-הבנות כאן גוררות תוצאות רעות בהמשך.
Solution design
(תכנון הפתרון)		הצעד הראשון לעבר הפתרון: לתכנן פתרון שיעמוד בדרישות. משתמשים בדיאגרמות, טבלאות וכו'.
Implementation
(מימוש)		צוותי הפיתוח כותבים את הקוד — לפי התכנון.
Testing
(בדיקות)		בודקים את הקוד כדי לוודא שהוא עומד בדרישות.
Delivery / deployment
(אספקה/פריסה)		הקוד נפרס/נמסר ללקוח.
Maintenance
(תחזוקה)		טיפול בבאגים, בעיות חומרה ושינויים לאורך זמן.

השאלה הגדולה היא: באיזה סדר ובאיזה קצב עוברים בין השלבים? כל "מודל פיתוח" עונה על זה אחרת.

19. מודל המפל (Waterfall)

🧭
קונטקסט המודל הראשון והפשוט ביותר (Dr. Winston W. Royce, 1970). הרעיון: לרדת במדרגות, שלב אחרי שלב, כמו מפל מים — בכיוון אחד בלבד. 
 Requirements
      ↓
   Solution design
        ↓
     Implementation
          ↓
        Testing
            ↓
         Delivery/deployment
              ↓
           Maintenance
  • נעים בכיוון אחד, ועוברים לשלב הבא רק כשהנוכחי הושלם.
  • ומה אם מתגלים פגמי תכנון או שדרישות משתנות? חוזרים אחורה, מתקנים, ומתקדמים שוב למטה.
⚠️
חסרונות (Discussion) • כל שלב אורך יותר מדי זמן — עד שהפתרון מוכן, הרבה כבר השתנה, והושקע מאמץ בתכנון שכבר לא רלוונטי.
• הלקוח מעורב רק בסוף — וכשהוא סוף-סוף רואה את התוצר, פידבק קריטי כבר חסר תועלת (מאוחר מדי לשנות).

20. מודל V

🧭
קונטקסט מודל V נבנה על גבי מודל המפל, אבל מוסיף דגש חזק על בדיקות — צורת ה-V מציגה ירידה (פיתוח) ועלייה (בדיקות). 
  Requirements                              System & acceptance
    analysis                                     testing
        ↓                                           ↑
     Solution                                  Integration
      design                                     testing
          ↓                                       ↑
       Implementation ───▶ Unit testing ─────────┘
  • בונה על מודל המפל, עם דגש על בדיקות.
  • כל בדיקה מאמתת את השלב המקביל לה (הצד השמאלי).
  • הבדיקות מוגדרות בצד שמאל (בזמן הפיתוח) אבל מורצות בצד ימין.
  • התוצאה: מוצרים טובים יותר, פחות בעיות במעבר לשלב הבא.

21. מודלים איטרטיביים ואינקרמנטליים

  • הפרויקט מתקדם בהדרגה (incrementally) — מוסיפים פונקציונליות מעט-מעט.
  • כל איטרציה היא כמו "מפל קטן" בפני עצמו.
  • בכל איטרציה מתקרבים יותר לפתרון הרצוי.
  • קל לשתף את ההתקדמות עם הלקוח → לקבל פידבק יקר-ערך → ולממש את השינויים הנדרשים.
⚠️
חסרונות • קשה יותר "לסגור" ולסיים את הפרויקט (תמיד יש עוד איטרציה). מי משלם על העלויות?
• לפעמים מגלים בעיות (גדולות מדי) מאוחר מדי בדרך.

22. מודל הפרוטוטייפ (Prototype)

🧪
הרעיון המרכזי להגיע לפרוטוטייפ עובד מהר ככל האפשר. ברגע שללקוח יש משהו מוחשי לראות — הוא מבין את הצרכים שלו טוב יותר, וכתוצאה מכך מתאר את הדרישות טוב יותר.

יש שתי גישות מה לעשות עם הפרוטוטייפ:

  • Evolutionary (אבולוציוני) — משפרים את הפרוטוטייפ שוב ושוב עד שהוא הופך למוצר.
  • Throwaway (חד-פעמי) — אחרי שהבנו את הדרישות, זורקים את הפרוטוטייפ וממשיכים במודל "רגיל" (למשל Waterfall).
⚖️
דיון — יתרונות וחסרונות יתרון: תוצאות טובות יותר — הדרישות וציפיות הלקוח נענות טוב יותר.
חסרונות: משקיעים הרבה זמן בפרוטוטייפ "חסר תועלת"; מפתחים מתאהבים בפרוטוטייפ (קל "לנצל" אותו); והלקוח (בטעות) מתייחס לפרוטוטייפ כאל המוצר הסופי.

23. Agile — המניפסט והעקרונות

🏃
מניפסט ה-Agile התנועה שמעריכה (מבלי לפסול את הצד הימני):
• אנשים ואינטראקציות > תהליכים וכלים
• תוכנה עובדת > תיעוד מקיף
• שיתוף פעולה עם הלקוח > משא-ומתן חוזי
• תגובה לשינוי > היצמדות לתוכנית

מאחורי המניפסט עומדים 12 עקרונות:

  1. העדיפות העליונה היא לספק את הלקוח באמצעות אספקה מוקדמת ורציפה של תוכנה בעלת ערך.
  2. לקבל בברכה שינויי דרישות, גם בשלב מאוחר — Agile רותם את השינוי ליתרון תחרותי של הלקוח.
  3. לספק תוכנה עובדת בתדירות גבוהה (משבועות עד חודשים, עדיף הקצר).
  4. אנשי עסקים ומפתחים חייבים לעבוד יחד יום-יומי לאורך הפרויקט.
  5. לבנות פרויקטים סביב אנשים מוטיבציוניים — לתת להם סביבה ותמיכה, ולתת בהם אמון.
  6. הדרך היעילה ביותר להעביר מידע היא שיחה פנים-אל-פנים.
  7. תוכנה עובדת היא המדד העיקרי להתקדמות.
  8. תהליכי Agile מקדמים פיתוח בר-קיימא — קצב קבוע שאפשר לשמר לאורך זמן.
  9. תשומת לב מתמדת למצוינות טכנית ולעיצוב טוב משפרת את הזריזות.
  10. פשטות — אומנות מקסום כמות העבודה שלא נעשתה — היא חיונית.
  11. הארכיטקטורות, הדרישות והעיצובים הטובים ביותר צומחים מצוותים שמארגנים את עצמם.
  12. במרווחים קבועים, הצוות מהרהר איך להשתפר ומכוונן את התנהגותו בהתאם.
🔁
מהו Agile בפועל לחלק את הפרויקט לבנייות אינקרמנטליות קטנות, באיטרציות קצרות, שכל אחת כוללת: תכנון → מימוש → אספקה ופידבק. (שימו לב — זה בדיוק "המודל האיטרטיבי" מהפרק הקודם, מורם לרמת פילוסופיה.)

24. Scrum

Scrum הוא framework (מסגרת עבודה) של Agile. מושגי המפתח שלו:

קטגוריהפריטים
Roles (תפקידים)Product Owner, Scrum Master, Development Team.
Events (אירועים)Sprint, Sprint Planning, Daily Scrum (ה"stand-up" היומי), Sprint Review.
Artifacts (תוצרים)Product Backlog, Sprint Backlog.
💡
בקצרה (משלי) העבודה מחולקת ל-Sprints (איטרציות של ~שבועיים). ה-Product Backlog הוא רשימת כל המשימות; בתחילת כל ספרינט בוחרים מתוכה ל-Sprint Backlog. בכל בוקר יש stand-up קצר לסנכרון, ובסוף הספרינט — Review להצגת מה שנעשה.

25. Extreme Programming (XP)

XP היא מתודולוגיית פיתוח Agile, המתמקדת בפרקטיקות הנדסיות. הפרקטיקות המרכזיות:

  • Pair Programming — תכנות בזוגות.
  • Test-Driven Development (TDD) — שראינו בחלק א'!
  • Continuous Integration — אינטגרציה רציפה.
  • Small Releases — שחרורים קטנים ותכופים.
👥
Pair Programming — תכנות בזוגות שני מתכנתים עובדים יחד מול מחשב אחד: אחד כותב קוד (Driver), השני סוקר ומציע הצעות (Observer). מתחלפים בתפקידים בתדירות גבוהה. התוצאה: שיפור באיכות הקוד ובשיתוף ידע בין חברי הצוות.
🔗
הכל מתחבר שימו לב שחלק ב' חוזר אל חלק א': TDD שבנינו בו את מחלקת Article הוא בעצם פרקטיקה מרכזית של XP. ו-refactoring שעשינו לאורך כל חלק א' הוא בדיוק מה שעיקרון Agile מס' 9 ("מצוינות טכנית ועיצוב טוב") מבקש. כתיבת קוד טוב (חלק א') ותהליך פיתוח טוב (חלק ב') הם שני צדדים של אותו מטבע.

📌 סיכום ההרצאה

🎯
בקצרה חלק א': קוד שעובד ≠ קוד טוב. משפרים אותו ב-refactoring, מונחים ע"י עקרונות (DRY/KISS/SOLID) וזיהוי code smells, נעזרים בתבניות עיצוב (Command), ב-Dependency Injection, וב-בדיקות ו-TDD כרשת ביטחון.
חלק ב': פרויקט תוכנה מנוהל לפי מודל פיתוח. עברנו מ-Waterfall הנוקשה (לקוח רק בסוף), דרך V (דגש בדיקות), איטרטיבי ו-פרוטוטייפ, ועד Agile (Scrum, XP) — שמעמיד במרכז תוכנה עובדת, פידבק תכוף מהלקוח, ותגובה לשינוי.
