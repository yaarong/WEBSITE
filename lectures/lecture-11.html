חלק א' · יסודות

1. אתגרי המובייל + רכיבי Android

🧭
קונטקסט פיתוח למובייל מציב אתגרים שאין ב-web/desktop, ו-Android בנויה כדי להתמודד איתם.
📱
אתגרי המובייל
  • גרסאות OS רבות + הפצות שונות של יצרנים (Samsung, Xiaomi...).
  • גדלי מסך ורזולוציות מגוונים.
  • יעילות — סוללה ומשאבים מוגבלים; ה-OS יכול לסגור תהליכים בכל רגע כדי לפנות משאבים.
  • קלט מגוון — מגע, מחוות (gestures), חיישנים.

Android היא מערכת הפעלה מבוססת Linux לטלפונים, טאבלטים, שעונים, טלוויזיות ועוד. מגיעה עם SDK (אמולטורים, ספריות, תיעוד). רכיבי הליבה של אפליקציה:

רכיבתפקיד
Activityמסך בודד באפליקציה
Fragmentחלק מודולרי לשימוש חוזר בתוך Activity
Intent"הודעה" לתקשורת בין רכיבים (למשל פתיחת מסך)
Serviceלוגיקה ברקע (בלי UI)
Broadcast Receiverמאזין לאירועי מערכת (למשל סיום אתחול)
Content Providerשיתוף נתונים בין אפליקציות

הבסיס תמיד: Java/Kotlin + XML — לוגיקה בקוד, layout ב-XML (ההפרדה מהרצאה 8).

2. Activity ומחזור החיים

🧭
קונטקסט Activity = מסך בודד. יש לו UI (בדרך כלל) ולוגיקה (במחלקה). ה-Activity טוען את ה-layout שלו ב-onCreate.
java · MainActivity.java
public class MainActivity extends Activity {

    @Override
    protected void onCreate(Bundle savedInstanceState) {
        super.onCreate(savedInstanceState);
        setContentView(R.layout.activity_main);
    }
}

וה-layout המתאים (כאן עם ConstraintLayout — מסדר רכיבים לפי אילוצים יחסיים):

xml · res/layout/activity_main.xml
<?xml version="1.0" encoding="utf-8"?>
<androidx.constraintlayout.widget.ConstraintLayout
    xmlns:android="http://schemas.android.com/apk/res/android"
    xmlns:app="http://schemas.android.com/apk/res-auto"
    android:layout_width="match_parent"
    android:layout_height="match_parent">

    <TextView
        android:layout_width="wrap_content"
        android:layout_height="wrap_content"
        android:text="Hello World!"
        app:layout_constraintBottom_toBottomOf="parent"
        app:layout_constraintLeft_toLeftOf="parent"
        app:layout_constraintRight_toRightOf="parent"
        app:layout_constraintTop_toTopOf="parent" />

</androidx.constraintlayout.widget.ConstraintLayout>

ארבעת ה-constraint (top/bottom/left/right ל-parent) "נועלים" את ה-TextView במרכז המסך.

🔄
מחזור החיים (Activity Life-cycle) ה-OS קורא ל-Activity דרך שרשרת callbacks לפי מצבו. זה קריטי במובייל — כי ה-OS יכול לעצור/להרוס את ה-Activity בכל רגע (שיחה נכנסת, מעבר לאפליקציה אחרת). 
      onCreate ──▶ onStart ──▶ onResume ──▶ [Activity בחזית, פעיל]
                       ▲                          │
                   onRestart                   onPause
                       │                          ▼
                       └────────  onStop  ◀───────┘
                                    │
                                 onDestroy
java · מימוש ה-callbacks
public class MainActivity extends AppCompatActivity {
    @Override protected void onCreate(Bundle b) {
        super.onCreate(b);
        setContentView(R.layout.activity_main);
        Log.i("MainActivity", "onCreate");
    }
    @Override protected void onStart()   { super.onStart();   Log.i("MainActivity", "onStart"); }
    @Override protected void onResume()  { super.onResume();  Log.i("MainActivity", "onResume"); }
    @Override protected void onPause()   { super.onPause();   Log.i("MainActivity", "onPause"); }
    @Override protected void onStop()    { super.onStop();    Log.i("MainActivity", "onStop"); }
    @Override protected void onDestroy() { super.onDestroy(); Log.i("MainActivity", "onDestroy"); }
}
🔤
מתי כל callback onCreate (נוצר), onStart (נראה), onResume (אינטראקטיבי) → המשתמש פועל → onPause (מאבד פוקוס), onStop (לא נראה), onDestroy (נהרס). onRestart מחזיר Activity שנעצר. כאן שומרים/משחזרים מצב — למשל לעצור ניגון ב-onPause. (להשוואה: ב-React Native זה useEffect.)

3. Resources ו-R

🧭
קונטקסט ב-Android, נכסים שאינם קוד (layouts, מחרוזות, צבעים, תמונות) הם resources בתיקיית res/. ניגשים אליהם דרך מחלקה אוטומטית בשם R.
🔤
הפניה ל-resources
  • מ-Java: R.layout.activity_main, R.string.foo, R.id.btnUpdate.
  • מ-XML: @string/foo, @color/black, @drawable/icon.
  • R היא הפניה ל-resource, לא ה-resource עצמו (טעינה בפועל עשויה לדרוש קוד נוסף).
  • R נוצרת ומתעדכנת אוטומטית ע"י כלי הבנייה.

למשל קובץ המחרוזות (זוכרים את ה-i18n מהרצאה 8? קובץ strings.xml נפרד לכל שפה):

xml · res/values/strings.xml
<resources>
    <string name="app_name">My Application</string>
    <string name="button">foo</string>
    <string name="checkbox">bar</string>
    <string name="hello">hello</string>
</resources>

תיקייה כמו values-land/ (landscape) או values-es/ (ספרדית) תספק ערכים חלופיים לאותם שמות — וכך ה-UI מסתגל אוטומטית למצב/שפה.

חלק ב' · Views ולוגיקה

4. Views, ViewGroup ו-Layouts

🧭
קונטקסט כל רכיב חזותי ב-Android יורש מהמחלקה View. אפשר ליצור Views ב-code, אבל עדיף ב-XML (הפרדת layout מלוגיקה — שוב, הרצאה 8).
java · יצירת View ב-code מול XML
// ב-Java:
TextView hello = new TextView(this);
hello.setText("Hello world");

// ב-XML (מועדף):
// <TextView android:text="Hello world" />
🔤
ViewGroup ו-Layouts ViewGroup = View שיכול להכיל Views אחרים. ה-Layouts הם תת-מחלקות של ViewGroup שמסדרות את הילדים בצורה מסוימת: LinearLayout (טור/שורה), ConstraintLayout (אילוצים יחסיים), FrameLayout ועוד. זה אותו רעיון של עץ רכיבים שראינו ב-View של React Native וב-div של HTML.

דוגמה — מגוון רכיבים בתוך LinearLayout:

xml · רכיבים שונים
<LinearLayout
    xmlns:android="http://schemas.android.com/apk/res/android"
    android:layout_width="match_parent"
    android:layout_height="match_parent"
    android:orientation="vertical">

    <Button      android:text="@string/button" ... />
    <ImageButton app:srcCompat="@mipmap/ic_launcher" ... />
    <CheckBox    android:text="@string/checkbox" ... />
    <RadioGroup ...>
        <RadioButton android:text="@string/radio1" ... />
        <RadioButton android:text="@string/radio2" ... />
    </RadioGroup>
</LinearLayout>
🔤
gravity מול layout_gravity — מלכודת נפוצה android:gravity — מיישר את התוכן בתוך ה-View (למשל הטקסט בתוך ה-TextView). android:layout_gravity — מיישר את ה-View עצמו בתוך ההורה. בלבול ביניהם הוא מקור תקלות קלאסי.

5. חיבור layout ללוגיקה

🧭
קונטקסט ה-XML מגדיר איך זה נראה; ה-Java מגדיר מה זה עושה. הגשר ביניהם: android:id ב-XML, ו-findViewById ב-Java שמאתר את ה-View לפי ה-id.
xml · layout עם id-ים
<EditText android:id="@+id/etName"  ... android:hint="@string/etName" />
<Button   android:id="@+id/btnUpdate" ... android:text="@string/btnUpdate" />
<TextView android:id="@+id/tvName"   ... />
java · הלוגיקה
public class MainActivity extends AppCompatActivity {
    @Override
    protected void onCreate(Bundle savedInstanceState) {
        super.onCreate(savedInstanceState);
        setContentView(R.layout.activity_main);

        Button btn = findViewById(R.id.btnUpdate);

        btn.setOnClickListener(view -> {
            EditText edName = findViewById(R.id.etName);
            TextView tvName = findViewById(R.id.tvName);
            tvName.setText(edName.getText().toString().toUpperCase());
        });
    }
}
🔤
\u200E@+id\u200E, findViewById, ו-lambda
  • @+id/etName — ה-+ אומר "צור id חדש בשם הזה". בלי + = הפניה ל-id קיים.
  • findViewById(R.id.btnUpdate) — מאתר את ה-View בזמן ריצה.
  • btn.setOnClickListener(view -> {...}) — Listener pattern מהרצאה 8! כאן בתחביר lambda של Java (view -> {...}) — קצר יותר מאשר new View.OnClickListener(){...}. שוב: מעבירים פונקציה, לא קוראים לה.

חלק ג' · Intents וניווט

6. Intent — מעבר בין Activities

🧭
קונטקסט Intent = "כוונה" — דרך לתקשר בין רכיבים ב-Android, למשל לפתוח Activity אחר או לבקש מהמערכת לבצע פעולה. שני סוגים: explicit (מציינים בדיוק איזה Activity) ו-implicit (מציינים פעולה, והמערכת בוחרת מי יטפל).
java · Explicit מול Implicit
// Explicit — פתח Activity ספציפי שלי
Intent intent = new Intent(this, ActivityName.class);
startActivity(intent);

// Implicit — בקש מהמערכת "לפתוח כתובת" (כל דפדפן שמותקן)
Intent intent = new Intent(Intent.ACTION_VIEW,
                           Uri.parse("https://www.foo.com"));
startActivity(intent);

העברת נתונים בין Activities עם putExtra / getExtras:

java · Activity שולח (putExtra)
btn.setOnClickListener(view -> {
    EditText edName = findViewById(R.id.etName);
    Intent intent = new Intent(this, MainActivity2.class);
    intent.putExtra("text", edName.getText().toString());
    startActivity(intent);
});
java · Activity מקבל (getExtras)
public class MainActivity2 extends AppCompatActivity {
    @Override
    protected void onCreate(Bundle savedInstanceState) {
        super.onCreate(savedInstanceState);
        setContentView(R.layout.activity_main2);

        TextView tvName = findViewById(R.id.tvName);
        Intent intent = getIntent();
        String name = intent.getExtras().getString("text");
        tvName.setText(name.toUpperCase());
    }
}
🔤
putExtra / getExtras putExtra("text", value) מצרף נתון ל-Intent תחת מפתח. ב-Activity המקבל, getIntent().getExtras().getString("text") שולף אותו. זה ה-mechanism להעברת data בין מסכים — מקביל ל-props בהרצאה 8/10, אבל בין Activities.

7. Manifest ו-Activities Stack

🧭
קונטקסט ה-AndroidManifest.xml הוא "תעודת הזהות" של האפליקציה: הוא מצהיר על כל ה-Activities, ההרשאות, והאייקון. בין השאר, הוא מגדיר intent-filters — אילו Intents כל Activity יודע לטפל בהם.
xml · AndroidManifest.xml (קטע)
<activity android:name=".MainActivity" android:exported="true">
    <intent-filter>
        <action   android:name="android.intent.action.MAIN" />
        <category android:name="android.intent.category.LAUNCHER" />
    </intent-filter>
</activity>
🔤
ה-intent-filter המיוחד ה-action.MAIN + category.LAUNCHER מסמנים את ה-Activity שנפתח כשלוחצים על אייקון האפליקציה — נקודת הכניסה. Activity עם intent-filter של action.VIEW + data scheme="http" יכול לטפל בקישורים — וכך implicit intent של "פתח כתובת" עשוי להגיע אליו.
📚
Activities Stack (Back Stack) כל startActivity דוחף Activity חדש לערימה; כפתור ה-Back מוציא (pop) את העליון וחושף את הקודם. זו בדיוק ערימת ה-Stack מ-Expo Router (הרצאה 10) — אותו רעיון של ניווט-ערימה. 
 Foreground:  [ FooActivity      ]  ◀── המשתמש רואה
              [ AnotherActivity  ]
              [ MainActivity     ]  ◀── תחתית הערימה
                  Back Stack       (Back מוציא את העליון)

חלק ד' · רשימות

8. ListView ו-Adapter

🧭
קונטקסט — תבנית ה-Adapter להצגת רשימה צריך שני דברים: ListView (ה-View שמציג את הרשימה) ו-Adapter (ה"מתאם" שמחבר בין הנתונים ל-Views של כל פריט). זו תבנית עיצוב מרכזית ב-Android: ה-ListView לא יודע כלום על הנתונים — ה-Adapter מתרגם נתון → View.
xml · ListView
<ListView
    android:id="@+id/lstFeed"
    android:layout_width="match_parent"
    android:layout_height="match_parent"
    android:dividerHeight="5dp" />
java · ArrayAdapter פשוט + onItemClick
public class MainActivity extends Activity {
    @Override
    protected void onCreate(Bundle savedInstanceState) {
        super.onCreate(savedInstanceState);
        setContentView(R.layout.activity_main);

        ListView lst = findViewById(R.id.lstFeed);
        ArrayAdapter<String> adapter = new ArrayAdapter<String>(this,
            android.R.layout.simple_list_item_1,
            getResources().getStringArray(R.array.menu));
        lst.setAdapter(adapter);

        lst.setOnItemClickListener((parent, view, position, id) -> {
            Toast.makeText(MainActivity.this,
                ((TextView) view).getText().toString(),
                Toast.LENGTH_LONG).show();
        });
    }
}
🔤
ArrayAdapter ArrayAdapter<String> הוא adapter מובנה למערך מחרוזות. הוא מקבל את ה-context, layout לכל שורה (simple_list_item_1 — מובנה), ואת מערך הנתונים. setOnItemClickListener מאזין ללחיצה על פריט (Listener pattern, שוב). Toast = הודעה קופצת קצרה.

9. Adapter מותאם + ViewHolder

🧭
קונטקסט לרשימה עם פריטים מורכבים (תמונה + שם + כפתורים — כמו פיד חברתי) צריך Adapter מותאם אישית שיורש מ-BaseAdapter. נתחיל ממחלקת המודל Post:
java · Post.java (מודל)
public class Post {
    String name;
    String whenPosted;
    int profileImage;
    int img;

    public Post(String name, String whenPosted, int profileImage, int img) {
        this.name = name; this.whenPosted = whenPosted;
        this.profileImage = profileImage; this.img = img;
    }
    public String getName() { return name; }
    public String getWhenPosted() { return whenPosted; }
    // ... getters
}
java · FeedAdapter + ViewHolder
public class FeedAdapter extends BaseAdapter {
    List<Post> posts;

    private class ViewHolder {
        TextView name;
        TextView when;
        ImageView profile;
        ImageView img;
    }

    public FeedAdapter(List<Post> posts) { this.posts = posts; }

    @Override public int getCount() { return posts.size(); }
    @Override public Object getItem(int position) { return posts.get(position); }

    @Override
    public View getView(int position, View convertView, ViewGroup parent) {
        if (convertView == null) {
            convertView = LayoutInflater.from(parent.getContext())
                .inflate(R.layout.feed_post_layout, parent, false);

            ViewHolder viewHolder = new ViewHolder();
            viewHolder.name = convertView.findViewById(R.id.feed_post_name);
            viewHolder.when = convertView.findViewById(R.id.feed_post_when);
            viewHolder.profile = convertView.findViewById(R.id.feed_post_profile_img);
            viewHolder.img = convertView.findViewById(R.id.feed_post_img);
            convertView.setTag(viewHolder);
        }

        Post p = posts.get(position);
        ViewHolder viewHolder = (ViewHolder) convertView.getTag();
        viewHolder.name.setText(p.getName());
        viewHolder.when.setText(p.getWhenPosted());
        viewHolder.profile.setImageResource(p.getProfileImage());
        viewHolder.img.setImageResource(p.getImg());

        return convertView;
    }
}
🔤
תבנית ה-ViewHolder ו-convertView (חשוב לביצועים!)
  • getView נקרא לכל שורה. LayoutInflater.inflate "מנפח" XML ל-View אמיתי — פעולה יקרה.
  • convertView — Android ממחזר Views של שורות שיצאו מהמסך. אם convertView != null, אפשר לעשות בו שימוש חוזר במקום לנפח מחדש.
  • ViewHolder — שומר הפניות ל-Views (עם setTag/getTag) כדי לא לקרוא ל-findViewById בכל גלילה. זו אופטימיזציה קריטית — בלעדיה הרשימה "מקרטעת". (RecyclerView יאכוף את זה.)

10. RecyclerView

🧭
קונטקסט RecyclerView הוא היורש המודרני של ListView — יעיל יותר ומאכוף את תבנית ה-ViewHolder (לכן השם: הוא "ממחזר" Views). זה המקביל ל-FlatList מ-React Native (הרצאה 10).
java · MainActivity — הגדרת RecyclerView
RecyclerView lstGrid = findViewById(R.id.lstGrid);
GridAdapter gridAdapter = new GridAdapter(this, images);
lstGrid.setAdapter(gridAdapter);
lstGrid.setLayoutManager(
    new StaggeredGridLayoutManager(2, LinearLayoutManager.VERTICAL));
java · GridAdapter
public class GridAdapter extends RecyclerView.Adapter<GridAdapter.ViewHolder> {
    private int[] images;

    public class ViewHolder extends RecyclerView.ViewHolder {
        ImageView img;
        ViewHolder(View itemView) {
            super(itemView);
            img = itemView.findViewById(R.id.grid_image);
        }
    }

    GridAdapter(Context context, int[] images) { this.images = images; }

    @Override
    public ViewHolder onCreateViewHolder(ViewGroup parent, int viewType) {
        View view = LayoutInflater.from(parent.getContext())
            .inflate(R.layout.image_grid_layout, parent, false);
        return new ViewHolder(view);
    }

    @Override
    public void onBindViewHolder(ViewHolder holder, int position) {
        holder.img.setImageResource(images[position]);
    }

    @Override
    public int getItemCount() { return images.length; }
}
🔤
שלוש המתודות של RecyclerView.Adapter
  • onCreateViewHolder — מנפח את ה-layout ויוצר ViewHolder (נקרא רק כשצריך View חדש).
  • onBindViewHolder — ממלא ViewHolder קיים בנתונים של position מסוים (נקרא הרבה — בכל גלילה).
  • getItemCount — כמה פריטים.
  • LayoutManager (כמו StaggeredGridLayoutManager) קובע את סידור הפריטים — רשימה, רשת, וכו'.
ההפרדה בין create ל-bind היא בדיוק מה ש-ViewHolder ב-ListView עשה ידנית — כאן זה מובנה ומאולץ.

חלק ה' · רכיבים מתקדמים

11. Fragments

🧭
קונטקסט Fragment = חתיכת UI+לוגיקה מודולרית ולשימוש חוזר, שחיה בתוך Activity (או fragment אחר). למה צריך? כדי לבנות מסכים מורכבים מחלקים, ולהתאים בין מסכי טלפון לטאבלט (למשל שני fragments זה לצד זה בטאבלט, ומסך-מסך בטלפון).
💡
מאפיינים ל-fragment יש מחזור חיים משלו (מושפע מזה של ה-Activity), back-stack נפרד, ואפשר כמה fragments באותו Activity — יחד או מחליפים זה את זה. (רעיון מוכר? זה ממש כמו components ב-React — חלקי UI לשימוש חוזר.)
java · MainFragment.java
public class MainFragment extends Fragment {
    @Nullable @Override
    public View onCreateView(@NonNull LayoutInflater inflater,
                             @Nullable ViewGroup container,
                             @Nullable Bundle savedInstanceState) {
        return inflater.inflate(R.layout.main_fragment, container, false);
    }

    @Override
    public void onViewCreated(@NonNull View view, @Nullable Bundle savedInstanceState) {
        super.onViewCreated(view, savedInstanceState);
        TextView message = view.findViewById(R.id.message);
        message.setText("Foobar");
    }
}

מציבים fragment ב-Activity דינמית, דרך ה-FragmentManager (transaction):

java · החלפת fragment בקוד
if (savedInstanceState == null) {
    getSupportFragmentManager().beginTransaction()
        .replace(R.id.container, MainFragment.newInstance())
        .commit();
}
🔤
העברת ארגומנטים ל-fragment בדומה ל-Intent של Activity, ל-fragment מעבירים נתונים דרך Bundle ב-setArguments/getArguments (בדרך כלל במתודת factory בשם newInstance). שוב — אותו רעיון של props/extras בניב אחר.

12. Broadcast Receivers ו-Notifications

🧭
קונטקסט Broadcast Receiver מאזין ל"שידורים" של אירועי מערכת (סיום אתחול, שינוי חיבור רשת...) ומגיב להם — גם כשהאפליקציה לא פתוחה.
java · BroadcastReceiver
public class Foo extends BroadcastReceiver {
    @Override
    public void onReceive(Context context, Intent intent) {
        Toast.makeText(context, "Boot completed!", Toast.LENGTH_LONG).show();
    }
}

רושמים אותו ב-Manifest עם intent-filter (כאן: לאירוע סיום אתחול):

xml · רישום ב-Manifest
<receiver android:name=".Foo" android:exported="true">
    <intent-filter>
        <action android:name="android.intent.action.BOOT_COMPLETED" />
    </intent-filter>
</receiver>

Notifications — בונים התראה עם NotificationCompat.Builder:

java · בניית התראה
Intent intent = new Intent(this, MainActivity.class);
PendingIntent pendingIntent = PendingIntent.getActivity(
    this, 0, intent, PendingIntent.FLAG_IMMUTABLE);

NotificationCompat.Builder builder =
    new NotificationCompat.Builder(this, CHANNEL_ID)
        .setSmallIcon(R.drawable.notification_icon)
        .setContentTitle("My notification")
        .setContentText("Hello World!")
        .setPriority(NotificationCompat.PRIORITY_DEFAULT)
        .setContentIntent(pendingIntent)
        .setAutoCancel(true);
🔤
PendingIntent ו-Channels PendingIntent — "Intent דחוי" שהמערכת תפעיל מאוחר יותר בשמכם (למשל כשהמשתמש לוחץ על ההתראה). מ-Android 8+ חובה גם NotificationChannel — "ערוץ" שמקבץ התראות דומות ומאפשר למשתמש לשלוט בהן. ה-.setXxx(...) בשרשרת הוא Builder pattern — בונים אובייקט מורכב שלב-שלב.

13. Services

🧭
קונטקסט בעוד Activity מבצע לוגיקה לפי UI, Service מבצע לוגיקה ברקע — בלי UI, ולא קשור ל-Activity מסוים (למשל הורדת קובץ, ניגון מוזיקה). שימו לב: כברירת מחדל הוא רץ על ה-thread הראשי — לעבודה כבדה צריך thread נפרד.
java · MyService
public class MyService extends Service {
    @Override
    public IBinder onBind(Intent intent) { return null; }

    @Override
    public int onStartCommand(Intent intent, int flags, int startId) {
        // ... לוגיקת רקע
        return START_STICKY;   // ה-OS יפעיל מחדש את ה-service אם נהרג
    }
}
java · הפעלת service מ-Activity
button.setOnClickListener(view -> {
    Intent intent = new Intent(this, MyService.class);
    startForegroundService(intent);
});

14. Room — מסד נתונים מקומי

🧭
קונטקסט Room היא ספריית persistence שבנויה מעל SQLite (מסד רלציוני מוטמע במכשיר). זה בעצם ORM — בדיוק מה שלמדנו בהרצאה 9! (Hibernate היה ל-Java שרת; Room הוא ה-ORM של Android.) שלושה חלקים: @Entity (טבלה), @Dao (פעולות), @Database (ניהול).
java · Entity
@Entity
public class Post {
    @PrimaryKey(autoGenerate = true)
    private int id;
    private String content;

    public Post(String content) { this.content = content; }
    public int getId() { return id; }
    public String getContent() { return content; }
    public void setId(int id) { this.id = id; }
    public void setContent(String content) { this.content = content; }
}
java · DAO ו-Database
@Dao
public interface PostDao {
    @Query("SELECT * FROM post")            List<Post> index();
    @Query("SELECT * FROM post WHERE id = :id") Post get(int id);
    @Insert  void insert(Post... posts);
    @Update  void update(Post... posts);
    @Delete  void delete(Post... posts);
}

@Database(entities = {Post.class}, version = 1)
public abstract class AppDB extends RoomDatabase {
    public abstract PostDao postDao();
}
🔤
Annotations של Room
  • @Entity + @PrimaryKey(autoGenerate=true) — מחלקה→טבלה, ומפתח ראשי שמתמלא אוטומטית (בדיוק @Entity/@Id של Hibernate מהרצאה 9!).
  • @Dao — Data Access Object: ממשק עם פעולות. @Query מאפשר לכתוב SQL ידני; @Insert/@Update/@Delete נוצרים אוטומטית (CRUD).
  • @Database — מחלקה אבסטרקטית שמרכזת את כל ה-DAOs ומגדירה גרסה.
java · שימוש
AppDB db = Room.databaseBuilder(getApplicationContext(), AppDB.class, "FooDB")
    .allowMainThreadQueries().build();
PostDao postDao = db.postDao();

postDao.insert(new Post("Hello"));   // Create
List<Post> all = postDao.index();    // Read

15. Threads ו-AsyncTask

🧵
הבעיה — ה-UI Thread כברירת מחדל האפליקציה רצה על ה-thread הראשי (UI thread). אם מריצים עליו קוד ארוך (רשת, מסד), ה-UI נתקע — כי ה-thread עסוק ולא יכול לטפל בציור ובמגע. הכלל: דוחים משימות ארוכות ל-thread אחר. אבל — אסור לגעת ב-UI מ-thread שאינו ה-UI thread (צריך לחזור אליו, למשל עם runOnUiThread).
🧭
AsyncTask — הפתרון הקלאסי AsyncTask מפריד את העבודה: doInBackground רץ על thread נפרד, ו-onPostExecute רץ על ה-UI thread (אז בטוח לעדכן בו את המסך).
java · AsyncTask בסיסי
private class FooTask extends AsyncTask<String, Void, String> {
    protected String doInBackground(String... s) {
        // עבודה כבדה — רץ על thread נפרד
        return result;
    }
    protected void onPostExecute(String result) {
        // עדכון UI — רץ על ה-UI thread (בטוח)
    }
}

new FooTask().execute("Hello", "world");

דוגמה מעשית — בקשת HTTP ברקע (צריך הרשאת אינטרנט ב-Manifest):

java · HTTP request דרך AsyncTask
<!-- ב-Manifest: -->
<uses-permission android:name="android.permission.INTERNET" />

class FooTask extends AsyncTask<URL, String, String> {
    @Override
    protected String doInBackground(URL... urls) {
        try {
            HttpURLConnection connection = (HttpURLConnection) urls[0].openConnection();
            if (connection.getResponseCode() == HttpsURLConnection.HTTP_OK) {
                // קריאת ה-response מה-connection
            }
        } catch (IOException e) { }
        return responseString;
    }

    @Override
    protected void onPostExecute(String response) {
        // עדכון ה-UI עם התשובה
    }
}
🔤
\u200EAsyncTask<Params, Progress, Result>\u200E שלושת הטיפוסים הגנריים: סוג הקלט ל-execute, סוג עדכון ההתקדמות (onProgressUpdate + publishProgress), וסוג התוצאה שעוברת ל-onPostExecute. (הערה: AsyncTask הוקצה כ-deprecated בגרסאות חדשות לטובת פתרונות מודרניים, אבל הוא ממחיש מצוין את עקרון ה-threading.)

★ סיכום ההרצאה

🎯
התמונה המלאה
  • Activity = מסך, עם מחזור חיים (onCreate→onStart→onResume→onPause→onStop→onDestroy) — קריטי כי ה-OS שולט.
  • Views/Layouts — עץ רכיבים ב-XML; findViewById + setOnClickListener מחברים ללוגיקה (Listener pattern). Resources דרך R.
  • Intent — מעבר בין Activities (explicit/implicit), העברת נתונים ב-putExtra. Manifest מצהיר הכל; ניווט = back stack.
  • רשימות — תבנית Adapter: ListView+ArrayAdapter, Adapter מותאם עם ViewHolder (מיחזור Views), ו-RecyclerView המודרני (onCreate/onBindViewHolder).
  • Fragments — UI מודולרי לשימוש חוזר (≈ components). Services — לוגיקת רקע. Broadcast Receivers + Notifications.
  • Room — ORM מעל SQLite (@Entity/@Dao/@Database) — בדיוק ה-ORM מהרצאה 9, על המכשיר.
  • Threads — לא חוסמים את ה-UI thread; AsyncTask (doInBackground/onPostExecute) לעבודה ברקע.
🔗
חיבורים native מול cross-platform (הרצאה 8): כאן הגרסה ה-native — יותר כוח, יותר קוד, מול React Native (הרצאה 10) הקצר יותר. ה-Listener pattern (הרצאה 8) בכל מקום. ה-back stack = ה-Stack של Expo Router. RecyclerView = FlatList. Room = ORM מהרצאה 9. ה-XML layouts וה-string resources ראינו כבר בהרצאה 8.
➡️
