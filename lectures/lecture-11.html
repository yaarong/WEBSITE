1. אתגרי המובייל + רכיבי Android

🧭
קונטקסט פיתוח למובייל מציב אתגרים שאין ב-web/desktop, ו-Android בנויה כדי להתמודד איתם.
📱
אתגרי המובייל
  • גרסאות OS רבות + הפצות שונות של יצרנים (Samsung, Xiaomi...).
  • גדלי מסך ורזולוציות מגוונים.
  • יעילות — סוללה ומשאבים מוגבלים; ה-OS יכול לסגור תהליכים בכל רגע כדי לפנות משאבים.
  • קלט מגוון — מגע, מחוות (gestures), חיישנים.

Android היא מערכת הפעלה מבוססת Linux לטלפונים, טאבלטים, שעונים, טלוויזיות ועוד. מגיעה עם SDK (אמולטורים, ספריות, תיעוד). רכיבי הליבה של אפליקציה:

רכיבתפקיד
Activityמסך בודד באפליקציה
Fragmentחלק מודולרי לשימוש חוזר בתוך Activity
Intent"הודעה" לתקשורת בין רכיבים (למשל פתיחת מסך)
Serviceלוגיקה ברקע (בלי UI)
Broadcast Receiverמאזין לאירועי מערכת (למשל סיום אתחול)
Content Providerשיתוף נתונים בין אפליקציות

הבסיס תמיד: Java/Kotlin + XML — לוגיקה בקוד, layout ב-XML (ההפרדה מהרצאה 8).

2. Activity ומחזור החיים

🧭
קונטקסט Activity = מסך בודד. יש לו UI (בדרך כלל) ולוגיקה (במחלקה). ה-Activity טוען את ה-layout שלו ב-onCreate.
java · MainActivity.java
public class MainActivity extends Activity {

    @Override
    protected void onCreate(Bundle savedInstanceState) {
        super.onCreate(savedInstanceState);
        setContentView(R.layout.activity_main);
    }
}

וה-layout המתאים (כאן עם ConstraintLayout — מסדר רכיבים לפי אילוצים יחסיים):

xml · res/layout/activity_main.xml
<?xml version="1.0" encoding="utf-8"?>
<androidx.constraintlayout.widget.ConstraintLayout
    xmlns:android="http://schemas.android.com/apk/res/android"
    xmlns:app="http://schemas.android.com/apk/res-auto"
    android:layout_width="match_parent"
    android:layout_height="match_parent">

    <TextView
        android:layout_width="wrap_content"
        android:layout_height="wrap_content"
        android:text="Hello World!"
        app:layout_constraintBottom_toBottomOf="parent"
        app:layout_constraintLeft_toLeftOf="parent"
        app:layout_constraintRight_toRightOf="parent"
        app:layout_constraintTop_toTopOf="parent" />

</androidx.constraintlayout.widget.ConstraintLayout>

ארבעת ה-constraint (top/bottom/left/right ל-parent) "נועלים" את ה-TextView במרכז המסך. אבל רגע — מה כל שורה כאן בעצם אומרת? בואו נפרק את ה-XML שורה-שורה, כי הסינטקס הזה חוזר בכל קובץ layout בקורס:

🔤
אנטומיה של קובץ layout — כל שורה
  • <?xml version="1.0" encoding="utf-8"?> — שורת הצהרה סטנדרטית שאומרת "זהו קובץ XML". מופיעה בראש כל קובץ XML, תמיד אותו דבר.
  • <androidx.constraintlayout...ConstraintLayout> — האלמנט השורש. הוא ה-ViewGroup שעוטף את כל השאר. השם המלא עם הנקודות הוא ה-נתיב המלא של המחלקה (package + שם).
  • xmlns:android="..." — מגדיר את הקידומת (namespace) android:. בלי השורה הזו, כל תכונה שמתחילה ב-android: לא תזוהה. xmlns:app="..." עושה את אותו דבר לקידומת app: (תכונות שמגיעות מספריות, כמו ה-constraints).
  • android:layout_width / android:layout_height — חובה בכל View: כמה רוחב/גובה הוא תופס. הערכים בטבלה למטה.
  • android:text="Hello World!" — תכונה ספציפית ל-TextView: הטקסט שיוצג.
  • app:layout_constraint... — כללי המיקום (רק ב-ConstraintLayout). מסבירים אותם מיד.
  • /> בסוף תג בלי ילדים = "תג שנסגר על עצמו". תג עם ילדים נסגר ב-</TagName> נפרד.

הערך של layout_width/layout_height הוא אחד משלושה — וזו אחת המלכודות הכי נפוצות למתחילים:

ערךמשמעות
match_parent"תמלא את כל ההורה" — קח את כל הרוחב/גובה הזמין.
wrap_content"בדיוק כמה שצריך" — התכווץ לגודל התוכן (למשל רוחב הטקסט עצמו).
16dp (מספר קבוע)גודל מדויק ביחידות dp (ראו הסבר על יחידות בהמשך).
🧲
איך עובדים ה-Constraints (ב-ConstraintLayout) ConstraintLayout ממקם כל View לפי "עוגנים" — אתה אומר "הצד הזה של ה-View צמוד לצד הזה של משהו אחר". התבנית: app:layout_constraintשלי_toשלוOf="מי".
  • layout_constraintTop_toTopOf="parent" — "החלק העליון שלי צמוד לחלק העליון של ההורה".
  • layout_constraintBottom_toBottomOf="parent" — והתחתון שלי לתחתון של ההורה.
  • כשגם עליון וגם תחתון מעוגנים להורה — ה-View "נמתח" בין שניהם ומתמרכז אנכית. אותו דבר עם left+right → מרכוז אופקי. זה הטריק שממרכז את הטקסט.
  • אפשר לעגן גם ל-View אחר לפי ה-id שלו: app:layout_constraintTop_toBottomOf="@id/title" = "מתחת ל-title".
🔄
מחזור החיים (Activity Life-cycle) ה-OS קורא ל-Activity דרך שרשרת callbacks לפי מצבו. זה קריטי במובייל — כי ה-OS יכול לעצור/להרוס את ה-Activity בכל רגע (שיחה נכנסת, מעבר לאפליקציה אחרת). אתה לא קורא לפונקציות האלה — המערכת קוראת להן בשבילך ברגעים המתאימים, ואתה רק "ממלא" אותן במה שצריך לקרות. 
      onCreate ──▶ onStart ──▶ onResume ──▶ [Activity בחזית, פעיל]
                       ▲                          │
                   onRestart                   onPause
                       │                          ▼
                       └────────  onStop  ◀───────┘
                                    │
                                 onDestroy
java · מימוש ה-callbacks
public class MainActivity extends AppCompatActivity {
    @Override protected void onCreate(Bundle b) {
        super.onCreate(b);
        setContentView(R.layout.activity_main);
        Log.i("MainActivity", "onCreate");
    }
    @Override protected void onStart()   { super.onStart();   Log.i("MainActivity", "onStart"); }
    @Override protected void onResume()  { super.onResume();  Log.i("MainActivity", "onResume"); }
    @Override protected void onPause()   { super.onPause();   Log.i("MainActivity", "onPause"); }
    @Override protected void onStop()    { super.onStop();    Log.i("MainActivity", "onStop"); }
    @Override protected void onDestroy() { super.onDestroy(); Log.i("MainActivity", "onDestroy"); }
}

עכשיו החלק החשוב — מה כל אחת מ-6 המתודות האלה בעצם אומרת, מתי המערכת קוראת לה, ומה הגיוני לשים בתוכה:

Callbackמתי המערכת קוראת לומה שמים בו בדרך כלל
onCreate()פעם אחת, כשה-Activity נוצר מאפס. זו נקודת הכניסה.setContentView (טעינת ה-layout), findViewById, חיבור listeners, אתחול משתנים.
onStart()כשה-Activity הופך גלוי על המסך (אבל עדיין לא בהכרח בפוקוס).פעולות שצריכות שהמסך יהיה נראה — בדרך כלל פחות נפוץ.
onResume()כשה-Activity הופך אינטראקטיבי — בחזית, המשתמש יכול ללחוץ. זה ה"מצב הפעיל".הפעלת דברים שצורכים משאבים: מצלמה, GPS, אנימציות, חיישנים.
onPause()כשה-Activity מאבד פוקוס אבל עדיין חלקית נראה (קופץ דיאלוג, שיחה נכנסת, מסך חצי-שקוף מעליו).לעצור את מה שהפעלת ב-onResume: השהיית וידאו, ניתוק מצלמה. צריך להיות מהיר.
onStop()כשה-Activity כבר לא נראה כלל (המשתמש עבר לאפליקציה אחרת או לחץ Home).שמירת נתונים (DB), שחרור משאבים כבדים.
onDestroy()כשה-Activity נהרס סופית (המשתמש סגר אותו, או שה-OS הרג אותו לפינוי זיכרון).ניקוי אחרון — שחרור כל מה שנותר.
onRestart()כשמשתמש חוזר ל-Activity שהיה ב-onStop (לחץ על האפליקציה שוב). אחריו נקרא שוב onStart.רענון נתונים שאולי השתנו בזמן שהיינו ברקע.
🎬
תרחיש אמיתי — נעקוב אחרי הסדר
  • פותחים את האפליקציה: onCreate → onStart → onResume → המסך פעיל.
  • נכנסת שיחת טלפון: onPause (המסך עוד נראה ברקע) → אם השיחה ממשיכה והמסך מתכסה: onStop.
  • חוזרים לאפליקציה: onRestart → onStart → onResume → פעיל שוב.
  • לוחצים Back / סוגרים: onPause → onStop → onDestroy. נגמר.
  • מסובבים את המסך (portrait↔landscape): ה-Activity נהרס ונוצר מחדש! onPause→onStop→onDestroy ואז onCreate→onStart→onResume. לכן חשוב לשמור מצב — אחרת המשתמש מאבד את מה שהקליד.
🔗
הקבלה ל-React זה אותו רעיון של useEffect ב-React Native: קוד שרץ "כשהרכיב מופיע" (≈ onResume) ופונקציית ניקוי שרצה "כשהוא נעלם" (≈ onPause/onStop). פה זה רק מפורש יותר — שש מתודות נפרדות במקום effect אחד.

3. Resources ו-R

🧭
קונטקסט ב-Android, נכסים שאינם קוד (layouts, מחרוזות, צבעים, תמונות) הם resources בתיקיית res/. ניגשים אליהם דרך מחלקה אוטומטית בשם R.
🔤
הפניה ל-resources
  • מ-Java: R.layout.activity_main, R.string.foo, R.id.btnUpdate.
  • מ-XML: @string/foo, @color/black, @drawable/icon.
  • R היא הפניה ל-resource, לא ה-resource עצמו (טעינה בפועל עשויה לדרוש קוד נוסף).
  • R נוצרת ומתעדכנת אוטומטית ע"י כלי הבנייה.

למשל קובץ המחרוזות (זוכרים את ה-i18n מהרצאה 8? קובץ strings.xml נפרד לכל שפה):

xml · res/values/strings.xml
<resources>
    <string name="app_name">My Application</string>
    <string name="button">foo</string>
    <string name="checkbox">bar</string>
    <string name="hello">hello</string>
</resources>

תיקייה כמו values-land/ (landscape) או values-es/ (ספרדית) תספק ערכים חלופיים לאותם שמות — וכך ה-UI מסתגל אוטומטית למצב/שפה.

4. Views, ViewGroup ו-Layouts

🧭
קונטקסט כל רכיב חזותי ב-Android יורש מהמחלקה View. אפשר ליצור Views ב-code, אבל עדיף ב-XML (הפרדת layout מלוגיקה — שוב, הרצאה 8).
java · יצירת View ב-code מול XML
// ב-Java:
TextView hello = new TextView(this);
hello.setText("Hello world");

// ב-XML (מועדף):
// <TextView android:text="Hello world" />
🔤
ViewGroup ו-Layouts ViewGroup = View שיכול להכיל Views אחרים. ה-Layouts הם תת-מחלקות של ViewGroup שמסדרות את הילדים בצורה מסוימת. זה אותו רעיון של עץ רכיבים שראינו ב-View של React Native וב-div של HTML — Layout בתוך Layout בתוך Layout.

סוגי ה-Layout הנפוצים — לכל אחד "אסטרטגיית סידור" משלו:

Layoutאיך מסדר את הילדיםמתי משתמשים
LinearLayoutבשורה אחת — אנכית או אופקית (orientation), אחד אחרי השני.טפסים, רשימת כפתורים, סידור פשוט בטור/שורה.
ConstraintLayoutלפי אילוצים — כל View מעוגן יחסית לאחרים או להורה (ראינו למעלה).מסכים מורכבים ושטוחים (בלי קינון מיותר) — הברירת-מחדל המודרנית.
FrameLayoutמערים את הילדים אחד על השני (כמו שכבות).כשרוצים View אחד על גבי אחר (תמונה עם טקסט מעליה).
RelativeLayoutכל View ממוקם יחסית לאחרים ("מתחת ל-", "מימין ל-").גרסה ותיקה יותר — היום מועדף ConstraintLayout במקומו.

דוגמה — מגוון רכיבים בתוך LinearLayout אנכי (orientation="vertical" → כל רכיב מתחת לקודם):

xml · רכיבים שונים
<LinearLayout
    xmlns:android="http://schemas.android.com/apk/res/android"
    android:layout_width="match_parent"
    android:layout_height="match_parent"
    android:orientation="vertical">

    <Button      android:text="@string/button" ... />
    <ImageButton app:srcCompat="@mipmap/ic_launcher" ... />
    <CheckBox    android:text="@string/checkbox" ... />
    <RadioGroup ...>
        <RadioButton android:text="@string/radio1" ... />
        <RadioButton android:text="@string/radio2" ... />
    </RadioGroup>
</LinearLayout>
🔤
הרכיבים (Widgets) שבדוגמה
  • Button — כפתור עם טקסט (android:text).
  • ImageButton — כפתור עם תמונה במקום טקסט (app:srcCompat מצביע על תמונה ב-res/).
  • CheckBox — תיבת סימון (מסומן / לא מסומן, עצמאי).
  • RadioGroup + RadioButton — קבוצת בחירה שרק אחד מתוכה יכול להיות מסומן בכל רגע. ה-RadioGroup הוא ה-ViewGroup שאוכף את הבלעדיות הזו.
  • EditText — שדה קלט טקסט (ראו בסעיף הבא); TextView — טקסט לתצוגה בלבד.

וזה ה-"סינטקס העיצובי" שמופיע שוב ושוב — התכונות (attributes) הנפוצות שכל View מקבל. כדאי להכיר אותן בעל-פה:

תכונהמה עושהערך לדוגמה
android:idשם ייחודי כדי לאתר את ה-View מ-Java (סעיף הבא)."@+id/btnOk"
android:layout_width / _heightגודל (חובה!) — match_parent/wrap_content/מספר."wrap_content"
android:textהטקסט שיוצג (עדיף הפניה ל-@string/)."@string/title"
android:hintטקסט "רמז" אפור שנעלם כשמתחילים להקליד (ב-EditText)."הכנס שם"
android:textSizeגודל הטקסט — ביחידות sp."18sp"
android:textColor / backgroundצבע טקסט / צבע רקע."@color/black"
android:paddingריווח פנימי — בין גבול ה-View לתוכן שלו."16dp"
android:layout_marginריווח חיצוני — בין ה-View לשכנים שלו."8dp"
android:orientation(LinearLayout) כיוון הסידור."vertical" / "horizontal"
📏
dp מול sp — למה לא פשוט פיקסלים? מסכים שונים = צפיפות פיקסלים שונה. אם נשתמש בפיקסלים גולמיים (px), כפתור ייראה ענק במסך אחד וזעיר באחר. לכן:
  • dp (density-independent pixels) — יחידה ש"מתכווננת" אוטומטית לצפיפות המסך, כך שגודל פיזי דומה בכל מכשיר. משתמשים בה לכל מידה (רוחב, padding, margin).
  • sp (scale-independent pixels) — כמו dp, אבל גם מתחשב בהגדרת גודל הגופן של המשתמש. משתמשים בה רק ל-textSize, כדי לכבד משתמשים שהגדילו את הפונט במכשיר.
🔤
gravity מול layout_gravity — מלכודת נפוצה android:gravity — מיישר את התוכן בתוך ה-View (למשל הטקסט בתוך ה-TextView). android:layout_gravity — מיישר את ה-View עצמו בתוך ההורה. בלבול ביניהם הוא מקור תקלות קלאסי. טריק זיכרון: כל מה שמתחיל ב-layout_ מדבר על היחס של ה-View להורה שלו; השאר מדבר על ה-View עצמו.

5. חיבור layout ללוגיקה

🧭
קונטקסט ה-XML מגדיר איך זה נראה; ה-Java מגדיר מה זה עושה. הגשר ביניהם: android:id ב-XML, ו-findViewById ב-Java שמאתר את ה-View לפי ה-id.
xml · layout עם id-ים
<EditText android:id="@+id/etName"  ... android:hint="@string/etName" />
<Button   android:id="@+id/btnUpdate" ... android:text="@string/btnUpdate" />
<TextView android:id="@+id/tvName"   ... />
java · הלוגיקה
public class MainActivity extends AppCompatActivity {
    @Override
    protected void onCreate(Bundle savedInstanceState) {
        super.onCreate(savedInstanceState);
        setContentView(R.layout.activity_main);

        Button btn = findViewById(R.id.btnUpdate);

        btn.setOnClickListener(view -> {
            EditText edName = findViewById(R.id.etName);
            TextView tvName = findViewById(R.id.tvName);
            tvName.setText(edName.getText().toString().toUpperCase());
        });
    }
}
🔤
\u200E@+id\u200E, findViewById, ו-lambda
  • @+id/etName — ה-+ אומר "צור id חדש בשם הזה". בלי + = הפניה ל-id קיים.
  • findViewById(R.id.btnUpdate) — מאתר את ה-View בזמן ריצה.
  • btn.setOnClickListener(view -> {...}) — Listener pattern מהרצאה 8! כאן בתחביר lambda של Java (view -> {...}) — קצר יותר מאשר new View.OnClickListener(){...}. שוב: מעבירים פונקציה, לא קוראים לה.

6. Intent — מעבר בין Activities

🧭
קונטקסט Intent = "כוונה" — דרך לתקשר בין רכיבים ב-Android, למשל לפתוח Activity אחר או לבקש מהמערכת לבצע פעולה. שני סוגים: explicit (מציינים בדיוק איזה Activity) ו-implicit (מציינים פעולה, והמערכת בוחרת מי יטפל).
java · Explicit מול Implicit
// Explicit — פתח Activity ספציפי שלי
Intent intent = new Intent(this, ActivityName.class);
startActivity(intent);

// Implicit — בקש מהמערכת "לפתוח כתובת" (כל דפדפן שמותקן)
Intent intent = new Intent(Intent.ACTION_VIEW,
                           Uri.parse("https://www.foo.com"));
startActivity(intent);
🔤
פירוק השורות
  • new Intent(this, ActivityName.class) — בונים "מעטפה" עם שני פרטים: this = מאיפה (ה-Activity הנוכחי, ה-context), ו-ActivityName.class = לאן (אובייקט המתאר את מחלקת היעד — לא יוצרים מופע, רק מצביעים על הטיפוס).
  • startActivity(intent) — מוסרים את המעטפה למערכת, והיא פותחת את ה-Activity.
  • ב-implicit: במקום מחלקה נותנים פעולה (ACTION_VIEW) + נתון (Uri.parse(...) ממיר מחרוזת לכתובת). המערכת מחפשת מי יודע לטפל בפעולה כזו (ראו Manifest בהמשך).

העברת נתונים בין Activities עם putExtra / getExtras:

java · Activity שולח (putExtra)
btn.setOnClickListener(view -> {
    EditText edName = findViewById(R.id.etName);
    Intent intent = new Intent(this, MainActivity2.class);
    intent.putExtra("text", edName.getText().toString());
    startActivity(intent);
});
java · Activity מקבל (getExtras)
public class MainActivity2 extends AppCompatActivity {
    @Override
    protected void onCreate(Bundle savedInstanceState) {
        super.onCreate(savedInstanceState);
        setContentView(R.layout.activity_main2);

        TextView tvName = findViewById(R.id.tvName);
        Intent intent = getIntent();
        String name = intent.getExtras().getString("text");
        tvName.setText(name.toUpperCase());
    }
}
🔤
putExtra / getExtras putExtra("text", value) מצרף נתון ל-Intent תחת מפתח. ב-Activity המקבל, getIntent().getExtras().getString("text") שולף אותו. זה ה-mechanism להעברת data בין מסכים — מקביל ל-props בהרצאה 8/10, אבל בין Activities.

7. Manifest ו-Activities Stack

🧭
קונטקסט ה-AndroidManifest.xml הוא "תעודת הזהות" של האפליקציה: הוא מצהיר על כל ה-Activities, ההרשאות, והאייקון. בין השאר, הוא מגדיר intent-filters — אילו Intents כל Activity יודע לטפל בהם.
xml · AndroidManifest.xml (קטע)
<activity android:name=".MainActivity" android:exported="true">
    <intent-filter>
        <action   android:name="android.intent.action.MAIN" />
        <category android:name="android.intent.category.LAUNCHER" />
    </intent-filter>
</activity>
🔤
ה-intent-filter המיוחד ה-action.MAIN + category.LAUNCHER מסמנים את ה-Activity שנפתח כשלוחצים על אייקון האפליקציה — נקודת הכניסה. Activity עם intent-filter של action.VIEW + data scheme="http" יכול לטפל בקישורים — וכך implicit intent של "פתח כתובת" עשוי להגיע אליו.
📚
Activities Stack (Back Stack) כל startActivity דוחף Activity חדש לערימה; כפתור ה-Back מוציא (pop) את העליון וחושף את הקודם. זו בדיוק ערימת ה-Stack מ-Expo Router (הרצאה 10) — אותו רעיון של ניווט-ערימה. 
 Foreground:  [ FooActivity      ]  ◀── המשתמש רואה
              [ AnotherActivity  ]
              [ MainActivity     ]  ◀── תחתית הערימה
                  Back Stack       (Back מוציא את העליון)

★ סיכום ההרצאה

🎯
התמונה המלאה
  • Activity = מסך, עם מחזור חיים (onCreate→onStart→onResume→onPause→onStop→onDestroy) — קריטי כי ה-OS שולט.
  • Views/Layouts — עץ רכיבים ב-XML; findViewById + setOnClickListener מחברים ללוגיקה (Listener pattern). Resources דרך R.
  • Intent — מעבר בין Activities (explicit/implicit), העברת נתונים ב-putExtra. Manifest מצהיר הכל; ניווט = back stack.
🔗
חיבורים native מול cross-platform (הרצאה 8): כאן הגרסה ה-native — יותר כוח, יותר קוד, מול React Native (הרצאה 10) הקצר יותר. ה-Listener pattern (הרצאה 8) בכל מקום. ה-back stack = ה-Stack של Expo Router. Intent + putExtra = ניווט + props בין מסכים ב-RN. ה-XML layouts וה-string resources ראינו כבר בהרצאה 8.
