ארכיטקטורת Android — ViewModel, LiveData, Repository ו-Retrofit

בהרצאה 11 בנינו את אבני הבניין של Android, אבל ערבבנו הכל ב-Activity: גם UI, גם נתונים, גם רשת. ההרצאה הזו עוסקת בארכיטקטורה נכונה — איך לסדר את החלקים כך שהאפליקציה תהיה ניתנת לתחזוקה ועמידה לשינויי מצב. נכיר את Architecture Components של Google: ViewModel (מחזיק נתונים ל-UI), LiveData (נתונים נצפים), Repository ("מקור אמת יחיד"), ולבסוף Retrofit לצריכת API — וכך נחבר את האפליקציה לשרת מהרצאה 9.

🗺️
קונטקסט כללי — למה צריך ארכיטקטורה ב-Android, ה-Activity נהרס ונבנה מחדש בכל שינוי מצב (סיבוב מסך!). אם הנתונים יושבים ב-Activity — הם אובדים. בנוסף, Activity שמכיל גם UI, גם גישה לנתונים וגם רשת מפר את ה-SRP (הרצאה 2) והופך לבלתי-ניתן לתחזוקה. ה-Architecture Components פותרים את שתי הבעיות: מפרידים אחריות, ומחזיקים נתונים במקום ששורד שינויי-מצב.

חלק א' · רקע

1. Room — תזכורת קצרה

🧭
קונטקסט ראינו את Room בהרצאה 11 — נחזור עליו בקצרה כי הוא הבסיס לאחסון המקומי בארכיטקטורה. Room = ספריית מיפוי אובייקטים-SQL מעל SQLite (כלומר ORM, כמו בהרצאה 9). שלושה חלקים:
🔤
שלושת החלקים
  • @Entity — מחלקה שמוגדרת כטבלה.
  • @Dao — Data Access Object: ממשק עם פעולות שליפה/עדכון.
  • @Database — מחלקה ייעודית שמנהלת את הגישה ל-DB.
java · Entity + Dao + Database
@Entity
public class Post {
    @PrimaryKey(autoGenerate = true)
    private int id;
    private String content;
    public Post(String content) { this.content = content; }
    // getters / setters ...
}

@Dao
public interface PostDao {
    @Query("SELECT * FROM post")            List<Post> index();
    @Query("SELECT * FROM post WHERE id = :id") Post get(int id);
    @Insert void insert(Post... posts);
    @Update void update(Post... posts);
    @Delete void delete(Post... posts);
}

@Database(entities = {Post.class}, version = 1)
public abstract class AppDB extends RoomDatabase {
    public abstract PostDao postDao();
}

חלק ב' · Architecture Components

2. ViewModel — מחזיק הנתונים ל-UI

🧭
קונטקסט — הבעיה ה-Activity/Fragment נהרס ונוצר מחדש בכל configuration change (סיבוב מסך, שינוי שפה). אם הנתונים יושבים בו — הם אובדים, וצריך לטעון אותם מחדש. ViewModel פותר זאת: הוא מחזיק את הנתונים של ה-UI, ושורד את שינויי המצב (הוא לא נהרס עם ה-Activity). 
   ┌────────────────────────┐   operation (אירוע UI)   ┌─────────────┐
   │  Activity / Fragment   │ ─────────────────────────▶│  ViewModel  │
   │  (מציג את הנתונים)     │ ◀───────────────────────── │ (מחזיק data)│
   └────────────────────────┘   notification (data שונה) └─────────────┘
💡
מאפיינים ה-ViewModel הוא ה-"model עבור ה-UI": הוא שורד configuration changes, וניתן לשיתוף בין fragments של אותו Activity. (הרעיון מזכיר את ה-state בהרצאה 8 — מקום מרכזי לנתוני המסך — אבל כאן הוא חי מחוץ למחזור החיים של ה-UI.)

3. LiveData — נתונים נצפים (observable)

🧭
קונטקסט LiveData נותן תצפיתיות (observability) לערך מאוחסן: כשהנתון משתנה — כל ה-observers מקבלים הודעה אוטומטית ומתעדכנים. MutableLiveData מאפשר גם לשנות את הערך. זו תבנית ה-Observer (קרובת משפחה של ה-Listener מהרצאה 8): נרשמים לערך, ומקבלים עדכון כשהוא משתנה.
java · ViewModel עם LiveData
public class SampleViewModel extends ViewModel {
    private MutableLiveData<String> foo;

    public MutableLiveData<String> getFoo() {
        if (foo == null) {
            foo = new MutableLiveData<String>();
        }
        return foo;
    }
}
java · ה-Activity נרשם כ-observer
public class MainActivity extends AppCompatActivity {
    private SampleViewModel viewModel;

    @Override
    protected void onCreate(@Nullable Bundle savedInstanceState) {
        super.onCreate(savedInstanceState);
        setContentView(R.layout.activity_main);

        viewModel = ViewModelProviders.of(this).get(SampleViewModel.class);

        // נרשמים לשינויים: כש-foo משתנה, onChanged ירוץ אוטומטית
        viewModel.getFoo().observe(this, new Observer<String>() {
            @Override
            public void onChanged(@Nullable final String foo) {
                getSupportActionBar().setTitle(foo);
            }
        });

        FloatingActionButton btnAdd = findViewById(R.id.btnAdd);
        btnAdd.setOnClickListener(view -> {
            String currentDateTime = DateFormat.getDateTimeInstance().format(new Date());
            viewModel.getFoo().setValue(currentDateTime);   // משנים → ה-observer מתעדכן
        });
    }
}
🔤
observe / setValue
  • ViewModelProviders.of(this).get(...) — מקבל את ה-ViewModel (אותו מופע שורד אם ה-Activity נבנה מחדש).
  • liveData.observe(owner, observer) — רושם observer. onChanged ירוץ בכל שינוי, וגם הוא מודע ל-lifecycle (לא יעדכן UI שנהרס).
  • liveData.setValue(x) — משנה את הערך → כל ה-observers מקבלים onChanged. זה בדיוק הרעיון "UI = פונקציה של state" מהרצאה 8: משנים נתון, וה-UI מתעדכן לבד.

4. התמונה המלאה — Repository כ"מקור אמת יחיד"

🧭
קונטקסט מחברים את הכל לארכיטקטורה מומלצת. ה-Repository הוא שכבה שמרכזת את כל מקורות הנתונים (מסד מקומי + שרת מרוחק) ומציגה אותם כ"מקור אמת יחיד" (single source of truth). ה-UI לא יודע מאיפה הנתונים מגיעים — הוא רק צופה ב-LiveData. 
  ┌─────────────────┐  operation   ┌───────────┐   ┌──────────────┐
  │ Activity/Fragment│ ────────────▶│ ViewModel │──▶│  Repository  │
  │  (RecyclerView)  │ ◀──────────  │           │   │ (מקור אמת   │
  └─────────────────┘  Live<List>  └───────────┘   │   יחיד)      │
                                                     └──────┬───────┘
                                              ┌─────────────┴─────────────┐
                                              ▼                           ▼
                                       ┌────────────┐            ┌────────────────┐
                                       │ Room(SQLite)│            │  Web-service   │
                                       │ אחסון מקומי │            │  (API מרוחק)   │
                                       └────────────┘            └────────────────┘
🔗
השוו לארכיטקטורת השרת מהרצאה 9 זה אותו רעיון בדיוק של שכבות! בהרצאה 9 בצד השרת: controller → service → model. כאן בצד הלקוח: UI → ViewModel → Repository → (Room / Web-service). בכל מקרה — כל שכבה אחראית לדבר אחד, וההפרדה מאפשרת להחליף חלקים בלי לשבור את השאר (SRP מהרצאה 2).

חלק ג' · אפליקציית הדגמה

🧱
מה נבנה אפליקציית פוסטים: RecyclerView שמציג רשימה, עם משיכה-לרענון (pull-to-refresh) וכפתור הוספה. הנתונים זורמים מ-Repository דרך LiveData, מגובים ב-Room מקומית ומסונכרנים עם API מרוחק דרך Retrofit.

5. ה-UI — RecyclerView + ViewModel

ה-layout: RecyclerView בתוך SwipeRefreshLayout (משיכה-לרענון) + כפתור צף להוספה:

xml · activity_main.xml
<androidx.swiperefreshlayout.widget.SwipeRefreshLayout
    android:id="@+id/refreshLayout"
    android:layout_width="match_parent"
    android:layout_height="match_parent">

    <androidx.recyclerview.widget.RecyclerView
        android:id="@+id/lstPosts"
        android:layout_width="match_parent"
        android:layout_height="match_parent" />
</androidx.swiperefreshlayout.widget.SwipeRefreshLayout>

ה-ViewModel חושף את הרשימה כ-LiveData ומאציל הכל ל-Repository:

java · PostsViewModel.java
public class PostsViewModel extends ViewModel {
    private PostsRepository repository;
    private LiveData<List<Post>> posts;

    public PostsViewModel() {
        repository = new PostsRepository();
        posts = repository.getAll();
    }

    public LiveData<List<Post>> get() { return posts; }
    public void add(Post post)    { repository.add(post); }
    public void delete(Post post) { repository.delete(post); }
    public void reload()          { repository.reload(); }
}

ה-Activity מחבר הכל: יוצר ViewModel, נרשם ל-LiveData, ומחבר את הרענון:

java · MainActivity.java
public class MainActivity extends AppCompatActivity {
    private PostsViewModel viewModel;

    @Override
    protected void onCreate(Bundle savedInstanceState) {
        super.onCreate(savedInstanceState);
        setContentView(R.layout.activity_main);

        viewModel = new ViewModelProvider(this).get(PostsViewModel.class);

        RecyclerView lstPosts = findViewById(R.id.lstPosts);
        final PostsListAdapter adapter = new PostsListAdapter(this);
        lstPosts.setAdapter(adapter);
        lstPosts.setLayoutManager(new LinearLayoutManager(this));

        SwipeRefreshLayout refreshLayout = findViewById(R.id.refreshLayout);
        refreshLayout.setOnRefreshListener(() -> viewModel.reload());

        // ה-observer: כשהרשימה משתנה — מעדכנים את ה-adapter, אוטומטית
        viewModel.get().observe(this, posts -> {
            adapter.setPosts(posts);
            refreshLayout.setRefreshing(false);
        });
    }
}
🔤
הזרימה ה-Activity לא מחזיק נתונים ולא יודע על רשת/מסד — הוא רק (1) יוצר ViewModel, (2) צופה ב-LiveData ומעדכן את ה-adapter ב-onChanged, (3) מבקש reload() במשיכה. כל הלוגיקה עברה החוצה. זו הפרדת אחריות במיטבה.

6. Repository ו-LiveData

🧭
קונטקסט ה-Repository מחזיק את ה-PostDao (Room) ואת ה-LiveData. שימו לב שהשליפה מה-DB רצה על thread נפרד (אסור לחסום את ה-UI thread — הרצאה 11!), ומעדכנת את ה-LiveData עם postValue.
java · PostsRepository.java
public class PostsRepository {
    private PostDao dao;
    private PostListData postListData;

    public PostsRepository() {
        LocalDatabase db = LocalDatabase.getInstance();
        dao = db.postDao();
        postListData = new PostListData();
    }

    // LiveData מותאם שטוען מה-DB ברגע שמתחילים לצפות בו
    class PostListData extends MutableLiveData<List<Post>> {
        public PostListData() {
            super();
            setValue(new LinkedList<Post>());
        }
        @Override
        protected void onActive() {           // כשמישהו מתחיל לצפות
            super.onActive();
            new Thread(() -> {                 // thread נפרד — לא חוסם UI
                postListData.postValue(dao.get());
            }).start();
        }
    }

    public LiveData<List<Post>> getAll() { return postListData; }
}
🔤
setValue מול postValue, ו-onActive
  • setValue — מעדכן את ה-LiveData מה-UI thread. postValue — מעדכן מ-thread אחר (בטוח לקרוא מ-background). ההבחנה קריטית.
  • onActive() — נקרא כשמתחילים לצפות ב-LiveData; כאן זו ההזדמנות לטעון נתונים מה-DB ברקע.

7. Retrofit — צריכת API מרוחק

🧭
קונטקסט — סוגרים את המעגל הגענו לחיבור עם השרת! Retrofit היא ספרייה לצריכת REST APIs ב-Android. מגדירים ממשק שמתאר את ה-endpoints עם annotations, ו-Retrofit מייצר את המימוש. זה ה-client שמדבר עם ה-API שבנינו בהרצאה 9 (Express + MongoDB).
java · WebServiceAPI.java — ממשק ה-API
public interface WebServiceAPI {
    @GET("posts")
    Call<List<Post>> getPosts();

    @POST("posts")
    Call<Void> createPost(@Body Post post);

    @DELETE("posts/{id}")
    Call<Void> deletePost(@Path("id") int id);
}
🔤
Annotations של Retrofit = REST @GET("posts"), @POST("posts"), @DELETE("posts/{id}") — כל מתודה ממופה ל-endpoint ולמתודת HTTP. @Body שולח אובייקט כגוף הבקשה (יומר ל-JSON), @Path מזריק ערך ל-URL. זה בדיוק ה-REST API מהרצאה 9 — רק מהצד שצורך אותו (client) במקום מהצד ששירת (server).

בונים מופע Retrofit ומבצעים קריאה אסינכרונית עם enqueue:

java · PostAPI.java
public class PostAPI {
    private MutableLiveData<List<Post>> postListData;
    private PostDao dao;
    Retrofit retrofit;
    WebServiceAPI webServiceAPI;

    public PostAPI(MutableLiveData<List<Post>> postListData, PostDao dao) {
        this.postListData = postListData;
        this.dao = dao;

        retrofit = new Retrofit.Builder()
            .baseUrl(MyApplication.context.getString(R.string.BaseUrl))
            .addConverterFactory(GsonConverterFactory.create())
            .build();
        webServiceAPI = retrofit.create(WebServiceAPI.class);
    }

    public void get() {
        Call<List<Post>> call = webServiceAPI.getPosts();
        call.enqueue(new Callback<List<Post>>() {
            @Override
            public void onResponse(Call<List<Post>> call, Response<List<Post>> response) {
                new Thread(() -> {
                    dao.clear();
                    dao.insertList(response.body());        // שמירה מקומית
                    postListData.postValue(dao.get());      // עדכון ה-LiveData
                }).start();
            }
            @Override
            public void onFailure(Call<List<Post>> call, Throwable t) { }
        });
    }
}
🔤
enqueue, GsonConverter, ו-onResponse
  • GsonConverterFactory — ממיר אוטומטית JSON ↔ אובייקטי Java (כמו ש-Mongoose עשה ל-documents בהרצאה 9).
  • call.enqueue(callback) — מבצע את הבקשה אסינכרונית (לא חוסם UI). כשמגיעה תשובה — onResponse (הצלחה) או onFailure (כשל). שוב Listener/Callback pattern.
  • בהצלחה: שומרים מקומית ל-Room, ואז מעדכנים את ה-LiveData עם postValue → ה-UI מתעדכן אוטומטית. זה ה-"single source of truth": גם offline (מ-Room) וגם online (מהשרת) — דרך אותו LiveData.

ולבסוף — ה-Repository מאחד מסד מקומי + API מאחורי ממשק אחד:

java · PostsRepository — מאחד הכל
public class PostsRepository {
    private PostDao dao;
    private PostListData postListData;
    private PostAPI api;

    public PostsRepository() {
        LocalDatabase db = LocalDatabase.getInstance();
        dao = db.postDao();
        postListData = new PostListData();
        api = new PostAPI(postListData, dao);
    }

    public LiveData<List<Post>> getAll() { return postListData; }
    public void add(final Post post)    { api.add(post); }
    public void delete(final Post post) { api.delete(post); }
    public void reload()                { api.get(); }
}

★ סיכום ההרצאה

🎯
התמונה המלאה
  • הבעיה: Activity נהרס בשינויי-מצב (אובדן נתונים) + ערבוב אחריות (UI+data+רשת ביחד).
  • ViewModel — מחזיק את נתוני ה-UI ושורד configuration changes; ניתן לשיתוף בין fragments.
  • LiveData — ערך נצפה: משתנה → ה-observers מתעדכנים אוטומטית (Observer pattern). setValue (UI thread) מול postValue (background).
  • Repository — "מקור אמת יחיד" שמאחד Room (מקומי) + Web-service (מרוחק) מאחורי ממשק אחד.
  • Retrofit — client ל-REST API: ממשק עם @GET/@POST/@DELETE, המרת JSON עם Gson, קריאה אסינכרונית עם enqueue.
  • הזרימה: UI צופה ב-ViewModel → Repository מביא מ-Room/Retrofit → מעדכן LiveData → ה-UI מתעדכן לבד. הכל על threads נכונים.
🔗
חיבורים — סוגרים את כל הקורס הארכיטקטורה כאן = אותן שכבות מהשרת בהרצאה 9 (controller→service→model), בצד הלקוח (UI→ViewModel→Repository). ה-Retrofit צורך בדיוק את ה-REST API (Express+MongoDB) מהרצאה 9. ה-LiveData הוא "UI = פונקציה של state" מהרצאה 8, בניב Android. Room = ה-ORM מהרצאה 9. וההפרדה לשכבות = ה-SRP מהרצאה 2. Foo News / פוסטים ליווה אותנו לאורך כל הדרך — web, server, מסד נתונים, React, מובייל cross-platform ו-native.
