חלק א' · Docker

🐳
קונטקסט — מה Docker פותר Docker אורז את האפליקציה יחד עם כל סביבת ההרצה (מערכת הפעלה, Node, ספריות) ל"קופסה" אחת — image — שרצה זהה בכל מחשב. זה מחסל את "אצלי זה עובד ואצלך לא". Image = התבנית הקפואה · Container = מופע פעיל של ה-image. נראה את הרעיונות המרכזיים על אפליקציית Express קטנה.

1. ה-Dockerfile — המתכון

נתחיל מאפליקציית Express מינימלית (app.js שמחזיר "Hello world" על פורט 3000). ה-dockerfile מתאר שלב-שלב איך לבנות את ה-image:

dockerfile
FROM node:lts-alpine3.21    # תמונת בסיס — Node על Alpine (Linux קטן)
WORKDIR /app               # תיקיית העבודה בתוך הקופסה
COPY . .                   # העתק את כל הקוד מהמחשב לקופסה
RUN npm install            # התקן תלויות (בזמן הבנייה)
EXPOSE 3000                # הצהר שהאפליקציה מאזינה ל-3000
CMD ["node", "app.js"]     # הפקודה שמריצה את האפליקציה
🔤
ההוראות העיקריות
  • FROM — מאיזו תמונת בסיס מתחילים (alpine = הפצת Linux זעירה → image קטן).
  • WORKDIR — קובע את הספרייה הפעילה בתוך הקופסה.
  • COPY — מעתיק קבצים מהמחשב אל ה-image.
  • RUN — מריץ פקודה בזמן הבנייה (התקנות, קומפילציה).
  • EXPOSE — תיעוד של הפורט שהקונטיינר מאזין לו.
  • CMD — הפקודה שתרוץ כשהקונטיינר עולה.

2. build, run ומיפוי פורטים

terminal
$ docker build -t best_app_image .   # בונה image בשם best_app_image
[+] Building ... FINISHED

$ docker run -p 4000:3000 best_app_image  # מריץ container
🔌
-p 4000:3000 — מיפוי פורטים (host:container) הקונטיינר מאזין בפנים ל-3000, אבל אנחנו "מחברים" אותו לפורט 4000 של המחשב. לכן בדפדפן ניגשים ל-localhost:4000 ורואים "Hello world". השמאלי = פורט המחשב, הימני = פורט הקונטיינר.

אפשר לראות ב-Docker Desktop את ה-image (עם גודלו, למשל ~232MB), ולצפות בקונטיינר הרץ — סטטיסטיקות (CPU, זיכרון, רשת), ואפילו את מערכת הקבצים בתוכו (תיקיות לינוקס סטנדרטיות bin, etc, app... — קופסה שלמה!).

3. Layers — שכבות וה-cache

🥞
הרעיון המרכזי
  • כל פקודה ב-Dockerfile = שכבה (layer).
  • אם שכבה משתנה — כל השכבות שאחריה חייבות להיבנות מחדש.

בבנייה חוזרת, Docker מציג CACHED על שכבות שלא השתנו ומדלג עליהן — מה שמאיץ מאוד. אבל זה אומר שסדר הפקודות קריטי לביצועים.

⚡
מלכודת ה-cache — ואיך לתקן ב-Dockerfile הראשון כתבנו COPY . . ואז RUN npm install. בעיה: כל שינוי בקוד מבטל את ה-cache של COPY, וגורם ל-npm install לרוץ מחדש (איטי!) גם אם התלויות לא השתנו. הפתרון: להעתיק קודם רק את package.json, להתקין, ורק אז להעתיק את הקוד:
dockerfile · ממוטב ל-cache
FROM node:lts-alpine3.21
WORKDIR /app
COPY package*.json ./    # 1. רק קבצי התלויות
RUN npm install          # 2. שכבה זו תשתמש ב-cache כל עוד package.json לא השתנה
COPY ./src ./src         # 3. הקוד — שינוי כאן לא מבטל את ה-npm install
CMD ["node", "./src/app.js"]
עכשיו שינוי בקוד מבטל רק את שכבת ה-COPY ./src — וה-npm install היקר נשאר CACHED.

4. .dockerignore ו-Multi-stage builds

🚫
.dockerignore כמו .gitignore — אומר ל-Docker מה לא להעתיק ל-image (ל-build context). בעיקר node_modules (נבנה מחדש בתוך הקופסה ב-npm install), build, .git וכו'. מקטין את ה-context ומזרז.
.dockerignore
node_modules
build
.dockerignore
.gitignore
Dockerfile
README.md

Multi-stage build — לאפליקציית React, ה-image עם כל ה-node_modules ענק (~756MB). הטריק: שלב בנייה שמייצר את קבצי ה-static, ואז שלב סופי קטן (nginx) שמגיש רק אותם — וזורק את כל כלי הבנייה:

dockerfile · multi-stage (frontend)
FROM node:lts-alpine3.21 AS build    # ← שלב 1: "build"
WORKDIR /app
COPY package*.json ./
RUN npm install
COPY . .
RUN npm run build                    # מייצר /app/build (static)

FROM nginx:alpine                    # ← שלב 2: image סופי קטן
COPY --from=build /app/build /usr/share/nginx/html   # מעתיק רק את התוצר
EXPOSE 80
📉
התוצאה COPY --from=build מעתיק רק את התוצר הסופי מהשלב הראשון אל nginx. כל הכלים, ה-node_modules וקבצי הביניים נשארים מאחור. ה-image יורד מ-~756MB ל-~81MB.

5. docker-compose — כמה שירותים יחד

אפליקציה אמיתית = כמה קונטיינרים (backend, frontend, DB). במקום להריץ כל אחד ידנית, docker-compose.yml מתאר את כולם, ו-docker-compose up --build בונה ומריץ הכל בפקודה אחת:

yaml · docker-compose.yml
services:
  backend:
    build: ./backend         # בנה מה-Dockerfile שבתיקייה
    ports:
      - "4000:3000"
  frontend:
    build: ./frontend
    ports:
      - "5000:80"
    depends_on:
      - backend              # frontend עולה אחרי backend
🌐
מלכודת ה-CORS ה-frontend (ב-localhost:5000) עושה fetch ל-backend (localhost:4000) — origin שונה → הדפדפן חוסם עם CORS error. הפתרון בצד השרת: להתקין ולהפעיל את חבילת cors.
javascript · backend
const cors = require('cors');
const app = express();
app.use(cors());        // ← מתיר בקשות cross-origin

6. Mongo ב-compose — networking, volumes, env

מוסיפים שירות db עם image מוכן של mongo. שלוש נקודות חשובות:

א. רשת — מתחברים לפי שם השירות, לא localhost

🔌
ECONNREFUSED על localhost:27017 ה-backend ניסה mongoose.connect('mongodb://localhost:27017/news') ונכשל — כי בתוך הקונטיינר, localhost זה הקונטיינר עצמו, לא ה-DB. ב-compose, כל שירות נגיש לאחרים לפי שמו. התיקון: mongodb://db:27017/news.

ב. Volumes — שמירת מידע (persistence)

💾
בלי volume — המידע נמחק עם הקונטיינר קונטיינר הוא חולף; כשמוחקים אותו, כל מה שבתוכו נעלם. כדי שה-DB ישרוד, ממפים named volume לתיקיית הנתונים של Mongo (/data/db). ה-volume חי בנפרד מהקונטיינר ונשאר גם אחרי מחיקתו.

ג. Init script + env_file

אפשר לזרוע נתונים התחלתיים: סקריפט init-mongo.js שממופה ל-/docker-entrypoint-initdb.d/ ירוץ פעם אחת כשה-DB נוצר. ו-env_file: .env מזריק משתני סביבה (כמו MONGO_URI) — כך אין סודות בקוד.

yaml · docker-compose.yml מלא
services:
  backend:
    build: ./backend
    ports: ["4000:3000"]
    env_file:
      - .env                 # מזריק MONGO_URI וכו'
    depends_on:
      - db
  frontend:
    build: ./frontend
    ports: ["5000:80"]
    depends_on:
      - backend
  db:
    image: mongo:latest
    restart: always
    volumes:
      - ./init/db/init-mongo.js:/docker-entrypoint-initdb.d/init-mongo.js  # seed
      - mongo_data:/data/db  # ← persistence

volumes:
  mongo_data:                # named volume
javascript · init/db/init-mongo.js
db = db.getSiblingDB('news');
db.articles.insertMany([
    { title: "foo" },
    { title: "bar" },
    { title: "caz" }
]);
🎯
סיכום Docker בשורה Dockerfile בונה image בשכבות (סדר חכם = cache טוב), multi-stage מקטין אותו, ו-compose מריץ כמה שירותים שמדברים לפי שם, עם volumes לשמירת מידע ו-env_file לסודות.

חלק ב' · מודל ה-Event Loop

המצגת מסכמת עם מודל חזותי של איך JavaScript מטפלת בקוד אסינכרוני. ארבעה שחקנים:

Stack / Heapהקוד הסינכרוני שרץ עכשיו
→
Web APIsטיימרים, רשת — רצים "בצד"
→
Task queuecallbacks של setTimeout וכו'
🔁
איך זה עובד — הדוגמה setTimeout(fn, 2000)
  1. ה-setTimeout רץ ב-stack, אבל ה-callback שלו נשלח ל-Web APIs (שם רץ הטיימר) — וה-stack ממשיך הלאה (מדפיס bar).
  2. כשהטיימר מסתיים, ה-callback עובר ל-Task queue.
  3. ה-Event loop בודק: אם ה-stack ריק — הוא לוקח משימה מהתור ומריץ אותה.

▸ מחזור החיים של Promise

ל-Promise יש אובייקט ב-heap עם State (pending → fulfilled), OnFulfilled (ה-callback שנרשם ב-.then) ו-Result. כשה-resolve(42) רץ, ה-state הופך ל-fulfilled, ה-result נשמר (42), וה-callback של .then נשלח ל...Microtasks queue:

⚡
שני התורים — וסדר העדיפויות
  • ל-Microtasks queue (callbacks של Promises/.then) יש עדיפות על ה-Task queue (setTimeout).
  • ה-event loop מרוקן את כל ה-microtasks; רק כשהוא ריק — לוקחים משימה אחת מה-task queue.
  • אחרי שאותה משימה הסתיימה — בודקים שוב את ה-microtasks. וחוזר חלילה.
🎯
רוצה לתרגל את זה לעומק? המודל הזה הוא בדיוק מה שפותר את "חידות הפלט" (מה הסדר של setTimeout מול Promise.then מול await). יש עמוד נפרד עם תיאוריה מלאה + 5 חידות פתורות שורה-שורה + 5 דוגמאות נוספות: ⏳ Promise · async · await — Event Loop.

חלק ג' · CI/CD ⚙️

🔄
קונטקסט — מה הבעיה ש-CI/CD פותר כשעובדים בצוות, כל שינוי קוד חייב להיבדק (build, tests) לפני שמתמזג, ובסוף — להגיע לפרודקשן. לעשות את זה ידנית בכל פעם זה איטי ומועד לטעויות. CI/CD = להפוך את כל הצינור הזה לאוטומטי: כל push בונה, בודק, ומדווח — ובסוף אפילו פורס לבד.

8. GitHub Actions — Workflows

ב-GitHub, ה-CI/CD מוגדר בקובצי workflow מסוג YAML שיושבים ב-.github/workflows/. כל קובץ מתאר: מתי לרוץ ומה לעשות. דוגמה — בדיקת אפליקציית Python:

yaml · .github/workflows/python-app.yml
name: Python application

on:                          # ← מתי להריץ (triggers)
  push:
    branches: [ "main" ]
  pull_request:
    branches: [ "main" ]

jobs:
  build:
    runs-on: ubuntu-latest   # ← על איזה "מחשב" (runner) לרוץ
    steps:
      - uses: actions/checkout@v3            # מושך את הקוד
      - name: Set up Python 3.10
        uses: actions/setup-python@v3
        with:
          python-version: "3.10"
      - name: Install dependencies
        run: |
          pip install flake8 pytest
          if [ -f requirements.txt ]; then pip install -r requirements.txt; fi
      - name: Lint with flake8
        run: flake8 .
      - name: Test with pytest
        run: pytest
🔤
מבנה ה-workflow
  • name — שם ה-workflow.
  • on — הטריגרים: מתי לרוץ (כאן: push או pull_request לענף main).
  • jobs — מה לעשות. כל job רץ על runner (runs-on: ubuntu-latest).
  • steps — הצעדים בתוך ה-job. uses: = פעולה מוכנה מה-Marketplace (כמו checkout); run: = פקודת shell.

אחרי commit, ה-workflow רץ אוטומטית. בלשונית Actions רואים אותו עובר: Queued → In progress → ✓ Success (או ✗ נכשל). אפשר לפתוח כל job ולראות את הפלט של כל צעד (למשל 1 passed מ-pytest).

🖥️
Runners המחשבים שמריצים את ה-jobs. GitHub-hosted (סטנדרטי, מנוהל ע"י GitHub — Linux/Windows/Mac) או Self-hosted (מחשב משלך). יש מכסת דקות חינמית חודשית.

9. ה-CI בפעולה — תופס באגים

הערך האמיתי: כשמשהו נשבר, ה-CI עוצר את זה לפני המיזוג. נדגים שני מקרים מתוך ה-demo:

א. טסט נכשל (דרך Pull Request)

משנים את hello() שיחזיר 'world2' במקום 'world', ופותחים PR. ה-checks רצים על ה-PR ונכשלים:

Test with pytest — FAILED
    def test_hello():
>       assert hello() == 'world'
E       AssertionError: assert 'world2' == 'world'
test_demo.py:6: AssertionError
1 failed
Error: Process completed with exit code 1.
🛑
"All checks have failed" ב-PR מופיע ✗ אדום — הקוד הבעייתי לא ימוזג ל-main. כך באג נתפס אוטומטית לפני שהוא מגיע לקוד הראשי. (אפשר אפילו לדרוש שכל ה-checks יעברו כתנאי למיזוג.)

ב. תלות חסרה

מוסיפים import numpy as np בלי להתקין — ה-CI נכשל עם ModuleNotFoundError: No module named 'numpy'. התיקון: קובץ requirements.txt (שה-workflow מתקין ממנו):

requirements.txt
numpy
💡
הלקח ה-CI רץ בסביבה נקייה בכל פעם — אז הוא חושף "אצלי זה עבד" שנובע מתלויות לא-מוצהרות. כל תלות חייבת ב-requirements.txt (או package.json).

10. ה-CD בפעולה — push image ל-Docker Hub

החלק השני: אחרי שהקוד תקין, בונים image ודוחפים אותו ל-Docker Hub אוטומטית. לשם כך צריך אישורי התחברות — ואסור לשים אותם בקוד.

🔑
Secrets — סודות מוצפנים ב-Settings → Secrets and variables → Actions מגדירים סודות מוצפנים (למשל DOCKER_USERNAME, DOCKER_PASSWORD = access token מ-Docker Hub). ה-workflow ניגש אליהם דרך ${{ secrets.NAME }} — הם אף פעם לא מופיעים בקוד או בלוגים.
yaml · .github/workflows/push-to-hub.yml
name: Publish Docker image

on:
  release:
    types: [published]       # ← רץ כשמפרסמים Release חדש

jobs:
  push_to_registry:
    runs-on: ubuntu-latest
    steps:
      - name: Check out the repo
        uses: actions/checkout@v4
      - name: Log in to Docker Hub
        uses: docker/login-action@...
        with:
          username: ${{ secrets.DOCKER_USERNAME }}    # ← מהסודות
          password: ${{ secrets.DOCKER_PASSWORD }}
      - name: Extract metadata (tags, labels)
        id: meta
        uses: docker/metadata-action@...
        with:
          images: <user>/demo
      - name: Build and push Docker image
        uses: docker/build-push-action@...
        with:
          context: .
          file: ./Dockerfile
          push: true                            # ← דוחף ל-registry
          tags: ${{ steps.meta.outputs.tags }}
          labels: ${{ steps.meta.outputs.labels }}
🚀
הזרימה המלאה יוצרים Release ב-GitHub (תגית כמו v1.0.0) → ה-trigger on: release מפעיל את ה-workflow → הוא מתחבר ל-Docker Hub, בונה את ה-image ודוחף אותו. אחר כך ב-Docker Hub מופיע ה-image עם התגיות latest ו-v1.0.0. זהו Continuous Deployment בפעולה — מגרסה ל-image באוויר, אוטומטית.
🔤
Release ו-Semantic Versioning Release = "צילום" רשמי של גרסה, עם תגית (tag). נהוג v1.0.0 (major.minor.patch — "semantic versioning"), ולגרסאות לא-יציבות סיומת כמו v0.2.0-alpha.

קונטקסט — למה צריך אסטרטגיית ענפים כשהרבה מפתחים עובדים על אותו קוד, לא יכולים כולם לדחוף ישר ל-main — זה כאוס ומסוכן ללקוחות. אסטרטגיית branching מגדירה אילו ענפים קיימים, מה התפקיד של כל אחד, ואיך הקוד זורם ביניהם — מהמפתח הבודד ועד הפרודקשן. 11. אסטרטגיות נפוצות — סקירה

אסטרטגיההרעיון
Trunk-basedכולם עובדים על ענף אחד ("trunk") עם commits קטנים ותכופים. פשוט, אבל דורש משמעת ובדיקות אוטומטיות חזקות.
Feature branchכל פיצ'ר מקבל ענף משלו, שמתפצל מ-develop וחוזר אליו במיזוג. כך עבודות לא דורכות אחת לשנייה.
Forkכל מתורם משכפל את כל המאגר לחשבון שלו, עובד אצלו, ושולח Pull Request. נפוץ בקוד פתוח.
GitHub flowגרסה פשוטה: main + ענפי פיצ'ר קצרי-חיים, כל מיזוג דרך PR. מתאים ל-deploy תכוף.

12. Git Flow — חמשת הענפים והתהליך

האסטרטגיה המפורטת ביותר. יש בה חמישה סוגי ענפים, וזה מה שכל שם אומר:

ענףמה הואמתפצל מ-ממוזג ל-
main / masterהקוד ה"אמיתי" — הגרסה שהלקוחות משתמשים בה. יציב תמיד.——
developענף האינטגרציה — "קוד שבפיתוח". כאן מצטברים הפיצ'רים לפני שחרור.main(דרך release)
feature/*ענף לכל פיצ'ר בודד (למשל feature/add-button). יכולים להתקיים כמה במקביל.developdevelop
releaseהכנת גרסה לשחרור. מייצבים, מתקנים באגים אחרונים — בלי לחסום פיתוח חדש על develop.developmain + develop
hotfixתיקון דחוף לבאג קריטי שהתגלה בפרודקשן. עוקף את develop כדי לתקן מהר.mainmain + develop
Git Flow — תרשים הזרימה
🔄
התהליך בגרף — שלב-שלב
  1. main מחזיק את הקוד היציב, אבל אי אפשר לקפוץ עליו מגרסה לגרסה (יותר מדי מפתחים, סיכון ללקוחות).
  2. פותחים develop לקוד שבפיתוח. אבל גם שם הרבה מפתחים — אז כל אחד פותח feature branch משלו מ-develop.
  3. מסיימים פיצ'ר → ממזגים חזרה ל-develop. אם develop לא השתנה בינתיים — מיזוג fast-forward (פשוט מזיזים את המצביע קדימה, בלי commit מיזוג).
  4. אם develop כן השתנה (פיצ'ר אחר מוזג) — אי אפשר fast-forward. מושכים (pull) את השינויים לתוך ה-feature branch, ממזגים, ורק אז חוזרים ל-develop.
  5. כש-develop מוכן → פותחים release branch עם הפיצ'רים. מייצבים ומתקנים שם באגים — ובמקביל develop ממשיך להתקדם בלי הפרעה.
  6. הגרסה אושרה → ממזגים את release ל-main (שחרור!) וגם בחזרה ל-develop (כדי שהתיקונים לא יאבדו).
  7. באג קריטי בפרודקשן → פותחים hotfix מ-main, מתקנים, וממזגים ישירות ל-main (מהר!) וגם ל-develop.
🔤
"אל תשכח את develop" הכלל החוזר: כל מה שמתמזג ל-main (release או hotfix) חייב להתמזג גם בחזרה ל-develop — אחרת develop "פיגר" אחרי הקוד שבפרודקשן, והבאג יחזור בגרסה הבאה.

13. סביבות (Environments)

מפתחים כותבים, מריצים ובודקים קוד מקומית. אבל כשהקוד "מוכן" — איך הוא מגיע ללקוחות? לפרוס ישירות מהמחשב המקומי לפרודקשן זה מסוכן: הסביבות לרוב לא זהות, יש באגים, וצריך בדיקות ידניות. לכן מכניסים סביבות ביניים:

Localהמחשב של המפתח
→
Development / Testingבדיקות אינטגרציה
→
Staging / Pre-prod"חזרה גנרלית" — זהה לפרודקשן
→
Productionהלקוחות
🔗
איך זה מתחבר ל-Git Flow הסביבות ממופות לענפים: feature branches → Local, develop → Testing, release → Staging, main → Production. כל מעבר קדימה הוא הזדמנות לתפוס בעיות לפני שהן מגיעות ללקוח.

★ סיכום ההרצאה

🎯
מה לקחנו
  • Docker: Dockerfile בונה image ב-שכבות (סדר חכם = cache); multi-stage מקטין; compose מריץ כמה שירותים שמדברים לפי שם, עם volumes (persistence), env_file (סודות), ופתרון CORS.
  • Event Loop: stack → Web APIs → task queue, עם microtasks (Promises) בעדיפות על tasks (setTimeout).
  • CI/CD: אוטומציה של Code→Build→Test→Integrate (CI) ועד Release→Deploy (CD). ב-GitHub Actions: workflow ב-.github/workflows/*.yml עם on (טריגרים) ו-jobs/steps; ה-CI תופס באגים לפני מיזוג; secrets לאישורים; release → build & push image ל-Docker Hub.
  • Branching: Git Flow עם 5 ענפים — main (פרודקשן), develop (אינטגרציה), feature/* (פיצ'ר), release (הכנת גרסה), hotfix (תיקון דחוף). מה שמתמזג ל-main חוזר גם ל-develop. + 4 סביבות (Local→Testing→Staging→Production).
