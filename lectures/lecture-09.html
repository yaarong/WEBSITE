גישה לנתונים — ממסדים רלציוניים (SQL) ו-ORM ועד NoSQL, MongoDB ו-Mongoose

בהרצאה 7 ה-Model של אתר ה-Foo News היה מערך כתבות קשיח בזיכרון — הבטחנו שנחליף אותו במסד נתונים אמיתי. הגיע הרגע. ההרצאה הזו עוסקת בגישה לנתונים מתמשכים (persistence): נתחיל ממסדים רלציוניים ו-SQL, נראה את הבעיה של חיבור קוד מונחה-עצמים למסד מונחה-טבלאות (JDBC, ORM והביקורת עליו), ומשם נעבור ל-NoSQL ול-MongoDB — ונבנה את שכבת הנתונים המלאה של Foo News ב-Mongoose (ה-MERN stack).

חלק א' · המוטיבציה

1. דפים דינמיים — ולמה צריך מסד נתונים

🧭
קונטקסט אתר אמיתי הוא דינמי: כתובת כמו news.foo.com/article/1 צריכה להציג כתבה מס' 1 — תמונה, כותרת, גוף, ותגובות — והכל מגיע מנתונים שמורים. הכתבות נוצרות, נערכות ונמחקות, וצריך שהן יישמרו גם אחרי שהשרת נכבה. מערך בזיכרון (כמו בהרצאה 7) נעלם ברגע שמפעילים מחדש. צריך persistence — מסד נתונים.
🔗
המשך ישיר מהרצאה 7 זוכרים את ה-models/articles.js מהרצאה 7? הוא החזיק מערך כתבות בקוד, עם getArticle(id). אמרנו אז: "ה-controller לא יודע אם זה מערך, קובץ או מסד נתונים — הוא רק קורא לפונקציה". בזכות אותה הפרדה (ה-DAL), נחליף עכשיו את המערך במסד נתונים אמיתי — בלי לגעת ב-controller או ב-routes. זה בדיוק התשלום על ההשקעה בארכיטקטורה נקייה.

חלק ב' · מסדים רלציוניים (SQL)

2. טבלאות, שורות ו-SQL

🧭
קונטקסט הגישה הקלאסית והנפוצה ביותר היא מסד נתונים רלציוני (Relational DB): הנתונים מאוחסנים בטבלאות. כל טבלה היא אוסף שורות (rows), וכל שורה היא רשומה אחת עם עמודות (columns) קבועות. שואלים את המסד בשפה ייעודית — SQL (Structured Query Language).

למשל, טבלת כתבות:

IdTitleBody...
1FooBar bar...
2HelloWorld...

\u200FTable: Articles\u200F

שולפים נתונים עם פקודת SELECT, ומסננים עם WHERE:

sql · שליפת כתבה לפי id
SELECT Title, Body FROM Articles
WHERE Id = 1;
🔤
אנטומיה של שאילתת SQL
  • SELECT Title, Body — אילו עמודות להחזיר (* = הכל).
  • FROM Articles — מאיזו טבלה.
  • WHERE Id = 1 — תנאי הסינון: אילו שורות.
התוצאה היא טבלה (תת-קבוצה של השורות והעמודות) — כאן שורה אחת: Foo | Bar bar.

3. קשרים — one-to-many ו-many-to-many

🧭
קונטקסט — מאיפה ה-"רלציוני" העוצמה של המסד הרלציוני היא בקשרים בין טבלאות. במקום לשכפל נתונים, מקשרים שורות בעזרת מפתחות (id-ים). שני סוגי קשר נפוצים:

One-to-many (אחד-לרבים) — לכתבה אחת יש הרבה תגובות. כל תגובה מחזיקה ArticleId שמצביע על הכתבה שלה:

IdArticleIdTitle
11Yes I agree
21Bla bla

\u200FTable: Comments — שתי התגובות שייכות לכתבה 1\u200F

sql · כל התגובות של כתבה 1
SELECT Id, Title FROM Comments
WHERE ArticleId = 1;

Many-to-many (רבים-לרבים) — לכתבה יכולים להיות כמה מחברים, ולמחבר כמה כתבות. אי אפשר לייצג זאת בעמודה אחת, אז משתמשים בטבלת קישור (join table):

ArticleIdAuthorId
11
12
21

\u200FTable: Articles-Authors — כתבה 1 נכתבה ע"י מחברים 1 ו-2\u200F

כל שורה בטבלת הקישור היא "צמד" שמקשר כתבה למחבר. כדי לקבל את מחברי כתבה 1, מבצעים JOIN בין שלוש הטבלאות. זה גמיש מאוד — אבל גם מתחיל להיות מסורבל. נחזיק את הדוגמה הזו, כי בהמשך נראה איך MongoDB מייצג את אותם קשרים אחרת לגמרי.

חלק ג' · לחבר קוד למסד — JDBC ו-ORM

4. ה-mismatch בין OOP לטבלאות + Driver/API

🧭
קונטקסט — הבעיה המרכזית השרת שלנו (ה-client של המסד) מדבר בשאילתות SQL, מבצע אותן במסד, ומקבל בחזרה טבלאות. אבל הקוד שלנו מונחה-עצמים — אנחנו רוצים לעבוד עם אובייקטים (Article), לא עם שורות וטבלאות. הפער הזה נקרא "object-relational impedance mismatch", וכל הסיפור של חלק ג' הוא איך לגשר עליו. 
  ┌──────────────────────┐   "SELECT * FROM Article..."   ┌──────────┐
  │   Client (web server)│ ──────────────────────────────▶│ Database │
  │   קוד מונחה-עצמים    │ ◀──────────────────────────────│          │
  └──────────────────────┘     טבלה: Id | Title | Body    └──────────┘
            │
            ▼   צריך להמיר את הטבלה לאובייקטים:  List<Article>
🔤
Driver ו-API כדי שהקוד יוכל לדבר עם המסד, צריך driver — ספרייה שמתרגמת בין הקוד למסד. ה-driver ממומש מול API סטנדרטי. בעולם ה-Java/Windows: JDBC (Java DataBase Connectivity), ODBC (Open DB Connectivity), OLEDB. כל driver הוא מימוש ספציפי של ה-API עבור טכנולוגיית מסד מסוימת (MySQL, PostgreSQL...).

5. JDBC — גישה גולמית למסד

🧭
קונטקסט JDBC היא ה-API של Java לתקשורת עם מסדים. זו הגישה ה"גולמית": אנחנו כותבים את ה-SQL כמחרוזת, מריצים אותו, ועוברים על התוצאות שורה-שורה כדי לבנות אובייקטים. נתחיל מהמחלקה Article (POJO — אובייקט פשוט):

java · Article.java
public class Article {
    private int id;
    private String title;
    private String body;

    public Article(int id, String title, String body) {
        this.id = id;
        this.title = title;
        this.body = body;
    }
    public int getId()     { return id; }
    public String getTitle() { return title; }
    public String getBody()  { return body; }
}

קריאה (Read) — מתחברים, מריצים שאילתה, וממירים כל שורה לאובייקט:

java · JDBC — שליפה
Connection connection = DriverManager.getConnection(
    "jdbc:mysql://localhost/news",
    "username",
    "password");

Statement statement = connection.createStatement();
ResultSet results = statement.executeQuery(
    "SELECT id, title, body FROM Article");

List<Article> articles = new ArrayList<>();
while (results.next()) {
    int id       = results.getInt(1);
    String title = results.getString(2);
    String body  = results.getString(3);
    Article article = new Article(id, title, body);
    articles.add(article);
}
/* יש לסגור כראוי את האובייקטים (Connection/Statement/ResultSet) —
   הושמט כאן לקיצור. */

עדכון (Update) — עם PreparedStatement וסימני שאלה ? כ-placeholders:

java · JDBC — עדכון מאובטח
Article article = new Article(1, "Yoo hoooo!", "yada yada");

String query = "UPDATE Article SET title = ?, body = ? WHERE id = ?";
PreparedStatement statement = connection.prepareStatement(query);
statement.setString(1, article.getTitle());
statement.setString(2, article.getBody());
statement.setInt(3, article.getId());
statement.execute();
🔤
למה PreparedStatement עם ? ה-? הם מקומות שמורים לערכים, שממלאים בנפרד עם setString/setInt. זה לא רק נוח — זה קריטי לאבטחה: אם נשרשר את הקלט ישירות למחרוזת ה-SQL, תוקף יוכל להזריק SQL זדוני (SQL Injection). ה-driver "בורח" (escapes) את הערכים בבטחה. כלל ברזל: לעולם לא לשרשר קלט משתמש לתוך שאילתה.
😣
למה זה כואב שימו לב לכמות ה"בּוֹילֶרְפְּלֵייט": חיבור ידני, לולאה ידנית, המרה ידנית שורה→אובייקט, וסגירת משאבים. וגרוע מזה — ה-SQL הוא מחרוזת טקסט: טעות כתיב או שם עמודה שגוי יתגלו רק ב-זמן ריצה, בלי בדיקת-טיפוסים והשלמה אוטומטית. זו בדיוק הבעיה שה-ORM בא לפתור.

6. ORM — מיפוי אובייקטים ↔ טבלאות

🧭
קונטקסט ORM = Object-Relational Mapping. הרעיון: ספרייה שתמפה אוטומטית בין אובייקטים בקוד לשורות בטבלה, ותייצר את ה-SQL בשבילנו. במקום לכתוב SQL ולהמיר ידנית — עובדים עם אובייקטים, וה-ORM מתרגם. דוגמאות: Hibernate ו-JPA (Java), Entity Framework (.NET).

מסמנים את המחלקה כ-Entity עם annotations — וה-ORM יודע למפות אותה לטבלה:

java · Hibernate + JPA — Entity
@Entity
public class Article {
    @Id
    private int id;
    private String title;
    private String body;

    public Article() { }
    public Article(int id, String title, String body) {
        this.id = id;
        this.title = title;
        this.body = body;
    }
    public void setTitle(String title) { this.title = title; }
    // ...
}
🔤
Annotations @Entity אומר ל-ORM "המחלקה הזו ממופה לטבלה". @Id מסמן את המפתח הראשי (primary key). מכאן ה-ORM יודע ליצור/לקרוא/לעדכן שורות בטבלת Article בלי שנכתוב SQL.

CRUD (Create / Read / Update / Delete) דרך EntityManager, בתוך טרנזקציות:

java · Create
EntityManagerFactory emf = Persistence.createEntityManagerFactory("News");
EntityManager em = emf.createEntityManager();

em.getTransaction().begin();
Article article1 = new Article(1, "Foo", "Bar bar");
Article article2 = new Article(2, "Hello", "World");
em.persist(article1);     // ← הוספה
em.persist(article2);
em.getTransaction().commit();

em.close();
emf.close();
java · Read / Update / Delete
// Read
Article article = em.find(Article.class, 1);    // לפי id

// Update — פשוט משנים את האובייקט, וה-ORM מזהה ושומר
em.getTransaction().begin();
Article a = em.find(Article.class, 1);
a.setTitle("Yoo hoooo!");
em.getTransaction().commit();

// Delete
em.getTransaction().begin();
Article del = em.find(Article.class, 1);
em.remove(del);
em.getTransaction().commit();
✨
היופי שימו לב ב-Update: רק שינינו שדה באובייקט (setTitle) ו-commit — וה-ORM זיהה את השינוי וייצר את ה-UPDATE בעצמו. אין SQL בקוד שלנו. נראה אידיאלי. אבל...

7. ביקורת על ORM — האם זה anti-pattern?

⚖️
הטענה: ORM הוא anti-pattern Anti-pattern = פתרון נפוץ ופופולרי לבעיה שכיחה, שנראה כמו רעיון טוב — אבל בפועל בעייתי. קשה להתווכח עם הפופולריות של ORM, אבל יש עליו ביקורת רצינית:
  • מבטיח להפשיט את ה-SQL — אבל לא באמת: כדי לכתוב שאילתות לא-טריוויאליות עדיין צריך להבין SQL, ולעיתים ללמוד שפת שאילתות נוספת של ה-ORM עצמו.
  • מאבדים שליטה: ה-ORM עדיין מייצר SQL מאחורי הקלעים — אבל אתם לא רואים אותו ולא שולטים בו. זה יכול לפגוע בביצועים בצורה משמעותית (למשל בעיית ה-"N+1 queries").
  • תרגום מייגע: ההמרה הלוך-ושוב בין ייצוג-אובייקטים לייצוג-נתונים מסובכת מטבעה.
💡
המסקנה — ומכאן הגשר ל-MongoDB ההצעה של מבקרי ה-ORM: במקום להילחם בפער בין אובייקטים לטבלאות עם שכבת תרגום — השתמשו במסד שמתאים מלכתחילה לאובייקטים. וזה בדיוק מה ש-NoSQL ו-MongoDB מציעים: לאחסן נתונים כמסמכים דמויי-JSON, שכבר נראים כמו האובייקטים שלנו. נעבור לשם.

חלק ד' · NoSQL ו-MongoDB

8. NoSQL — גישה לא-רלציונית

🧭
קונטקסט NoSQL (Not Only SQL) הוא שם כולל למסדים שלא מאחסנים נתונים בצורה רלציונית (טבלאות עם סכמה נוקשה). יש כמה גישות שונות (document, key-value, graph, column). המכנה המשותף: גמישות (אין סכמה קשיחה) וביצועים (מותאמים לסקייל גדול). MongoDB הוא מסד document-oriented — אחד הפופולריים ביותר.

9. MongoDB — Document ו-Collection

🧭
קונטקסט — המילון כדי לעבוד עם MongoDB צריך להחליף את אוצר המילים הרלציוני. ההקבלה כמעט אחד-לאחד:
רלציוני (SQL)MongoDBמה זה
Row (שורה)Documentיחידת נתונים — ייצוג בינארי של אובייקט JSON (נקרא BSON)
Table (טבלה)Collectionקבוצת documents
Column (עמודה)Field (שדה)זוג מפתח-ערך בתוך document
🪄
ההבדל הגדול — Schema-less בטבלה רלציונית כל השורות חייבות אותן עמודות. ב-MongoDB, documents באותו collection יכולים להיות בעלי שדות שונים! כתבה אחת יכולה להחזיק שדה video ואחרת לא. זו הגמישות של NoSQL — מצד אחד נוחה מאוד, מצד שני מעבירה את האחריות על העקביות אלינו (לכן Mongoose יוסיף סכמה רכה מרצוננו, בהמשך).

document לדוגמה (זה פשוט נראה כמו אובייקט JSON — בדיוק מה ש-ORM ניסה להגיע אליו בעמל):

json · document בקולקציית articles
{
    _id: 1,
    title: "Foo",
    body: "Bar bar"
}

ה-_id הוא מזהה ייחודי שכל document מקבל (MongoDB מייצר אותו אוטומטית אם לא סיפקנו).

10. מודלינג — Embedding מול Referencing

🧭
קונטקסט זוכרים את הקשרים מחלק ב' (כתבות↔תגובות, כתבות↔מחברים)? ב-MongoDB אין JOIN רלציוני. במקום זאת יש שתי דרכים לייצג קשרים, וזו אחת ההחלטות החשובות במודלינג של MongoDB.

One-to-many (כתבה ↔ תגובות):

json · Referencing — תגובות נפרדות עם הצבעה
// document בקולקציית articles
{ _id: 1, title: "Foo", body: "Bar bar" }

// documents בקולקציית comments — כל אחד "מצביע" על הכתבה
{ _id: 1, articleId: 1, title: "Yes I agree" }
{ _id: 2, articleId: 1, title: "Bla bla" }
json · Embedding — התגובות בתוך הכתבה
{
    _id: 1,
    title: "Foo",
    body: "Bar bar",
    comments: [
        { title: "Yes I agree" },
        { title: "Bla bla" }
    ]
}
🔤
מתי כל אחד?
  • Embedding (הטמעה) — שומרים את הנתונים הקשורים בתוך ה-document. מהיר לקריאה (הכל במקום אחד, בלי JOIN), אבל מנפח את ה-document. מתאים כשהנתונים "שייכים" ונקראים יחד (כמו תגובות של כתבה).
  • Referencing (הפנייה) — שומרים רק הפניה (id) לנתונים שיושבים בקולקציה אחרת. דומה למסד רלציוני. מתאים כשהנתונים גדולים, משותפים, או נקראים בנפרד.

Many-to-many (כתבות ↔ מחברים) — עם referencing דו-כיווני (כל צד מחזיק מערך id-ים), בלי טבלת קישור נפרדת:

json · many-to-many ב-MongoDB
// authors
{ _id: 1, name: "Alice", articles: [1, 2] }
{ _id: 2, name: "Bob",   articles: [2] }

// articles
{ _id: 1, title: "Foo",   body: "Bar bar", author: [1, 2] }
{ _id: 2, title: "Hello", body: "World",   author: [1]    }

במקום טבלת קישור (כמו ב-SQL), כל צד פשוט מחזיק מערך של ה-id-ים הקשורים. ככה הקשר רבים-לרבים מיוצג ישירות בתוך ה-documents.

11. CRUD ב-MongoDB

🧭
קונטקסט הפעולות הבסיסיות ב-MongoDB מקבלות אובייקטי JSON כפרמטרים — גם הנתונים וגם תנאי הסינון. כל הפעולות מתבצעות על collection (אובייקט הקולקציה). נעבור על ארבע פעולות ה-CRUD.

Create — insertOne / insertMany:

js · MongoDB — Create
// מוסיף document אחד שבו השדה foo שווה 1
collection.insertOne({ foo: 1 })

// מוסיף שני documents
collection.insertMany([ { foo: 1 }, { foo: 2 } ])

Read — find. הפרמטר הוא אובייקט שאילתה (filter):

js · MongoDB — Read
collection.find({})                  // כל ה-documents בקולקציה
collection.find({ foo: 1 })          // כל ה-documents שבהם foo == 1
collection.find({ foo: 1, bar: "2" })// foo == 1 וגם bar == "2"  (AND)
collection.find({ $or: [ { foo: 1 }, { bar: "2" } ] })  // foo==1 או bar=="2"
🔤
תחביר השאילתות
  • שדות מרובים באובייקט אחד = AND (כל התנאים חייבים להתקיים).
  • $or = OR — מקבל מערך תנאים, מספיק שאחד יתקיים. ה-$ מסמן אופרטור של MongoDB.
  • שימו לב: bar: "2" (מחרוזת) שונה מ-bar: 2 (מספר) — הטיפוס חשוב.

Update — updateOne / updateMany. שני ארגומנטים: מי לעדכן, ומה לשנות (עם $set):

js · MongoDB — Update
// בעדכון הראשון שבו foo==1, קבע את השדה bar למחרוזת "3"
collection.updateOne({ foo: 1 }, { $set: { bar: "3" } })

// אותו דבר על כל ה-documents שבהם foo==1
collection.updateMany({ foo: 1 }, { $set: { bar: "3" } })

Delete — deleteOne / deleteMany:

js · MongoDB — Delete
collection.deleteOne({ foo: 1 })    // מוחק את הראשון שבו foo==1
collection.deleteMany({ foo: 1 })   // מוחק את כל אלו שבהם foo==1
🔁
התבנית החוזרת שמו לב לסימטריה: כל פעולה מגיעה בגרסת ...One (הראשון שמתאים) ו-...Many (כל המתאימים). ותמיד הפרמטרים הם אובייקטי JSON — לא מחרוזות SQL. אין שאילתה טקסטואלית, אין mismatch, אין שכבת תרגום — הנתונים כבר אובייקטים.

חלק ה' · Mongoose וה-MERN stack

🧱
מה נבנה בחלק הזה עכשיו נחבר הכל: נבנה את שכבת הנתונים של Foo News על MongoDB דרך Mongoose, בתוך מבנה ה-MVC מהרצאה 7 — אבל הפעם עם שכבת service נוספת ופעולות אסינכרוניות. נעלה את הקוד מלמטה למעלה: Model → Service → Controller → Routes → app.js, ולסיום נחבר את צד ה-React. זהו ה-MERN stack: Mongo, Express, React, Node.

12. Mongoose (ODM) — Schema ו-Model

🧭
קונטקסט Mongoose היא ספריית ODM (Object-Document Mapping) ל-Node — המקבילה של MongoDB ל-ORM, אבל למסמכים. היא נותנת לנו להגדיר Schema (סכמה "רכה" מרצוננו, על אף ש-MongoDB עצמו schema-less) ולקבל Model — אובייקט שדרכו מבצעים את כל פעולות הנתונים.
js · models/article.js
const mongoose = require('mongoose');
const Schema = mongoose.Schema;

const Article = new Schema({
    title : {
        type: String,
        required: true
    },
    published: {
        type: Date,
        default: Date.now
    }
});

module.exports = mongoose.model('Article', Article);
🔤
פירוק ה-Schema
  • new Schema({...}) — מגדיר את מבנה ה-document: אילו שדות, מאיזה טיפוס, ואילו אילוצים.
  • title: { type: String, required: true } — שדה מסוג מחרוזת, חובה (Mongoose יזרוק שגיאה אם ננסה לשמור בלעדיו).
  • published: { type: Date, default: Date.now } — תאריך, וברירת מחדל "עכשיו" אם לא סופק. (שימו לב: Date.now בלי סוגריים — מעבירים את הפונקציה, היא תיקרא בעת היצירה. המלכודת המוכרת מהרצאות 6-8!)
  • mongoose.model('Article', Article) — יוצר ומייצא את ה-Model בשם Article. דרכו נבצע find, save וכו'.
ה-Schema נותן לנו את היתרונות של ORM (בדיקות, מבנה) — אבל מרצון, בלי לאבד את הגמישות של MongoDB.

13. שכבת ה-Service — ה-DAL החדש

🧭
קונטקסט — שכבה חדשה בהרצאה 7 ה-controller פנה ישירות ל-Model. עכשיו, כשהגישה לנתונים מורכבת יותר (אסינכרונית, לוגיקה), מוסיפים שכבת service ביניהם. ה-service היא ה-DAL האמיתי: היחיד שמכיר את Mongoose ומבצע את פעולות המסד. ה-controller רק יקרא לו.
js · services/article.js
const Article = require('../models/article');

const createArticle = async (title, published) => {
    const article = new Article({ title : title });
    if (published) article.published = published;
    return await article.save();
};

const getArticleById = async (id) => {
    return await Article.findById(id);
};

const getArticles = async () => {
    return await Article.find({});
};

const updateArticle = async (id, title) => {
    const article = await getArticleById(id);
    if (!article) return null;
    article.title = title;
    await article.save();
    return article;
};

const deleteArticle = async (id) => {
    await article.remove();
};

module.exports = { createArticle, getArticleById, getArticles,
                   updateArticle, deleteArticle };
🔤
async / await ופעולות Mongoose
  • כל פעולות המסד הן אסינכרוניות (לוקחות זמן, לא חוסמות): מסמנים את הפונקציה async ומשתמשים ב-await כדי לחכות לתוצאה. (זוכרים אסינכרוניות מ-JavaScript בהרצאה 6.)
  • new Article({...}) + .save() — יצירה ושמירה (Create).
  • Article.findById(id) — שליפה לפי _id (Read).
  • Article.find({}) — שליפת הכל (Read) — בדיוק כמו collection.find({}) שראינו, אבל דרך ה-Model.
  • עדכון: שולפים, משנים שדה, ו-save() שוב — בדיוק כמו ה-ORM (Hibernate), אבל ל-documents.
🔗
זה בדיוק ה-getArticle מהרצאה 7 — רק "התבגר" השוו: בהרצאה 7 היה getArticle = (id) => articles.filter(...)[0] על מערך. עכשיו זה getArticleById = async (id) => await Article.findById(id) על מסד. אותה חתימה רעיונית — קלט id, פלט כתבה — והשכבות שמעל (controller, routes) לא מרגישות בהבדל. זה התשלום על ה-DAL.

14. Controller ו-Routes — API מסוג RESTful

🧭
קונטקסט ה-controller מתאם בין הבקשה ל-service, ומחזיר JSON (כי עכשיו אנחנו API שמשרת את ה-React, לא מרנדר HTML). ה-routes ממפים מתודות HTTP לפעולות — בדיוק לפי עקרונות REST מהרצאה 5.

ה-controller — אסינכרוני, עוטף את ה-service ומחזיר JSON:

js · controllers/article.js
const articleService = require('../services/article');

const createArticle = async (req, res) => {
    res.json(await articleService.createArticle(req.body.title));
};

const getArticles = async (req, res) => {
    res.json(await articleService.getArticles());
};

const getArticle = async (req, res) => {
    const article = await articleService.getArticleById(req.params.id);
    if (!article) {
        return res.status(404).json({ errors: ['Article not found'] });
    }
    res.json(article);
};
// ... updateArticle, deleteArticle

module.exports = { createArticle, getArticles, getArticle,
                   updateArticle, deleteArticle };
🔤
נקודות תחביר
  • res.json(...) — שולח את התשובה כ-JSON (במקום res.render שהחזיר HTML בהרצאה 7).
  • await articleService... — ה-controller מחכה ל-service. ה-service הוא היחיד שנוגע במסד.
  • res.status(404).json({...}) — אם הכתבה לא נמצאה, מחזירים קוד סטטוס 404 Not Found עם הודעת שגיאה. טיפול שגיאות נכון = חלק מ-API טוב.

ה-routes — שימו לב לתחביר router.route() ששרשר כמה מתודות לאותו נתיב:

js · routes/article.js
const express = require('express');
var router = express.Router();
const articleController = require('../controllers/article');

router.route('/')
    .get(articleController.getArticles)      // GET    /articles      → כל הכתבות
    .post(articleController.createArticle);  // POST   /articles      → יצירה

router.route('/:id')
    .get(articleController.getArticle)       // GET    /articles/:id  → כתבה אחת
    .patch(articleController.updateArticle)  // PATCH  /articles/:id  → עדכון
    .delete(articleController.deleteArticle);// DELETE /articles/:id  → מחיקה

module.exports = router;
🔗
CRUD ↔ HTTP ↔ REST ראו את ההתאמה המושלמת: Create→POST, Read→GET, Update→PATCH, Delete→DELETE. זה לב ה-REST מהרצאה 5: אותו משאב (/articles/:id), פעולות שונות לפי מתודת ה-HTTP. router.route('/:id') פשוט מקבץ את כל המתודות של אותו נתיב במקום אחד — קריא ונקי.

15. app.js — חיבור למסד, env ו-CORS

🧭
קונטקסט נקודת הכניסה מרכיבה את הכל: מתחברת ל-MongoDB דרך Mongoose, טוענת קונפיגורציה לפי סביבה, רושמת middleware (כולל CORS), ומחברת את ה-router. זה ה-app.js מהרצאה 7 — עם תוספות לעולם ה-API והמסד.
js · app.js
const express = require('express');
const bodyParser = require('body-parser');
const cors = require('cors');
const mongoose = require('mongoose');
const articles = require('./routes/article');

require('custom-env').env(process.env.NODE_ENV, './config');

mongoose.connect(process.env.CONNECTION_STRING, {
    useNewUrlParser: true,
    useUnifiedTopology: true
});

var app = express();
app.use(cors());
app.use(bodyParser.urlencoded({ extended: true }));
app.use(express.json());
app.use('/articles', articles);

app.listen(process.env.PORT);
🔤
מה כל שורה עושה
  • mongoose.connect(CONNECTION_STRING, ...) — פותח חיבור למסד. ה-connection string מצביע על שרת ה-MongoDB (מקומי או בענן, למשל MongoDB Atlas).
  • require('custom-env').env(process.env.NODE_ENV, './config') — טוען משתני סביבה לפי הסביבה (development / production). זה מאפשר connection string וסיסמאות שונים בלי לכתוב אותם בקוד — וזו גם הסיבה שמשתמשים ב-process.env.X במקום ערכים קשיחים.
  • app.use(cors()) — מאפשר CORS: בקשות מ-origin אחר. צריך אותו כי ה-React (למשל על פורט 3000) הוא דומיין נפרד מה-API (פורט 8080), והדפדפן חוסם cross-origin כברירת מחדל.
  • app.use(express.json()) — מפענח גוף בקשות JSON ל-req.body (המקבילה של urlencoded מהרצאה 7, אבל ל-JSON — כי ה-React שולח JSON).
🔐
למה env וקונפיגורציה? סיסמת המסד וה-connection string הם סודות — אסור לכתוב אותם בקוד שעולה ל-git. מחזיקים אותם בקבצי קונפיגורציה/סביבה נפרדים (שלא נכנסים ל-git), וטוענים דרך process.env. בונוס: אותו קוד רץ בפיתוח ובפרודקשן עם הגדרות שונות — רק הסביבה משתנה.

16. צד ה-React — fetch מה-API

🧭
קונטקסט — סוגרים את המעגל בהרצאה 8 ה-React הציג נתונים מקובץ db.json מקומי. עכשיו הנתונים אמיתיים — צד ה-React מושך אותם מה-API שבנינו, עם fetch. נחבר שני Hooks חדשים: useEffect (להרצת קוד אחרי הרינדור — כמו קריאת רשת) ו-useParams (לקרוא את ה-id מה-URL בצד הלקוח).
jsx · Article.js (צד React)
import { useState, useEffect } from "react";
import { useParams } from 'react-router-dom';

function Article() {
    const { id } = useParams();
    const [article, setArticle] = useState({
        title: '', published: '', author: '', content: ''
    });

    useEffect(() => {
        const fetchArticles = async () => {
            try {
                const response = await fetch('http://localhost:8080/articles/' + id);
                const article = await response.json();
                setArticle(article);
            } catch (error) {
                // handle error
            }
        };
        fetchArticles();
    }, [id]);

    return (
        <article>
            <h1>{ article.title }</h1>
            <p>Published on: { article.published }</p>
            <p>Author: { article.author }</p>
            <p>{ article.content }</p>
        </article>
    );
}

export default Article;
🔤
שני ה-Hooks החדשים + fetch
  • const { id } = useParams() — קורא פרמטר מה-URL בצד הלקוח (react-router). מקביל ל-req.params.id בצד השרת מהרצאה 7 — אבל זה ה-routing של ה-React.
  • useEffect(fn, [id]) — מריץ את fn אחרי הרינדור, ושוב בכל פעם ש-id משתנה (זה ה-"dependency array"). כאן: כשנכנסים לכתבה או שה-id משתנה — מביאים את הנתונים. (בלי זה, fetch בתוך הרכיב היה רץ בכל רינדור ויוצר לולאה.)
  • fetch(url) — בקשת HTTP מהדפדפן ל-API. אסינכרוני, לכן await. response.json() ממיר את התשובה לאובייקט.
  • setArticle(article) — מעדכן את ה-state → React מרנדר מחדש עם הנתונים האמיתיים (בדיוק מנגנון ה-state מהרצאה 8).
🌐
התמונה המלאה של MERN הנה כל המסע של בקשה: משתמש פותח כתבה ב-React → useEffect עושה fetch ל-localhost:8080/articles/1 → Express route מנתב ל-controller → ה-controller קורא ל-service → ה-service שואל את MongoDB דרך Mongoose → ה-document חוזר כ-JSON עד ה-React → ה-state מתעדכן → המסך מצייר את הכתבה. זה ה-MERN stack המלא, ובדיוק מה שבנינו חתיכה-חתיכה לאורך הרצאות 7-9.

★ סיכום ההרצאה

🎯
התמונה המלאה — שרשרת של בעיות ופתרונות
  • צריך persistence — מערך בזיכרון (הרצאה 7) נעלם; צריך מסד נתונים.
  • מסד רלציוני (SQL) — טבלאות, שורות, SELECT/WHERE, וקשרים: one-to-many (FK) ו-many-to-many (join table).
  • ה-mismatch — קוד מונחה-עצמים מול מסד מונחה-טבלאות. מגשרים דרך driver/API (JDBC/ODBC).
  • JDBC — גישה גולמית: SQL כמחרוזת, המרה ידנית, PreparedStatement נגד SQL injection. מסורבל ולא בטוח-טיפוסים.
  • ORM (Hibernate/JPA, Entity Framework) — ממפה אובייקטים↔טבלאות אוטומטית (@Entity, EntityManager CRUD). נוח — אבל יש הטוענים שזה anti-pattern (מאבדים שליטה ב-SQL, ביצועים).
  • NoSQL / MongoDB — פתרון object-friendly: Document (=row, JSON/BSON) ב-Collection (=table), schema-less. מודלינג ב-embedding מול referencing. CRUD עם אובייקטי JSON: insertOne/find/$or/updateOne/$set/deleteOne.
  • Mongoose (ODM) — Schema + model ל-Node. שכבת service (DAL אסינכרוני) → controller (res.json, status) → routes (router.route, RESTful: POST/GET/PATCH/DELETE) → app.js (mongoose.connect, env, cors, json).
  • צד React — fetch מה-API עם useEffect + useParams → state → render. זהו ה-MERN stack המלא.
🔗
חיבורים להרצאות קודמות זו הרצאת ה"איחוד": ה-Model מהרצאה 7 הוחלף סוף-סוף במסד אמיתי, בלי לגעת בשאר השכבות (כוח ה-DAL). ה-REST וה-GET/POST מהרצאה 5 הם ה-routes כאן. ה-async מהרצאה 6 הוא ה-service. ה-React/state/props מהרצאה 8 הם צד הלקוח. ו-Date.now בלי סוגריים ב-Schema — שוב המלכודת "מעבירים פונקציה, לא קוראים לה".
