חלק א' · משרת גולמי ל-Framework

1. שרת web — הרעיון

🧭
קונטקסט ביסוד כל אתר עומדת אינטראקציה פשוטה: הדפדפן (client) שולח בקשה (request) אל השרת (web server), והשרת מחזיר תשובה (response). הבקשה נשלחת בפרוטוקול HTTP מעל החיבור (TCP) שכבר הכרנו.

כשמקלידים http://www.foo.com/index.html בדפדפן, נשלחת בקשת HTTP שנראית כך:

Request:
  GET /index.html HTTP/1.1        ← שורת הבקשה (method, path, version)
  Accept-Language: en-us          ┐
  Connection: Keep-Alive          ├ כותרות (headers)
  ...                             ┘

הדפדפן הוא הלקוח, והמכונה בקצה השני שמאזינה ומגיבה היא השרת.

2. לכתוב שרת HTTP ידנית — והבעיה

🧭
קונטקסט — מחברים לסוקטים מההרצאה הקודמת טכנית, אפשר לבנות שרת HTTP בדיוק כמו שבנינו שרת echo: סוקט TCP שמאזין על פורט 80, מקבל בקשות, ועונה. אבל מהר מאוד נגלה למה זה לא רעיון טוב.
python · שרת HTTP גולמי
from socket import socket, AF_INET, SOCK_STREAM

s = socket(AF_INET, SOCK_STREAM)
s.bind(('', 80))
s.listen(5)

while True:
    client, addr = s.accept()

    ##### repeat for each request ####
    request = client.recv(1024)

    # parse request
    # prepare relevant response

    client.send(response)

    ##### done repeat ####

    client.close()

זה אותו דפוס TCP מההרצאה הקודמת (socket→bind→listen→accept→recv→send→close) — רק על פורט 80 ועם תוכן HTTP. אבל יש בעיות:

⚠️
"להמציא את הגלגל מחדש בכל פעם"
  • מייגע ונוטה לשגיאות (tedious, error-prone).
  • צריך לממש את כל פרוטוקול HTTP בעצמך — לפענח שורת בקשה, headers, body, status codes...
  • הביצועים תלויים במימוש מקביליות טוב — צריך לטפל בהרבה לקוחות בו-זמנית (זוכרים threads מההרצאה הקודמת?).

הפתרון: במקום להמציא את הגלגל לכל אפליקציה — להשתמש בטכנולוגיות web-server קיימות.

3. איך שרת מודרני מטפל בבקשה

🧭
קונטקסט כשמשתמשים ב-framework, אנחנו לא נוגעים ב-headers הגולמיים. ה-framework "עוטף" את כל זה ונותן לנו עבודה עם אובייקטים נוחים. נראה את ארבעת השלבים.
  1. הדפדפן שולח בקשת HTTP גולמית (טקסט).
  2. השרת יוצר אובייקט שמכיל את מידע הבקשה:
    js · request object
    Request = {
    method : 'GET',
    url : '/index.html',
    httpVersion : '1.1'
}
  3. השרת מפעיל פונקציה ומעביר לה את אובייקט הבקשה (ואובייקט תשובה):
    js · handler
    function handleRequest(request, response) {
    // ...
    response.write('<html><head>...');
    response.end();
}
  4. הפונקציה מכינה אובייקט תשובה, וה-framework מתרגם אותו חזרה לפרוטוקול HTTP ושולח לדפדפן:
    js → http · response
    Response = { statusCode : '200', statusMessage : 'OK', ... }

   ↓ מתורגם ל:

HTTP/1.1 200 OK
Content-Length: 250
Content-Type: text/html
...
<html><head>...
🏗️
שתי דרכים לארח HTTP server 1. שרת חיצוני (למשל Apache Tomcat): השרת רץ בנפרד, ומזהה איזה Servlet/JSP צריך לטפל בבקשה ("front-end" סטטי + "back-end" קוד).
2. Framework מוטמע (Flask/Express/Jetty): שרת ה-HTTP מוטמע ישירות בתוך הקוד שלנו — אנחנו כותבים את הלוגיקה, וה-framework מטפל ב-HTTP מסביב. זו הגישה המודרנית והנפוצה.

4. NodeJS ו-Express

🧭
קונטקסט NodeJS מאפשר להריץ JavaScript בצד השרת. מודול ה-http המובנה כבר נותן שרת בסיסי, ו-Express הוא framework שמייפה ומפשט את זה עוד יותר.

NodeJS גולמי (מודול http)

js · http מובנה
const http = require("http")

function handleRequest(request, response) {
    response.write("Hello world")
    response.end()
}

const server = http.createServer(handleRequest)

server.listen(80)

שים לב כמה זה קצר לעומת הסוקט הגולמי — אין accept/recv/parsing. רק "מה לעשות עם בקשה" + listen.

עם Express

js · Express
const express = require('express')
const app = express()
const port = 12345

app.get('/', (req, res) => {
    res.send('<html><body><h1>Hello World!</h1></body></html>')
})

app.listen(port, () => {
    console.log(`Server is running on http://localhost:${port}`)
})
🔤
הסבר סינטקס
  • require('express') — טוען את ספריית Express (כמו import/#include).
  • app.get('/', (req, res) => {...}) — "כשמגיעה בקשת GET לנתיב /, הרץ את הפונקציה הזו". זו arrow function (פונקציית חץ) — הצורה המקוצרת של JS.
  • res.send(...) — שולח תשובה (Express דואג ל-Content-Type, Content-Length וכו' אוטומטית).
  • `...${port}...` — template literal (גרשיים אחוריים): מאפשר להזריק משתנים למחרוזת עם ${...}.

5. ניתוב (Routing)

🧭
קונטקסט אתר אמיתי מחזיר תוכן שונה לכתובות שונות. ניתוב = להחליט מה להחזיר לפי ה-URL וה-method של הבקשה.

ניתוב ידני עם מודול http (בודקים את request.url):

js · ניתוב לפי url
const http = require("http")

function handleRequest(request, response) {
    if (request.url == '/foo')
        response.write('<html><body><b>foo</b></body></html>')
    else if (request.url == '/bar')
        response.write('<html><body><u>bar</u></body></html>')
    else
        response.statusCode = 404

    response.end()
}

const server = http.createServer(handleRequest)
server.listen(80)

אם ה-URL לא מוכר — מחזירים 404 (Not Found). שים לב כמה ה-if/else הזה היה הופך למסורבל באתר אמיתי.

ניתוב נקי עם Express — כל method+נתיב הוא handler נפרד:

js · Express routing
import express from 'express'
const app = express()

app.get('/', (req, res) => {
    res.send('<html><body><h1>GET</h1></body></html>')
})

app.post('/', (req, res) => {
    res.send('<html><body><h1>POST</h1></body></html>')
})

app.listen(80)

שים לב — אותו נתיב / אבל שני handlers נפרדים ל-GET ול-POST. וזה מוביל אותנו לשאלה: מה ההבדל בין GET ל-POST?

חלק ב' · פרוטוקול HTTP

6. GET מול POST — safe ו-idempotent

🧭
קונטקסט ל-HTTP יש כמה "methods" (פעלים). ההבדל ביניהם מוגדר ע"י שני מושגים תיאורטיים חשובים: safe ו-idempotent.
  • Safe — אם ה-method משמש רק לפעולת קריאה בלבד (read-only), הוא נחשב "בטוח" (לא משנה את מצב השרת).
  • Idempotent — אם הפעלת ה-method כמה פעמים לאותה בקשה נותנת אותה תוצאה על השרת כמו הפעלה אחת, הוא "אידמפוטנטי".
MethodSafe?Idempotent?
GET✓ כן (קריאה בלבד)✓ כן (ולכן גם אידמפוטנטי)
POST✗ לא✗ לא
🛒
דוגמה משלי — למה זה חשוב GET "תן לי את דף המוצר" — אפשר לרענן 100 פעם, שום נזק (safe + idempotent). POST "בצע הזמנה" — כל הפעלה יוצרת הזמנה חדשה! לכן הדפדפן מזהיר "האם לשלוח שוב את הטופס?" כשמרעננים אחרי POST. זו הסיבה שטפסים שמשנים נתונים משתמשים ב-POST, וקישורים/חיפושים ב-GET.

7. דוגמאות בקשה/תשובה

בקשת GET רגילה + תשובה מוצלחת:

Request:                          Response:
  GET /index.html HTTP/1.1          HTTP/1.1 200 OK
  Accept-Language: en-us            Content-Length: 250
  Connection: Keep-Alive            Content-Type: text/html
  ...                               ...
                                    < html >< head >...

GET עם פרמטרים (query string) — נשלח דרך הטופס ב-URL, וכאן השרת מחזיר 404:

Request:                          Response:
  GET /foo.php?bar=6 HTTP/1.1        HTTP/1.1 404 Not Found
  Accept-Language: en-us            ...
  ...

ה-?bar=6 הוא ה-query string — נתונים שנשלחים בתוך ה-URL (גלויים). ב-HTML הטופס היה: <form action="foo.php" method="GET">.

בקשת POST — הנתונים נשלחים ב-גוף הבקשה (body), לא ב-URL:

Request:                          Response:
  POST /login.php HTTP/1.1           HTTP/1.1 200 OK
  Accept-Language: en-us             Content-Length: 250
  Connection: Keep-Alive             Content-Type: text/html
  ...                                ...
  username=foo&password=bar         < html >< head >...
🔤
GET vs POST — איפה הנתונים? ב-GET הפרמטרים ב-URL (?bar=6) — גלויים, נשמרים בהיסטוריה, מוגבלים באורך. ב-POST הנתונים ב-body (username=foo&password=bar) — לא ב-URL, מתאים לסיסמאות ולנתונים גדולים. (זו עוד סיבה ש-login הוא POST.)

8. Web Services ו-REST

🧭
קונטקסט Web service = חשיפת נתונים/פונקציונליות דרך API ייעודי. הוא יכול לעמוד בפני עצמו או להיות חלק מאפליקציה, ונגיש גם מצד לקוח וגם מצד שרת. הסגנון הנפוץ ביותר היום לבניית web service הוא REST.

הגישה הקלאסית — RPC מבוסס חוזה (Contract): הלקוח "משיג חוזה" (interface), ואז קורא לפונקציות מרחוק לפי החוזה (request/response). דוגמה: WCF. גישה כבדה יותר.

REST (Representational State Transfer) — אילוצים פשוטים וקלים:

  • המצב מיוצג כמשאבים (resources).
  • לכל משאב יש מזהה אחיד (URI) — למשל foo.com/Articles/5.
  • המשאבים מיוצגים, למשל, ב-JSON.
  • ממשק אחיד — דרך מתודות HTTP: GET, POST, PUT, PATCH, DELETE.

RESTful API טיפוסי סביב משאב "Articles":

פעולהבקשהתשובה
קבל את כל הכתבותGET /api/Articles200 OK + מערך JSON
קבל כתבה מסוימתGET /api/Articles/5200 OK + {id:5, ...}
צור כתבה חדשהPOST /api/Articles + body201 Created + מיקום
עדכן כתבהPUT /api/Articles/5 + body204 No content
מחק כתבהDELETE /api/Articles/5204 No content
💡
הרעיון היפה של REST במקום להמציא שמות פונקציות (getArticle, deleteArticle...), משתמשים בפעלי HTTP הקיימים על משאב מזוהה. אותו URI (/api/Articles/5) + פועל שונה = פעולה שונה. זה אחיד, צפוי, וקל לתעד. שים לב גם ל-status codes המדויקים: 201 ליצירה, 204 לפעולה שהצליחה בלי תוכן להחזיר.

חלק ג' · תקשורת ו-Push מהשרת

9. סינכרוני מול אסינכרוני

🧭
קונטקסט איך הדף מתעדכן? בגישה הישנה כל פעולה טוענת מחדש את כל הדף. בגישה המודרנית מעדכנים רק את החלק שהשתנה. זה ההבדל בין תקשורת סינכרונית לבין אסינכרונית.
Synchronous (מסורתי)Asynchronous
מקור המידעהלקוח שולף משאבים מהשרתשולף מידע נוסף בלי לטעון מחדש
מידע חדשדורש בקשה/חיבור חדששולף מידע ספציפי ומשלב בדף הקיים
הדףנטען מחדש (flicker / הבהוב)לא נטען מחדש
 
Synchronous:  כל חיפוש → GET /search?q=foo → 200 OK <html>...</html>
              → הדפדפן מחליף את כל ה-HTML (הדף "מהבהב")

Asynchronous: כל חיפוש → GET /search?q=foo → 200 OK [{id:1,title:"foo"}]
              → מחליפים רק את תוצאות החיפוש (הדף לא מהבהב)
🔄
למה JSON שינה הכל בגישה האסינכרונית המודרנית, השרת מחזיר נתונים גולמיים ב-JSON (למשל [{id:1, title:"foo", content:"bar"}]) במקום HTML שלם, והלקוח בונה בעצמו את התצוגה ומשלב אותה בדף. זה גם חוסך רוחב-פס וגם מאפשר חוויית משתמש חלקה (בלי הבהוב). זה בדיוק מה שמאפשר אפליקציות web "חיות" כמו Gmail.

10. Polling, Long-polling, Streaming

🧭
קונטקסט — הבעיה: השרת רוצה "לדחוף" בתקשורת קלאסית הלקוח יוזם בקשות, והשרת מחכה להן. אבל מה אם לשרת יש מידע חדש (התראה, התקדמות משימה) והוא רוצה לדחוף (push) אותו ללקוח? HTTP לא בנוי לזה ישירות — לכן יש כמה תחבולות.

Polling (תשאול)

הלקוח שואל את השרת שוב ושוב אם יש עדכון (למשל כל 30 שניות). אם יש מידע חדש — מחזירים אותו, אחרת — מחזירים את התשובה הקודמת.

Client                          Server
  │── "תן לי תוצאה" ──────────────▶│
  │◀──────────────── 0:0 ──────────│
  │  (ממתין זמן מה)                 │
  │── "תן לי תוצאה" ──────────────▶│
  │◀──────────────── 0:1 ──────────│
  │  (ממתין זמן מה) ...             │
⚠️
החיסרון של polling טעינות חוזרות תכופות — רע ל-UX ולביצועים, ה-UI עלול "להבהב", במיוחד כשהבקשה מבוזבזת כי אין מידע חדש (שאלנו, אבל שום דבר לא השתנה).

Long-polling (תשאול ארוך)

הלקוח שואל, אבל השרת מעכב את התשובה עד שיש מידע חדש. ברגע שיש — עונה, והלקוח שואל מיד שוב.

Client                          Server
  │── "תן לי תוצאה" ──────────────▶│  השרת מחזיק את
  │       (השרת ממתין...)          │  הבקשה פתוחה עד
  │◀──────────────── 0:1 ──────────│  שמשהו משתנה

היתרון: פחות בקשות מבוזבזות — השרת הוא זה שמעכב את התשובה, ולא הלקוח שמתשאל לריק.

Streaming (הזרמה)

הלקוח שולח בקשה אחת, והשרת לא סוגר את החיבור — הוא ממשיך לכתוב לתוכו עדכונים לאורך זמן.

Client                          Server
  │── "תן לי תוצאה" ──────────────▶│  חיבור אחד נשאר פתוח,
  │◀──────────────── 0:0 ──────────│  והשרת ממשיך לכתוב:
  │◀──────────────── 0:1 ──────────│  parent.update(10)
  │◀──────────────── 1:1 ──────────│  parent.update(20) ...
📊
מתי כל אחד? — דוגמה משלי Polling: עדכונים נדירים ולא דחופים (מזג אוויר כל 10 דק'). Long-polling: צ'אט/התראות (רוצים מיידיות בלי להציף בבקשות). Streaming: זרם רציף ומהיר (מחיר מניה, מד התקדמות). היום קיימים גם WebSockets ו-Server-Sent Events שמשכללים את הרעיונות האלה.

חלק ד' · Async ב-JavaScript

11. AJAX — fetch ו-polling

🧭
קונטקסט AJAX (Asynchronous JavaScript And XML) הוא הטכניקה שמאפשרת ל-JS לשלוח בקשות לשרת ברקע ולעדכן את הדף בלי לטעון אותו מחדש. ב-JS המודרני עושים זאת עם fetch.

דוגמה — מד התקדמות שמתשאל את השרת כל 3 שניות:

html + js · AJAX polling
<h1>Progress:</h1>
<div class="progressBar" style="background-color:green;width:0px">0%</div>
<script>
  var current = 0;

  function fetchProgress() {
      fetch("http://localhost:8080/Progress")
          .then(response => response.json())
          .then(data => {
              if (data.progress != current) {
                  current = data.progress;
                  const progressBar = document.querySelector('.progressBar');
                  progressBar.style.width = `${data.progress}%`;
                  progressBar.textContent = `${data.progress}%`;
              }
          })
          .catch(error => {
              console.error('Error fetching progress:', error);
          });
  }

  setInterval(fetchProgress, 3000);
</script>
🔤
הסבר סינטקס
  • fetch(url) — שולח בקשת HTTP ומחזיר Promise (הבטחה לתוצאה עתידית — מיד נסביר).
  • .then(callback) — "כשה-Promise יסתיים בהצלחה, הרץ את ה-callback". משרשרים: קודם הופכים את התשובה ל-JSON, אחר כך משתמשים בנתונים.
  • .catch(...) — מטפל בשגיאה אם הבקשה נכשלה.
  • setInterval(fn, 3000) — מריץ את fn כל 3000 מילישניות = polling.
  • document.querySelector('.progressBar') — מאתר את האלמנט ב-DOM כדי לעדכן אותו.

12. Promise

🧭
קונטקסט Promise הוא אובייקט שמייצג תוצאה עתידית של פעולה אסינכרונית — משהו שעוד לא קרה אבל יקרה (או ייכשל). זה הבסיס לכל ה-async ב-JS.
js · Promise
let foo = new Promise((resolve, reject) => {
    setTimeout(() => {
        resolve('42');
    }, 2000);
    console.log('Hiiiiush')
})

function bar(data) {
    console.log(data)
    return data + data
}

function bar2(data) {
    console.log(data)
}

foo.then(bar).then(bar2);

מה קורה כאן:

  • ה-Promise מתחיל לרוץ מיד — מדפיס Hiiiiush, ומתזמן resolve('42') בעוד 2 שניות.
  • אחרי 2 שניות ה-Promise "נפתר" (resolved) עם הערך '42'.
  • .then(bar) — מקבל את '42', מדפיס אותו, ומחזיר '4242' (שרשור מחרוזות).
  • .then(bar2) — מקבל את '4242' (מה ש-bar החזיר) ומדפיס.
🔤
המהות של then then "מצרף callbacks לפתרון/דחייה של ה-Promise". הוא מקבל פונקציה ל-onfulfilled (הצלחה) ואופציונלית ל-onrejected (כישלון), ומחזיר Promise חדש — ולכן אפשר לשרשר .then().then(). הערך שמחזירים מ-then אחד עובר ל-then הבא.

13. Event Loop

🧭
קונטקסט — איך JS עושה async עם thread אחד? JavaScript הוא single-threaded — thread אחד בלבד! אז איך הוא מטפל בכמה דברים "במקביל"? התשובה היא ה-Event Loop.
js · מה יודפס קודם?
setTimeout(function (){
    console.log('foo');
}, 2000);

console.log('bar');

הפלט הוא bar ואז foo — למרות ש-setTimeout נכתב ראשון! כי הוא מתוזמן ל"אחר כך", וה-console.log('bar') רץ מיד.

 ┌──────────────┐         ┌──────────────┐
 │  Stack/Heap  │ ───────▶│     APIs     │   (setTimeout, fetch,
 │ (קוד שרץ עכשיו)│        │ (פעולות אסינכ')│    טיימרים — רצים "בצד")
 └──────────────┘         └──────┬───────┘
        ▲                        │ כשמסתיים, ה-callback
   Event loop                    ▼ נכנס לתור
 ┌──────────────────────────────────────────┐
 │              Task queue                    │
 └──────────────────────────────────────────┘
   ה-Event loop לוקח מהתור ומחזיר ל-Stack
   רק כשה-Stack ריק
⚙️
איך זה עובד הקוד הסינכרוני רץ על ה-Stack. פעולות אסינכרוניות (טיימרים, בקשות רשת) נמסרות ל-APIs שמטפלים בהן "בצד". כשהן מסתיימות, ה-callback שלהן נכנס ל-Task queue. ה-Event loop מעביר משימות מהתור ל-Stack רק כשה-Stack ריק. לכן bar (סינכרוני) תמיד יודפס לפני foo (callback בתור), גם אם ה-timeout היה 0.

14. async / await

🧭
קונטקסט שרשור .then().then() נעשה מסורבל. async/await הוא סוכר תחבירי (syntactic sugar) מעל Promises — מאפשר לכתוב קוד אסינכרוני שנראה סינכרוני וקריא.

אותו מד התקדמות מקודם, עכשיו עם async/await:

js · async/await
let current = 0;

async function fetchProgress() {
    try {
        const response = await fetch("http://localhost:8080/Progress");

        const data = await response.json();

        if (data.progress !== current) {
            current = data.progress;
            const progressBar = document.querySelector('.progressBar');
            progressBar.style.width = `${data.progress}%`;
            progressBar.textContent = `${data.progress}%`;
        }
    } catch (error) {
        console.error('Error fetching progress:', error);
    }
}

setInterval(fetchProgress, 3000);
🔤
הסבר סינטקס — השוו ל-then
  • async function — מסמן שהפונקציה אסינכרונית (תמיד מחזירה Promise).
  • await fetch(...) — "חכה שה-Promise ייפתר, ותן לי את הערך" — בלי .then! במקום callback מקונן, פשוט const response = await ....
  • try/catch — מחליף את .catch(...) לטיפול בשגיאות, בדיוק כמו בקוד סינכרוני רגיל.
אותה התנהגות כמו גרסת ה-.then מקודם — רק קריא בהרבה. await "מחביא" את ה-Promise ונותן תחושת קוד שורה-אחרי-שורה (אבל מתחת למכסה המנוע, ה-event loop עדיין עובד).
🔗
חיבור להרצאה הקודמת זוכרים threads ו-race conditions מהרצאה 4? JavaScript נמנע מהם דרך מודל single-thread + event loop: אין שני threads שניגשים לזיכרון בו-זמנית, ולכן אין race conditions קלאסיים. המחיר: צריך לחשוב באופן אסינכרוני (Promises/await) במקום לחסום את ה-thread היחיד.

חלק ה' · ארכיטקטורה

15. Layers ו-Tiers

🧭
קונטקסט ככל שאפליקציה גדלה, צריך לארגן אותה. הרעיון: הפרדת אחריות (separation of responsibilities). שני מושגים שנשמעים דומים אבל שונים: layers (שכבות) ו-tiers (רבדים).
Layers (שכבות)Tiers (רבדים)
סוג ההפרדהלוגית (בקוד)פיזית (מכונות נפרדות)
היכן רציםיכולות לרוץ על אותה מכונהרצים על מכונות שונות

כל layer יכול לרוץ על מכונה נפרדת, או כמה layers על אותה מכונה. לגבי האינטראקציה:

  • Closed layer architecture — שכבה מדברת רק עם השכנה הישירה.
  • Open layer architecture — שכבה יכולה לדבר עם כל שכבה אחרת.

למה בכלל לחלק? (Rationale):

  • פיתוח — אנשים שונים עובדים על שכבות שונות במקביל.
  • תלות — אפשר להחליף מימוש של שכבה בלי לשנות את האחרות.
  • Scalability — כל tier ניתן להגדלה (scale) באופן עצמאי.
  • Testability — קל לבדוק לוגיקה של שכבה ספציפית.

הארכיטקטורה הקלאסית — 3-Tier:

 Presentation Tier        Application Tier         Data Tier
 ┌──────────────────┐    ┌──────────────────┐    ┌──────────────┐
 │ Web UI (browser) │◀──▶│   Web server     │◀──▶│   Database   │
 │ GUI (desktop)    │    │ (ASP.NET/servlet)│    │(MySQL/Mongo) │
 │ GUI (mobile)     │    │                  │    │              │
 └──────────────────┘    └──────────────────┘    └──────────────┘

ובתוך כל tier יש layers לוגיים:

Layerאחריות
PL — Presentation Layerהצגת נתונים, טיפול באירועי UI, אינטראקציות משתמש.
BL — Business Logic Layerמה האפליקציה עושה — אלגוריתמים, עיבוד נתונים.
DAL — Data Access Layerשמירה/שליפה/עדכון של נתונים מתמשכים (persisted).
💡
layers ≠ tiers — הנקודה החשובה אותם layers לוגיים (PL/BL/DAL) יכולים להתפרס על tiers שונים. למשל באפליקציית client בלבד — כל ה-PL+BL+DAL על מכשיר המשתמש (tier אחד). באפליקציית web — ה-PL בדפדפן, BL+DAL בשרת. כלומר: "layer" זה איך הקוד מאורגן, "tier" זה איפה הוא רץ פיזית. (וכפי שהמצגת מציינת — לפעמים צריך גם קצת לוגיקה/תצוגה בצד "הלא צפוי".)

16. MVC — Model, View, Controller

🧭
קונטקסט — המוטיבציה ככל שה-GUI נעשה עשיר ומורכב, הוא דורש לוגיקה. איפה לשים אותה? ב-BL? ב-PL? איך מאפשרים לחלקים שונים להתפתח בקצב שונה בלי לצמד (couple) אותם בחוזקה? תבנית MVC נותנת תשובה.

המטרה: הפרדת אחריות (separation of concerns) לשלושה חלקים:

רכיבאחריות
Modelייצוג ועיבוד הנתונים.
Viewהצגה (ויזואליזציה) של נתוני ה-Model.
Controllerמתווך את האינטראקציות בין Model, View, והמשתמש.
 
                  ┌────────────┐
                  │ Controller │
                  └─┬────────┬─┘
       User         │        │  Manipulates
     interaction    │        ▼
            ┌────────┴─┐   ┌───────┐
            │   View   │◀──│ Model │
            └──────────┘   └───────┘
                  Updates

הזרימה ב-web (MVC variation):

 Browser                    Controller
   │── 1. request ──────────────▶│
   │                       2. invoke ↓
   │                            Model ── 3. data ──▶ Controller
   │                       4. data ↓
   │                            View ── 5. data visualization ──▶ Controller
   │◀── 6. response (data visualization) ──│
🎭
אנלוגיה משלי — מסעדה Model = המטבח והמזווה (הנתונים והכנתם). View = הצלחת שמוגשת ללקוח (התצוגה). Controller = המלצר — מקבל את ההזמנה מהלקוח, מעביר למטבח, ומחזיר את הצלחת. הלקוח לא מדבר ישירות עם המטבח, והמטבח לא מעצב צלחות — כל אחד עושה את שלו. כך אפשר להחליף את העיצוב של הצלחת בלי לגעת במתכון.

17. MVC ב-NodeJS

🧭
קונטקסט נראה איך MVC מתממש בפועל בפרויקט NodeJS — מבנה תיקיות שמשקף בדיוק את שלושת הרכיבים. 
MVC/
├── controllers/
│   └── articles.js     ← ה-Controller
├── models/
│   └── article.js      ← ה-Model
├── routes/
│   └── articles.js     ← הניתוב
├── views/
│   └── articles.ejs    ← ה-View (תבנית)
├── app.js
└── package.json

Model — הנתונים ופעולותיהם:

js · models/article.js
const articles = [
    { id : 1, title : "Hello",  content : "world" },
    { id : 2, title : "Hello2", content : "world2" },
    { id : 3, title : "Hello3", content : "world3" }
];

exports.getArticles = () => {
    return articles;
};

Controller — מקבל בקשה, שולף מה-Model, ומבקש מה-View לרנדר:

js · controllers/articles.js
const Article = require('../models/article');

const index = (req, res) => {
    res.render("../views/articles", { articles: Article.getArticles() });
}

module.exports = {
    index
};

View — תבנית EJS שמכניסה את הנתונים ל-HTML:

ejs · views/articles.ejs
<!DOCTYPE html>
<html lang="en">
<head>
    <meta charset="utf-8" />
    <title></title>
</head>
<body>
    <main role="main">
        <%for (var i = 0; i < articles.length; i++) {%>
        <article>
            <span><%=articles[i].id%></span>
            <span><%=articles[i].title%></span>
            <span><%=articles[i].content%></span>
        </article>
        <%}%>
    </main>
</body>
</html>
🔤
הסבר סינטקס — EJS templating
  • exports.getArticles / module.exports — כך NodeJS "מייצא" פונקציות ממודול אחד כדי ש-require במודול אחר יוכל להשתמש בהן.
  • res.render("view", data) — אומר ל-View לבנות HTML מהתבנית + הנתונים.
  • ב-EJS: <% ... %> = קוד JS (לולאה, תנאי), ו-<%= ... %> = הדפסת ערך לתוך ה-HTML. כך הלולאה מייצרת <article> לכל כתבה.
זו בדיוק תבנית ה-"UI template" מהדיאגרמות בתחילת ההרצאה — ה-View מקבל נתונים מה-Controller ובונה מהם את התצוגה.
🏛️
חיבור הכל יחד — מ-ASP.NET ועד Android את אותה תבנית MVC רואים בכל מקום: ASP.NET Core MVC (השרת מרנדר View ומחזיר HTML), ASP.NET Web API (השרת מחזיר JSON, והלקוח בונה את ה-View), וAndroid (View → ViewModel → Repository → Local data). כולן וריאציות על אותו רעיון: הפרדת המידע מהתצוגה מהבקרה — בדיוק העקרונות (SOLID, separation of concerns) שראינו בהרצאה 2, מיושמים בקנה מידה של מערכת שלמה.

★ סיכום ההרצאה

🎯
התמונה המלאה
  • אפשר לכתוב שרת HTTP ידנית (סוקטים) אבל זה מייגע ונוטה לשגיאות → משתמשים ב-frameworks (NodeJS, Express).
  • שרת מטפל בבקשה ב-4 שלבים: request object → handler function → response object → תרגום ל-HTTP.
  • HTTP: GET (safe + idempotent, פרמטרים ב-URL) מול POST (לא safe, נתונים ב-body). REST = משאבים + URI + פעלי HTTP + JSON.
  • תקשורת סינכרונית (טעינה מלאה, מהבהב) מול אסינכרונית (עדכון חלקי, JSON). לדחיפה מהשרת: polling / long-polling / streaming.
  • JS async: fetch מחזיר Promise; .then משרשר; ה-event loop מאפשר async עם thread יחיד; async/await = כתיבה קריאה מעל Promises.
  • ארכיטקטורה: layers (לוגי: PL/BL/DAL) מול tiers (פיזי: Presentation/Application/Data). MVC מפריד Model/View/Controller — מיושם בכל פלטפורמה.
