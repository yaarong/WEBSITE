חלק א' · העמסת אופרטורים

1. Friend functions

🧭
מה זה friend friend function/class = פונקציה או מחלקה שאינה חלק מהמחלקה, אבל מורשית לגשת לממברים הפרטיים שלה. friend שובר אנקפסולציה! — לכן משתמשים בו במשורה: בעיקר עבור אופרטורים של המחלקה ומחלקות "מנהלות" (design patterns). לא נאפשר לאף אחד אחר.
cpp · friend
class Point {
public:
    Point(int x, int y) : m_x(x), m_y(y) {}
    // מתירים לאופרטור הפלט לגשת לשדות הפרטיים:
    friend std::ostream& operator<<(std::ostream& os, const Point& p);
private:
    int m_x, m_y;
};

std::ostream& operator<<(std::ostream& os, const Point& p) {
    return os << "(" << p.m_x << ", " << p.m_y << ")";  // גישה ל-private
}

2. העמסת אופרטורים — בסיס וכללים

🧭
קונטקסט בניגוד ל-Java, ב-C++ אפשר להגדיר מה אופרטורים (+, *, ==, [], <<...) עושים על המחלקות שלנו. למשל למספרים מרוכבים, מטריצות ווקטורים יש אריתמטיקה טבעית.
cpp · חתימות אופרטורים
Point  operator+(const Point& ob);     // חיבור
Point& operator*=(const Point& ob);    // הכפלה-והשמה
Matrix operator/(std::int32_t num);    // חילוק בסקלר
bool   operator!();                    // אונארי
📏
כללים — מתי ואיך
  • הפעולה צריכה להיות מובנת מאליה — אל תמציאו פונקציות מוזרות כאופרטורים.
  • אם מימשת אופרטור ממשפחה מוכרת — ממש את כולם: לא +,*,/ בלי -; לא ==,<=,> בלי >=; ממש גם ++ prefix וגם postfix.
🚫
מגבלות אי אפשר: לשנות את הקדימות של אופרטור; לשנות את מספר הארגומנטים שהוא מקבל (פרט ל-operator()); ליצור אופרטורים חדשים (כמו **=).
↔️
אופרטורים "מימין ומשמאל" כשמממשים אופרטור כמתודה, האובייקט הוא תמיד צד שמאל. כדי לתמוך ב-2 + point (סקלר משמאל), מממשים אופרטור כ-פונקציה חופשית (לרוב friend) שמקבלת את שני האופרנדים.

3. אופרטורים מיוחדים — ++ ו-()

cpp · ++ prefix מול postfix
class Counter {
public:
    Counter& operator++() { ++m_v; return *this; }       // ++x (pre)
    Counter  operator++(int) {                            // x++ (post)
        Counter tmp = *this; ++m_v; return tmp;           // ה-int מבדיל ביניהם
    }
private:
    int m_v = 0;
};
🔤
איך מבדילים pre מ-post operator++() (בלי פרמטר) = prefix (++x), מחזיר רפרנס. operator++(int) (פרמטר int דמה, לא בשימוש) = postfix (x++), מחזיר עותק של הערך הישן.
🎛️
operator() — Functor העמסת אופרטור הקריאה הופכת אובייקט ל-"function object" (functor) — אובייקט שאפשר "לקרוא לו" כמו פונקציה. ניתן להעמיס בכל צורה (כל מספר ארגומנטים). למשל למטריצה: int operator()(size_t i, size_t j) const; → גישה עם m(2, 3).

חלק ב' · ספריית STL

4. Containers — vector, map, list

🧭
קונטקסט ה-STL (Standard Template Library) הוא מימוש C++ של אלגוריתמים ומבני-נתונים מוכרים. לבנות מבני-נתונים בעצמנו שוב ושוב = בזבוז זמן, באגים, וביצועים גרועים. במקום זה — להשתמש ב-STL (גם בטוח יותר: יש בדיקות גבולות, בניגוד למערכי C). לכל ה-containers יש size(), empty(), clear().
Containerמה זהסיבוכיות
vectorמערך דינמיגישה O(1), הוספה/הסרה בסוף O(1) amortized
mapאסוציאטיבי (key→value), כמו BSTהכל O(log n)
listרשימה מקושרת כפולההוספה/הסרה בקצוות O(1)
cpp · vector ו-map
#include <vector>
#include <map>

std::vector<int> v;
v.push_back(10);        // הוספה בסוף
v.push_back(20);
int x = v[0];           // גישה כמו מערך C
v.pop_back();           // הסרת האחרון (UB אם ריק!)

std::map<std::string, int> ages;
ages["Cohen"] = 30;     // הכנסה/עדכון
if (ages.count("Cohen")) { /* קיים — count מחזיר 0 או 1 */ }
⚠️
שתי מלכודות חשובות operator[] על map מכניס איבר אם הוא לא קיים — להשתמש בזהירות (אם רק בודקים קיום, להשתמש ב-count/find). ועל vector אין בדיקת גבולות (UB מחוץ לטווח) — להשתמש ב-.at() לבדיקה. ולגבי list — הטיפ מהמצגת: פשוט אל תשתמשו בו (ברוב המקרים vector מהיר יותר בפועל).

5. Iterators

🧭
מה זה iterator ב-C++, iterator הוא בעצם מצביע לנתונים + סט קטן של פעולות. השתיים הנפוצות: dereference (*it = הערך) ו-increment (++it = האיבר הבא). כל container תומך ב-begin() (iterator לאיבר הראשון) ו-end() (iterator מיוחד אחרי האחרון — לא ניתן ל-dereference, כמו null).
cpp · מעבר על container
// עם iterator מפורש:
for (auto it = v.begin(); it != v.end(); ++it)
    std::cout << *it;

// for-each (מועדף!):
for (const auto& x : v)
    std::cout << x;

// מעבר על map:
for (const auto& [key, value] : ages)   // structured binding (C++17)
    std::cout << key << "=" << value;
🔤
היררכיית ה-iterators מהחלש לחזק: input/output (קריאה/כתיבה חד-פעמית) → forward (linked list) → bidirectional (doubly-linked, תומך --) → random-access (vector, תומך +, <, גישה ישירה). ה-iterators של STL מובטחים להיות לפחות forward. הכוח: אלגוריתם עובד על טווח (range) של iterators — ללא תלות ב-container הספציפי.

6. Algorithms

🧭
קונטקסט הספרייה <algorithm> = אוסף פונקציות שעובדות על טווחים. טווח = זוג iterators [begin, end) — כולל את ההתחלה, לא כולל את הסוף. [begin, end) = כל ה-container; [it, it) = טווח ריק.
cpp · אלגוריתמים נפוצים
#include <algorithm>

auto it = std::find(v.begin(), v.end(), 20);   // מחפש 20
int n   = std::count(v.begin(), v.end(), 5);   // סופר מופעים של 5
std::sort(v.begin(), v.end());                 // ממיין
std::sort(v.begin(), v.end(),                  // עם השוואה מותאמת:
          [](int a, int b) { return a > b; }); // סדר יורד (lambda)
🎯
אלגוריתמים מול לולאות למה להעדיף אלגוריתמי STL על לולאות? הם פחות נוטים לשגיאות, בד"כ יעילים יותר, והקוד ברור יותר (האלגוריתם מתעד את עצמו — std::sort אומר בדיוק מה קורה). כלל אצבע: נמנעים מלולאות; אם בכל זאת — שתהיה לפחות for-each.

7. מלכודות נפוצות ב-STL

🪤
לשים לב
  • STL לא thread-safe על האובייקטים שלכם — לקרוא את התיעוד.
  • STL לא משחרר מה שהוא לא הקצה.
  • vector[i] — אין בדיקת גבולות (UB). vector.at(i) — יש.
  • map[key] — מבצע הכנסה אם המפתח לא קיים!
  • חלק מהפונקציות (כמו remove_if) לא עושות מה שנדמה — לקרוא תיעוד לפני שימוש.
  • לא להשתמש במערכי C כלל — std::array הוא תחליף טוב.

חלק ג' · Templates

8. מה זה template (מול Java/C)

🧭
קונטקסט Templates = דרך לייצר קוד גנרי. שלא כמו Java generics (שרק עושים auto-cast, אותו קוד לכולם, רק על אובייקטים), ב-C++ הקומפיילר מייצר קוד חדש לכל טיפוס שבו משתמשים. זה נותן יותר גמישות (עובד על כל טיפוס, כולל פרימיטיביים) — אבל מגדיל את הקובץ הבינארי.
C++ TemplatesJava GenericsC (void*)
מנגנוןקוד חדש לכל טיפוסאותו קוד, auto-castאותו קוד, cast ידני
עובד עלכל הטיפוסיםרק אובייקטיםרק מצביעים
שגיאותבזמן קומפילציהבזמן קומפילציהבזמן ריצה
⏱️
Templates = פולימורפיזם בזמן קומפילציה ירושה+virtual = פולימורפיזם בזמן ריצה (גמיש, איטי, שגיאות בריצה). Templates = פולימורפיזם בזמן קומפילציה (מהיר, גנרי, שגיאות בקומפילציה, אבל קשה יותר לכתוב). שתי דרכים להשיג "חוזה שטיפוסים שונים ממלאים".

9. תחביר — function ו-class templates

cpp · function template
template <typename T>          // אפשר גם class במקום typename
T max(T a, T b) {
    return (a > b) ? a : b;
}

max(3, 7);          // T מוסק כ-int
max(2.5, 1.0);      // T מוסק כ-double
max<double>(2, 3);   // ציון מפורש כשהקומפיילר לא יכול להסיק
cpp · class template + non-type param
template <typename T>
class Box {
    T m_value;
public:
    Box(T v) : m_value(v) {}
    T get() const { return m_value; }
};
Box<int> b(42);

// פרמטר תבנית מטיפוס פרימיטיבי (non-type) — למשל מערך בגודל קבוע:
template <typename T, int SIZE>
class FixedArray { T data[SIZE]; };
FixedArray<int, 10> arr;
🔤
‎typename‎ מול ‎class‎ ברוב המקרים template <typename T> ו-template <class T> זהים. יש מקרים מיוחדים שבהם חייבים אחד מהם. ניתן גם פרמטרים מטיפוס פרימיטיבי (int SIZE) — שימושי למערכים בגודל קבוע.

10. Template Specializations

🧭
קונטקסט לפעמים רוצים שתבנית תעבוד אחרת עבור טיפוס מסוים — למשל פונקציה גנרית שמתנהגת מיוחד עבור int, או מחלקה שעובדת אחרת עם std::string. זה specialization.
cpp · specialization
template <typename T>
std::string describe(T x) { return "generic"; }

template <>                                  // התמחות מלאה ל-bool
std::string describe<bool>(bool x) { return x ? "true" : "false"; }

describe(42);     // "generic"
describe(true);   // "true"

11. constexpr, type_traits ו-static_assert

🧭
חישוב בזמן קומפילציה התגלה שמנגנון ה-templates הוא Turing-complete — אפשר לחשב כל דבר בזמן קומפילציה (נקרא TMP — Template Meta Programming). בעקבות זאת נוספה המילה constexpr: מסמנת שמשתנה/פונקציה/מחלקה יכולים (משתנה — חייב) להיות מחושבים בזמן קומפילציה. משתנה constexpr מחושב בקומפילציה ולא תופס מקום בריצה — לכן תמיד נגדיר קבועי-קומפילציה כ-constexpr.
cpp · constexpr (פיבונאצ'י בקומפילציה)
constexpr int fib(int n) {
    return (n < 2) ? n : fib(n - 1) + fib(n - 2);
}
constexpr int f10 = fib(10);   // 55 — מחושב בזמן קומפילציה!
🔤
type_traits, static_assert, if constexpr
  • <type_traits> — בדיקות זמן-קומפילציה על טיפוסים. למשל std::is_pointer<T>::value (או std::is_pointer_v<T> מ-C++14). ממומשות דרך template specialization.
  • static_assert(condition, "msg") — מוודא תנאי בזמן קומפילציה. מונע הודעות שגיאה באורך 4000 שורות כשטיפוס לא מתאים נכנס לתבנית.
  • if constexpr (C++17) — תנאי שנבדק בזמן קומפילציה; רק הענף הפעיל מתקמפל. מחליף את ה-#if/#elif המכוער של C, ונוח לשילוב עם type_traits.
📚
נושאים מתקדמים (להמשך) Variadic templates (מספר משתנה של פרמטרים), SFINAE, Concepts (C++20), ו-std::variant — מעבר להיקף התרגול, אבל שווה להכיר את השמות.

★ סיכום

🎯

  • Templates: קוד גנרי שמיוצר בקומפילציה (פולימורפיזם בזמן קומפילציה); function/class templates, specializations; constexpr, type_traits, static_assert.
