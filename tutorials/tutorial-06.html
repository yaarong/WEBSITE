חלק א' · מושגים

1. Process מול Thread

🧭
קונטקסט זוכרים שלתוכנית יש 4 איזורי זיכרון (Data, Text, Stack, Heap — תרגול 1)? בפישוט: הנתונים (איזורי הזיכרון) הם ה-process, והיחידה שמריצה פקודות היא ה-thread.
🔤
מי מחזיק מה
  • Process = תוכנית שרצה. מחזיק את ה-code, heap ו-data (משותפים).
  • Thread = יחידת הרצה. לכל thread יש registers, stack ו-program counter משלו.
  • process מכיל לפחות thread אחד (ה-main thread, שנקודת הכניסה שלו ב-C/C++ היא main), ויכול להחזיק רבים.
  • ה-threads לא מסונכרנים — הם יכולים לרוץ בכל סדר אפשרי. (זה בדיוק מקור הבעיות בהמשך.)

2. מושגים, Join, ויעילות

מושגפירוש
CPU-boundזמן הסיום נקבע ע"י מהירות המעבד
IO-boundזמן הסיום נקבע ע"י מהירות פעולות הקלט/פלט (המתנה)
User threadthread תוכנתי, לא נתמך ע"י ה-OS
Kernel threadנתמך ע"י ה-OS, יכול להריץ שתי פקודות בו-זמנית
Context-switchהמעבר מ-thread אחד לאחר
Atomic operationפעולה שלא ניתנת להפרעה באמצע
⚛️
למה i++ מסוכן i++ נראה אטומי אבל אינו — הוא מתפרק ל-3 פעולות: (1) שמירת הערך של i, (2) הגדלה, (3) החזרת הערך הישן. context-switch יכול לקרות בין הפעולות! ב-C/C++ מניחים שכל הפעולות לא אטומיות.
🔗
Join — המתנה ל-thread thread אחד יכול לחכות ל-thread אחר שיסיים — זה join. בעת join אפשר גם לקבל את ערך ההחזרה של ה-thread. (זה פותר את הבעיה: thread 1 שיצר את thread 2 רוצה את התוצאה שלו → עושה join.)
⚖️
יתרונות וחסרונות יתרונות: ריצה מקבילית (multicore), כש-thread ממתין ל-I/O אחרים עובדים, וארגון קוד ליחידות. חסרונות: קוד ודיבוג קשים בהרבה, overhead של context-switch, וקשה לאופטם. תוכנית יעילה כשהיא CPU-bound (משימות IO-bound מבזבזות זמן בהמתנה).

חלק ב' · הבעיות

3. Critical Section ו-Race Conditions

🧭
Critical Section קטע קריטי = כל קטע קוד שניגש למשאב משותף (קובץ, משתנה) — לקריאה או לכתיבה. גישה מקבילית למשאבים משותפים עלולה להוביל להתנהגות לא צפויה, ולכן נחשבת לא בטוחה. (קריאה בלבד בד"כ בטוחה במקביל — אבל אלא אם מובטח, כל פעולה על משאב משותף נחשבת לא בטוחה.)
🏁
Race Condition המצב שבו שתי משימות (או יותר) רצות בו-זמנית, ועלולות להתבצע בסדר שהמתכנת לא התכוון אליו, מה שמוביל לתוצאות לא דטרמיניסטיות ולא רצויות. זה באג.
דוגמה · משיכה מחשבון בנק
// amount = 1000, שני threads מושכים 500 כל אחד במקביל:
//
// מצופה:                       race condition:
//  T1: read  amount (1000)      T1: read amount (1000)
//  T1: amount -= 500 → 500      T2: read amount (1000)   ← לפני ש-T1 כתב!
//  T2: read  amount (500)       T1: amount -= 500 → 500
//  T2: amount -= 500 → 0        T2: amount -= 500 → 500
//  תוצאה: 0  ✅                 תוצאה: 500  ❌ (כסף "נוצר"!)

4. Mutex, Deadlock ו-Starvation

🔒
Mutual Exclusion — הפתרון הפתרון הפשוט לבעיית הקטע הקריטי: mutual exclusion — שני threads לא יכולים להריץ את הקטע הקריטי במקביל. הכלי: mutex (lock). ה-mutex נעול/לא-נעול; אם thread מנסה לנעול mutex נעול — הוא ימתין עד שישתחרר. "כרטיס כניסה" יחיד לקטע הקריטי — רק מי שמחזיק את הכרטיס נכנס.
🔤
סוגי mutex recursive (ניתן לנעול כמה פעמים), fast (רק התכונות הבסיסיות), shared (לקריאה/כתיבה). שחרור mutex לא-נעול = undefined behavior.
🔴
Deadlock המצב שבו שני threads מחכים זה לזה (בד"כ למשאב משותף) — אף אחד לא ימשיך והתוכנית "תיתקע".
deadlock — סדר נעילה הפוך
// Thread A          // Thread B
lock(m2);            lock(m1);
lock(m1);  ←┐    ┌─→ lock(m2);    ← כל אחד מחכה ל-mutex שהשני מחזיק!
...         └────┘   ...
unlock(m1);          unlock(m2);
unlock(m2);          unlock(m1);
מניעה: לנעול mutexes תמיד באותו סדר.
🍽️
Starvation מצב שבו thread אחד נמנע ממנו לקבל את המשאבים הדרושים לו לעבודתו. בד"כ מניחים שזה לא קורה; אם כן — מיישמים מנגנון הוגנות (fairness).

חלק ג' · POSIX threads (C)

5. pthreads ו-mutexes

🧭
קונטקסט מערכות Unix לא מכירות "threading" מובנה, ולכן יש מימוש תוכנתי — pthread (POSIX threads). זה ה-low-level שעליו בנויות ספריות C++. צריך #include <pthread.h> ולקשר עם libpthread. הספרייה מחזירה 0 בהצלחה וקוד שגיאה אחרת.
c · יצירה ו-join
#include <pthread.h>

void* worker(void* arg) {        // נקודת כניסה: void* → void*
    int n = *(int*)arg;
    // ... עבודה
    return nullptr;
}

int main() {
    pthread_t tid;
    int arg = 42;
    pthread_create(&tid, nullptr, worker, &arg);  // (id, attr, fn, arg)
    pthread_join(tid, nullptr);                    // ממתין לסיום (retval=null)
    return 0;
}
🔤
הקריאות החשובות
  • pthread_create(&tid, attr, start_routine, arg) — attr בד"כ nullptr; start_routine חייבת להיות פונקציה void* f(void*).
  • pthread_join(tid, &retval) — ממתין ל-thread ומקבל את ערך ההחזרה (או nullptr להתעלם).
  • pthread_exit / return — סיום thread. pthread_detach — thread ש"מנקה אחרי עצמו" (לא צריך join).
c · mutex
pthread_mutex_t mutex;
pthread_mutex_init(&mutex, nullptr);   // לפני השימוש

pthread_mutex_lock(&mutex);            // נכנס לקטע הקריטי (ממתין אם נעול)
// ... critical section
pthread_mutex_unlock(&mutex);          // יוצא

pthread_mutex_destroy(&mutex);         // בסיום
🔤
חוקי ה-mutex חובה init לפני שימוש ו-destroy בסיום (ואסור לאתחל פעמיים). lock נתקע אם נעול; trylock חוזר מיד עם EBUSY; unlock — רק ה-thread שנעל יכול לשחרר. מעבירים את ה-mutex by pointer (כדי שכולם ישתפו את אותו אובייקט).

חלק ד' · C++ threads

6. std::thread ו-locks

🧭
קונטקסט C++11 מציעה ממשק threading מובנה, נוח ו-cross-platform (בנוי על pthread ב-Linux). #include <thread>. יוצר thread כמעט מכל callable, ואפילו ממתודות.
cpp · std::thread
#include <thread>

void worker(int n) { /* ... */ }

std::thread t(worker, 42);   // יצירה + הרצה
t.join();                    // המתנה לסיום (או t.detach())
🔤
3 המתודות + מלכודת ארגומנטים join / detach (כמו POSIX), ו-joinable (האם רץ/הסתיים-אך-לא-detached). מלכודת: הארגומנטים מועברים כ-rvalue — אי אפשר להעביר lvalue reference ישירות; עוטפים ב-std::ref(var).
cpp · mutex + lock_guard
#include <mutex>

std::mutex m;

void safe_increment() {
    std::lock_guard<std::mutex> guard(m);   // נועל בבנאי
    // ... critical section
}   // ← משתחרר אוטומטית ב-destructor (סוף ה-scope)
🛡️
RAII על mutexes — lock_guard ו-unique_lock
  • std::mutex ישיר (lock/unlock) — low-level, לא RAII → נמנעים ממנו ישירות.
  • lock_guard — נועל בבנאי, משחרר ב-destructor. הדרך הקלה ביותר; לשימוש ב-scopes קצרים. (RAII — בדיוק כמו תרגול 2!)
  • unique_lock — גמיש יותר (אפשר לא לנעול מיד, יש owns_lock וכו'). דרוש ל-condition variables.

7. Atomic, Condition Variables ו-Thread Pool

⚛️
Atomic — בלי mutex ל-טיפוסים פרימיטיביים (int, char, bool), std::atomic (#include <atomic>) מספק פעולות אטומיות בלי mutex — יעיל מאוד. std::atomic<int> counter; counter++; בטוח בין threads.
🌀
Busy waiting — הדרך הרעה להמתין להמתין ל-thread אחר ע"י לולאה ריקה (while (!done) {}) = busy waiting — מבזבז את המעבד לחלוטין. יש דרך טובה יותר.
🔔
Condition Variable — הדרך הנכונה אובייקט סנכרון שאפשר להמתין עליו (wait) ולהעיר אותו (notify_one / notify_all). ה-thread הממתין "נרדם" (לא מבזבז CPU) עד שמישהו מעיר אותו. דורש unique_lock נעול — הוא משחרר אותו בזמן ההמתנה ונועל מחדש כשמסתיים. הערה: עלול להתעורר גם בלי שהתנאי התקיים ("spurious wakeup") — לכן בודקים את התנאי בלולאה.
🏊
Thread Pool — design pattern שרת מטפל בהרבה בקשות קצרות וחוזרות. גישה נאיבית: thread חדש לכל בקשה — אבל יצירה/מחיקה של אלפי threads = overhead אדיר ו-thrashing. הפתרון — Thread Pool: יוצרים מראש מספר קבוע של threads + תור משימות. כל thread ממתין שהתור יתמלא, שולף משימה, מבצע אותה, וחוזר להמתין. שימוש חוזר ב-threads במקום ליצור חדשים — חוסך את ה-overhead.

★ סיכום

🎯
מה לקחנו
  • Process (code/heap/data משותפים) מול thread (stack/registers/PC משלו). threads לא מסונכרנים.
  • הבעיות: קטע קריטי → race condition (תוצאות לא דטרמיניסטיות), deadlock (המתנה הדדית), starvation. i++ לא אטומי!
  • הפתרון: mutex (mutual exclusion) — כרטיס כניסה יחיד לקטע הקריטי.
  • POSIX: pthread_create/join, pthread_mutex_lock/unlock.
  • C++: std::thread, ו-RAII על mutexes (lock_guard/unique_lock), std::atomic, condition variables (במקום busy waiting), ותבנית thread pool.
