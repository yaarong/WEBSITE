תכנות מונחה-עצמים ב-C++ — מחלקות, ירושה ופולימורפיזם

חלק א' · מחלקות

1. מחלקות והרשאות גישה

מחלקה = אגד של שדות קשורים + פעולות עליהם. ב-C++ ההרשאות מוגדרות בבלוקים (לא פר-שדה כמו ב-Java):

cpp · class
class BankAccount {
public:                       // ממשק — נגיש מכל מקום
    void deposit(double sum);
    double balance() const;

private:                      // ברירת המחדל — נגיש רק למתודות המחלקה
    double m_balance = 0;     // מוסכמה: קידומת m_ לשדות
};
הרשאהמי ניגש
publicכל מקום בקוד (בהינתן גישה לאובייקט) — מגדיר את הממשק
protectedהמחלקה עצמה + כל המחלקות היורשות
privateרק המחלקה עצמה (ו-friends) — ברירת המחדל
🔤
class מול struct ב-C++, struct זהה ל-class פרט להבדל אחד: ב-struct ברירת המחדל היא public (לתאימות אחורה עם C), וב-class היא private. שתיהן יכולות להחזיק מתודות, בנאים ו-destructor. כלל אצבע: רק מתודות צריכות להיות public; שדות תמיד private.
🔁
אובייקטים כשדות מחלקה יכולה להחזיק שדה שהוא מחלקה אחרת ✅. אבל לא שדה מאותו טיפוס (רקורסיה אינסופית) ❌ — אלא מצביע לאותו טיפוס ✅ (מצביע הוא רק כתובת בגודל קבוע).

2. ממברים סטטיים ו-this

🧭
סטטי ממבר סטטי שייך לטיפוס ולא למופע — ניגשים אליו ישירות בלי אובייקט (ClassName::member). שדה סטטי ≈ משתנה גלובלי (להשתמש בזהירות, פרט ל-Singleton). פונקציה סטטית ≈ פונקציה גלובלית, אך יכולה לגשת לממברים הפרטיים של המחלקה.
cpp · static + this
class Counter {
public:
    static int count;                 // הצהרה
    static int get() { return count; } // פונקציה סטטית

    void setName(std::string name) {
        this->name = name;            // this — מצביע לאובייקט הקורא
    }
private:
    std::string name;
};
int Counter::count = 0;               // הגדרה — חייבת בקובץ .cpp
🔤
this מצביע לאובייקט הקורא (כמו בשפות אחרות). בד"כ לא נחוץ, אבל דרוש כשיש התנגשות שמות בין שדה לארגומנט. הערה: שדה סטטי לא-קבוע לא ניתן להגדיר ב-header (רק להצהיר) — מגדירים בקובץ .cpp (או inline static מ-C++17).

3. Qualifiers על מתודות

מצמידים מילות מפתח אחרי חתימת המתודה (וניתן לשרשר):

Qualifierמשמעות
constהמתודה לא משנה שדות — ניתן לקרוא לה מאובייקט const
overrideהמתודה דורסת מתודה של מחלקת אב
noexceptהמתודה לא זורקת חריגות
finalלא ניתן לדרוס את המתודה במחלקות יורשות
& / &&המתודה נקראת רק מ-lvalue / rvalue
cpp · qualifiers
double balance() const;            // לא משנה שדות
void process() const noexcept;     // שרשור

const BankAccount acc{};
acc.balance();   // מתקמפל רק כי balance מסומנת const
⭐
כלל זהב — const ה-qualifier הכי חשוב. כל מתודה שלא משנה שדות פנימיים חייבת להיות const — כך אפשר לקרוא לה גם מאובייקטים קבועים, וזה מתעד את הכוונה. ל-noexcept יש מתודות שחייבות אותו: destructor, move constructor, move assignment.

חלק ב' · ניהול משאבים

4. בנאים, destructor ו-Rule of 5

🔤
חמשת "מנהלי המשאבים" (Resource handlers)
  • בנאי (constructor) — מאתחל את האובייקט.
  • בנאי העתקה (copy constructor) — בונה אובייקט מאובייקט אחר מאותו סוג.
  • בנאי הזזה (move constructor, C++11) — בונה מאובייקט אחר שזמן חייו תם (move).
  • אופרטור = בהעתקה / בהזזה — השמה.
  • destructor — נקרא לפני שהאובייקט נהרס, משחרר משאבים. noexcept כברירת מחדל.
לכל החמישה יש מימוש ברירת-מחדל ב-C++.
5️⃣
Rule of 5 אם הגדרת אחת מחמש המתודות (destructor / copy ctor / copy assignment / move ctor / move assignment) — עליך להגדיר את כולן. (כי אם הגדרת אחת, סימן שאתה מנהל משאב ידנית, והאחרות צריכות להתאים.)
cpp · חתימות חמשת המתודות
class Buffer {
public:
    Buffer();                                  // בנאי רגיל
    Buffer(const Buffer& other);               // copy constructor
    Buffer(Buffer&& other) noexcept;           // move constructor
    Buffer& operator=(const Buffer& other);    // copy assignment
    Buffer& operator=(Buffer&& other) noexcept; // move assignment
    ~Buffer();                                 // destructor
};

5. RAII, =default/=delete, explicit

🧭
RAII — Resource Acquisition Is Initialization גישת תכנות איכותית: כובלים את אורך חייו של משאב לאורך חייו של אובייקט. המשאב נתפס בבנאי ומשוחרר ב-destructor — כך אין דליפות. דוגמאות מהספרייה: std::unique_ptr (זיכרון), std::fstream (קובץ), std::vector (מערך).
🔤
=default ו-=delete ברגע שמגדירים אחד ממנהלי המשאבים, ארבעת האחרים מתבטלים אוטומטית. כדי להחזיר מימוש ברירת-מחדל: = default. כדי לאסור פעולה מפורשות (כמו ב-unique_ptr שאוסר העתקה): = delete.
cpp · default / delete / explicit
class NoCopy {
public:
    NoCopy() = default;                       // בנאי ברירת מחדל
    NoCopy(const NoCopy&) = delete;           // אוסר העתקה
    NoCopy& operator=(const NoCopy&) = delete;

    explicit NoCopy(int x);   // explicit — מונע המרה אוטומטית מ-int ל-NoCopy
};
⚠️
explicit בנאי שמקבל פרמטר יחיד מאפשר לקומפיילר להמיר אוטומטית מהטיפוס הזה למחלקה — לרוב התנהגות לא רצויה. הוספת explicit לפני הבנאי מונעת את ההמרה האוטומטית.

חלק ג' · ירושה ופולימורפיזם

6. ירושה — תחביר והרשאות

ירושה = הרחבת מחלקה ע"י מחלקה אחרת. כל המתודות, השדות והמאפיינים הסטטיים של האב נמצאים ביורשת:

cpp · syntax
class B : public A { };   // B יורשת מ-A (ירושה פומבית)
// אפשר גם:  class B : private A { };   או   protected
🔤
C++ מול Java
  • C++ מאפשרת ירושה מרובה → אין צורך ב-interface מובנה.
  • ב-C++ לא כל מתודה ניתנת לדריסה כברירת מחדל (צריך virtual).
  • C++ תומכת בירושה פרטית ומוגנת (לא רק פומבית).
⚖️
ירושה — טוב ורע טוב: שימוש חוזר בקוד, הגדרת ממשק והיררכיה, Liskov substitution. רע: בעיית yo-yo, שבירת אנקפסולציה, ירושת מתודות מיותרות, וקוד שקשה יותר להבין. ירושה היא כלי חזק — להשתמש במידה.

7. מתודות וירטואליות

🧭
מה זה virtual מתודה וירטואלית = מתודה שמותר לדרוס במחלקות יורשות. הקומפיילר לא יודע בזמן קומפילציה איזו לקרוא, ולכן מבצע dynamic dispatch — מחליט בזמן ריצה. ה-"virtualness" עוברת בירושה (יורשת לא חייבת לציין שוב virtual). לא ניתן ל-constructor.
cpp · virtual + override
class Animal {
public:
    virtual void speak() const { std::cout << "..."; }
    virtual ~Animal() = default;     // destructor וירטואלי — חשוב בירושה!
};

class Dog : public Animal {
public:
    void speak() const override { std::cout << "Woof"; }  // override
};
🔤
למה תמיד override ה-keyword override (C++11) מסמן שהמתודה דורסת. היתרון: אם בטעות לא דרסת באמת (שם/חתימה לא תואמים) — הקוד לא יתקמפל, ומונע באגים קשים לאיתור. גם קריא יותר. לכן: בכל דריסה — להשתמש ב-override.
🏗️
סדר קריאה לבנאים (1) בנאי מחלקת האב, (2) בנאי כל השדות לפי סדר ההצהרה, (3) בנאי המחלקה עצמה. ה-destructor — בסדר הפוך. אם לאב/שדה אין בנאי ברירת-מחדל, חובה לספק ארגומנטים דרך initialization list: Dog() : Animal("dog"), m_age(0) { }.

8. Pure virtual, abstract ו-interface

cpp · pure virtual
class Shape {
public:
    virtual double area() const = 0;   // pure virtual — חייב מימוש ביורשת
    virtual ~Shape() = default;
};
🔤
pure virtual ו-abstract
  • Pure virtual — מתודה בלי גוף, שחייבים לדרוס. תחביר: = 0 אחרי החתימה (כמו מתודת interface ב-Java).
  • מחלקה עם לפחות מתודה pure virtual אחת היא abstract — לא ניתן ליצור ממנה מופע, אבל כן להשתמש בה כטיפוס פולימורפי.
  • יורשת של abstract חייבת לדרוס את כל ה-pure virtuals — אחרת היא עצמה הופכת ל-abstract.
🔌
interface ב-C++ אין keyword מיוחד. בפועל, interface הוא מחלקה ש: (1) כל המתודות שלה pure virtual, (2) אין לה שדות לא-סטטיים, (3) יש לה destructor וירטואלי (כדי שמחיקה דרך מצביע לבסיס תשחרר נכון).

9. פולימורפיזם ו-casting

🧭
פולימורפיזם = "רב-צורתיות" היכולת להסתכל על אובייקטים רבים דרך "משקפיים" של טיפוס אחד (האב). ב-C++ משיגים זאת רק דרך מצביע או רפרנס ל-base class — לא דרך הטיפוס ה"חשוף". כשטיפוס פולימורפי קורא למתודה וירטואלית, מתבצעת בדיקה בזמן ריצה איזו מתודה להפעיל.
cpp · polymorphism
Animal* a = new Dog();   // מצביע ל-base, אובייקט מסוג derived
a->speak();              // "Woof" — dynamic dispatch

Base* bPtr = new Derived();              // ✅ מתקמפל ועובד
Derived* dPtr = new Base();              // ❌ לא מתקמפל — Base אינו Derived
🔤
\u200Estatic_cast\u200E מול \u200Edynamic_cast\u200E
cpp
Base* b1 = new Derived();
Base* b2 = new Base();

// C-style — מסוכן, undefined behavior אם לא תואם:
Derived* d = (Derived*)b2;                   // מתקמפל אבל UB!

// dynamic_cast — בטוח, מחזיר nullptr אם ההמרה לא חוקית:
Derived* d1 = dynamic_cast<Derived*>(b1);    // עובד
Derived* d2 = dynamic_cast<Derived*>(b2);    // nullptr (בטוח)
dynamic_cast עובד רק על מחלקות פולימורפיות (עם לפחות מתודה וירטואלית אחת), ומוגדר היטב. נמנעים מ-C-style cast ומ-UB ככל האפשר.

חלק ד' · העשרה — ירושה מרובה

10. Virtual table ובעיית היהלום

🧭
Virtual table (vtable) כל מחלקה פולימורפית (עם לפחות מתודה וירטואלית) מקבלת טבלה וירטואלית — טבלה (פר-מחלקה) שמפרטת לכל מתודה וירטואלית איפה המימוש שלה. בקריאה למתודה וירטואלית דרך מצביע/רפרנס, ניגשים לטבלה, בודקים איזו מתודה להפעיל, ומפעילים. זה המנגנון מאחורי ה-dynamic dispatch.
cpp · ירושה מרובה
class Cartoon { /* ... */ };
class Bear    { /* ... */ };

class WinnieThePooh : public Cartoon, public Bear { };  // יורש משתיהן
💎
בעיית היהלום ("deadly diamond of death") אם B ו-C יורשות מ-A, ו-D יורשת משתיהן — אז A מופיעה פעמיים בתוך D (עותק דרך B ועותק דרך C). זו הבעיה המרכזית בירושה מרובה (ולכן Java אסרה אותה).
cpp · virtual inheritance — הפתרון
class B : virtual public A { };   // ירושה וירטואלית
class C : virtual public A { };
class D : public B, public C { }; // עכשיו רק עותק אחד של A!
🔤
virtual inheritance ומושגי העשרה
  • ירושה וירטואלית (virtual בהצהרת הירושה) מבטיחה שרק עותק אחד של מחלקת הבסיס יישמר. ("virtual" כאן = כתובת הבסיס לא ידועה בזמן קומפילציה.)
  • mutable — qualifier על שדה: ניתן לשנותו גם במתודה const. רק לשדות עזר (cache), אף פעם לא ל-state. (כלל M&M: mutex יכול וצריך להיות mutable.)
  • ירושה פרטית/מוגנת — לפתרון שכפול קוד בלי הסיכון של פולימורפיזם שגוי; מגבילה הרשאות.

★ סיכום

🎯
מה לקחנו
  • מחלקות: הרשאות (public/protected/private בבלוקים), class≈struct, ממברים סטטיים, this.
  • Qualifiers: const (החשוב!), override, noexcept, final.
  • ניהול משאבים: חמשת מנהלי המשאבים → Rule of 5, RAII, =default/=delete, explicit.
  • ירושה: : public Base, מתודות virtual + override, pure virtual → abstract → interface, סדר בנאים + init list.
  • פולימורפיזם: רק דרך מצביע/רפרנס; dynamic_cast בטוח מול C-style. העשרה: vtable, בעיית היהלום + virtual inheritance.
