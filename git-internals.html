Git — מהבסיס ועד Git Internals

חלק א' · בסיס וטרמינל

1. מה זה Git?

Git היא מערכת בקרת גרסאות מבוזרת (distributed version control). תפקידה: לעקוב אחרי שינויים בקבצים לאורך זמן, לאפשר לחזור לכל נקודה בעבר, ולעבוד בכמה אנשים על אותו פרויקט.

📸
הרעיון המרכזי: Snapshots, לא Diffs רוב מערכות בקרת הגרסאות הישנות שמרו הפרשים (diffs) — "בקובץ הזה השתנו השורות האלו". Git שונה: בכל commit הוא שומר תצלום (snapshot) של כל הפרויקט באותו רגע. קובץ שלא השתנה — Git לא משכפל אותו, אלא מצביע לאותו תוכן קודם (כבר נראה איך, ב-internals). זה הבסיס למהירות ולאמינות של Git.
🌍
"מבוזרת" לכל מפתח יש עותק מלא של כל ההיסטוריה על המחשב שלו (לא רק "העתק עבודה" כמו במערכות מרכזיות). אפשר לבצע commit, לדפדף בהיסטוריה ולעבוד — בלי חיבור לרשת. השרת (כמו GitHub) הוא רק "עוד עותק" שמסנכרנים מולו.

2. שלושת האזורים

כל קובץ בפרויקט Git חי באחד משלושה "אזורים". להבין אותם = להבין את כל זרימת העבודה:

Working Directoryהקבצים שאתה עורך כרגע בתיקייה
→
Staging Area (Index)"רשימת ההמתנה" — מה ייכנס ל-commit הבא
→
Repository (.git)ההיסטוריה הקבועה — כל ה-commits
אזורמה זההפקודה שמעבירה אליו
Working Directoryהתיקייה שלך עם הקבצים, כפי שהם עכשיו. כאן עורכים.(עריכה ידנית)
Staging Area (Index)אזור ביניים. "בחרתי את השינויים האלה להיכנס ל-commit הבא".git add
Repositoryמסד הנתונים הקבוע (.git). ברגע שעשית commit — השינוי שמור לתמיד.git commit
🎯
למה צריך Staging Area? כדי שתוכל לבחור בדיוק מה נכנס לכל commit. ערכת 5 קבצים אבל רוצה commit שמכיל רק 2 מהם? git add רק על השניים → git commit. ה-Staging הוא ה"טיוטה" של ה-commit הבא.

3. הפקודות הבסיסיות בטרמינל

זרימת העבודה הטיפוסית, מההתחלה:

terminal · זרימה בסיסית
$ git init                  # יוצר מאגר חדש (תיקיית .git) בתיקייה הנוכחית
$ git config user.name "Guy"   # מגדיר מי אתה (פעם אחת)
$ git config user.email "g@x.com"

$ git status                # מה מצב הקבצים? (modified / staged / untracked)
$ git add file.txt         # מעביר קובץ ל-Staging
$ git add .                 # מעביר את כל השינויים ל-Staging
$ git commit -m "message"   # שומר snapshot מה-Staging ל-Repository

$ git log                   # מציג את היסטוריית ה-commits
$ git log --oneline         # גרסה קצרה: hash + הודעה לכל commit
$ git diff                  # מה השתנה ב-Working אבל עוד לא ב-Staging
$ git diff --staged         # מה נמצא ב-Staging (ייכנס ל-commit הבא)
פקודהמה עושה
git initיוצר מאגר חדש — תיקיית .git ריקה.
git clone <url>מוריד עותק מלא של מאגר קיים (כולל כל ההיסטוריה).
git statusמציג מה השתנה, מה ב-Staging, ומה לא במעקב.
git add <file>מעביר שינויים ל-Staging Area.
git commit -m "..."שומר את ה-Staging כ-commit חדש בהיסטוריה.
git logמציג את היסטוריית ה-commits.
git diffמראה הבדלים בין האזורים.
🚫
.gitignore קובץ טקסט שמפרט מה לא לעקוב אחריו. כל שורה = תבנית. למשל node_modules, .env, *.log. קבצים שתואמים לא יופיעו ב-git status ולא ייכנסו ל-commit — כך לא מעלים תלויות ענק או סודות לגיט.

חלק ב' · Git Internals — הלב 🫀

🔑
המעבר מ"איך משתמשים" ל"איך זה עובד" עד כאן למדנו פקודות. עכשiv נפתח את הקופסה השחורה. כל מה שראינו (add, commit, branches) הוא רק "ממשק נוח" מעל מנגנון אחד פשוט להפליא. כשתבין אותו — תפסיק לשנן פקודות ותתחיל להבין מה כל אחת עושה.

4. הרעיון הגדול — מסד נתונים ממוען-תוכן

בלבו, Git הוא פשוט key-value store (מסד נתונים מסוג מפתח→ערך):

🗃️
Content-Addressable Store אתה נותן ל-Git תוכן (bytes), והוא: (1) מחשב את ה-SHA-1 שלו — מחרוזת בת 40 תווים הקסדצימליים, (2) שומר את התוכן בקובץ ששמו הוא ה-hash הזה, ו-(3) מחזיר לך את ה-hash. כדי לקבל את התוכן בחזרה — אתה נותן את ה-hash. המפתח הוא ה-hash של התוכן עצמו — מכאן "ממוען-תוכן" (content-addressable).
💡
ההשלכה המיידית אם המפתח נגזר מהתוכן, אז תוכן זהה → hash זהה → נשמר פעם אחת בלבד. שני קבצים זהים (או אותו קובץ ב-100 commits) חולקים אובייקט אחד. זה ה-deduplication שמאחורי "snapshots" יעילים — וזה הסוד למה Git לא מתנפח.

5. תיקיית .git

כש-git init רץ, הוא יוצר תיקייה אחת — .git — שמכילה את כל המאגר. זהו "המוח" של Git:

.git/ — המבנה החשוב
.git/
├── objects/      # מסד הנתונים — כאן יושבים כל האובייקטים (לפי hash)
├── refs/         # מצביעים: refs/heads/ (ענפים), refs/tags/ (תגיות)
├── HEAD          # "איפה אני עכשיו" — מצביע לענף הנוכחי
├── index         # ה-Staging Area (קובץ בינארי)
└── config        # הגדרות המאגר
🧭
שלושת השחקנים כמעט הכל ב-Git מסתכם ב-3 אלה: objects/ (התוכן — האובייקטים), refs/ (מצביעים בעלי-שם ל-commits), ו-HEAD (איפה אני עומד). נכיר כל אחד.

6. SHA-1 — איך תוכן הופך לכתובת

נראה את זה בעיניים. git hash-object היא פקודת ה-plumbing ששומרת תוכן ומחזירה את ה-hash שלו:

terminal · שמירת blob
$ printf 'hello git\n' | git hash-object -w --stdin
8d0e41234f24b6da002d962a26c2495ea16a425f     # ← ה-SHA-1 של התוכן
  • -w = "write" (באמת תשמור, לא רק תחשב). בלעדיו Git רק מחשב את ה-hash.
  • --stdin = קרא את התוכן מהקלט (במקום מקובץ).

איך ה-hash מחושב? Git לא מערבב את ה-bytes הגולמיים — הוא מוסיף כותרת קטנה:

פורמט אובייקט (לפני SHA-1 + דחיסה)
"<type> <size>\0<content>"

# לדוגמה, עבור 'hello git\n' (10 בייטים):
"blob 10\0hello git\n"
        └────┬───┘ └┬┘ └────┬────┘
           type   size   התוכן עצמו
# SHA-1 של המחרוזת הזו = 8d0e4123...

ואיפה זה נשמר על הדיסק? לפי ה-hash, מחולק ל-2 חלקים:

terminal · מיקום על הדיסק
$ ls .git/objects/8d/
0e41234f24b6da002d962a26c2495ea16a425f

# 2 התווים הראשונים = שם התיקייה   → 8d/
# 38 התווים הנותרים  = שם הקובץ      → 0e41234f...
🔤
חקירת אובייקטים — git cat-file הפקודה לקריאת אובייקטים מהמסד:
  • git cat-file -t <hash> — הtype של האובייקט (blob/tree/commit/tag).
  • git cat-file -p <hash> — התוכן (pretty-print).
  • git cat-file -s <hash> — הגודל בבייטים.
terminal · cat-file
$ git cat-file -t 8d0e4123
blob
$ git cat-file -p 8d0e4123
hello git
$ git cat-file -s 8d0e4123
10
✂️
לא צריך את כל ה-40 תווים מספיק קידומת ייחודית (בד"כ 7 תווים, כמו 8d0e412). Git ישלים את השאר כל עוד אין שני אובייקטים עם אותה קידומת.

7. 4 סוגי האובייקטים

זה הלב של ה-internals. כל מה שיש ב-Git הוא אחד מ-4 סוגי אובייקטים — וכולם נשמרים באותו אופן (תוכן → SHA-1 → objects/):

🟦 Blob
תוכן של קובץ אחד. בלי שם, בלי תאריך — רק ה-bytes.
🟩 Tree
תיקייה: רשימה של שמות → blobs/trees + הרשאות.
🟨 Commit
snapshot: מצביע ל-tree אחד + הורה + מחבר + הודעה.
🟪 Tag
תגית מוערת: מצביע לאובייקט (בד"כ commit) + הודעה.
אובייקטמייצגמכילמה הוא לא מכיל
Blobתוכן קובץרק את ה-bytes של הקובץשם הקובץ, תאריך, הרשאות
Treeתיקייהרשומות: מצב + שם + hash (לכל blob/tree)את התוכן עצמו (רק מצביע)
Commitsnapshot בזמןtree אחד + הורה/הורים + author + committer + הודעהאת הקבצים (רק מצביע ל-tree)
Tagתגית קבועהאובייקט-יעד + type + שם + tagger + הודעה—

🟦 Blob — תוכן קובץ

ה-blob שכבר יצרנו (8d0e4123) הוא בדיוק זה — התוכן hello git. שים לב: ל-blob אין שם! הוא לא יודע שהוא "hello.txt". הוא רק תוכן. השם יגיע מה-tree.

🟩 Tree — תיקייה

ה-tree נותן שמות ל-blobs ובונה היררכיה. נבנה אחד: נכניס את ה-blob ל-Staging תחת השם hello.txt, ונכתוב tree:

terminal · בניית tree
$ git update-index --add --cacheinfo 100644 8d0e4123 hello.txt
$ git write-tree
07ed5a7aebb914e3a02edf6d622b82d364037e3c

$ git cat-file -p 07ed5a7
100644 blob 8d0e41234f24b6da002d962a26c2495ea16a425f	hello.txt
       └─┬─┘ └─┬─┘ └──────────────┬──────────────┘ └───┬───┘
       mode  type        hash (מצביע ל-blob)         שם

כל שורה ב-tree היא רשומה: מצב (mode) + type + hash + שם. ה-tree לא מכיל את התוכן — רק מצביע ל-blob לפי ה-hash שלו. תיקיית-משנה? פשוט רשומה נוספת מסוג tree שמצביעה ל-tree אחר.

modeמשמעות
100644קובץ רגיל
100755קובץ הרצה (executable)
040000תיקייה (tree)
120000symlink

🟨 Commit — snapshot בזמן

ה-commit עוטף tree אחד ומוסיף לו הקשר: מי, מתי, הודעה, ומה ה-commit הקודם. נבנה אחד שמצביע ל-tree שלנו:

terminal · בניית commit
$ echo "first commit" | git commit-tree 07ed5a7
7237a5c6304c71e23be9eb1d038efb63bca4febe

$ git cat-file -p 7237a5c
tree 07ed5a7aebb914e3a02edf6d622b82d364037e3c     # ← ה-snapshot
author    demo <a@b.c> 1782651228 +0300           # מי כתב, מתי
committer demo <a@b.c> 1782651228 +0300           # מי ביצע commit

first commit                                    # ההודעה
🔤
מבנה ה-commit
  • tree — מצביע ל-tree אחד (השורש של ה-snapshot). זהו "צילום המסך" של כל הפרויקט.
  • parent — ה-commit הקודם. ל-commit הראשון אין; לרגיל יש אחד; ל-merge יש 2+ (לא נעסוק בזה).
  • author מול committer — בד"כ זהים. author = מי כתב את השינוי במקור; committer = מי הכניס אותו בפועל.
  • ההודעה — אחרי שורה ריקה.
🔗
שרשרת ה-commits = ההיסטוריה כל commit מצביע ל-parent שלו, וזה ל-parent שלו... עד ה-commit הראשון. ההיסטוריה היא פשוט שרשרת מקושרת של commits, שכל אחד מצביע אחורה. git log בסך הכל הולך אחורה לאורך ה-parents.

🟪 Tag — תגית מוערת

אובייקט tag (annotated) מצביע ל-commit ונותן לו שם קבוע + הודעה (למשל v1.0). הוא נוצר ב-git tag -a v1.0 -m "...", ומבנהו דומה ל-commit: object (היעד) + type + tag (השם) + tagger + הודעה.

8. איך הכל מתחבר — גרף האובייקטים

כל אובייקט מצביע לאובייקטים אחרים לפי ה-hash שלהם. התוצאה היא גרף יפהפה:

גרף האובייקטים
  commit  7237a5c
    │ tree
    ▼
  tree  07ed5a7
    │ hello.txt
    ▼
  blob  8d0e412   →  "hello git"

# commit → tree (snapshot) → blobs (תוכן הקבצים)
# commit הבא → parent → commit הקודם ...
🧩
זה הכל. באמת. commit → tree → (subtrees +) blobs. ושרשרת ה-commits אחורה דרך parent. כל מה ש-Git עושה הוא ליצור אובייקטים כאלה ולקשר ביניהם. אין שום דבר מעבר לזה במודל הנתונים.
📸
למה זה "snapshots" ולא "diffs" — עכשiv זה ברור כל commit מצביע ל-tree מלא של כל הפרויקט. אבל אם קובץ לא השתנה בין שני commits — שני ה-trees מצביעים לאותו blob (אותו hash!). אז יש snapshot מלא בכל commit, אבל בלי שכפול — קבצים שלא השתנו נשמרים פעם אחת ומשותפים. snapshot מלא + dedup = הטוב משני העולמות.

9. Refs, HEAD וענפים — רק קבצים עם hash

בעיה: ה-hashes הם 40 תווים מקריים. בני אדם לא זוכרים 7237a5c.... הפתרון: refs — שמות קריאים שמצביעים ל-commits.

🏷️
ענף (branch) הוא פשוט קובץ טקסט ענף בשם main הוא קובץ .git/refs/heads/main שמכיל שורה אחת — ה-hash של ה-commit האחרון בענף. זהו. "ליצור ענף" = ליצור קובץ קטן עם hash. "להזיז ענף קדימה" (commit חדש) = לכתוב hash חדש לקובץ.
terminal · refs הם קבצים
$ cat .git/refs/heads/main
7237a5c6304c71e23be9eb1d038efb63bca4febe     # ← רק hash של commit

$ cat .git/HEAD
ref: refs/heads/main                       # ← HEAD מצביע לענף (לא ל-commit)
שחקןמה זהמצביע ל...
branch (refs/heads/x)קובץ עם hashה-commit האחרון בענף
tag (refs/tags/x)קובץ עם hashcommit קבוע (גרסה)
HEAD"איפה אני עכשיו"בד"כ לענף (ref: refs/heads/main)
🧭
השרשרת המלאה HEAD → refs/heads/main → commit 7237a5c → tree 07ed5a7 → blob 8d0e412. כשאתה עושה git commit: נוצרים tree+commit חדשים, וה-Git מעדכן את הקובץ refs/heads/main כך שיצביע ל-commit החדש. "להחליף ענף" (git checkout) = לשנות לאן HEAD מצביע.

10. בונים commit ביד (plumbing)

נחבר הכל. כל מה שעושה git commit -m "..." מאחורי הקלעים הוא רצף הפקודות האלה — ועכשiv תראה שאין בו שום קסם:

terminal · commit ביד, מ-A עד ת
# 1. שומרים תוכן כ-blob
$ printf 'hello git\n' | git hash-object -w --stdin
8d0e4123...

# 2. שמים אותו ב-Staging תחת שם, וכותבים tree
$ git update-index --add --cacheinfo 100644 8d0e4123 hello.txt
$ git write-tree
07ed5a7...

# 3. עוטפים את ה-tree ב-commit
$ echo "first commit" | git commit-tree 07ed5a7
7237a5c...

# 4. מצביעים את הענף main ל-commit החדש
$ git update-ref refs/heads/main 7237a5c

# זהו! בדיוק כמו git add + git commit.

Porcelain מול Plumbing Git מחלק את הפקודות לשתיים: Porcelain (האינסטלציה היפה — add, commit, log — מה שמשתמשים בו יומיום) ו-Plumbing (הצנרת הנמוכה — hash-object, cat-file, write-tree — שעליהם בנויים הראשונים). הבנת ה-plumbing = הבנת Git.

11. העקרונות הגדולים

🏛️
4 העקרונות שנובעים ממיעון-לפי-תוכן
  • Deduplication — תוכן זהה = hash זהה = נשמר פעם אחת. קבצים שלא השתנו משותפים בין commits.
  • Integrity (שלמות) — ה-hash מחושב מהתוכן. אם בייט אחד משתנה (אפילו תקלת דיסק) — ה-hash כבר לא יתאים, ו-Git מזהה שחיתות מיד.
  • Immutability (אי-שינוי) — אובייקטים אף פעם לא משתנים. "לשנות היסטוריה" = ליצור אובייקטים חדשים ולהזיז ref. הישנים נשארים עד שמנקים אותם.
  • Snapshots יעילים — כל commit הוא תצלום מלא, אבל בזכות ה-dedup זה לא בזבזני.
🎯
המשפט אחד לזכור Git הוא מסד נתונים שממען תוכן לפי ה-SHA-1 שלו. יש 4 אובייקטים: blob (תוכן קובץ), tree (תיקייה: שמות→hashes), commit (tree אחד + הורה + מטא), tag. הם מצביעים זה לזה ב-hashes ויוצרים גרף: commit → tree → blobs. refs (ענפים) הם קבצים עם hash, ו-HEAD אומר איפה אתה. זהו כל Git.
🧪
הערה — Loose objects מול Packfiles האובייקטים שראינו נשמרים כקבצים נפרדים ("loose"), כל אחד מכווץ ב-zlib. כשמצטברים הרבה, Git אורז אותם ל-packfile אחד דחוס (ב-git gc / push) — שם הוא כן משתמש בדלתות (deltas) לחיסכון מקום. אבל זו רק אופטימיזציית אחסון; המודל הלוגי תמיד נשאר 4 האובייקטים + snapshots.
