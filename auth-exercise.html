שכבת הזדהות (JWT) מעל תרגיל Node.js + MongoDB

קיבלת תרגיל Express + MongoDB רגיל (CRUD על משאב כלשהו) — ועכשיו חמי דורש גם שכבת הזדהות: חלק מהפקודות מותרות רק למשתמש מחובר. כאן נראה בדיוק מה מתווסף לקוד: מודל User במונגו, קונטרולר הרשמה, קונטרולר טוקן (login שמחזיר JWT), ושכבת middleware ששומרת על הראוטים המוגנים.

🧭
הרעיון ב-3 משפטים
Registerיוצר משתמש ב-DB (שומר username + password).
Loginבודק username+password, ואם תקין → מחזיר טוקן JWT חתום.
Middlewareשומר על ראוטים מוגנים: מוודא שהטוקן תקין לפני שנותן להיכנס.
🪙
מה זה בכלל JWT? JSON Web Token — מחרוזת בת 3 חלקים מופרדים בנקודה: header.payload.signature. ה-payload מכיל מידע על המשתמש (id, username); ה-signature נוצר בעזרת סוד (JWT_SECRET) שרק השרת מכיר. הטוקן חתום, לא מוצפן — כלומר כל אחד יכול לקרוא את ה-payload, אבל אף אחד לא יכול לזייף אותו בלי הסוד. כשהשרת מקבל טוקן הוא מאמת את החתימה — אם היא תקינה, סימן שהשרת עצמו הנפיק אותו ואפשר לסמוך עליו.

1️⃣ .env — הסוד לחתימה

החתימה של ה-JWT מבוססת על מחרוזת סוד. שומרים אותה ב-.env (אף פעם לא בקוד ולא בגיט!), יחד עם שאר ההגדרות:

env · .env
JWT_SECRET=mysecret
PORT=3000
MONGO_URI=mongodb://localhost:27017/mydb
📝
במבחן בכתב מספיק להזכיר ש-JWT_SECRET יושב ב-.env (ושהקובץ ב-.gitignore). לעולם לא לכתוב את הסוד ישירות בקוד.

2️⃣ models/User.js — מודל המשתמש

זו ה"מגירה" החדשה במונגו: collection בשם users, שכל מסמך בה הוא משתמש עם username ו-password. בדיוק כמו כל Mongoose Schema אחר:

javascript · models/User.js
const mongoose = require('mongoose');

const userSchema = new mongoose.Schema({
    username: { type: String, required: true },
    password: { type: String, required: true }   // exam: plain. (אמיתי: hash)
});

module.exports = mongoose.model('User', userSchema);
🐛
הבאגים החוזרים שלך כאן
  • type: String / type: Number — לא int.
  • required: true — לא require.
  • module.exports — לא export.
  • סוגריים נכונים: new mongoose.Schema({ ... }).
🔓
סיסמה ב-plain text במבחן שומרים את הסיסמה כמו שהיא (פשטות). במציאות אף פעם לא — מאחסנים hash (למשל עם bcrypt) ומשווים עם bcrypt.compare. אם חמי מבקש אבטחה אמיתית — להזכיר את זה.

3️⃣ controllers/userController.js — Register

הקונטרולר שיוצר משתמש חדש. הלוגיקה: ולידציה → לבדוק שאין כבר משתמש כזה → ליצור. register הוא ה-handler של POST /api/users.

javascript · controllers/userController.js
const User = require('../models/User');

async function register(req, res) {
    const { username, password } = req.body;
    if (!username || !password)
        return res.status(400).json({ error: "missing fields" });

    const existing = await User.findOne({ username });
    if (existing)
        return res.status(409).json({ error: "user exists" });

    await User.create({ username, password });
    return res.status(201).json({ message: "created" });
}

module.exports = { register };
🐛
הבאגים החוזרים שלך כאן
  • הסדר הוא (req, res) — לא (res, req)!
  • req.body — לא req.body() (זה לא פונקציה).
  • מחרוזות שגיאה במרכאות: { error: "..." }.
  • async + await כי findOne/create פונים ל-DB.
🔢
קודי הסטטוס כאן 400 שדה חסר · 409 Conflict (המשתמש כבר קיים) · 201 Created (נוצר בהצלחה).

4️⃣ controllers/tokenController.js — Login (כאן ה-JWT)

זה קונטרולר הטוקן. מאמת username+password מול ה-DB, ואם תקין — חותם טוקן JWT ומחזיר אותו ללקוח:

javascript · controllers/tokenController.js
const jwt = require('jsonwebtoken');
const User = require('../models/User');

async function login(req, res) {
    const { username, password } = req.body;

    const user = await User.findOne({ username });
    if (!user || user.password !== password)
        return res.status(401).json({ error: "invalid credentials" });

    const token = jwt.sign(
        { id: user._id, username: user.username },   // payload
        process.env.JWT_SECRET                         // הסוד
    );
    return res.status(200).json({ token });
}

module.exports = { login };
🔤
jwt.sign(payload, secret) מקבל את ה-payload (המידע שנרצה לשמור בתוך הטוקן — בד"כ ה-id והשם של המשתמש) ואת ה-secret, ומחזיר את הטוקן החתום כמחרוזת. לא לשים סיסמה ב-payload (הוא קריא לכולם!).
🌐
למה login הוא POST /api/tokens? ב-REST כל פעולה היא "יצירה/קריאה/מחיקה של משאב". התחברות = "יצירת טוקן" → לכן POST על המשאב tokens. זה הסגנון שחמי אוהב (בדיוק כמו POST /api/restaurants שיוצר מסעדה).
🔐
למה 401 אחד גם ל"אין משתמש" וגם ל"סיסמה שגויה" מחזירים 401 invalid credentials בשני המקרים בכוונה — כדי לא לחשוף לתוקף אם שם המשתמש קיים. שים לב לתנאי המאוחד: if (!user || user.password !== password).

5️⃣ middleware/authMiddleware.js — השומר

זו שכבת ההזדהות עצמה: פונקציה שרצה לפני ה-controller. היא שולפת את הטוקן מה-header, מאמתת אותו, ורק אם הוא תקין נותנת להמשיך:

javascript · middleware/authMiddleware.js
const jwt = require('jsonwebtoken');

function authMiddleware(req, res, next) {
    const header = req.headers.authorization;        // "Bearer <token>"
    if (!header)
        return res.status(401).json({ error: "no token" });

    const token = header.split(' ')[1];              // קח רק את הטוקן
    try {
        const payload = jwt.verify(token, process.env.JWT_SECRET);
        req.user = payload;                          // מצרף את המשתמש לבקשה
        next();                                       // ממשיך לראוט
    } catch {
        return res.status(401).json({ error: "invalid token" });
    }
}

module.exports = authMiddleware;
⭐
שני הדברים הקריטיים
  • req.user = payload; — כדי שה-controller שאחרי יידע מי המשתמש המחובר.
  • next(); — מעביר את הבקשה הלאה ל-handler. בלעדיו הבקשה תיתקע ולא תגיע לעולם לקונטרולר.
🔤
פירוק ה-header ו-jwt.verify ה-header מגיע בפורמט "Bearer <token>" — לכן header.split(' ')[1] מבודד רק את הטוקן עצמו. jwt.verify(token, secret) מאמת את החתימה: אם תקין → מחזיר את ה-payload; אם מזויף/פג → זורק שגיאה, ולכן עוטפים ב-try/catch ומחזירים 401.

6️⃣ routes/ — חיווט ותקיעת ה-middleware

שני ראוטים חדשים — אחד למשתמשים (הרשמה) ואחד לטוקנים (התחברות):

javascript · routes/userRoutes.js + tokenRoutes.js
// routes/userRoutes.js
const router = require('express').Router();
const { register } = require('../controllers/userController');
router.post('/', register);                 // POST /api/users
module.exports = router;

// routes/tokenRoutes.js
const router = require('express').Router();
const { login } = require('../controllers/tokenController');
router.post('/', login);                    // POST /api/tokens
module.exports = router;

וכאן הקסם — הגנה על ראוט קיים: תוקעים את ה-middleware כ"שכבה" בין הנתיב ל-handler. רצים משמאל לימין; auth רץ ראשון, ורק אם קרא next() מגיעים ל-controller:

javascript · routes/restaurantRoutes.js
const auth = require('../middleware/authMiddleware');

router.post('/',   auth, createRestaurant);   // ← רק מחובר יכול ליצור
router.get('/',          getRestaurants);     // ← פתוח לכולם
router.delete('/:id', auth, deleteRestaurant); // ← גם מחיקה מוגנת
🧱
"שכבה לפני ה-handler" router.post('/', auth, createRestaurant) = "כשמגיע POST, קודם הרץ את auth; אם הוא אישר (next()) — הרץ את createRestaurant". אפשר לשרשר כמה middleware. ראוטים בלי auth (כמו get('/')) — פתוחים לכולם.

🔌 app.js — חיבור הכל

javascript · app.js
require('dotenv').config();                  // טוען את .env ל-process.env
const express = require('express');
const mongoose = require('mongoose');
const app = express();
app.use(express.json());                     // לקרוא body כ-JSON

app.use('/api/users',       require('./routes/userRoutes'));
app.use('/api/tokens',      require('./routes/tokenRoutes'));
app.use('/api/restaurants', require('./routes/restaurantRoutes'));

mongoose.connect(process.env.MONGO_URI);
app.listen(process.env.PORT || 3000);
🐛
אל תשכח require('dotenv').config() בשורה הראשונה (אחרת process.env.JWT_SECRET יהיה undefined) · app.use(express.json()) (אחרת req.body ריק).
