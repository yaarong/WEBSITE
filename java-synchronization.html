Threads ו-Synchronization ב-Java

כשכמה threads (חוטי ריצה) נוגעים באותו משתנה בו-זמנית, מתרחש באג עדין ומסוכן — race condition: התוצאה שגויה ומשתנה מהרצה להרצה. כאן נבין למה זה קורה, איך synchronized פותר את זה, ולמה לא כדאי פשוט לעטוף הכל ב-synchronized (רמז: ביצועים). הכל סביב התרגיל שמצורף.

🧵
קונטקסט — מה זה Thread ולמה זה מסובך Thread = "חוט ריצה" עצמאי בתוך התוכנית. כמה threads רצים במקביל (על ליבות שונות) וחולקים את אותו זיכרון. זה נהדר לביצועים — אבל ברגע ששניים כותבים לאותו משתנה בלי תיאום, הם דורכים אחד על השני. זה הנושא של העמוד.

1️⃣ Threads — איך מריצים קוד במקביל

ב-Java יוצרים thread עם המחלקה Thread, ונותנים לו קוד להריץ (כ-lambda מסוג Runnable). שלוש פעולות מרכזיות:

פעולהמה עושה
new Thread(() -> {...})יוצר thread עם הקוד שירוץ בתוכו (עדיין לא רץ).
.start()מתחיל את ה-thread — הקוד מתחיל לרוץ במקביל ל-thread הראשי.
.join()"חכה ל-thread הזה שיסיים" — ה-thread הראשי נעצר עד שזה מסתיים.
🔤
למה start() ולא קריאה ישירה? קריאה ישירה לקוד הייתה מריצה אותו ב-thread הנוכחי (טורי). start() פותח thread חדש שרץ בנפרד. וה-join() חשוב כדי שה-thread הראשי לא ידפיס את התוצאה לפני ש-thread-ים אחרים סיימו לעבוד.

2️⃣ הבעיה — Race Condition

הקוד מהתרגיל: מחלקה Foo עם מונה bar. שני threads, כל אחד מגדיל את bar ב-baz() עשרת-אלפים פעם. נצפה לקבל בסוף 20000:

java · race condition
class Foo {
    private int bar = 0;
    public void baz() {
        bar++;
    }
    public int getBar() {
        return bar;
    }
    public static void main(String[] args) {
        Foo f = new Foo();
        Thread[] arr = new Thread[2];
        for (int i = 0; i < arr.length; i++) {
            arr[i] = new Thread(() -> {
                for (int j = 0; j < 10000; j++) {
                    f.baz();
                }
            });
            arr[i].start();        // מריץ את שני ה-threads במקביל
        }
        for (int i = 0; i < arr.length; i++) {
            arr[i].join();         // מחכה לשניהם
        }
        System.out.println(f.getBar());
    }
}
פלט (משתנה מהרצה להרצה!)
17342     // ולא 20000... בריצה אחרת: 15981, 19004 ...
💥
למה התוצאה קטנה מ-20000? — bar++ אינו אטומי השורה bar++ נראית כמו פעולה אחת, אבל למעשה היא שלוש פעולות:
  1. קורא את הערך של bar מהזיכרון.
  2. מוסיף 1 לערך שנקרא.
  3. כותב את התוצאה בחזרה ל-bar.
כששני threads עושים את זה במקביל, הם יכולים לקרוא את אותו ערך לפני שאחד מהם הספיק לכתוב — ואז שתי ההגדלות "נדרסות" לאחת. זה נקרא lost update (עדכון אבוד).
Thread A read bar → 41
+ 1 → 42
(נעצר רגע)
write 42 → bar=42
Thread B (במקביל) read bar → 41 // אותו ערך!
+ 1 → 42
write 42 → bar=42
// שתי הגדלות → +1 בלבד

↑ שתי הגדלות שהפכו לאחת. לכן הסכום הסופי נמוך מ-20000, ומשתנה בכל הרצה לפי הטיימינג.

3️⃣ הפתרון — synchronized

השינוי מהתרגיל: מוסיפים את המילה synchronized למתודות:

java · synchronized methods
public synchronized void baz() {
    bar++;
}
public synchronized int getBar() {
    return bar;
}
פלט (תמיד!)
20000     // נכון, ויציב בכל הרצה
🔒
מה synchronized עושה — מנעול (lock / monitor) לכל אובייקט ב-Java יש מנעול פנימי (intrinsic lock / monitor). כשמתודה מסומנת synchronized, thread שנכנס אליה תופס את המנעול של האובייקט, וכל thread אחר שמנסה להיכנס למתודה synchronized של אותו אובייקט חייב לחכות עד שהראשון יוצא. כך bar++ רץ בלעדית — שלושת הצעדים (read-add-write) קורים בלי שאף אחד מתערב באמצע.
👁️
למה לסנכרן גם את getBar()? לא רק בגלל ה-race — אלא בגלל visibility (נראוּת). בלי סנכרון, thread אחד עלול לראות ערך ישן של bar (מ-cache של הליבה שלו) ולא את הערך העדכני. synchronized מבטיח שהשינויים של thread אחד נראים ל-threads האחרים. לכן מסנכרנים גם את הקריאה וגם את הכתיבה.

4️⃣ synchronized(this) — בלוק במקום מתודה

וריאציה דומה (אך מעט שונה) מהתרגיל — לסנכרן בלוק בתוך המתודה במקום את כולה:

java · synchronized block
public void baz() {
    synchronized (this) {
        bar++;
    }
}
🔤
אותה תוצאה, יותר שליטה public synchronized void baz() הוא בדיוק קיצור של עטיפת כל גוף המתודה ב-synchronized(this){...} — אותו מנעול (של this), אותה התנהגות (הפלט שוב 20000). היתרון של הבלוק: אפשר לנעול רק את החלק הקריטי ולהשאיר את שאר המתודה בלי נעילה — מה שמקצר את הזמן שבו המנעול תפוס (וזה משפר ביצועים, כי threads אחרים מחכים פחות).
🎯
הכלל ננעל כמה שפחות קוד — רק את הגישה למשתנה המשותף. כל שורה מיותרת בתוך הבלוק מאריכה את זמן ההמתנה של כל שאר ה-threads.

5️⃣ אז למה לא תמיד synchronized? — Performance

אם synchronized מתקן את הבאג — למה לא לעטוף הכל? כי לנעילה יש מחיר. נמדוד אותו: 10 threads, כל אחד מגדיל 10,000,000 פעם (סה"כ 100 מיליון), עם מדידת זמן.

5.1 בלי סנכרון — מהיר אבל שגוי

java · ללא synchronized
public void baz() { bar++; }          // ← ללא synchronized
// ... 10 threads, כל אחד 10,000,000 הגדלות ...
System.out.println(f.getBar() + " " + (System.currentTimeMillis() - start));
פלט (מייצג — משתנה לפי מחשב)
38274915 142     // סכום שגוי (הרבה פחות מ-100000000), אבל מהיר (~142ms)

5.2 עם synchronized — נכון אבל איטי בהרבה

java · עם synchronized(this)
public void baz() {
    synchronized (this) {
        bar++;
    }
}
// ... אותם 10 threads × 10,000,000 ...
פלט (מייצג — משתנה לפי מחשב)
100000000 4380   // סכום נכון, אבל איטי פי כמה (~4380ms)
⚡
למה synchronized כל-כך מאט?
  • Serialization (טוריות): המנעול מאלץ את ה-threads לעבוד בתור, אחד-אחד — מבטל את היתרון של המקביליות. בדיוק החלק שרצינו במקביל הפך לטורי.
  • Contention (תחרות): 10 threads נלחמים על אותו מנעול. רובם חוסמים וממתינים רוב הזמן.
  • Overhead: כל תפיסה/שחרור של מנעול עולה זמן בפני עצמה — וכאן זה קורה 100 מיליון פעם.
⚖️
ה-trade-off בלי סנכרון: מהיר אבל שגוי. עם סנכרון: נכון אבל איטי. לכן הכלל: לסנכרן רק כשחייבים (גישה משותפת לנתון משתנה), ורק את החלק המינימלי. סנכרון מיותר = איטיות בלי תועלת.
🧪
העשרה — חלופות ל-synchronized לדוגמה הספציפית הזו (מונה), Java מציעה את AtomicInteger עם incrementAndGet() — מבצע את ה-read-add-write כפעולה אטומית אחת ברמת החומרה, בלי מנעול מלא — מהיר יותר מ-synchronized ועדיין נכון. (מעבר לתרגיל, אבל טוב לדעת שיש כלים קלים יותר.)

6️⃣ סיכום ומה לזכור

🎯
המשפט אחד לזכור כש-threads כותבים לאותו משתנה, x++ אינו אטומי (read-add-write) → race condition ותוצאה שגויה. synchronized נותן מנעול שמכריח גישה בלעדית (ומבטיח visibility) — מתקן את הבאג, אבל מאט כי הוא מטריל את המקביליות. הכלל: לסנכרן רק את החלק הקריטי, ורק כשצריך.
גרסהתוצאהמהירות
ללא synchronized❌ שגויה (lost updates)⚡ מהירה
synchronized (מתודה / בלוק)✅ נכונה🐢 איטית
AtomicInteger (העשרה)✅ נכונה⚡ מהירה יחסית
