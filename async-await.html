Promise, async, await — וה-Event Loop

JavaScript היא שפה חד-תהליכית (single-threaded) — היא יכולה לעשות רק דבר אחד בכל רגע. אז איך היא מטפלת בפעולות שלוקחות זמן (בקשת רשת, קריאת קובץ, טיימר) בלי "להיתקע"? התשובה: Promises, התחביר async/await, וה-Event Loop. כאן נבין את התיאוריה לעומק, ואז נפצח חידות פלט שורה-שורה.

🧭
למה זה חשוב למבחן חמי אוהב לשאול "מה הפלט של הקוד הבא?" עם ערבוב של setTimeout, Promise.then ו-await. כדי לענות נכון צריך להבין סדר אחד: קוד סינכרוני → ואז כל ה-microtasks → ואז macrotask אחד → וחוזר חלילה. נלמד את הכלל הזה, ואז הוא יפתור לך כל חידה.

1️⃣ הבעיה — JS חד-תהליכית

ל-JavaScript יש thread אחד — "תור" אחד של פקודות שרץ מלמעלה למטה. אם פעולה לוקחת זמן (למשל בקשת רשת של 2 שניות), ובאמת היינו מחכים לה — כל הדף היה קופא ל-2 שניות (אי אפשר ללחוץ, לגלול, כלום).

הפתרון: פעולות שלוקחות זמן הן אסינכרוניות. במקום לעצור ולחכות, JS "שולחת אותן הצידה", ממשיכה בקוד, וכשהן מסתיימות — מטפלת בתוצאה. השאלה היחידה היא: איך מתארים "מה לעשות כשהתוצאה תגיע"?

😵
הדרך הישנה — Callbacks (ולמה היא בעייתית) פעם העבירו פונקציה ("callback") שתופעל כשהפעולה תסתיים. הבעיה: כשמשרשרים כמה פעולות תלויות זו בזו, מקבלים "Callback Hell" — פונקציות מקוננות עמוק, קוד שקשה לקרוא ולתחזק. Promises נולדו כדי לפתור בדיוק את זה.

2️⃣ מה זה Promise?

Promise הוא אובייקט שמייצג תוצאה עתידית — "שובר" לערך שעוד לא הגיע. אפשר לחשוב עליו כמו קבלה שמקבלים בבית קפה: עדיין אין לך הקפה, אבל יש לך הבטחה שהוא יגיע (או שיגידו לך שנגמר). ל-Promise יש בדיוק 3 מצבים:

Pendingממתין — עוד לא הסתיים
Fulfilledהצליח — יש תוצאה (resolve)
Rejectedנכשל — יש שגיאה (reject)
🔒
כלל הברזל Promise מתחיל ב-pending, ועובר פעם אחת בלבד ל-fulfilled או ל-rejected — ואז הוא "נעול" (settled) ולא משתנה יותר. אי אפשר לקרוא ל-resolve פעמיים, ואי אפשר לעבור מ-fulfilled ל-rejected.

2.1 יצירת Promise

javascript · new Promise
const p = new Promise((resolve, reject) => {
  // ה"executor" — רץ מיד, סינכרונית
  const ok = true;
  if (ok) {
    resolve('הצליח!');   // → המצב הופך ל-fulfilled עם הערך 'הצליח!'
  } else {
    reject('נכשל!');     // → המצב הופך ל-rejected עם השגיאה
  }
});
🔤
ה-executor רץ מיד הפונקציה שמעבירים ל-new Promise(...) רצה סינכרונית, ברגע היצירה. resolve ו-reject הן פונקציות שה-JS נותן לך כדי "לסגור" את ה-Promise. (זה חשוב לחידות בהמשך — ה-console.log בתוך ה-executor רץ מיד!)

2.2 צריכת Promise — .then / .catch / .finally

javascript
p
  .then(value => console.log('הצלחה:', value))   // רץ אם fulfilled
  .catch(err  => console.log('שגיאה:', err))      // רץ אם rejected
  .finally(() => console.log('תמיד רץ'));          // בכל מקרה
מתודהמתי רצהמקבלת
.then(fn)כשה-Promise הצליח (fulfilled)את הערך מ-resolve
.catch(fn)כשה-Promise נכשל (rejected)את השגיאה מ-reject
.finally(fn)בכל מקרה, אחרי שהסתייםכלום
🔗
שרשור (chaining) .then מחזיר Promise חדש — לכן אפשר לשרשר: fetch(url).then(res => res.json()).then(data => ...). כל שלב מקבל את התוצאה של הקודם. זה מה ש-Promises פתרו לעומת ה-callback hell — שרשרת שטוחה במקום קינון עמוק.

3️⃣ async / await — סוכר תחבירי

async/await הם דרך נוחה יותר לעבוד עם Promises — אותו דבר מתחת למכסה, אבל הקוד נראה סינכרוני וקריא. שתי מילות קסם:

מילהמה היא עושה
asyncתווית על פונקציה. הופכת אותה כך שתמיד תחזיר Promise. חובה כדי שמותר יהיה להשתמש ב-await בתוכה.
await"עצור כאן, חכה שה-Promise יסתיים, ותן לי את התוצאה האמיתית" (לא את ה-Promise). אם ה-Promise נכשל — await זורק את השגיאה.
javascript · אותו קוד, שתי דרכים
// עם .then
function getData() {
  return fetch('/api/data').then(res => res.json());
}

// עם async/await — קריא יותר
async function getData() {
  const res = await fetch('/api/data');
  const data = await res.json();
  return data;
}
⭐
3 כללים קריטיים
  • כל פונקציית async מחזירה Promise. return 42 בתוכה = Promise שנפתר עם 42.
  • await רק בתוך async. מחוץ לפונקציית async זו שגיאת תחביר.
  • אם קוראים לפונקציית async ממקום אחר — גם שם צריך await (אחרת מקבלים את ה-Promise, לא את הערך).
🧠
הנקודה הכי חשובה ל-event loop await מפצל את הפונקציה לשניים: מה שלפני ה-await רץ סינכרונית; מה שאחריו נהפך ל"המשך" שמתוזמן כ-microtask (בדיוק כמו .then). זה הסוד לפתרון כל החידות בהמשך.

4️⃣ ה-Event Loop — החוק שפותר הכל

יש שני תורים שבהם JS שומרת משימות אסינכרוניות, ולהם עדיפות שונה:

התורמי נכנס אליועדיפות
Microtasks (תור מהיר)Promise.then/.catch, וההמשך אחרי await🥇 גבוהה
Macrotasks (תור איטי)setTimeout, setInterval, אירועי I/O🥈 נמוכה
🔁
הכלל — שננו אותו!
  1. הרץ את כל הקוד הסינכרוני (מלמעלה למטה) עד הסוף.
  2. כשהמחסנית ריקה → רוקן את כל תור ה-microtasks (כולל microtasks חדשים שנוצרים תוך כדי!).
  3. קח macrotask אחד (למשל setTimeout אחד), הרץ אותו.
  4. חזור לשלב 2 — שוב רוקן את כל ה-microtasks.
  5. חזור על 3–4 עד שהכל ריק.
💡
המסקנה בשורה אחת סינכרוני → כל ה-microtasks → macrotask אחד → כל ה-microtasks → ... כלומר: כל ה-Promise.then ו-await תמיד רצים לפני כל setTimeout — גם אם ה-setTimeout הוא 0ms!

5️⃣ 5 חידות פלט — פתורות שורה-שורה

חידה 1 setTimeout מול Promise

javascript
console.log(1);
setTimeout(() => {
  console.log(2);
}, 0);
Promise.resolve().then(() => {
  console.log(3);
});
console.log(4);
פלט
1
4
3
2
🔍
למה
  • סינכרוני: console.log(1) → 1. ה-setTimeout נכנס לתור ה-macrotasks. ה-.then נכנס לתור ה-microtasks. console.log(4) → 4.
  • microtasks: המחסנית ריקה → רץ ה-.then → 3.
  • macrotask: עכשיו ה-setTimeout → 2.
הלקח: Promise.then תמיד לפני setTimeout(...,0).

חידה 2 microtask שמייצר עוד משימות

javascript
setTimeout(() => console.log(1), 0);
Promise.resolve().then(() => {
  console.log(2);
  setTimeout(() => console.log(3), 0);
  Promise.resolve().then(() => {
    console.log(4);
  });
});
פלט
2
4
1
3
🔍
למה
  • סינכרוני: נרשם setTimeout(1) → macrotask A. נרשם ה-.then החיצוני → microtask.
  • microtasks: רץ ה-.then → 2. בתוכו: נרשם setTimeout(3) → macrotask B (אחרי A!), ונרשם .then(4) → microtask חדש. ממשיכים לרוקן microtasks → 4.
  • macrotasks לפי הסדר: A קודם → 1, ואז B → 3.
הלקח: תור ה-microtasks מתרוקן לגמרי (כולל חדשים) לפני שנוגעים ב-macrotask. וה-macrotasks רצים לפי סדר הרישום.

console.log(4);
demo();
console.log(5);

🔍
למה
  • סינכרוני: console.log(4) → 4. קוראים ל-demo(): console.log(1) → 1. מגיעים ל-await → מה שאחריו (log(2)...) הופך ל-microtask, והשליטה חוזרת לקורא. console.log(5) → 5.
  • microtasks: ההמשך הראשון → 2, ואז await שני → ההמשך הבא נכנס כ-microtask → רץ → 3.
הלקח: כל מה שאחרי await הוא microtask. הקוד שאחרי קריאת demo() (כלומר 5) רץ לפני ההמשכים.

למה
  • שלב סינכרוני (יורדים פנימה): level1 → 6, קורא ל-level2 → 4, קורא ל-level3 → 1. מגיעים ל-new Promise — ה-executor רץ מיד → 2 ו-resolve(). ה-await ב-level3 עוצר אותו; וההמתנה מתפשטת כלפי מעלה — כל הרמות "תקועות" על ה-await שלהן.
  • microtasks (עולים בחזרה): ההמשך של level3 → 3; level3 סיים → משחרר את ה-await של level2 → 5; level2 סיים → משחרר את level1 → 7.
הלקח: כשכל רמה await-ת על הבאה, ה"ירידה" סינכרונית (6,4,1,2) וה"עלייה" דרך microtasks (3,5,7).

למה — וההבדל מחידה 4
  • סינכרוני: level1 → 6. קורא ל-level2() בלי await: level2 → 4, level3 → 1, ה-executor → 2+resolve, ו-level3 נעצר ב-await. השליטה חוזרת ל-level2 (שנעצר ב-await), שמחזיר Promise ל-level1 — אבל level1 לא מחכה לו! ממשיך מיד → console.log(7) → 7.
  • microtasks: ההמשך של level3 → 3; level3 סיים → ההמשך של level2 → 5. (אף אחד לא מחכה ל-level2, אז אין "7" נוסף — הוא כבר רץ.)
הלקח: await = "חכה לי". בלעדיו (level2() סתם), הקוד שאחרי הקריאה (7) לא ממתין ורץ בשלב הסינכרוני. זה ההבדל היחיד — והוא מזיז את ה-7 קדימה.

6️⃣ 5 דוגמאות נוספות — מושגים חדשים

החידות לימדו אותך סדר. עכשיו 5 דוגמאות שמכסות מושגים שחשובים בקוד אמיתי ובמבחן.

דוגמה א' async תמיד מחזירה Promise

javascript
async function getNumber() {
  return 42;                 // לא "מחזיר 42" — מחזיר Promise שנפתר עם 42
}

console.log(getNumber());                  // אובייקט Promise
getNumber().then(n => console.log(n));     // 42 (שולפים את הערך)
console.log('end');
פלט
Promise { 42 }
end
42
🔍
למה סינכרוני: getNumber() מחזיר Promise (לא 42!) → מודפס Promise { 42 }. הקריאה השנייה רושמת .then כ-microtask. 'end' מודפס. ואז ה-microtask → 42. הלקח: כדי לקבל את הערך — await או .then, אף פעם לא ישירות.

דוגמה ב' טיפול בשגיאות — throw + try/catch

javascript
async function risky() {
  throw new Error('boom');     // throw ב-async = Promise שנדחה (rejected)
}

async function main() {
  try {
    await risky();             // await על rejected → זורק כאן
  } catch (e) {
    console.log('caught:', e.message);
  }
  console.log('after');
}
main();
פלט
caught: boom
after
🔍
למה throw בתוך async = ה-Promise חוזר rejected. כש-await פוגש Promise דחוי, הוא זורק את השגיאה — ולכן אפשר לתפוס אותה ב-try/catch רגיל. הלקח: עם async/await מטפלים בשגיאות בדיוק כמו בקוד סינכרוני (במקום .catch).

דוגמה ג' טור מול מקביל — מלכודת הביצועים

שתי פונקציות שכל אחת "לוקחת זמן". נשווה: await אחד-אחרי-השני מול Promise.all.

javascript
function wait(ms, label) {
  return new Promise(res => setTimeout(() => { console.log(label); res(); }, ms));
}

// ❌ טורי — ~300ms (מחכה ל-A ורק אז מתחיל את B)
async function sequential() {
  await wait(100, 'A');
  await wait(200, 'B');
  console.log('sequential done');
}

// ✅ מקבילי — ~200ms (מתחיל את שניהם יחד)
async function parallel() {
  await Promise.all([ wait(100, 'A'), wait(200, 'B') ]);
  console.log('parallel done');
}
⚡
למה זה חשוב ב-sequential, await wait(100) חוסם עד שהוא מסתיים, ורק אז מתחיל את השני — סה"כ 100+200=300ms. ב-parallel, Promise.all מתחיל את שני הטיימרים בו-זמנית וממתין לשניהם — סה"כ 200ms (האיטי שבהם). הלקח: אם פעולות אסינכרוניות לא תלויות זו בזו — אל תעשה להן await בטור! הרץ אותן יחד עם Promise.all.
🔤
Promise.all([...]) מקבל מערך של Promises, ומחזיר Promise אחד שנפתר כשכולם נפתרו — עם מערך התוצאות (לפי הסדר). אם אחד נדחה — כל ה-all נדחה מיד.

דוגמה ד' Promise.all מול Promise.race

javascript
const fast = new Promise(res => setTimeout(() => res('fast'), 100));
const slow = new Promise(res => setTimeout(() => res('slow'), 300));

Promise.all([fast, slow]).then(r => console.log('all:', r));    // מחכה לשניהם
Promise.race([fast, slow]).then(r => console.log('race:', r));  // הראשון שמסתיים
פלט
race: fast          // אחרי ~100ms
all: [ 'fast', 'slow' ]   // אחרי ~300ms
🔍
למה race נפתר עם הראשון שמסתיים (fast, אחרי 100ms) — לכן הוא מודפס קודם. all ממתין לכולם ומחזיר מערך (אחרי 300ms — האיטי). הלקח: all = "כולם", race = "הראשון". (race שימושי ל-timeout: מירוץ בין בקשה לבין טיימר.)

דוגמה ה' ה-await שנשכח — הבאג הנפוץ

javascript
async function save() {
  console.log('saving...');
  return 'saved';
}

function handler() {
  const result = save();              // ❌ שכחנו await!
  console.log('result =', result);    // לא 'saved' — אלא Promise!
}
handler();
פלט
saving...
result = Promise { 'saved' }
🐛
למה — והתיקון בלי await, save() מחזיר Promise, לא את המחרוזת. result הוא אובייקט Promise — ולכן result === 'saved' יהיה false, וכל לוגיקה שתסתמך עליו תישבר בשקט. התיקון: להפוך את handler ל-async ולכתוב const result = await save();. הלקח: אם משתנה מכיל "Promise { ... }" — שכחת await.
🎯
המשפט אחד לזכור Promise = תוצאה עתידית (pending → fulfilled/rejected). async = פונקציה שמחזירה Promise; await = חכה לתוצאה (ומפצל את הפונקציה ל-microtask). וה-event loop: סינכרוני → כל ה-microtasks (then/await) → macrotask אחד (setTimeout) → וחוזר חלילה.
