Cookies — איך שרת "זוכר" מי אתה

פרוטוקול HTTP הוא חסר-מצב (stateless): כל בקשה עומדת בפני עצמה, והשרת לא זוכר אותך מבקשה לבקשה. Cookies הם המנגנון שפותר את זה — פיסת מידע שהשרת שומר אצל הדפדפן, והדפדפן מצרף אותה אוטומטית לכל בקשה עתידית. כך נבנות התחברויות (login), העדפות משתמש ועוד. כאן נבין לעומק מה זה, איך זה עובד מאחורי הקלעים, ואת הקוד שצריך להכיר.

🎯
קונטקסט — למה בכלל צריך קוקיז דמיין שאתה מתחבר לאתר עם שם משתמש וסיסמה. ה-HTTP request הבא שתשלח (לחיצה על קישור, רענון) הוא בקשה חדשה לגמרי — השרת כבר לא יודע שזה אתה. בלי מנגנון "זיכרון" היית צריך להזין סיסמה מחדש בכל קליק. הקוקי הוא ה"תג זיהוי" שהדפדפן נושא איתו ומראה לשרת בכל פעם.

1. מה זה Cookie? — התיאוריה

קוקי הוא בסך הכל זוג שם=ערך (טקסט קצר) שהשרת מבקש מהדפדפן לשמור. ההגדרה המדויקת:

🍪
HTTP Cookie פיסת מידע ששרת אינטרנט שומר על הדפדפן של המשתמש. מרגע שנשמר, הדפדפן שולח אותו אוטומטית לשרת בכל בקשה לאותו אתר — בלי שהקוד צריך לעשות משהו מיוחד.

שתי התכונות שהופכות אותו לשימושי כל כך:

  • מאוחסן בצד הלקוח — הקוקי יושב בדפדפן של המשתמש, לא בזיכרון השרת. כך שרת אחד יכול לשרת מיליוני משתמשים בלי לשמור לכל אחד "סשן" בזיכרון.
  • נשלח אוטומטית — זה הקסם. הדפדפן מצרף את הקוקי ל-header של כל בקשה אל אותו דומיין, כך שהשרת "מזהה" את המשתמש בכל פעם מחדש.
🧭
למה זה פותר את בעיית ה-statelessness HTTP לא שומר זיכרון בין בקשות. הקוקי הוא ה"חוט" שמחבר בין הבקשות של אותו משתמש: ברגע שהשרת שותל קוקי (למשל user=guest), הדפדפן יחזיר אותו בכל בקשה הבאה — והשרת יוכל לדעת "אה, זה המשתמש שכבר התחבר".

שימושים נפוצים: התחברות וסשנים (לזכור שמשתמש מחובר), העדפות (שפה, מצב כהה/בהיר), ומעקב/אנליטיקות.

2. מחזור החיים — Set-Cookie ו-Cookie

כל הסיפור מסתכם בשני HTTP headers — אחד שהשרת שולח, ואחד שהדפדפן מחזיר:

Headerמי שולח אותומתימה הוא עושה
Set-Cookieהשרת (בתוך ה-response)כשרוצים לשתול/לעדכן קוקיאומר לדפדפן "צור קוקי בשם X עם הערך Y"
Cookieהדפדפן (בתוך ה-request)בכל בקשה עתידית לאותו אתר — אוטומטיתמצרף את הקוקי כדי שהשרת יקרא אותו

הזרימה בארבעה צעדים:

1. תגובת השרתresponse עם Set-Cookie: user=guest
←
2. הדפדפן שומרהדפדפן יוצר את הקוקי ושומר אותו אצלו
←
3. בקשה הבאההדפדפן מצרף אוטומטית Cookie: user=guest
←
4. השרת קוראהשרת מזהה את המשתמש מתוך הבקשה
⭐
הנקודה החשובה ביותר המתכנת לא שולח את הקוקי בבקשות הבאות — הדפדפן עושה זאת לבד. כל מה שהקוד שלנו עושה הוא (א) פעם אחת לשתול את הקוקי עם res.cookie(...), ו-(ב) לקרוא אותו מבקשות נכנסות עם req.cookies. השליחה החוזרת — באחריות הדפדפן.

4. קוד השרת — app.js

השרת מגדיר שני נתיבים: POST /login (מאמת ושותל קוקי) ו-GET /private (עמוד מוגן שדורש קוקי). שים לב כיצד cookie-parser נטען כ-middleware בראש.

javascript · app.js
const express = require('express');
const cookieParser = require('cookie-parser');
const app = express();

app.use(express.urlencoded({ extended: true })); // לקרוא body של טפסים (form)
app.use(cookieParser());                          // לקרוא קוקיז → req.cookies
app.use(express.static('public'));                // להגיש קבצים סטטיים מ-public

// --- כניסה: מאמת, ואם תקין — שותל קוקי ומפנה לעמוד הפרטי ---
app.post('/login', (req, res) => {
    const { username, password } = req.body;
    if (username === 'guest' && password === '1234') {
        res.cookie('user', username);   // ← שותל קוקי (Set-Cookie בתשובה)
        res.redirect('/private');
    } else {
        res.send('Invalid username or password.');
    }
});

// --- עמוד פרטי: נגיש רק אם יש קוקי 'user' ---
app.get('/private', (req, res) => {
    const user = req.cookies.user;      // ← קורא את הקוקי מהבקשה
    if (user) {
        res.send(`Welcome, ${user}! This is your private page.`);
    } else {
        res.redirect('/');              // אין קוקי? חזרה לטופס הכניסה
    }
});

app.listen(8080);
🔤
שלושת ה-middleware בראש
  • express.urlencoded({ extended: true }) — מפענח את גוף הטופס (form) כדי שנוכל לגשת ל-req.body.username.
  • cookieParser() — מפענח את ה-header Cookie מהבקשה ושם אותו ב-req.cookies. בלעדיו req.cookies לא קיים.
  • express.static('public') — מגיש אוטומטית קבצים מתיקיית public (כך index.html נטען בכניסה ל-/).

5. טופס הכניסה — public/index.html

טופס פשוט ששולח POST אל /login. שמות השדות (name="username", name="password") הם בדיוק מה שהשרת קורא מ-req.body.

html · public/index.html
<!DOCTYPE html>
<html lang="en">
<body>
    <form action="/login" method="POST">
        <input type="text"     id="username" name="username" placeholder="username">
        <input type="password" id="password" name="password" placeholder="password">
        <button type="submit">Login</button>
    </form>
</body>
</html>
🔗
איך הטופס מתחבר לשרת action="/login" + method="POST" אומרים לדפדפן לשלוח בקשת POST אל /login כשלוחצים על הכפתור. ה-name של כל input הופך למפתח ב-req.body — לכן חייבת להיות התאמה בין name="username" לבין req.body.username.

6. הזרימה המלאה — מה קורה צעד-צעד

נעקוב אחרי מה שמתרחש מרגע שגולשים לעמוד הפרטי בלי קוקי, ועד שמתחברים ומקבלים גישה:

6.1 ניסיון גישה לעמוד פרטי לפני התחברות

נכנסים ל-http://localhost:8080/private. בשרת רצה השורה:

javascript
const user = req.cookies.user;   // undefined — אין עדיין שום קוקי!

מכיוון שאין קוקי, user הוא undefined, התנאי נכשל, והשרת מבצע res.redirect('/') — חזרה לטופס הכניסה. אי אפשר לראות את העמוד הפרטי בלי להתחבר תחילה.

6.2 התחברות שגויה

אם נזין guest / guest — הסיסמה לא תואמת (1234), התנאי ב-/login נכשל, ומקבלים בחזרה את ההודעה Invalid username or password. — שום קוקי לא נשתל.

6.3 התחברות תקינה — הקוקי נולד

נזין guest / 1234. הפעם התנאי מתקיים ורצה:

javascript
res.cookie('user', username);   // מוסיף Set-Cookie: user=guest לתשובה
res.redirect('/private');       // מפנה את הדפדפן לעמוד הפרטי

שני דברים קורים: (1) התשובה כוללת Set-Cookie → הדפדפן יוצר ושומר את הקוקי. (2) ה-redirect גורם לדפדפן לשלוח אוטומטית בקשת GET /private — והפעם, מכיוון שכבר יש קוקי, הדפדפן מצרף אותו לבקשה. השרת קורא req.cookies.user === "guest", התנאי מתקיים, ומוצג העמוד הפרטי.

🔁
למה עכשיו זה עובד ואז לא? ההבדל היחיד בין הניסיון הראשון (6.1) לשני (6.3) הוא קיום הקוקי. אותה שורת קוד בדיוק (req.cookies.user) החזירה undefined בלי קוקי, ו-"guest" עם קוקי. הדפדפן צירף את הקוקי אוטומטית — לכן השרת יכול לזהות אותנו.

7. לראות את הקוקי בעיניים — כלי הפיתוח של הדפדפן

אפשר לצפות בכל המנגנון הזה ישירות ב-DevTools (לחיצה על F12). שני המקומות החשובים:

7.1 טאב Application — הקוקיז השמורים

ב-Application → Cookies → http://localhost:8080 רואים את הקוקיז ששמורים בדפדפן. לפני התחברות — הרשימה ריקה (ולכן req.cookies.user היה undefined). אחרי התחברות תקינה — מופיע קוקי בשם user עם הערך guest:

NameValueDomainPathExpires
userguestlocalhost/Session
⏳
Expires: Session הקוקי שלנו הוא session cookie — נמחק כשסוגרים את הדפדפן, כי לא הגדרנו לו תאריך תפוגה. אפשר להגדיר אורך חיים עם אופציות, למשל res.cookie('user', username, { maxAge: 3600000 }) (שעה במילישניות).

7.2 טאב Network — הכותרות בבקשה ובתשובה

ברענון העמוד, בטאב Network רואים את הבקשות. לוחצים על בקשת private ומסתכלים על הכותרות:

  • ב-Request Headers מופיע Cookie: user=guest — הדפדפן צירף אותו אוטומטית. זה בדיוק מה שמאפשר לשרת לעשות req.cookies.user.
  • ב-Response Headers של בקשת ה-login מופיע Set-Cookie: user=guest — זו ההוראה מהשרת לדפדפן ליצור את הקוקי.

נשים לב גם לקודי הסטטוס בשרשרת ההתחברות:

בקשהסטטוססוגהסבר
login302document / Redirectה-res.redirect מחזיר 302 ומפנה ל-/private
private200documentהעמוד הפרטי הוצג בהצלחה

7.3 מחיקת הקוקי — חוזרים להתחלה

אם מוחקים את הקוקי (דרך הדיאלוג Cookies and site data שליד כתובת האתר, או מטאב Application) ומרעננים — שוב אי אפשר לראות את העמוד הפרטי. בטאב Network רואים שבבקשה כבר אין כותרת Cookie. מחקנו את "תג הזיהוי", והשרת שוב לא מזהה אותנו.

⚠️
אבטחה — קוקיז על HTTP לא מאובטח הדפדפן מתריע "Your connection to this site is not secure" כי האתר רץ על http (לא https). קוקיז נשלחים בכל בקשה — ועל חיבור לא-מוצפן, תוקף יכול ליירט אותם ולגנוב את הזהות (session hijacking). בעולם האמיתי: HTTPS, ועל קוקיז רגישים מסמנים HttpOnly (לא נגיש ל-JS) ו-Secure (נשלח רק על HTTPS).

8. סיכום ה-API שצריך להכיר

קודמה עושה
app.use(cookieParser())מפעיל את ה-middleware שקורא קוקיז מהבקשה לתוך req.cookies
res.cookie('user', value)שותל קוקי — מוסיף Set-Cookie לתשובה. הדפדפן ישמור אותו
req.cookies.userקורא את ערך הקוקי user מהבקשה הנכנסת (undefined אם אין)
res.redirect('/path')מחזיר 302 ומפנה את הדפדפן לכתובת אחרת
res.clearCookie('user')מבקש מהדפדפן למחוק את הקוקי (התנתקות / logout)
🎯
המשפט אחד שצריך לזכור השרת שותל קוקי פעם אחת (res.cookie → Set-Cookie), הדפדפן שומר ומצרף אותו אוטומטית לכל בקשה (Cookie), והשרת קורא אותו (req.cookies) כדי לזהות את המשתמש. זה כל הסיפור של מצב (state) מעל פרוטוקול חסר-מצב.
